# 规则升级流程

## 什么时候读本文件

- 看到 R-NNN 同主题出现 ≥ 2 次 / 方法论铁规被引用 ≥ 3 次 的升级信号时
- 用户说"这条以后都要遵守 / 加到项目规则 / 写进宪章 / 变成硬规矩 / 这是硬规 / 升级成铁规"时
- 需要把 `.kdev/memory/方法论铁规.md` 的规则搬到项目根宪章（`constitution.md` 等）时

## 我不负责什么

- **切档（按时间搬家）** → `references/切档与归档.md`
- **改进建议.md 的写入格式** → `references/六类记录-schema.md` §5

---

## 为什么有这个流程

改进建议.md 里的 R-NNN、方法论铁规.md 里的条目，有些积累到一定程度就该**升级**到项目级硬规则（对外宣称、人类也读的那种）。升级**不自动执行**——Claude 发现信号 → 征得用户同意 → 用户选择目标位置 → Claude 搬家（**标记而非删除**）。

---

## 升级触发信号（满足任一，Claude 主动建议）

- **R-NNN 同主题出现 ≥ 2 次**：同一个问题（如"字段对齐脱钩"）在 R-001 和 R-003 都记过，说明不是偶发
- **方法论铁规被 Claude 引用 ≥ 3 次**：Claude 在对话里反复拿这条规则做判断依据，说明已经是稳定规则了
- **用户明确表达**"这条以后都要遵守 / 加到项目规则 / 写进宪章 / 变成硬规矩"等意图

这是**信号不是硬阈值**——Claude 判断相关就主动问，不要死抠次数。

---

## 升级时 Claude 必须问用户三件事

**严禁替用户拍板**——规则升级涉及"项目资产"级别承诺，必须用户显式同意。

### 问 1：现在升级还是先观察？

- 立刻升级 → 进入问 2/3
- 先观察 → 保持现状，但在 R-NNN 条目末尾标注 `> 候选升级：YYYY-MM-DD 信号命中`，避免遗忘

### 问 2：目标位置放哪？

Claude 按项目现状给**默认推荐**，用户可覆盖：

| 项目现状 | 默认推荐 | 语义 |
|---|---|---|
| 已有 `constitution.md` / `AGENTS.md`（项目根）| 放那里 | 对外宣称的项目宪章，人类也读 |
| 已有 `.specify/constitution.md`（Spec Kit 项目）| 放那里 | 纳入 Spec Kit 宪法体系 |
| 已有 `docs/constitution.md` / `docs/principles.md` | 放那里 | 归入文档体系 |
| 只有 CLAUDE.md、无 spec 文件 | **CLAUDE.md** | Claude 每次会话必读 |
| 规则只给 AI 用、不想对外宣称 | `.kdev/memory/方法论铁规.md` | 保留在 kdev 命名空间内 |

**这 7 个约定路径**（`constitution.md` / `spec.md` / `principles.md` / `AGENTS.md` / `.specify/constitution.md` / `docs/constitution.md` / `docs/principles.md`）**都已被 UserPromptSubmit hook 扫描**——无论放哪个都能参与 triggers 召回。方法论铁规.md 也被扫。

### 问 3：要不要加 `triggers:` 关键词？

- 规则有明确触发场景（如"commit 不带 Co-Authored-By"）→ 加 `triggers: ["co-authored-by", "署名", "提交消息"]`，召回更精准
- 规则是通用原则（如"测试优先"）→ 不加 triggers 靠 Claude 常规 Read 也够

---

## 升级执行（用户同意后 Claude 做的事）

1. **在目标位置追加规则**（不要覆盖已有内容；放在现有章节的合适位置，不要乱插）
2. **加 `triggers: [...]` 行**（如果用户同意）
3. **在 R-NNN 源条目（或方法论铁规原条目）末尾加「升级状态」标记**——原文完整保留：

   ```markdown
   ## R-003: 智能体高速产出与细节对齐脱钩
   日期：2026-04-15
   项目：token-statistics
   触发：执行日志 Step 8 评分差值 -2
   模型自评原话："代码一次跑通..."
   用户评价原话："subagent 写得快但字段名对不上 PRD..."
   ...（原有内容完整保留，一字不改）

   > **升级状态**：2026-04-21 升级到 `constitution.md §「字段对齐」`，triggers: ["字段对齐", "PRD 字段"]
   ```

4. **严禁删除源条目**——理由有二：
   - **原料库完整性**：未来 skill 作者 review 时，"这条规则来自 2026-04-15 Step 8 评分差值 -2"这个完整因果链只在 R-NNN 里有，目标位置只有结论没有证据
   - **归档托底**：改进建议.md 如果未来要切档（目前不切，但不排除），"升级状态"标记是追溯入口

5. **向用户汇报迁移结果**：新位置的规则编号/章节、加了哪些 triggers、源 R-NNN 已加升级标记

---

## 规则被废弃时怎么办

升级后规则如果后续被项目废弃（如"这条现在不合时宜了"），在源 R-NNN 的升级标记下**再加一行废弃记录**：

```markdown
> **升级状态**：2026-04-21 升级到 `constitution.md §「字段对齐」`
> **废弃**：2026-08-10，原因：PRD 改为 schema-first，字段对齐由工具保证，不再需要人工检查
```

保留完整轨迹——"曾经是什么 / 为什么升级 / 为什么废弃"对未来 skill 作者有用。目标位置的规则条目可以删，源 R-NNN 的废弃标记必须保留。

---

## 和切档的关系

切档是按**时间**搬家（月/季度），规则升级是按**重要性**搬家（候选 → 宪章）。两者正交：

- 一条 R-NNN 可以先升级、再切档（升级状态标记随 R 条目一起进归档）
- 一条没升级的 R-NNN 也能切档（原始信号保留到改进建议归档里）
