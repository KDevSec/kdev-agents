# 蒸馏触发机制（auto / manual 两档）

## 什么时候读本文件

- 用户问"自动蒸馏 / 定时蒸馏 / 何时触发 / distill mode / 蒸馏阈值"
- 配置 `.kdev/memory/config.yaml` 的 `distill.*` 字段
- 设计 / 修改 `hooks/lib/distill_trigger.py` 阈值检测逻辑
- 排查"为什么自动蒸馏没跑 / 跑得太频繁"

## 设计哲学

`.kdev/memory/` 积累到一定时间 + 一定数据增长 → **自动**蒸馏（默认）或**提醒**用户手动蒸馏（opt-out）。

**核心约束**：

1. **promote 阶段不能自动**——用户挑选 + 写 `docs/` 是版本控制相关高风险动作，必须人工
2. **dataset 阶段可以自动**——只写到 gitignore 的 `.kdev/memory/dataset/`，零侧效应、可重复跑
3. **触发条件用 AND 语义**——时间 ≥ N 天 **AND** 有新数据，避免没新数据时空跑 / 短期内疯狂跑
4. **失败显式 WARN**——绝不静默，避免数据集质量潜伏

## 配置 schema

`.kdev/memory/config.yaml`：

```yaml
# 落盘路径（既有）
record_mode: hybrid              # hybrid（默认） | inline

# 蒸馏触发（新增）
distill:
  mode: auto                     # auto（默认）| manual
  reminder_days: 7               # 时间阈值（天）
  reminder_new_f: 10             # F-NNN 新增条数阈值
  reminder_new_misalign: 3       # misalign Step 新增条数阈值
```

也支持 flat key 写法（`distill_mode: auto` / `distill_reminder_days: 7`）—— parser 两种都吃。

**未配置 = 视同全部默认**（auto / 7 天 / F≥10 / misalign≥3）。

## 两档行为对比

| 行为 | `auto`（默认） | `manual` |
|---|---|---|
| 阈值满足时 | SessionStart 后台 Popen `distill.py --auto-context --skip-promote` | brief 注入"建议蒸馏：[原因]，跑 `/kdev-memory-distill`" |
| promote 阶段 | **永远不自动** | 同样手动 |
| dataset 阶段 | 自动跑到 `.kdev/memory/dataset/` | 用户手动跑 |
| 跑完成功 | touch `.last-distill` + `.last-distill-auto`；下次 brief 提示"上次自动蒸馏完成于 X" | touch `.last-distill` |
| 跑完失败 | 写 `WARN-distill-failed-*.md`，下次 SessionStart 显眼提醒 | 同上 |
| 控制感 | brief 看见"已开始后台自动蒸馏"才知道跑了 | 用户决定何时跑 |
| 适合谁 | 想"AI 别打扰我，悄悄帮我跑" | 想看每一步、自己决策何时跑 |

## 触发逻辑（AND 语义）

```
should_trigger = time_ok AND data_ok

time_ok:
  - 从未蒸馏（无 .last-distill 也无 .last-promote） → True（首次触发）
  - 距上次蒸馏 ≥ reminder_days → True
  - 否则 → False

data_ok（任一满足即 True，OR 语义）:
  - F-NNN 新增条数 ≥ reminder_new_f
  - misalign Step（差值 ≥ 1.5）新增 ≥ reminder_new_misalign
  - R-NNN 新增 ≥ 5（硬编码兜底）

特殊：从未蒸馏 + 完全无数据 → 不触发（无数据可蒸馏）
特殊：从未蒸馏 + 有任意数据 → 触发（首次蒸馏，绕过时间检查）
```

## 数据增长检测的近似法

`distill_trigger.py` 用文件 mtime + frontmatter 的"日期"字段双重过滤：

1. 文件 mtime ≤ `.last-distill` mtime → 文件未动，新增 = 0
2. 文件 mtime > `.last-distill` mtime → 按 `## heading` 切条目，每条找"日期：YYYY-MM-DD"字段
   - 日期晚于上次蒸馏 → 计为"新增"
   - 没日期字段 → 保守计为"新增"（避免漏报）

这是近似法。更精准的 per-entry mtime 留给 v2。

## SessionStart 集成流程

```
SessionStart hook 触发 session-start-brief.py
    ↓
_distill_hint() 调用 distill_trigger.check_distill_trigger()
    ↓
拿到 TriggerCheck(should_trigger, mode, days_since_distill, new_f, new_misalign, new_r, reasons)
    ↓
分支：
  └─ should_trigger == False
     └─ 有 .last-distill-auto 且最近 24h 内 → brief 注入"上次自动蒸馏完成于 X"
     └─ 否则 → 不注入
  └─ should_trigger == True + mode == auto
     └─ subprocess.Popen(distill.py, --auto-context, detached) → brief 注入"已开始后台自动蒸馏"
  └─ should_trigger == True + mode == manual
     └─ brief 注入"建议蒸馏：[原因]，跑 /kdev-memory-distill"
```

**关键**：Popen 用 `start_new_session=True` detach 子进程，hook 立刻返回（SessionStart 有 5 秒 timeout）。

## 失败处理：WARN 文件

`distill.py` 用 try/except 包裹 `export_markdown_slices`，失败时：

1. 写 `WARN-distill-failed-<timestamp>.md` 到 `.kdev/memory/` 顶层
2. 文件含：错误信息 + 触发场景（auto/manual）+ 处理建议
3. 返回 exit code 4

下次 SessionStart brief 自动扫 `WARN-*.md`（既有的 `_glob_warn_files`），显眼列出。

用户处理完后**手动删 WARN 文件**（避免重复提醒）。

## 命令模板交互

`/kdev-memory-distill` 命令本身**手动跑时不受 mode 影响**——无论 `auto` 还是 `manual`，用户主动跑就两阶段都跑（promote + dataset）。

mode 字段只控制**SessionStart hook 是否后台自动跑 dataset 阶段**。

## 配置示例

### 默认（什么都不配）

```yaml
# 完全不配 config.yaml 或 config.yaml 不存在
# 等价于：
# distill.mode: auto
# distill.reminder_days: 7
# distill.reminder_new_f: 10
# distill.reminder_new_misalign: 3
```

### 用户偏好"别自动跑"（manual）

```yaml
distill_mode: manual
```

或嵌套形式：

```yaml
distill:
  mode: manual
```

### 用户希望长周期低频蒸馏

```yaml
distill:
  mode: auto
  reminder_days: 30        # 改成月度
  reminder_new_f: 30       # 等积累更多 F 才跑
```

### 用户希望短周期高频蒸馏（频繁迭代场景）

```yaml
distill:
  mode: auto
  reminder_days: 3
  reminder_new_f: 5
  reminder_new_misalign: 2
```

## 用户视角的体验

| 场景 | 体验 |
|---|---|
| **新项目首次开会话**（无 .last-distill + 数据稀少） | 不触发，安静开工 |
| **新项目几天后** + 写了 ≥10 条 F | mode=auto → 后台自动跑；brief 看见"已开始后台自动蒸馏" |
| **老项目** + 距上次蒸馏 14 天 + 写了 5 条 R | mode=auto → 后台自动跑（R 新增 ≥5 触发） |
| **手动想跑** | 无论 mode 是什么，跑 `/kdev-memory-distill` 都走完整两阶段（promote + dataset） |
| **后台跑失败** | 下次 SessionStart 看见 WARN 文件提醒，自己处理后删 WARN |

## 兼容老项目（无 .last-distill）

v0.x 老项目可能只有 `.last-promote`（旧 sentinel 文件）。`distill_trigger._last_distill_timestamp()` **取 .last-distill 和 .last-promote 中较新的**作为基线—— 平滑迁移。

`/kdev-memory-distill` 跑成功会创建 `.last-distill`，自然过渡到新机制。

## 未决问题（首批使用后再定）

1. Popen 后台跑失败时主会话无感知 → 是否需要 SessionStart 再扫一遍最近 24h 的 WARN 文件主动展示？（当前是被动等下次 SessionStart）
2. 短时间内多个会话连续启动时，会不会并发 Popen 同一个 distill？（建议加文件锁或 PID 锁）
3. 自动蒸馏跑完后是否要发系统通知（macOS osascript / Linux notify-send）？
4. 阈值默认值（7/10/3）需要 dogfood 数据验证 —— 当前是基于直觉
5. 用户从 manual 切到 auto 时，是否要立即跑一次（"补齐"）还是等下次 SessionStart？
