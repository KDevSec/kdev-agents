# 文件切档与归档

## 什么时候读本文件

- 看到 Stop hook 的 📦 归档提醒、用户同意切档时
- 用户说"切档 / 归档一下 / 归档执行日志 / 归档踩坑日志 / 切一下档 / 整理一下主文件"时
- 跨月/跨季度前后，主文件膨胀影响 Read 效率时

## 我不负责什么

- **R-NNN / 方法论铁规 升级到宪章的流程**（"按重要性搬家"，不是按时间）→ `references/规则升级流程.md`
- **归档后 triggers 召回的技术实现**（`trigger-match.py` 如何扫主文件+归档）→ `references/自动化机制-hooks.md`

---

## 为什么要切档

长项目主文件会持续膨胀（半年下来 `执行日志.md` 可能 1000+ 行），影响 Claude Read 效率和 trigger-match 扫描性能。kdev-memory 采用**人工触发 + Claude 提醒**的切档制度——不自动归档，由 Claude 看到跨期信号后提醒用户、征得用户同意后执行。

---

## 切档规则一览

所有归档文件统一放在 **`.kdev/memory/归档/`** 子目录下，保持顶层整洁。

| 文件 | 粒度 | 归档路径 | 是否参与 triggers 召回 |
|---|---|---|---|
| `执行日志.md` | 按月 | `归档/执行日志-YYYY-MM.md`（如 `归档/执行日志-2026-03.md`） | 归档后**不召回**（Step 本来就只召回今日/昨日） |
| `踩坑日志.md` | 按季度 | `归档/踩坑日志-YYYYQN.md`（如 `归档/踩坑日志-2026Q1.md`） | 归档后**仍召回**——老坑也要防重踩 |
| `决策日志.md` | 按季度 | `归档/决策日志-YYYYQN.md` | 不参与召回（决策走 Claude 主动 Read） |
| `改进建议.md` | **不切档** | —— | N/A |
| `每日汇总/YYYY-MM-DD.md` | 天然分文件 | —— | N/A |
| `当前状态.md` / `方法论铁规.md` | 无需切档 | —— | N/A |

改进建议不切档是**刻意选择**：它是给未来 skill 作者的原料库，完整性优先于可读性。未来如果真膨胀到影响效率，可以再讨论按项目分档（不是按时间）。

---

## Claude 何时提醒切档

Stop hook 每次 Claude 停下时扫主文件最早条目日期：

- `执行日志.md` 最早条目不在**当月** → 输出 📦 提醒
- `踩坑日志.md` / `决策日志.md` 最早条目不在**当季**（季度定义：Q1=1-3 月 / Q2=4-6 月 / Q3=7-9 月 / Q4=10-12 月） → 输出 📦 提醒

**不设行数阈值**——用日期边界判断更自然，不用定"500 行还是 1000 行"这种拍脑袋的数字。

---

## 切档操作（Claude 收到用户同意后执行）

收到用户同意切档后，**按每个需切档文件**执行以下步骤（以 `执行日志.md` 为例）：

1. **确保归档目录存在**：`.kdev/memory/归档/`（不存在就 mkdir）
2. **读主文件全部内容**
3. **按 `日期：YYYY-MM-DD` 识别每条 Step 属于哪个月**
4. **把非当月的条目按月分组**（如 2026-03 的所有 Step、2026-02 的所有 Step）
5. **对每个归档月**：把该月条目**追加**到 `归档/执行日志-YYYY-MM.md`（文件已存在则追加到末尾，新建则直接写入；**严禁覆盖**）
6. **从主文件删除被归档的条目**
7. **在主文件顶部维护一行索引**，例：

   ```markdown
   > 历史归档：归档/执行日志-2026-03.md、归档/执行日志-2026-02.md

   ## Step 45: 当月第一条...
   ```

   Claude 下次 Read 主文件看到这行就知道老数据去哪了。

8. **完成后向用户汇报迁移的条数**（如"迁出 12 条 Step 到 归档/执行日志-2026-03.md，主文件剩 8 条"）

踩坑日志 / 决策日志按同样流程，只是分组按季度、归档命名用 QN。

---

## 切档操作的铁规

- 🔴 **严禁删除条目**——切档是**搬家**，老条目完整迁到归档文件，总数不变
- 🔴 **归档文件追加而非覆盖**——同一月份如果已经有归档（极少见，除非手工操作），追加到末尾
- 🔴 **保留编号**：G-NNN / Step N / Q-NNN 的编号不因切档而改变，跨档引用仍然有效
- 🔴 **切档前必须征得用户同意**——不要收到 Stop hook 提醒就自动执行，询问用户"要现在切档吗"后再动手
- 🔴 **切档后跑一次 triggers 召回自测**（非强制）：验证踩坑/决策归档文件的 triggers 仍能被 glob 扫到

---

## 召回逻辑变化（0.4.0+）

`user-prompt-trigger.py` 和 `trigger-match.py` 从 0.4.0 起扫描改为"主文件 + 归档目录"两路：

- **踩坑日志**：扫 `.kdev/memory/踩坑日志.md` + `.kdev/memory/归档/踩坑日志-*.md` —— 老坑 triggers 照样命中
- **执行日志**：扫 `.kdev/memory/执行日志.md` + `.kdev/memory/归档/执行日志-*.md` —— 今日/昨日过滤会把归档里的老 Step 剔除，实际等效只扫主文件
- **决策日志**：不参与 triggers 召回（保持原状）

向后兼容：没有 `归档/` 子目录的项目（0.3.0 形态），扫描退化为单主文件，行为完全一致。
