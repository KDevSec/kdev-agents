# API 自动化测试编写规范（通用）

本 skill 是一套**与具体后端 / 框架 / 项目无关**的 API 自动化测试方法论，适用于任何用 **pytest + HTTP 客户端**测 RESTful 接口的项目。它沉淀的 6 大类规范在多个项目反复验证过——**绕过它们的代价已知**。

适配到你的项目时：把下文里的"接口封装 / 复合动作 / 用例 / 登录 fixture / 归档钩子"映射到你项目里对应的模块即可，规范本身不依赖任何特定模板或命名。前置输入（接口清单 + 接口用例 `.md`）视为已就绪。

---

## 第零原则（基座）— 测试脚本的目的：发现 BUG，不是刷通过率

所有规范都服务于这一条。当任意规范与"让报告变绿"冲突时，**永远选这条**。红是产出，不是失败。藏起来的红 = 任务失败。

### 三条直接推论（写脚本 / 改脚本 / 评审脚本时按这三条判）

1. **真缺陷保留红**：断言命中后端真实缺陷（字段缺失、格式不符、数据范围未生效、越权未拦截、唯一性未约束）→ 保留 FAIL，录入 `defects_<ts>.csv`，**禁止**伪造字段 / 弱化断言 / 改预期 / skip / xfail / try-except 吞异常来洗绿。
2. **只修"假红"，不动被测**：失败分诊三类——framework（基建）/ script（脚本写错）/ real-defect（真缺陷）。只有 framework + script 假红才允许改脚本；**禁止改被测系统（后端）让测试通过**（即使你确信后端是 bug，被测改动要用户授权 + 与失败用例分提交）。
3. **三态只有 PASS / FAIL / SKIP**：禁用 `@pytest.mark.xfail` 和 `pytest.xfail()`。spec carve-out → 改预期到 spec 真实最低边界；功能未实现 → 改 SKIP（带可审计理由）；真已知缺陷 → 留 FAIL + defects.csv。

### 写完 / 改完每段代码必须自问的三道判定

- 这个改动是让测试**更诚实**，还是让报告**更绿**？后者立刻停。
- 我把一条红改绿了——它是 framework/script 假红（合法），还是我在掩盖 real-defect（违规）？说不清就当 real-defect 保留红。
- 我加的断言**编码了"为什么这个行为重要"**（WHY）吗？还是只断了 `status_code == 200`（WHAT）？每条用例至少一个业务断言。

## 4 条铁规（任何项目接入时同步落地）

### R1 — 依赖数据走 API 预制，不走 UI

依赖资源（上级节点、父部门、角色等）用**复合动作**经 API 预制，不手点 UI。正向流复合动作必须做**轮询确认**——创建/删除后再查一次列表确认生效。后端常有几百毫秒~几秒写入延迟，紧跟一次同步查询会查不到；用"轮询到出现/消失再继续"，不要裸等固定 sleep。

### R2 — 依赖资源显式预制 + 用例后清理

每个用例的善后由 `cleanup` 栈收集（LIFO 逆序运行），不写 try/finally。**后端无 DELETE 接口的资源** → cleanup 只 `loud warn` 留痕（Fail Loud），**不假装清理**；并在汇总报告残留段写明"需 API/DBA 按前缀批量清"。

### R3 — 测试目的：发现 BUG，不刷通过率

见第零原则。交付声明必须附**四件 evidence**（`junit_<ts>.xml` + 自包含 HTML 报告 + `defects_<ts>.csv`（即使 0 条也建表头）+ 运行日志）。"测试已通过"四个字不附 evidence = 视为未执行。

### R4 — 用例之间独立，无执行依赖

每条用例自备前置（自己 add 依赖资源），不依赖别的用例先跑。除非是显式的"短→长→收尾"执行链（那种要在文件头注明声明顺序 = 执行顺序），否则用例**不能假设** collect 顺序。

## 五条不可妥协约束（即使你"看起来想到了更简单的写法"也不要破）

### 约束 1 — 每个接口封装方法 return 前必须过统一响应校验

每个单接口薄封装方法发请求后，统一过一个**响应校验出口**，不要在用例里裸 `assert resp['xxx']` 绕过它。校验出口应支持三态语义：**正向**（断言业务成功）/ **异常**（断言业务失败）/ **异常 + 错误码**（再校验错误码 + 文案）。统一出口保证全套用例的断言契约一致、可演进。

### 约束 2 — 正向封装与异常路径不能混用（**最易踩**）

负向用例（无权限 / 越权 / 参数非法，期望后端拒绝）**禁止**调用"内部默认断言业务成功"的封装。否则后端**正确的** 4xx/拒绝响应（业务失败）会在封装里被当成"正向校验失败"误判成红——这是 **script 假红**，不是 real-defect。

正确写法二选一：
- 用 **raw** 请求（`client.get(url)` / `client.post(url, data=...)`）拿原始响应，再断言"业务失败" + 校 code/msg；
- 或给封装传**显式异常模式参数**（切到异常流校验，断言业务失败）。

> 判别信号：同一模块里，create 侧负向用例用 raw 请求、query 侧负向却用了默认正向封装 → **不对称 = 陷阱**，对齐成 raw。详见 `references/positive-negative-assertions.md`。

### 约束 3 — 测试账号必须对齐真实预置，禁止自造账号名

受限画像账号（数据范围 / 越权用例用的账号）的**用户名必须对齐预置脚本真实创建的账号**。**禁止自造**预置从未创建的账号名——那会导致登录工厂永远登录失败 → 用例永远 SKIP，假装"已覆盖"。

**接账号前先 grep 预置脚本真实创建的账号全集**（`create_user(...)` 之类），只用清单里出现的名字。详见 `references/datascope-account-alignment.md`。

### 约束 4 — 否定性 / 数据范围断言必须实测保留红

当用例断言一个"应该有但后端没有"的字段，或"应该被裁剪/拦截但没有"的数据范围/越权 → 实测命中就是 real-defect，**保留红**。禁止为了绿掉而：伪造字段假装存在、把否定断言删掉、把强断言换成 `assert isinstance(data, list)` 这种永真弱断言。

数据范围用例若只断 `success + data 是 list`（不校裁剪本身）= **弱绿**，等于没验证——要补"看得到该看的、看不到不该看的"的忠实断言。

### 约束 5 — 用例池入口契约：只对"标记了该自动化"的用例写脚本

接入前先把接口用例 `.md` 过一遍，按"是否做 API 自动化"的标记字段切片：
```bash
grep -nE "是否API自动化[：:][[:space:]]*(是|Y|yes|true|✓)" 接口用例.md > api_candidates.txt
```
- 候选 0 → 告知用户"本批无 API 自动化用例"，不兜底；
- 候选 >0 → **仅对候选用例**写 pytest 脚本；
- 字段空缺 / 待定 → 回上游补字段，不自己脑补；
- 用户带可审计的 override 注释强行要求 → 放行，但记到汇总报告「override 清单」。

## pytest 工程工具箱（5 个与第零原则同向的借鉴点）

这几个 pytest 内置 / 生态细节能让测试**更诚实、更可审计、更能发现 BUG**——不是用来洗绿。每个都配了护栏，越过护栏就会滑回 kdev 禁区（mock 被测 / xfail / 弱绿），那时立刻停。详细代码 + 护栏见 `references/pytest-toolbox.md`，这里只给一句话索引：

1. **`monkeypatch.setenv/delenv`** — 覆盖"环境未设走默认 / 设了走覆盖"两条路径。护栏：只 patch 测试自己读的配置，**绝不 patch 被测后端**（那是 mock 被测）。
2. **`tmp_path` 内置 fixture** — 本地测试产物（含四件 evidence、临时文件）自动隔离 + 回收，补 R2 本地文件善后。护栏：**后端资源**善后仍走 cleanup 栈 / loud-warn，不能假装清干净。
3. **`--strict-markers` + 集中声明 marker** — 拼错的 marker 直接报错而非静默 skip，是 Fail Loud 的代码级拦截。护栏：**不解禁 xfail**，三态铁规不变。
4. **`pytest.param(..., id="业务语义名")`** — 参数化子用例在报告里显示可读名、可回溯。护栏：负向分支仍走 raw（约束 2），每组留业务断言（Rule 9）。
5. **`hypothesis @given`** — 对唯一性 / 数据范围 / 边界做模糊探测找真缺陷，与第零原则同向。护栏：找到的违反**保留红 + 录 defects.csv**，禁止因偶发就 xfail/skip 掩盖；引入新依赖前与用户确认。

> 这 5 个借鉴自通用 pytest 手册（python-testing-patterns）。该手册里的 `patch("requests.get")` mock 后端 + `@pytest.mark.xfail` 是 kdev 明确禁区，**不要照搬**——它们是反面教材。

## 标准开发流程（接入新业务模块 `<m>`，按顺序走）

> **前置 GATE（约束 5 落地，先于 step ① 执行）**：用例池过滤 → 候选 >0 才启动。
> **前置依赖**：接口清单（OpenAPI / inventory）+ 接口用例 `.md` 就绪。没有就先回上游侦察/派生流程，不自己拉接口、不自己写用例 .md。

```
①  bootstrap 测试骨架（仅新项目）：复用你项目的中立 API 测试模板
   - 改环境配置：base_url / 管理员账号密码 / 超时
②  接口字段以接口清单 / OpenAPI schema 为准，不以用例 .md 的字段名为准
   - schema 没有的字段 → 关联缺陷断言（real-defect），不伪造
③  写分层四件套（配方见 references/four-piece-recipe.md）：
   - 薄接口封装：单接口一方法，每方法过统一响应校验出口（约束 1）
   - 复合动作层：跨接口 add/edit + 列表轮询确认（R1）
   - 参数化数据：正向 + 负向输入集
   - pytest 用例：注入 复合动作 / cleanup / 受限登录 / 软断言 fixture
④  在 conftest 补该模块的 client fixture
⑤  用例 docstring 第一行 = "<用例编号> <用例名称>"（便于注入报告、回溯）
⑥  负向用例走 raw 请求（约束 2）；受限账号对齐预置（约束 3）；否定断言保留红（约束 4）
⑦  跑：pytest（归档钩子自动把四件 evidence 落到 reports/<ts>/）
⑧  看 reports/<ts>/defects_<ts>.csv 自动归类，按 references/failure-triage.md 分三类处理
⑨  生成跨用例汇总报告；更新 RUN_SUMMARY（含残留 loud-warn 段）
```

**为什么 ② 不能跳**：用例 .md 的字段名常滞后于后端 schema。以接口清单为准、schema 没有的字段关联缺陷断言——这是"测试验证真实后端"而不是"测试自圆其说"的分界。

## 当用户要"写一条新用例"时的标准动作清单

请求很具体时按顺序操作，**不要先写代码再补步骤**：

0. **用例池入口过滤（约束 5）**：该用例"是否做 API 自动化" = 是 → 继续；= 否 → refuse，告知走其它通道；待定/空白 → 回上游补字段；带 override 注释 → 放行 + 记汇总。
1. **确认前提**：项目里有 HTTP 客户端基类 + conftest（含复合动作 / 受限登录 fixture）吗？没有先 bootstrap 骨架。
2. **确认字段名 + 接口**：用户给的字段 / 路径与接口清单 + OpenAPI schema 一致吗？不一致以 schema 为准并提醒用户。
3. **确认接口封装已存在**：薄封装层有对应单接口方法吗？没有先加（过统一响应校验出口），再写用例。
4. **拿配方改字段**：按 `references/four-piece-recipe.md` 的用例骨架（正向 / 异常）改编号、字段、断言文案。负向用 raw 请求。
5. **跑一次确认**：`pytest <对应用例> -v`。**用户没让你跑就只建议命令，不擅自跑**。
6. **报告产出**：变了哪几个文件、新增编号、是否需要 cleanup。

## 当用户要"跑全量 + 三分"时的标准动作

1. 确认可达后端 + 依赖（`py_compile` + `--collect-only` 先过）。
2. 跑全量：`pytest`（归档钩子自动把四件 evidence 落到 `reports/<ts>/`）。
3. **三分结果来自 junit/CSV 真值，不靠会话转述**：PASS / FAIL / SKIP 计数 grep junit；失败三类靠 `references/failure-triage.md` 判。
4. 只修 framework/script 假红，real-defect 保留红（第零原则）。修完**重跑**验证转绿，不靠推断。
5. 产出：四件 evidence + 跨用例汇总报告 + 更新 RUN_SUMMARY（含残留 loud-warn 段）。

## 当用户要"诊断失败"时的标准动作（三分决策）

用户说"看下 defects_<ts>.csv" 或 "某用例为什么挂"，**不要凭印象瞎猜根因**：

1. 找 CSV → 看"原因分析"列（自动归类已分）+ "错误信息"列拿 assert 原文。
2. 找 junit `<failure>` / 运行日志看 captured 输出里**后端真实响应体**（code/msg/业务成功标志）。
3. 按 `references/failure-triage.md` 三分：
   - **framework**（登录抖动 / token 污染 / 连接 RST / 归档基建）→ 修基建，不动断言。
   - **script**（正向封装误用 / 账号名没对齐 / 字段路径写错）→ 修脚本，不弱化业务断言。
   - **real-defect**（字段缺失 / 格式不符 / 数据范围未生效 / 越权未拦截）→ 保留红 + 录 defects.csv + 不 xfail。
4. **拿不准 framework vs real-defect** → 先证"run 非 stale"（看时间戳 + 关联 PASS 用例），再看后端响应体定位崩在 harness 还是产品；说不清当 real-defect 保留红。

## 你必须知道的几个关键模块（映射到你项目的实际命名）

| 角色 | 作用 | 何时看 |
|---|---|---|
| HTTP 客户端基类 | 统一响应校验出口（三态）+ 自动登录 + retry | 写封装 / 改重试前必读 |
| 薄接口封装层 | 单接口一方法（每方法过响应校验出口） | 写用例前确认方法存在 |
| 复合动作层 | 跨接口 add/edit + 列表轮询确认 + 残留 loud-warn | 接入新资源类型时改 |
| conftest | session 登录 / 复合动作 / 受限登录 / 软断言 / 四件归档 / CSV 归类 | 一般不动；扩归类规则才改 |
| 环境配置 | base_url / 管理员账号密码 / 超时 / 受限账号名 | 接环境时改 |
| 接口清单（OpenAPI / inventory） | **接口路径/method/schema 权威源** | 字段/路径以它为准 |
| 跨用例汇总脚本 | 生成 RESULT 汇总报告 | 跑完全量后跑 |
| 预置脚本（`create_user(...)` 之类） | 环境真实创建的受限账号全集 | 写受限账号用例前 grep |

## 沟通节奏

- **用户没让你跑测试就不要跑**——只建议命令，让用户自己跑。
- **不要重写已存在的封装 / 复合动作方法**——先 grep 确认有没有，没有再加。
- **用户给的字段名跟接口清单 schema 不一致时直接说**，按 schema 为准并提醒，不擅自挑一边。
- **遇到红先分三类再动手**——别一上来就改脚本洗绿。改一条红前先答："这是假红还是真缺陷？"
- **修了被测才能过的红 → 停下找用户授权**，被测改动与失败用例分提交（禁止改被测让测试通过）。
