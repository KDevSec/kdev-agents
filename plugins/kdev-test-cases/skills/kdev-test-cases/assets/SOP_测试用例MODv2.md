# 测试用例模板（SOP_测试用例MODv2 — 实验版本）

> ⚠️ **本文件是 [SOP_测试用例MOD.md](SOP_测试用例MOD.md) 的 v2 实验版本**，与上游 [`SOP_测试点MODv2.md`](SOP_测试点MODv2.md) + skill [`kdev-test-points`](../../../.claude/skills/kdev-test-points/SKILL.md) 配套。**MODv1 仍为项目当前默认契约**——OPS 迭代正式产物继续以 MODv1 为准；本文件仅用于 A/B 对照评估期。
>
> **v2 与 v1 的差异（4 类，全部源自上游测试点 MODv2 的传导）**：
> ①  **AR 编号 spec 原样**：用例的「需求编号」字段直接 pass-through 上游 测试点 .md 的 `### AR-…` L3 标题（不锁前缀 `AR-SATP-`，不自造 `.01 / .02` 子段）；
> ②  **角色不锁默认**：用例「前置条件」第 1 条「操作人员：{角色}」的角色取自 spec 全局前提（如 spec 写「平台管理员」则全文用「平台管理员」，spec 写「超级管理员」则用「超级管理员」），MOD 不写死「超级管理员」默认；
> ③  **承载 h4 测试范围分组**：上游 测试点 .md 每个 AR 下挂 N 个 `#### 测试范围 N — …` 包装层（详见上游 MODv2 §1）；1:1 映射改为按「AR → 测试范围 → 行」三层渲染；用例编号公式从 v1 的 `TC-AR{8 位}-{NNN}` 扩展为 `TC-AR{数字段}-G{N}-{NNN}`，新增的 `G{N}` 段承载测试范围号；
> ④  **新增可选字段「测试范围分组」**：每个用例块可选写 `- 测试范围分组：测试范围 N`，便于审计反查 spec criteria 锚点（不携带也行，因为已编码在用例编号 `G{N}` 段中）。
>
> 用途：测试用例 `.md` 的**形状契约**——使每条用例可被 `playwrightmode` 脚本生成器直接消费（步骤可机器解析、预期可断言、测试数据齐全）。
> 位置：`/kdev-test-cases` 的标准 `--example`（v2 实验期）；典型链路 `spec.md →（kdev-test-points）→ 测试点.md（含 h4 测试范围分组）→（kdev-test-cases）→ 测试用例.md →（kdev-ui-autotest）→ Playwright 测试集`。
> 路径 A（从零起草）：复制 §二 空白模板填字段，按 §五 四条 Playwright 衔接约束写步骤与预期；多条用例之间以 `---` 分隔。
> 路径 B（从测试点 .md 派生，`/kdev-test-cases` 默认行为）：按 §六 1:1 映射机械渲染，**禁止改写优化**——用例名称逐字相等、编号公式派生（`TC-AR{数字段}-G{N}-{NNN}`）、用例类型/优先级 pass-through、用例总数恒等于上游 N 个测试范围下所有 # 行数之和（`grep -c "^- 用例编号：" 输出.md` 核验）。
> 自检：交付前照 §七 16 项清单（通用 8 + 1:1 映射 8）逐条勾过。
> 来源：从 [KDevSec-001功能测试用例v1.1.md](KDevSec/KDevSec-001功能测试用例v1.1.md) 反向抽取的标准化壳 + v2 上游 MODv2 改动传导。

## 一、字段定义

| 字段 | 必填 | 说明 | 取值/格式 |
|------|------|------|-----------|
| 用例编号 | 是 | 全局唯一，按 AR → 测试范围 → 行 三层确定性派生；上游为 SP15/SOP 测试点 .md 时**确定性派生**于 AR 编号 + 测试范围号 + 行号，使重跑幂等 | **v2 unified 公式**：`TC-AR{数字段}-G{N}-{NNN}`<br>- `{数字段}` = AR 编号剔除前缀与点号后的纯数字串联，保留 spec 原样段宽（如 `AR-PRL-FUN-01.001.00` → `01001 00` → `0100100`；`AR-SATP-04.001.001` → `04001001`）<br>- `G{N}` = 测试范围号（h4 命名编号，N=1~9）<br>- `{NNN}` = 该测试范围下 # 行号（zero-pad 3 位）<br>**示例**：`AR-PRL-FUN-01.001.00` 测试范围 1 第 3 行 → `TC-AR0100100-G1-003`；`AR-SATP-04.001.001` 测试范围 2 第 5 行 → `TC-AR04001001-G2-005`<br>**注**：v1 公式 `TC-AR{8 位}-{NNN}`（无 `G{N}` 段）保留向后兼容——若上游 测试点 .md 不带 h4 测试范围分组（v1 形态），生成器降级为 `TC-AR{数字段}-{NNN}` 直接派生 |
| 用例名称 | 是 | 一句话概括「动作 + 场景/校验点」；**上游为 SP15/SOP 测试点 .md 时必须原样摘抄「测试点标题」字段，不得改写**（否则 测试点 → 测试用例 → Playwright 的可追溯链断裂） | 中文；从零起草建议 ≤ 40 字；从测试点摘抄时与源逐字相等，上限继承上游（[SOP_测试点MODv2.md](SOP_测试点MODv2.md) §3.6：≤ 200 字符） |
| 需求编号 | 是 | 关联需求 ID — **v2 hard rule**：直接 pass-through 上游 测试点 .md 的 `### AR-…` L3 标题的完整 AR 编号（含 spec 原样前缀与三段数字），**禁止**自造 `.01 / .02` 子段（v2 Phase 6） | `AR-{spec 给定前缀}-{spec 段数字}`，如 `AR-PRL-FUN-01.001.00` / `AR-SATP-04.001.001` / `AR-KDevSec-001.01`；前缀来自 spec 不锁定（v2 Phase 5） |
| 需求点名称 | 是 | 关联需求点的中文名 | 与需求文档保持一致；摘抄上游 测试点 .md 的 `### AR-…` 破折号后部分（v1 + v2 同） |
| 测试范围分组 | 否（**v2 新增字段，可选携带**）| h4 测试范围号的人类可读形式；已编码在用例编号 `G{N}` 段中，写出便于审计反查 spec criteria 锚点 | `测试范围 N`（N = 1~9）；若上游 测试点 .md 不带 h4 分组（v1 形态），本字段省略 |
| 用例类型 | 是 | 用例所属流程类型 | `基本流` / `备选流` / `异常流` |
| 所属模块 | 是 | 一级功能模块 | 如 `系统管理` / `项目管理` / `版本管理` |
| 项目进程 | 是 | 所在迭代 | `Sprint N`，如 `Sprint 1` |
| 优先级 | 是 | 执行优先级（数字越小越高） | `1` / `2` / `3` |
| 前置条件 | 是 | 用例可执行前必须满足的角色/环境/数据条件 | 编号列表，**第 1 条固定写「操作人员：{角色}」**，{角色} 取自 spec 全局前提（v2 Phase 5：不锁默认「超级管理员」） |
| 测试步骤 | 是 | 逐步可复现的操作路径 | 编号列表，含菜单层级、按钮名、字段名等具体可定位描述 |
| 预期结果 | 是 | 步骤完成后的可观察结果 | 编号列表，覆盖 UI 提示 / 数据落库 / 状态变更 等可校验点 |
| 测试数据 | 否 | 执行所需的账号、密码、样例数据 | 子项缩进列表，至少包含账号/密码 |

### 字段约束补充

- **用例编号**：
  - v2 公式 `TC-AR{数字段}-G{N}-{NNN}` 中，`G{N}` 段是 v1 没有的新增段（承载 h4 测试范围号）；`G` 是字面常量，`N` 是 1-9 阿拉伯数字。
  - 同一 AR 不同测试范围下流水号各自从 001 起重新计数（与上游 MODv2 §1.1 "同一 AR 内跨分组的 `#` 测试点编号各自独立从 1 开始" 一致）。
  - 不复用、不跳号；废弃用例标注 `[已废弃]` 但保留编号。
  - **v1 兼容**：若上游不带 h4（v1 测试点 .md），降级用 `TC-AR{数字段}-{NNN}` 公式（无 `G{N}` 段）；不要把 v1 测试点强行套 v2 公式产出"幽灵 G1"。
- **需求编号（v2 Phase 6）**：**绝不**用 `.01 / .02 / .03` 后缀把上游 spec 1 个 AR 拆成多个伪 AR——上游 测试点 .md 多个测试范围分组本身就映射到同一个真实 AR 的多个 h4 子层，体现在用例编号的 `G{N}` 段，不在需求编号里。
- **测试步骤**：必须可复现 —— 出现「点击【菜单A】-【菜单B】」、「在弹出框点击【确定】」等具体可定位的描述，避免「进入页面后操作」这类抽象表述（Playwright 生成器无法把它转成选择器）。
- **预期结果**：必须可观察 —— 包括页面提示文案、数据库字段值、列表行变化等可被自动化或人工断言的现象，禁止「系统正常工作」这类无法判定的描述。
- **异常流**用例的预期结果必须包含「平台数据保持不变」或等价的副作用否定断言。
- 同一需求点下应至少包含一条 **基本流** 与一条 **异常流**（接口不可达 / 参数非法 / 权限缺失等）。

## 四、用例文档组织规范

1. 文件命名：`{项目代号}-{需求编号}功能测试用例v{x.y}.md`，如 `KDevSec-001功能测试用例v1.1.md`。
2. 文件首行：一级标题 `# {项目代号}-{需求编号} 功能测试用例 v{x.y}`。
3. 用例之间：使用单独一行的 `---` 分隔。
4. 同一「需求点名称」+「测试范围分组」下的用例集中编排，按 优先级升序 → 用例类型（基本流 → 备选流 → 异常流）排列。
5. 编号连续、不复用、不跳号；废弃用例标注 `[已废弃]` 但保留编号。
6. **v2 新增**：跨测试范围分组的用例之间也用 `---` 分隔；上游 测试点 .md 一个 `#### 测试范围 N` 分组对应输出 .md 内连续一组用例块，组与组之间显式 `---` 切换，便于人工评审分组完整性。

## 五、Playwright 脚本生成衔接约束

测试用例直接喂给 Playwright 脚本生成器，请遵守以下四条让生成器能正确抽取选择器与断言：

- **操作对象用 `【】` 或 `""` 包裹**：如 `【系统管理】`、`"触发轮换"按钮`，生成器据此抽取为选择器候选（`role=menuitem[name=系统管理]` / `text="触发轮换"`）。
- **登录步骤**：写作「登录{平台名}」或「使用账号 {account} / 密码 {password} 登录」即可，生成器会复用 `conftest.auth_state` 的登录态，无需逐字段拆开。
- **数据库校验步骤**：写明表名（如 `tb_key_metadata`、`dept`）与字段名（如 `status`、`key_version`），生成器会落到 `db_helper` 的 SQL 断言。
- **等待/提示步骤**：写作「等待…完成提示」或「观察弹出的…提示」，生成器会落到 toast / dialog 断言；不要写「等系统反应过来」这种无法定位的描述。

## 六、上游为 SP15/SOP 测试点 .md 时的 1:1 映射约束（v2 unified — 含 h4 测试范围分组）

当 测试用例 .md 的直接上游是已生成的 测试点 .md（典型来自 `/kdev-test-points` `feature-spec-lite` 模式产物，含 `### AR-…` L3 标题 + `#### 测试范围 N` h4 包装层 + 8 列编号表行），适用以下硬约束。这是把 测试用例文档当成"测试点的格式重写"而非"重新设计"——任何越权操作都会破坏 测试点 ↔ 用例 ↔ Playwright 三段可追溯链。

### 6.1 强制 1:1 映射（v2 unified — 按 AR → 测试范围 → 行 三层渲染）

- 测试点 .md 中**每一行**（即每个 `#### 测试范围 N` 包装层下的 `| # | 测试点标题 | ...` 8 列表格每行）转换为**恰好一条**测试用例代码块。
- **不得合并**两条测试点为一条用例（哪怕标题相似 / 预期重叠 / 跨测试范围分组）；**不得跳过**任何一条测试点；**不得新增**测试点之外的用例；**不得吃掉**任何 `#### 测试范围 N` 包装层的语义信息（用例编号 `G{N}` 段必须反映来源测试范围）。
- 输出文件应有 `grep -c "^- 用例编号：" 输出.md == ∑ᴺ (每个 AR 下每个测试范围下的 # 行数)` 的核验恒等式。
- **降级模式（向后兼容 v1 测试点 .md）**：若上游 测试点 .md 不含 `#### 测试范围 N` h4（v1 形态），生成器降级为"按 AR → 行 两层"渲染，用例编号公式自动退化为 `TC-AR{数字段}-{NNN}`（无 `G{N}` 段）；不要给 v1 测试点强行套 `G1`。

### 6.2 字段派生规则（按字段说明 — v2 unified）

| 测试用例字段 | 来源 | 派生规则 |
| --- | --- | --- |
| 用例编号 | AR 编号 + 测试范围号 + 测试点行号 | `TC-AR{数字段}-G{N}-{NNN}`，其中 `{数字段}` = AR 编号剔除前缀与点号后的纯数字串联（保留 spec 段宽，如 `AR-PRL-FUN-01.001.00` → `0100100`；`AR-SATP-04.001.001` → `04001001`），`G{N}` = 测试范围号（h4 命名编号 1-9），`{NNN}` = 该测试范围下 # 行号（zero-pad 3 位）；示例：`AR-PRL-FUN-01.001.00` 测试范围 1 第 3 行 → `TC-AR0100100-G1-003` |
| 用例名称 | 测试点「测试点标题」列 | **原样摘抄**，不缩写、不去前缀（包括「{spec 角色}登录，」前缀） |
| 需求编号 | AR 章节标题 `### AR-…` | 直接取 `AR-{spec 前缀}-{spec 段数字}` 全串（spec 原样，**不锁前缀**为 `AR-SATP-`，**不自造** `.01 / .02` 子段） |
| 需求点名称 | AR 章节标题的破折号后部分 | 含括号内 FR 引用，如 `查看产品线列表（FR1 criteria 1~11）` |
| 测试范围分组 | `#### 测试范围 N — …` h4 标题 | 写 `测试范围 N`（N 取自 h4 命名编号）；v1 上游无 h4 时省略本字段 |
| 用例类型 | 测试点「用例类型」列 | 直接 pass-through，不再重判（避免与上游不一致） |
| 优先级 | 测试点「优先级」列 | 直接 pass-through |
| 所属模块 | 测试点标题中"{模块}-{菜单}"片段，或 AR 章节归属 | 与上游 测试点 表头行第 2 段一致 |
| 项目进程 | 测试点「范围与参数」表迭代号 | 如 `迭代-OPS-001-PVM` → `Sprint 1` |
| 前置条件 | 第 1 条固定「操作人员：{角色}」（**角色取自上游 测试点 .md 的 "范围与参数" 表 / `## 模式与生成元数据` 段的 spec authoritative 角色字段，全文统一，v2 Phase 5 不锁默认「超级管理员」**）；其余从测试点标题与预期反推所需的数据预置 | — |
| 测试步骤 | 从测试点标题反推，按 Playwright 衔接约束（§五）格式化（菜单【】、按钮 ""、字段名直引） | 不得编造测试点未承诺的步骤 |
| 预期结果 | 测试点「预期结果」列 | 保留原编号列表；异常流必须含「平台数据保持不变」或等价否定断言（若上游缺失，由生成器**追加一条**而非改写已有断言） |
| 测试数据 | 默认账号 `admin`/`admin123` + 测试点标题中提及的业务参数 | 不得引入测试点未提及的字段 |

### 6.3 不得越权的事项（v2 unified — 含 h4 相关）

- ❌ 不得"修复"测试点的预期结果断言（哪怕觉得不够严谨）——回到 测试点 .md 修订后再生成。
- ❌ 不得改测试点的用例类型 / 优先级 / 准入字段。
- ❌ 不得合并相邻 AR 的相似行（一个 spec AR 一个 L3 标题、一行一用例）。
- ❌ 不得合并相邻**测试范围分组**的相似行（一个 h4 分组的所有行必须连续输出，组与组之间显式 `---` 分隔；不得把测试范围 1 与 测试范围 2 的行混编后省掉 h4 信息）。
- ❌ 不得**吃掉测试范围分组语义**（即不带 `G{N}` 段或不写「测试范围分组」字段就把多个测试范围的用例并入一个流水段——这会丢失 spec criteria 反查锚点）。
- ❌ 不得删除测试点中标记 OOS / 不单设 TC 的 SC 行——它们应当不在测试点 .md 里出现，若已出现说明上游有误，停下来反馈而非吃掉。
- ❌ 不得给需求编号追加 `.01 / .02 / .03` 子段把 1 个真实 AR 拆成多个伪 AR（v2 Phase 6 hard rule，与上游 MODv2 §1.5 一致）。
- ❌ 不得把「操作人员」角色默认设为「超级管理员」覆盖 spec 给定角色（v2 Phase 5）。

### 6.4 测试范围分组在用例编号中的承载（v2 新增子节）

`#### 测试范围 N — …` h4 标题是上游 MODv2 §1 强制的**统一包装层**（即便单 AR 下只有 1 张表也包成 `测试范围 1`），其语义必须在测试用例 .md 的两处承载：

1. **用例编号 `G{N}` 段**（强制）：`TC-AR{数字段}-G{N}-{NNN}` 中 N 直接对应 h4 命名编号；这是反查 spec criteria 锚点的硬指针，下游 Playwright 函数名 `test_TC_AR..._GN_...` 据此区分跨测试范围的同行号场景。
2. **可选字段「测试范围分组：测试范围 N」**（推荐携带，便于人工审计）：写在用例编号下方、用例名称下方的固定位置；不写不违规但失去人类可读的反查路径——只能靠拆解 `G{N}` 段还原。

**示例对应关系**：

| 上游测试点 .md 位置 | 输出用例编号 | 可选字段 |
| --- | --- | --- |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 1` 第 1 行 | `TC-AR0100100-G1-001` | `测试范围分组：测试范围 1` |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 1` 第 6 行 | `TC-AR0100100-G1-006` | `测试范围分组：测试范围 1` |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 2` 第 1 行 | `TC-AR0100100-G2-001` | `测试范围分组：测试范围 2` |
| `### AR-PRL-FUN-01.002.00` `#### 测试范围 3` 第 4 行 | `TC-AR0100200-G3-004` | `测试范围分组：测试范围 3` |

注意第 2 / 第 3 行：**测试范围 1 第 6 行**与**测试范围 2 第 1 行**之所以不冲突，正是因为 `G{N}` 段保证了同 AR 内跨分组的流水号空间各自独立（与上游 MODv2 §1 "同一 AR 内跨分组的 `#` 测试点编号各自独立从 1 开始" 完全对齐）。

### 6.5 grep 校验恒等式（v2 unified）

生成后必须满足下列恒等关系才视为合规交付：

```bash
# 用例总数 == 上游 N 个测试范围下所有 # 行数之和
test "$(grep -c '^- 用例编号：' 输出.md)" = "$(grep -cE '^\| [0-9]+ \|' 上游测试点.md)"

# 用例编号中 G{N} 段不超过上游测试点 .md 中实际出现的 h4 数量上限
test "$(grep -oE 'G[0-9]+' 输出.md | sort -u | wc -l)" -le "$(grep -c '^#### 测试范围' 上游测试点.md)"

# 需求编号种类数 == 上游 ### AR- 标题数（不多不少；多即自造伪 AR，少即合并 AR）
test "$(grep -oE 'AR-[A-Z0-9.-]+' 输出.md | sort -u | wc -l)" = "$(grep -c '^### AR-' 上游测试点.md)"
```

任一不等即视为 1:1 映射违约，须回查生成器或上游 测试点 .md 修订后重跑。

## 七、写用例自检清单

**通用项（任何上游都要核 — 8 项）**

- [ ] 用例编号唯一、格式 `TC-AR{数字段}-G{N}-{NNN}`（v2 上游）或 `TC-AR{数字段}-{NNN}`（v1 上游降级）之一，与 AR 编号段数对齐
- [ ] 用例类型与「步骤所驱动的真实场景」一致（异常流不能写成基本流）
- [ ] 前置条件第 1 条为「操作人员：{角色}」，**角色取自 spec 全局前提**（v2 Phase 5：不写死「超级管理员」默认），全文角色单一未中途切换
- [ ] 前置条件其余各项可被独立准备（不依赖未列出的隐式状态）
- [ ] 每个测试步骤都给出「操作动作 + 操作对象」，且操作对象用 `【】` 或 `""` 包裹
- [ ] 每个预期结果都可被人工或自动化断言（含 UI 文案、表字段值、列表变化）
- [ ] 测试数据足以执行该用例（账号、密码、必要业务参数齐全）
- [ ] 异常流包含数据副作用否定断言

**1:1 映射项（上游为 测试点 .md 时必核 — 8 项，v2 unified 含 h4 相关 3 项）**

- [ ] 用例总数 == 上游 N 个测试范围下所有 # 行数之和（`grep -c` 恒等式见 §6.5 第 1 条）
- [ ] 每条用例的「用例名称」字段与上游对应行「测试点标题」逐字相等（含「{spec 角色}登录，」前缀）
- [ ] 每条用例的「用例编号」AR 数字段 == 对应 AR 编号纯数字段宽串联；`G{N}` 段 == 来源 h4 测试范围编号；3 位流水号 == 该测试范围下对应 # 行号
- [ ] **需求编号种类数 == 上游 `### AR-` 标题数**（v2 Phase 6 hard rule，见 §6.5 第 3 条 grep）；**未**给需求编号追加 `.01 / .02` 子段把 1 个真实 AR 拆成多个伪 AR
- [ ] **「测试范围分组」字段编号上限**：`grep -oE 'G[0-9]+' 输出.md | sort -u` 出现的 G{N} 集合不超过上游实际 h4 数量（v2 Phase 6 unified，见 §6.5 第 2 条 grep）
- [ ] **跨测试范围分组用例之间显式 `---` 分隔**：一个 h4 分组的所有行连续输出后再切换下一组；未把多个测试范围混编
- [ ] 「用例类型」/「优先级」与上游一致，未被生成器重判
- [ ] 「预期结果」编号列表与上游同序、同断言；新增的「平台数据保持不变」行独立成项不覆盖原断言

## 八、v1 ↔ v2 兼容性对照

| 维度 | v1 (SOP_测试用例MOD.md) | v2 (本文件) |
| --- | --- | --- |

| AR 前缀 | 隐含锁定 `AR-SATP-` / `AR-KDevSec-`；自造前缀视为越权 | spec 给什么用什么（`AR-PRL-FUN-` / `AR-SATP-` / `AR-KDevSec-` / 其他都可，spec 原样） |

| 1:1 映射层级 | 两层（AR → 行） | 三层（AR → 测试范围 → 行） |
| grep 恒等式 | 用例总数 == 上游行数 | 用例总数 == 上游所有测试范围下行数之和 + G{N} 段不超上游 h4 数 + 需求编号种类数 == 上游 AR 数 |

**何时用 v1 / 何时用 v2**：
