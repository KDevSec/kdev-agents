# Test Case Renderer — 测试点 (含 h4 测试范围分组) → Playwright-Handoff Fielded Test Cases

This skill is **a renderer, not a designer**, mirroring [`kdev-test-cases-old`](../kdev-test-cases-old/SKILL.md)'s role but consuming the new upstream contract from [`kdev-test-points`](../kdev-test-points/SKILL.md) + [`SOP_测试点MODv2.md`](../../../project/ClaudeCodeTest/docs/SOP_测试点MODv2.md). The byte-equality contract is unchanged; the **only** v2-specific additions are h4 测试范围分组 承载 + 4 propagations from MODv2 (Phase 5 + Phase 6 + Phase 6 unified).

Mode / skill selection — when in doubt which renderer applies (kdev-test-cases-old is deprecated; prefer this skill, including its v1-downgrade mode):

| If upstream 测试点 .md… | Use |
| --- | --- |
| has `^#### 测试范围` h4 lines | **v2** (this skill) |
| has `AR-` prefix other than `AR-SATP-` (e.g. `AR-PRL-FUN-`) | **v2** (this skill) |
| user explicitly passed `--example SOP_测试用例MODv2.md` | **v2** (this skill) |
| was generated by `/kdev-test-points` | **v2** (this skill) |
| has no `#### 测试范围`, uses `AR-SATP-` prefix, was generated by `/kdev-test-points-old` | **this skill's v1-downgrade mode** (preferred); legacy [kdev-test-cases-old](../kdev-test-cases-old/SKILL.md) only to reproduce pre-v2 output |

Why this started as a fork instead of an in-place upgrade: at the time, v1 (now deprecated as kdev-test-cases-old) was production-default for OPS iterations and could not break; v2 carried 4 hard-rule propagations from MODv2 that change the AR-header regex, the ID formula, and the role defaulting — too much surface to dual-mode safely. v2 has since become the production default, and kdev-test-cases-old is retained only for migration compatibility. The same A/B-then-supersede pattern is mirrored by `kdev-test-points-old` ↔ `kdev-test-points`.

## 2. Arguments

```
/kdev-test-cases [--input <path-to-测试点.md>]      # required
                    [--example <path-to-fielded-template.md>]  # optional; defaults to bundled assets/SOP_测试用例MODv2.md
                    [--output <path>]                  # if absent, return inline
                    [--mod-version <auto|v1|v2>]       # default auto; force-override the h4 detection
                    [free-form prompt]
```

- `--input <path>` — the upstream 测试点 .md (typically `kdev-test-points` output with h4 sub-grouping, or `kdev-test-points-old` legacy output without h4).
- `--example <path>` — fielded-block template. Use `SOP_测试用例MODv2.md` for h4 上游 (carries the `测试范围分组` field convention); use `SOP_测试用例MOD.md` for v1-downgrade.
  - **Default when omitted (bundled example)**: if the user does **not** pass `--example`, default to the skill-bundled template `assets/SOP_测试用例MODv2.md` — **resolve this path relative to this SKILL.md's own directory**, never as an absolute or project-relative path, so the default travels with the skill to any install location. If the user passes any `--example <path>`, **the user's value always wins** — do not merge or fall back. Note: unlike a spec-design skill, the fielded-block template is structurally required (it defines the output schema), so there is **no "no-example" opt-out** — omitting `--example` simply uses the bundled default. The bundled default is the **v2** template; the v1-vs-v2 ID formula is still decided by `--mod-version` auto-detection on `--input`'s h4 (§1 / §2 `--mod-version`), **not** by which template is used — so the bundled v2 example works for both h4 and v1-downgrade 上游.
- `--output <path>` — output file. If absent, return inline.
- `--mod-version <auto|v1|v2>` — default `auto` (decided by §1 pre-flight h4 grep). Force-override only when auto-detection misclassifies (rare).

No `--mode` flag for renderer mode — this skill is single-mode by design (renderer only, no spec-design judgment).

## 3. The contract (v2 hard rules — these are correctness invariants, not stylistic preferences)

### 3.1 Cardinality (unchanged from v1)

```
count(output 用例编号 lines) == count(input 测试点 numbered rows across all AR × all 测试范围)
```

One numbered table row in the input → exactly one fielded code block in the output. No aggregation across rows. No skipping. No invented rows.

**Verifiable**:

```bash
# Output 用例编号 总数 == 上游所有 测试范围 下所有 # 行数总和
grep -c "^- 用例编号：" <output>.md == grep -cE "^\| [0-9]+ \|" <input>.md

# v2 mode: 输出的 G{N} 段不超过上游 #### 测试范围 总数
grep -oE 'G[0-9]+' <output>.md | sort -u | wc -l ≤ grep -c "^#### 测试范围" <input>.md

# v2 Phase 6: 需求编号种类数 == 上游 ### AR- 标题数（不多不少；多即自造伪 AR，少即合并 AR）
grep -oE 'AR-[A-Z][A-Z0-9-]*-[0-9.]+' <output>.md | sort -u | wc -l == grep -c "^### AR-" <input>.md
```

All three grep equalities must hold. If any diverges, the output is broken — stop and report.

### 3.2 Title preservation — byte-equal (unchanged from v1)

```
output[i].用例名称 == input[i].测试点标题   # byte-for-byte
```

**Do not paraphrase. Do not drop prefixes. Do not "fix" typos. Do not shorten parentheticals.**

The traceability chain (测试点 → 用例 → Playwright test name → defect ticket) depends on byte-equal name match. If the title looks suboptimal (the role-登录 prefix feels redundant; the parenthetical is verbose), that's an **upstream 测试点 issue** — stop, report it back to the user, and let them fix the 测试点 .md before re-running. Silently rewriting is the failure mode this whole skill exists to prevent.

### 3.3 用例编号 — deterministic derivation (v2 upgrade — adds G{N} segment)

**v2 mode (h4 present)**:

```
output[i].用例编号 == "TC-AR" + digits_concat(AR.segments) + "-G" + 测试范围_N + "-" + zfill(row_n, 3)
```

- `digits_concat(AR.segments)` = AR 编号 stripped of prefix and dots, segments concatenated keeping spec原样段宽 (no zero-padding, no length normalization).
  - `AR-SATP-04.001.001` → `04001001` (8 digits)
  - `AR-PRL-FUN-01.001.00` → `0100100` (7 digits)
  - `AR-KDevSec-001.01` → `00101` (5 digits)
- `测试范围_N` = the h4 number from `#### 测试范围 N — …` (1–9).
- `zfill(row_n, 3)` = the `| # |` column value in the table, zero-padded to 3 digits, **reset per-测试范围 group** (per [SOP_测试点MODv2.md](../../../project/ClaudeCodeTest/docs/SOP_测试点MODv2.md) §1 "同一 AR 内跨分组的 `#` 测试点编号各自独立从 1 开始").

Concrete:

| Upstream position | Output 用例编号 |
| --- | --- |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 1` row 1 | `TC-AR0100100-G1-001` |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 1` row 6 | `TC-AR0100100-G1-006` |
| `### AR-PRL-FUN-01.001.00` `#### 测试范围 2` row 1 | `TC-AR0100100-G2-001` (流水号 reset 因新 测试范围) |
| `### AR-PRL-FUN-01.002.00` `#### 测试范围 3` row 4 | `TC-AR0100200-G3-004` |

**v1-downgrade mode (no h4)**:

```
output[i].用例编号 == "TC-AR" + digits_concat(AR.segments) + "-" + zfill(row_n, 3)
```

— identical to v1's formula; emit without `G{N}` segment. Re-running the skill on the same v1 input must produce byte-identical IDs to what `kdev-test-cases-old` would emit.

**Verification regex**:

| Mode | Regex |
| --- | --- |
| v2 (h4) | `^TC-AR\d+-G\d-\d{3}$` |
| v1-downgrade | `^TC-AR\d+-\d{3}$` |

This makes IDs a **pure function of input** — re-running the skill produces byte-identical IDs. Downstream pipelines get a stable handle. Never introduce session-scoped, random, or model-generated IDs.

### 3.4 Pass-through fields — not re-judged (unchanged from v1)

These fields are copied verbatim from the input row to the output block:

```
output[i].用例类型       = input[i].用例类型       # 基本流 / 异常流
output[i].优先级         = input[i].优先级         # 1 / 2 / 3 / P0 / P1 / ...
output[i].是否准入       = input[i].是否准入       # 是 / 否
output[i].是否UI自动化   = input[i].是否UI自动化   # 是 / 否   ← 字段名必须与源 测试点 .md 列头一字不差
output[i].是否API自动化  = input[i].是否API自动化  # 是 / 否   ← 字段名必须与源 测试点 .md 列头一字不差
```

**Field names are byte-equal to source column headers, not paraphrased.** Downstream pipelines split UI / API automation lanes by grepping `^- 是否UI自动化：` and `^- 是否API自动化：` exactly — wrong prefix (`UI 自动化` vs `是否UI自动化`) makes the grep miss everything and silently fall back to "run all UI". These two field lines are **MANDATORY in every output block, never optional**.

If you disagree with the upstream judgment ("this should be P1, not P2"; "this clearly is 准入, not 否"), that's a 测试点 issue — surface it as a notice in the output footer, but **do not rewrite the field**. Re-judging is a class of silent drift that breaks downstream agreement between test design, test execution, and defect-rate accounting.

### 3.5 预期结果 — preserved, append-only (unchanged from v1)

```
output[i].预期结果 == input[i].预期结果   # same numbered items, same order
```

Keep the source's numbered list verbatim and in the same order.

**Only allowed mutation**: for 异常流 rows, if the source list omits the "平台数据保持不变" assertion, append it as a **new trailing bullet**. Never overwrite an existing bullet. Never reorder. Never collapse two bullets into one. Never add new bullets to 基本流 rows.

### 3.6 Generative fields (步骤 / 前置 / 数据) — tightly constrained (v2 Phase 5 role-pulling)

These three fields are the **only** fields you fill in generatively. Constraints:

- **Inferred from 测试点 标题 + 预期 only.** Do not pull from outside knowledge or from `--example`'s example rows.
- **Follow Playwright handoff conventions** (see [references/playwright-handoff.md](references/playwright-handoff.md)):
  - Menu items wrapped in `【...】`: `点击【项目管理】→【产品线管理】`.
  - Button labels in Chinese quotes: `点击"新增"按钮`, `点击"确认"按钮`.
  - Direct field names: `填写"名称"字段为 "测试产品线A"`.
- **Default credentials**: `admin` / `admin123` unless the 测试点 标题 explicitly names a different account.
- **角色 (v2 Phase 5 hard rule)**: 前置条件 第 1 条 `操作人员：{角色}` 的 `{角色}` **必须**取自上游 测试点 .md 的下列来源之一（按优先级）：
  1. `## 范围与参数` 表格中的 `角色（全文统一）` 行
  2. `## 模式与生成元数据` 段中的 `角色术语：…` 字段
  3. 上游测试点标题的开头角色（如 `平台管理员登录，…` → 抽 `平台管理员`；`超级管理员登录，…` → 抽 `超级管理员`）
  - **不得**默认 `超级管理员`（这是 v1 行为；v2 必须从 spec 取）。
  - 若上述 3 处都查不到角色，停下来 stop-and-ask 用户 explicit `--role <X>` override，不要硬猜。
- **Business parameters**: only what the 测试点 标题 mentions. If the title says "新增顶级产品线", use a plausible 产品线 name + 描述 — do **not** invent additional optional fields the title didn't promise.
- **Never** add a precondition or step that the 测试点 didn't promise.

When in doubt, prefer fewer steps over more. Empty 测试数据 is acceptable when the row doesn't need data.

### 3.7 No re-design + v2 Phase 6 no-fake-sub-AR

This skill **does not emit**:

- §6.1 Header / 范围与参数表 / 测试基础
- §6.2 Quality Coverage Matrix
- §6.3 COND tables
- §6.4 CI tables
- §6.7 Coverage Summary
- §6.8 Conformity Evaluation
- §6.9 RTM (full)
- §6.11 Risk segment

The 测试点 .md and its `<stem>-audit.md` companion already carry that ceremony upstream.

**v2 Phase 6 additional hard rule — no fake sub-AR**:

- ❌ **不得**给 `需求编号` 追加 `.01 / .02 / .03` 子段把 1 个真实 AR 拆成多个伪 AR（与上游 [SOP_测试点MODv2.md](../../../project/ClaudeCodeTest/docs/SOP_测试点MODv2.md) §1.5 v2 Phase 6 一致）。需求编号 must be byte-equal to the input `### AR-…` header（如 `AR-PRL-FUN-01.001.00`，不要写成 `AR-PRL-FUN-01.001.01`）。
- ❌ **不得**把不同 `#### 测试范围 N` 包装层的行混编后省掉 G{N} 段（这等于吃掉 h4 信息，让下游失去反查 spec criteria 锚点）。
- ❌ **不得**为了输出整齐而合并相邻 AR / 相邻 测试范围 的相似行。

The output's *only* non-block content is: section headers grouped by AR (一/二/三/...) — with v2-mode optional 测试范围 sub-headers (`### 测试范围 N — …`) — plus a final "用例集合统计" table and the 自检清单.

## 5. Workflow

0. **Resolve `--example` default (§2 bundled default)** — if the user passed `--example`, use their value verbatim. 2. **Index input**:
   - Parse all `^### AR-…` headers in order.
   - **v2 mode**: within each AR, parse `^#### 测试范围 N — …` h4 wrappers in order; within each h4, count + read numbered rows. Build a 3-level index: AR → 测试范围 → rows.

5. **Render**: for each input row in order (3-level traversal: AR → 测试范围 → row in v2-mode, 2-level in downgrade), emit one block. Apply §3.1–§3.7 contracts. Use the §4.A or §4.B skeleton per mode.

7. **Append self-check** verbatim from the example's §七 (if present).
