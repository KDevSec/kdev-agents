# 代码 + 质量评审标准（cap: code）

> **硬规（启动即执行）**：评审 agent 启动须先 `recall(scope=/staff/reviewer, subject:review:code)` 召回代码评审历史校准（过往 🔴 模式、评得松/紧），再据本标准打分（记忆底座 v1.0 §5.2）。
> **共用骨架**：百分制 schema / 双重通过条件 / 🔴🟡⚪ 分级 / 评分表输出格式 / 评审循环，一律遵循 [通用评分模板.md](通用评分模板.md)。

---

## 评审对象

- **被评产物**：`src/` + `tests/` + **diff**（dev-engineer 实现阶段产出的代码改动）。
- **caller gate**：`dev-engineer:g-code-review`。
- **上游锚点**：`spec` / `design.md` / `plan.md`（代码须与设计一致）；`tests/`（TDD 红绿证据）。
- **D2 并行**：本 gate 内代码正确性 + 质量两组维度可 fan-out 并行，再聚合（X3 D2 范式）。

---

## 评分维度（4 维 × 25 = 100）

| 维度 | 满分 | checklist |
|---|---|---|
| **spec 一致 + 正确性** | 25 | □ 实现与 spec/design 一致、无暗改 □ 逻辑正确、无明显 bug □ 关键路径行为符合预期 □ 无 spec 未覆盖的擅自功能 |
| **边界 + TDD 真过** | 25 | □ 边界/异常/空值路径处理 □ 错误处理得当（不吞异常）□ tests 真覆盖改动（非空跑）□ TDD 红→绿可信、无注释/跳过造假 |
| **风格 / 抽象 / 命名 / 复杂度 / 重复** | 25 | □ 命名表意、风格符合项目约定 □ 抽象层次合理、无过度/不足 □ 圈复杂度可控 □ 无明显复制粘贴重复（DRY）|
| **架构一致性（事后）** | 25 | □ 改动符合既定架构/分层 □ 依赖方向健康（不倒置、无环）□ 模块边界未被破坏 □ 未引入架构债 |

---

## 阈值

- **通过阈值 = 85**（关键产物，代码缺陷直接进生产）。
- 双重通过条件：`通过 = 总分 ≥ 85 AND 🔴阻断 = 0`。

---

## 问题分级（code 专属判定示例，标准见通用模板 §3）

- 🔴 **阻断**：正确性 bug / spec 违背 / 测试造假（注释掉断言、空跑充数）/ 数据丢失或破坏风险 / 架构红线被破。
- 🟡 **重要**：边界遗漏（非数据风险）、可维护性差、重复代码、抽象不当、命名混乱。
- ⚪ **轻微**：风格、注释、局部命名优化。

---

## 评分表

输出 `handoffs/reviewer/g-code-review.code.score.md`，格式照 [通用评分模板.md §4](通用评分模板.md)：

```yaml
cap: code
target: <diff_range 或 src/+tests/ 路径>
total: 82/100
dimensions:
  - {name: spec一致+正确性,        score: 22/25}
  - {name: 边界+TDD真过,           score: 18/25, issues: [🔴 ...]}
  - {name: 风格抽象命名复杂度重复, score: 21/25}
  - {name: 架构一致性,             score: 21/25}
issues:
  - {level: 🔴阻断, file: src/x.py, line: 42, desc: 边界未处理致空指针, suggest: ...}
  - {level: 🟡重要, ...}
verdict: FAIL        # 🔴≠0 → FAIL（即便 total 82 也不放行）
```
