# 安全评审标准（cap: security）

> **硬规（启动即执行）**：评审 agent 启动须先 `recall(scope=/staff/reviewer, subject:review:security)` 召回安全评审历史校准（过往漏洞模式），再据本标准打分（记忆底座 v1.0 §5.2）。
> **共用骨架**：百分制 schema / 双重通过条件 / 🔴🟡⚪ 分级 / 评分表输出格式 / 评审循环，一律遵循 [通用评分模板.md](通用评分模板.md)。

---

## 评审对象

- **被评产物**：`security.md`（安全自评/扫描报告）+ **diff**（dev-engineer 安全扫描阶段产出 + 代码改动）。
- **caller gate**：`dev-engineer:g-sec-review`。
- **上游锚点**：公司安全标准（如 kdev-secure-coding 八类）、OWASP Top 10、`src/` 改动面。

---

## 评分维度（4 维 × 25 = 100）

| 维度 | 满分 | checklist |
|---|---|---|
| **OWASP / 注入** | 25 | □ 无 SQL/命令/模板注入面 □ 无 XSS / SSRF / 反序列化漏洞 □ 依赖供应链无已知高危 CVE □ 无 eval/pickle 等危险调用 |
| **认证授权** | 25 | □ 认证逻辑无绕过 □ 授权检查到位、无越权 □ 会话/token 管理安全 □ 敏感操作有权限校验 |
| **数据安全** | 25 | □ 敏感数据加密存储/传输 □ 无密钥/凭据硬编码 □ 日志不泄露敏感信息 □ 数据最小化、无过度暴露 |
| **输入校验** | 25 | □ 外部输入全部校验/清洗 □ 边界/类型/长度校验 □ 文件上传/路径有白名单 □ 错误处理不泄露内部细节 |

---

## 阈值

- **通过阈值 = 85**（高风险产物，安全漏洞代价最高）。
- 双重通过条件：`通过 = 总分 ≥ 85 AND 🔴阻断 = 0`。

---

## 问题分级（security 专属判定示例，标准见通用模板 §3）

- 🔴 **阻断**：可利用漏洞（注入/越权/RCE）/ 密钥硬编码 / 敏感数据明文 / 认证绕过 —— 安全 🔴 一律强制 FAIL，不可 tech-debt 化。
- 🟡 **重要**：深度防御缺失、日志含半敏感信息、输入校验不全（无直接利用面）。
- ⚪ **轻微**：安全文档完整度、低风险加固建议。

---

## 评分表

输出 `handoffs/reviewer/g-sec-review.security.score.md`，格式照 [通用评分模板.md §4](通用评分模板.md)：

```yaml
cap: security
target: <security.md + diff_range>
total: 90/100
dimensions:
  - {name: OWASP注入, score: 23/25}
  - {name: 认证授权,  score: 22/25}
  - {name: 数据安全,  score: 23/25}
  - {name: 输入校验,  score: 22/25, issues: [🟡 ...]}
issues:
  - {level: 🟡重要, file: src/api.py, line: 18, desc: 输入长度未限, suggest: ...}
verdict: PASS        # total 90≥85 AND 🔴=0 → PASS
```
