# 测试设计评审员 checklist（T1 阻断 / T2 抽查）

> 评审对象（T1）：`.kdev/handoffs/test/test-points.md`
> 评审对象（T2）：`.kdev/handoffs/test/test-cases.md`
> 评审结论：PASS / FAIL（FAIL 必须列具体问题）

## T1 必检项（9 条 / 1 项 FAIL = 总体 FAIL）

1. [ ] **覆盖度按 AR 反查**：test-points.md 中每条测试点都标注对应 AR 编号；反查 AR.csv，100% AR 至少有 1 条测试点覆盖。
2. [ ] **边界态覆盖**：每个关键数值字段 / 列表接口有最小值、最大值、边界值测试点。
3. [ ] **异常态覆盖**：网络超时、服务不可用、非法入参、权限不足等异常路径均有对应测试点。
4. [ ] **等价类划分明确**：输入域按等价类分组，每组至少 1 个代表性测试点（不允许同等价类重复堆砌）。
5. [ ] **优先级排序**：测试点标注优先级（P0/P1/P2），P0 条数 ≥ AR 数 × 0.5（高优先级覆盖不足则测试计划无效）。
6. [ ] **自动化可行性标注**：每条测试点标注「自动化 / 手动」；标注「手动」须注明原因（视觉验证、物理交互等）。
7. [ ] **测试点原子性**：每条测试点描述单一行为，可独立执行，不依赖前序测试点的副作用。
8. [ ] **与 prototype 一致**：UI 相关测试点与 prototype 原型帧中的流转 / 状态保持一致，无冲突。
9. [ ] **无 TBD / TODO / FIXME** 占位。

## T2 抽查项（test-cases.md 回看）

- [ ] test-cases.md 中每条用例都有前置条件、步骤、预期结果三要素。
- [ ] 用例与对应 test-points.md 测试点编号可互查（双向可追溯）。
- [ ] 优先级 P0 用例的预期结果描述足够精确（含具体数值 / 状态码 / 响应字段）。

## 输出格式

写到 `.kdev/handoffs/test/<阶段>-测试设计评审员-review.md`（阶段为 T1 或 T2）：

```markdown
# 测试设计评审员 review — <阶段>

verdict: PASS | FAIL
date: <ISO-ts>

## 检查项结果
1. ✅ 覆盖度按 AR 反查
2. ❌ 边界态覆盖 — AR-PAY-03 最大金额边界测试点缺失
...

## 问题清单（FAIL 必填）
- AR-PAY-03 缺少 amount=MAX_DECIMAL 边界测试点
```
