# 架构评审员 checklist（F1 终审）

> 评审对象：F1 全套 — `handoffs/reqs/` + `handoffs/dev/` + `handoffs/test/`
> 评审结论：PASS / FAIL（FAIL 必须列具体问题）

## F1 必检项（10 条 / 1 项 FAIL = 总体 FAIL）

1. [ ] **长期可维护性**：模块拆分粒度合理，单一模块不超过「一个原因改变」；新成员可在 1 小时内理解模块结构。
2. [ ] **模块边界清晰**：模块间调用关系单向（无循环依赖），接口契约（入参/出参/异常）明确定义，不依赖内部实现细节。
3. [ ] **跨模块影响评估**：本 Feature 变更了哪些共享模块 / 基础设施？对其他 Feature 的潜在影响已在文档中列出。
4. [ ] **命名契约稳定**：公开 API 路径、领域实体名称、事件名称已定案，符合项目命名规范；不存在「上线后再改名」计划。
5. [ ] **扩展点设计合理**：预留的扩展点有明确的使用场景依据（YAGNI — 无场景的扩展点需删除），且扩展不破坏现有契约。
6. [ ] **数据一致性保障**：分布式写操作（多表更新 / 跨服务写）有事务策略（本地事务 / Saga / 最终一致性）说明，不能空白。
7. [ ] **可观测性设计**：关键路径有日志 / 指标 / 链路追踪设计；告警阈值或监控方案在 design.md 中说明。
8. [ ] **降级与熔断策略**：依赖外部服务的调用均有超时配置 + 降级 fallback 方案，不存在「单点不可用则全线崩」的设计。
9. [ ] **数据库设计无明显反模式**：schema 无明显反模式（宽表 / 多值字段 / 无索引高频查询列），已知热点查询有索引支撑。
10. [ ] **安全架构一致**：鉴权 / 授权边界与 security.md 一致；服务间通信有身份验证机制（不依赖网络隔离作为唯一防护）。

## 输出格式

写到 `.kdev/handoffs/review/架构评审员-review.md`：

```markdown
# 架构评审员 review — F1

verdict: PASS | FAIL
date: <ISO-ts>

## 检查项结果
1. ✅ 长期可维护性
2. ❌ 数据一致性保障 — order_service 多表写操作无事务策略说明
...

## 问题清单（FAIL 必填）
- design.md §4.2 order_service 的跨表写操作需补充事务策略（建议：本地事务或注明最终一致性理由）
```
