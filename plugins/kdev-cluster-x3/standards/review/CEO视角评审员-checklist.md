# CEO视角评审员 checklist（F1 终审）

> 评审对象：F1 全套 — `handoffs/reqs/` + `handoffs/dev/` + `handoffs/test/` + 前置评审结论
> 评审结论：PASS / FAIL（FAIL 必须列具体问题）

## F1 必检项（8 条 / 1 项 FAIL = 总体 FAIL）

1. [ ] **商业价值可量化**：sr.md 或 design.md 中明确陈述本 Feature 为业务带来的价值（留存率 / 转化率 / 效率提升等），不得仅写「提升用户体验」等空泛表述。
2. [ ] **用户感知清晰**：能用 ≤ 3 句话向非技术用户说清「这个功能让你能做什么 / 更快做什么」；如无法做到则需求描述需返工。
3. [ ] **战略对齐**：本 Feature 与产品路线图 / OKR 的对应关系在文档中有明确说明，不是孤立需求。
4. [ ] **ROI 合理**：Feature 的实现成本估算（人天）与预期业务收益对比合理；若成本显著超出收益须有决策说明。
5. [ ] **上线风险可控**：存在技术风险 / 数据迁移风险 / 用户感知风险时，均有对应缓解措施或灰度计划，无「上线后再说」。
6. [ ] **竞品参照说明**：若业界有类似功能，设计方案中需有竞品对照或差异化说明，避免低质量复制。
7. [ ] **可度量的成功指标**：至少 2 个可量化的上线成功指标（如：7日留存率 +5%、P99 响应 < 200ms），可供上线后回验。
8. [ ] **前置评审无遗留高风险**：reqs/dev/test 各组评审结论均已 PASS 或经审查组长决策降级，无跳过的阻断问题。

## 输出格式

写到 `.kdev/handoffs/review/CEO视角评审员-review.md`：

```markdown
# CEO视角评审员 review — F1

verdict: PASS | FAIL
date: <ISO-ts>

## 检查项结果
1. ✅ 商业价值可量化
2. ❌ 可度量的成功指标 — 文档中仅描述定性目标，缺少具体数值指标
...

## 问题清单（FAIL 必填）
- 需补充至少 2 个带数值的上线成功指标（建议：7日留存率基线 + P99 响应目标）
```
