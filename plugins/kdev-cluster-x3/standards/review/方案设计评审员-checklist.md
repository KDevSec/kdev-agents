# 方案设计评审员 checklist（R2 阻断）

> 评审对象：`.kdev/handoffs/reqs/design.md`
> 评审结论：PASS / FAIL（FAIL 必须列具体问题）

## R2 必检项（10 条 / 1 项 FAIL = 总体 FAIL）

1. [ ] **架构图存在且自洽**：design.md 含系统架构图（文本 / 图片），组件关系 / 数据流向与正文描述一致，无矛盾。
2. [ ] **模块拆分合理**：每个模块职责单一、边界清晰，无两个模块承担相同职责的情形。
3. [ ] **DB schema 完整**：所有实体 / 表结构 / 字段类型 / 主外键 / 索引均有说明；无孤立字段（有字段无归属实体）。
4. [ ] **API 列表覆盖 AR**：列举全部对外 API 端点（路径、方法、入参、出参），且 100% AR 中的接口需求可映射到某 API。
5. [ ] **风险列举**：至少 3 条技术风险，每条标注「概率 × 影响」并附缓解措施。
6. [ ] **关键决策记录**：选型决策（框架 / DB / 协议）各附 ≥ 2 个备选方案及放弃理由（ADR 格式或等价）。
7. [ ] **性能 / 扩展考量**：说明系统在 2 倍预期 QPS 下的应对策略（缓存 / 分库 / 队列等），不得空白。
8. [ ] **与 prototype 一致**：API 出参结构 / 数据字段与原型标注字段一致，UI 流程中依赖的 API 均已在列表中。
9. [ ] **部署拓扑清晰**：描述最终部署环境（容器 / 云 / 裸机）、服务数量、网络分区，不得仅写「部署到服务器」。
10. [ ] **无 TBD / TODO / FIXME** 占位。

## 输出格式

写到 `.kdev/handoffs/reqs/方案设计评审员-review.md`：

```markdown

verdict: PASS | FAIL

2. ❌ DB schema 完整 — users 表缺少 updated_at 字段定义
...

## 问题清单（FAIL 必填）
- ...
```
