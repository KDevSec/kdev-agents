# 质量评审员 checklist（D2 阻断）

> 评审对象：git diff（本次 D2 循环，由调用方 TDD实现员 提供 diff 范围）
> 评审结论：PASS / FAIL（FAIL 必须列具体问题）

## D2 必检项（9 条 / 1 项 FAIL = 总体 FAIL）

1. [ ] **命名表达力**：标识符命名可自文档化（读名字知意图），无需注释解释其含义；反例：`flag2`、`processData()`。
2. [ ] **抽象层级一致**：同一函数内操作保持同一抽象层级，不混用高层业务逻辑与底层 I/O 细节。
3. [ ] **DRY（不重复）**：diff 中无逻辑重复块（≥ 5 行实质相同逻辑出现 ≥ 2 次），重复逻辑应提取为函数 / 工具类。
4. [ ] **YAGNI（不过度设计）**：diff 中无当前 AR 未要求的扩展点、抽象层、配置开关；过度设计需说明依据。
5. [ ] **测试覆盖充分**：测试用例覆盖正常路径 + 至少 2 条异常路径；断言有意义（不是仅 `assert True`）。
6. [ ] **错误处理明确**：异常捕获范围精确（不裸 `except Exception`），错误消息含足够上下文，不吞掉错误。
7. [ ] **注释合理性**：注释解释「为什么」而非「是什么」；无过时注释（描述已不存在的行为）；无废话注释（`# 加 1`）。
8. [ ] **性能热点无明显劣化**：N+1 查询、全表扫描、嵌套循环大数据集等典型热点问题不在 diff 中新增。
9. [ ] **函数复杂度可控**：diff 引入的新函数圈复杂度 ≤ 10，单函数行数 ≤ 50（超出需有合理注释或拆分计划）。

## 输出格式

写到 `.kdev/handoffs/dev/<AR编号>-质量评审员-review.md`：

```markdown
# 质量评审员 review — <AR编号> D2

verdict: PASS | FAIL
date: <ISO-ts>
diff_range: <base>..<head>

## 检查项结果
1. ✅ 命名表达力
2. ❌ DRY — src/service.py:88-102 与 src/utils.py:34-48 逻辑重复，建议提取 _parse_response()
...

## 问题清单（FAIL 必填）
- 文件: src/service.py 行: 88-102 — 与 utils.py:34-48 重复，建议提取公共函数
```
