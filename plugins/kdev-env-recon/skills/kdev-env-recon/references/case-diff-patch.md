# 回写模式 — 测试用例 .md ↔ recon 实测的 diff/patch 算法

> 当用户传 `--cases <path>` 时，本 skill 进入"回写模式"。本文件描述如何把 recon 实测结果反向 diff 测试用例 .md，propose 补丁，**绝不直接编辑用户的源文件**。

---

## 为什么不直接改

补丁风险面：

| 情形 | 直接改的后果 |
|---|---|
| spec 对、UI 错（产品 bug） | 用例被改成"配合 UI bug"——bug 被掩盖，自动化跑绿，评审看不到 |
| 同名菜单不同路由 | 路由对错混改（例如两处都叫"产品线管理"但路由不同） |
| 用例里的菜单名是抽象指代（"该模块"），不是实际文案 | fuzzy match 抓错对象 |
| 用例已经在跑、改了名 PageObject 步骤宏与之解耦失败 | 立刻一红一片 |

**所以本 skill 永远只输出 `*.patch.md` 文件**，用户人工评审后再 apply。

---

## 输入与输出

### 输入

- `--cases <path>`：单个 .md 文件，或一个目录（递归扫所有 `*.md`）
- 隐式输入：本次刚生成的 `recon/menu_list.md` + 4 类 JSON

### 输出

```
recon/
├── case_diff.md             # 汇总报告：所有差异点 → 影响哪些 TC
└── patches/
    ├── SOP_测试用例-XXX.patch.md  # 单文件补丁
    └── SOP_测试用例-YYY.patch.md
```

---

## 解析测试用例 .md 中的 UI 锚点

用例 .md 通常按 `SOP_测试用例MOD` 模板，含 `【测试用例信息】` 块。要抽取的 UI 锚点 5 类：

| 锚点类型 | 正则 | 示例 |
|---|---|---|
| 菜单引用 | 点击【...】(- / →【...】)* | `点击【项目管理】-【产品线管理】` |
| 按钮引用 | 点击"..."按钮 / 点击"..." | `点击"新增"按钮` |
| 字段引用 | 填写"..."字段 / "..."输入框 | `填写"产品线名称"字段` |
| 弹窗引用 | 弹窗"..." / 对话框标题"..." | `弹窗"新增产品线"` |
| 列名引用 | 列"..." / "..."列 | `"挂载项目数"列` |

> 抽取时记录 (`TC-NNN`, 行号, 锚点类型, 锚点值)，这是后续输出"影响范围"的基础。

---

## diff 三类匹配

### 类型 A — 完全相等

用例锚点在 recon 实测里存在且文案相同 → 一致，**不出补丁行**。

### 类型 B — 完全不存在

用例锚点在 recon 实测里**完全找不到**（既无相等也无 fuzzy 命中）。

可能原因：
- UI 真的没有这个元素（spec 错、UI 没实装、菜单被删了）
- recon 没探到该页面（不在 `TARGET_PAGES` 里）
- 用例锚点表述过于抽象（如"配置中心"）

输出补丁条目：

````markdown
### TC-AR04001003-002 第 3 步

- **用例原文**：`点击"展开全部"按钮`
- **实测**：未在 `pages_productline.json.buttons` 中发现 `展开全部`
- **可能匹配**：`展开/折叠`（编辑距离 4）
- **建议**：
  ```diff
  -点击"展开全部"按钮
  +点击"展开/折叠"按钮（注：实测为一个 toggle 按钮，单击切换全部展开/折叠状态）
  ```
- **风险**：⚠️ 用例预期"展开全部 / 折叠全部"是两个独立按钮 → spec 与 UI 差异；如果这是产品需求未实装，**不要 apply**，应记 bug。
````

### 类型 C — Fuzzy 相似但不等

用例锚点 X，recon 里有 X'，且二者编辑距离 ≤2 或 X 是 X' 的 substring。

示例：
- 用例 `点击【系统管理】-【产品线管理】`，实测菜单是 `项目管理 > 产品线管理` → 一级菜单错（spec 把它放在了"系统管理"下）
- 用例 `弹窗"新增产品线"`，实测 dialog title 是 `添加产品线` → 文案差异
- 用例 `填写"产品线名"字段`，实测 form label 是 `产品线名称` → 用例简写了

输出补丁条目：

````markdown
### TC-AR05001001-002 第 4 步

- **用例原文**：`点击【系统管理】-【产品线管理】`
- **实测**：`产品线管理` 路由 `/pm/productline` 在【**项目管理**】下，不在【系统管理】下
- **建议**：
  ```diff
  -点击【系统管理】-【产品线管理】
  +点击【项目管理】-【产品线管理】
  ```
- **风险**：✅ 低 — UI 与 spec 间的"模块归属错误"，改用例对齐 UI 即可。
````

---

## 风险标签（每条补丁必须标）

| 标签 | 含义 | 建议处理 |
|---|---|---|
| ✅ 低 | 用例文案错、UI 正确（spec 笔误 / 用例写错）| 可直接 apply |
| ⚠️ 中 | 用例与 UI 都说得通（spec 模糊 / 用例多种写法都合理）| 人工评审，可能要回头改 spec |
| 🚨 高 | UI 可能错、spec 对（产品 bug 嫌疑）| **不要 apply**，立刻报 bug |

> **判断启发式**：
> - 用例锚点是 spec 直接复制 + UI 与 spec 矛盾 → 多半 spec 对，UI 错 → 高风险
> - 用例锚点是开发者自由发挥的简写 + UI 文案更完整 → 多半用例错 → 低风险
> - 用例锚点与 spec 都一致 + UI 唯一不同 → 中风险（评审决定）

```markdown
## 应用补丁建议

1. 先看每条的风险标签，🚨 高风险条目不要 apply，先与产品 / 开发对齐
2. ✅ 低风险条目可批量 apply（人工 copy 或 patch 工具）
3. ⚠️ 中风险条目逐条人工评审
4. apply 后建议在用例 .md 头部"测试环境与导航约定"节注明 "本次按 `recon/menu_list.md` (YYYY-MM-DD) 对齐"
```

## 算法实现要点

1. **预处理**：把 `recon/pages_*.json` + `dialogs_*.json` 的所有 `buttons` / `formLabels` / `tableHeaders` / `placeholders` 与 `menu_tree.json` 全部展开成一个 lookup set
2. **抽取用例锚点**：按上面 5 类正则扫所有 `*.md`，记录 (TC, 行号, 类型, 值)
3. **逐锚点匹配**：
   - 完全相等 → 跳过（类型 A）
   - fuzzy 候选搜索（编辑距离 ≤ 2 / substring / token overlap ≥ 80%）→ 类型 C
   - 全无候选 → 类型 B
4. **风险评分**：按上面启发式表打 ✅ / ⚠️ / 🚨
5. **聚合输出**：
   - 同一文件多条变更 → 同一 `patches/<file>.patch.md`
   - 汇总报告 `case_diff.md`

---

## 误报防止

- **保留量化报告**：所有"未匹配"的锚点都列出，让用户判断是 recon 漏采还是用例错
- **路由消歧**：菜单匹配时优先用路由，再用文案；用例 `点击【系统管理】-【产品线管理】` 但路由是 `/pm/productline` → 树里 `/pm/productline` 在【项目管理】下 → 用例的一级菜单错
- **抽象指代识别**：用例锚点形如"该模块"、"上一步的弹窗"、"刚才的按钮" → 跳过，不参与 diff
- **代码块内的代码不参与 diff**：用例 .md 里的 fenced code 段不视为 UI 锚点

---

## 何时跳过回写

- 用户传 `--cases <path>` 但 `<path>` 不存在或不是 .md → 写日志 + 跳过，不报错
- recon 阶段失败（如登录失败、菜单采空）→ 不写回写报告（因为对照基线不可信）
- 用例 .md 完全不符合标准模板（没有 `TC-AR` 编号、没有"测试步骤"段）→ 输出 `case_diff.md` 写明"用例格式无法解析，跳过 diff；建议先用 `kdev-test-cases` 渲染为标准 fielded 用例"
