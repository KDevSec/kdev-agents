> **v2.1 演进**（默认实践，承袭自 MODv1）：①  CRUD 基础默认拆 3 个 AR（新增 / 编辑 / 删除）；②  默认**不出**模块 99，跨实体约束分散承载——**完整规则见 §1.5**；③  名称字段新增/编辑 AR 必含 `<script>` 异常流（最小 XSS 子集）；④  编辑 AR 必含"修改非必填字段为空 → 接受 + NULL 持久化"行。

> **v2.1.1 收紧**（承袭自 MODv1）：性能 / 时间特性测试整体 OOS，不进本文件——**完整规则、数据档位、反模式、承诺转出措辞模板见 §7.6 主声明段**。

> **使用方式**：①  新迭代（OPS）按 §4 骨架填 §6 待填区；② 提交前逐项过 §7 自检清单 + §7.5 RTM 矩阵 + §7.6 残余风险；③ 一键转 xmind / xlsx / playwrightmode（见 §8）——`L7=是` → `tests/test_arNN_*.py::test_TC_NNN_*` Playwright 函数，`L8=是` → `pytest+httpx` 接口断言函数。

## 1. 层级契约（树状结构）

每条测试点对应固定的 10 层路径（L0~L9），缺一层视为不合格。L2 与 L3 之间**必有**一个 h4 子层 `#### 测试范围 N` 作为 8 列表格的统一包装层（N ≥ 1；即便 AR 下只有 1 张表也包成 `测试范围 1`）——它**不是**第 11 层、**不进**编号体系，只是 markdown 文档结构辅助，让「AR → 测试范围 → 表」永远是单一形态，下游 xmind / xlsx / pytest 解析器无需分支判断。

```
迭代号                                      L0  根节点（如：迭代16-OPS）
└── AR-XXX.YYY.ZZZ                          L1  需求编号 — 从需求文档原样摘录，禁止自造或加 .01/.02 子段
    └── 需求标题（功能描述）                 L2  需求名称 — 从需求文档原样摘录，不改写
        └── #### 测试范围 N — 摘要            [h4 辅助层，必有；N ≥ 1，AR 内连续编号；不进 §2 字段字典 / RTM AR 列]
            └── 测试点标题                    L3  一句话用例标题（句式见 §3.1）
                └── 预期结果                  L4  多行编号文本 "1. ... 2. ..."
                    └── 用例类型              L5  基本流 / 异常流 / 备选流 / 冒烟
                        └── 是否准入          L6  是 / 否
                            └── 是否UI自动化   L7  是 / 否
                                └── 是否API自动化  L8  是 / 否
                                    └── 优先级       L9  1 / 2 / 3
```

### 1.1 h4 命名规范

```
#### 测试范围 N — {中文摘要}（spec FRn criteria X / Y / Z）
```

- N 从 1 起递增，同 AR 内连续编号；不同 AR 各自重置（每个 AR 都有自己的 `测试范围 1`）
- 摘要写一句话，覆盖本组表格的测试重心（如"产品线列表展示、来源标签与默认排序"）
- 括注里的 spec criteria 编号是反向追溯锚点，评审反查时直接定位 spec 段落；若该分组覆盖某个完整 FR 则写 `spec FRn` 即可
- **绝不**写成 `#### AR-XXX.YYY.01 — …` 形式（变相自造伪 AR，违反 §1.5 Phase 6 hard rule）
- AR 下只有 1 张表时也必须包一层 `#### 测试范围 1 — {该 AR 唯一测试范围摘要}`——多 1 行 markdown 换取下游解析统一

### 1.2 反例（v2 Phase 6 之前的错误做法）

❌ **错误**：spec 给 `AR-PRL-FUN-01.001.00 查看产品线列表`（1 个 AR），文档把它写成 3 个 L3 标题 `AR-PRL-FUN-01.001.01 / .02 / .03`，并把每个标题改成测试范围中文摘要。

后果：
- L1 唯一性污染：下游用例平台 / xmind / xlsx 按 AR 编号建索引时出现 3 个"幽灵 AR"（spec 里查不到）
- L2 需求名称被挤掉：失去 spec 需求名"查看产品线列表"这一硬锚点 → 评审追溯回 spec 时少 1 层语义指针
- AR 概念混淆：AR 是 Acceptance Requirement（需求），不是测试点分组标识

✅ **正确**：L1 用 spec 原样 `AR-PRL-FUN-01.001.00`，L2 用 spec 原样"查看产品线列表"，下挂 3 个 `#### 测试范围 1 / 2 / 3 — …` h4。

## 1.5 模块编号约定（L1 第一段）

**v2 Phase 5 修订（A/B 实测反馈 F3 — "测试点编号不应固定，应根据需求文档中的需求编号来决定"）**：L1 AR 编号的**前缀**（如 `AR-SATP-` / `AR-PRL-FUN-` / `AR-KDevSec-` 等）和**编号本身**必须由 **spec（`--input`）提供**——通常 spec 第一条需求的 header（如 `# 需求 AR-PRL-FUN-01.001.00：...`）就声明了 namespace。**本 MOD 不锁前缀，也不写默认。** skill v2 严格按 spec 抽取（grep `--input`），不再让 MOD/example 兜底。若 spec 没有 AR-NNN 模式，skill 应 stop-and-ask 或要求用户传 `--ar-prefix` override。

**v2 Phase 6 修订（A/B 实测反馈 F4 — "AR 是需求，测试点不能叫 AR；不得用 `.01 / .02 / .03` 子段自造伪 AR"）**：L1 完整三段编号（含尾段 `.ZZ` 序号）必须 **spec 原样**整体摘录；**严禁**在 spec 给出的编号后追加 `.01 / .02 / .03` 子段把 1 个真实 AR 拆成多个伪 AR。

- **文档 L3 `### AR-…` 标题块的数量必须 == spec 真实 AR 数量**（grep `--input` spec 'AR-' header 行计数）；不得多、不得少。
- **单 AR 内 acceptance criteria 拆为 N 张表** → 用 §1 `#### 测试范围 N` h4 子层逐张包装（N ≥ 1；即便 N=1 也包），不得用伪 AR 编号代替。
- **CRUD-split** 的特殊情形：若 spec 明确把 新增 / 编辑 / 删除写成 3 条独立 AR-NNN（如 spec 同时存在 `AR-XXX.001.00` / `AR-XXX.002.00` / `AR-XXX.003.00`），文档可写 3 个 L3 标题块（一一对应），每个 L3 下挂 `#### 测试范围 1` 包装其单表；若 spec 只给 1 条 CRUD AR，则文档 1 个 L3 标题块 + 3 个 `#### 测试范围 1 / 2 / 3` h4 子层承载 C/U/D 拆分。判断依据**永远是 spec 实际编号数量**，不是 MOD 默认。
- **下游影响**：用例平台 / xmind / xlsx 按 L1 AR 建唯一索引，自造的 `.01 / .02` 会成为"幽灵 AR"——spec 里查不到、RTM 反查 criteria 时也找不到原始锚点，污染追溯链。

skill v2 应在生成后 grep 校验 `### AR-` 行数 == spec AR 数；不等即触发自检 fail。

本模块号语义表**仅作"已有模块的业务语义参考"**：当 spec 用 `AR-SATP-04.YYY.ZZZ` 时，下表帮助理解 `04` 对应"产品线 / 资源主数据"；当 spec 用其他前缀（如 `AR-PRL-FUN-01.NNN.NN`）时，本表不强制套用。

常见模块号语义（参考；具体由 spec 决定）：

| 模块号段 | 业务语义 | 历史 AR 入口示例 |
| --- | --- | --- |
| 04 | 产品线 / 资源主数据 | `AR-SATP-04.YYY.ZZZ` |
| 05 | 项目 / 工程主数据 | `AR-SATP-05.YYY.ZZZ` |
| 06 | 版本 / 发布 / 迭代 | `AR-SATP-06.YYY.ZZZ` |
| 07 | 关联 / 归属 / 跨实体引用 | `AR-SATP-07.YYY.ZZZ` |
| 09 | 安全活动主流程 | `AR-SATP-09.YYY.ZZZ` |
| 99 | **通用 / 跨模块约束**（操作日志、逻辑删除、认证、字典等真正跨实体的约束）| `AR-SATP-99.YYY.ZZZ` |

**模块 99 的纳入边界（v2.1 收紧 — 默认不出 99）：**

实践经验表明，把"跨实体约束"集中到 99 反而降低了可读性与可追溯性。本模板默认**不出**模块 99；下面列出的"跨实体约束"分散承载到对应业务 AR 的具体行内：

- ❌ **不进 99（v2.1）**：FR-004 操作日志四要素 → **inline 到每条 CRUD 行的预期结果末段**（"...N. 操作日志含 操作人/时间/目标实体/操作类型 4 要素"）；100% 覆盖率抽样 + 业务回滚不悬挂 → 集中放在最跨实体的某一业务 AR（典型为关联日志 AR），不另立 99。
- ❌ **不进 99（v2.1）**：FR-002 逻辑删除 → **分散到每个删除 AR**（写入断言：`del_flag=1 物理记录在`）+ **每个列表查询 AR**（隐式过滤断言：`已逻辑删除记录不返回`）；不另立 99 综合 AR。
- ❌ **不进 99（v2.1）**：跨实体冒烟链路（如 SC-001 "5 分钟产品线→项目→版本"）→ 放在链路**终点** AR（如"新增版本"AR 末行），不另立 99 链路 AR。
- ❌ **不进 99（v2.1）**：E2E 通过率（如 SC-003 "4 US 通过率 = 100%"）→ **不单设 TC**，由本测试集合的全部 G/W/T 用例集**整体执行结果**断言，仅在 RTM 与残余风险段记录承诺。
- ❌ **不进 99 / 不进任何业务 AR**：列表性能 / 任何时间特性承诺（如 SC-004 ≤Ns）整体 OOS——详见 §7.6"性能 / 时间特性整体 OOS"主声明段（本文件性能 OOS 仅在 §7.6 完整论述，其他位置只做一行引用以避免漂移）。
- ❌ **不进 99 但进业务 AR**：列表查询字段（FR-003）、UI 排版——**内联**到对应业务模块的列表查询 AR（如项目列表查询字段写在 `AR-SATP-05.001.007`），不要单独立"99 列表 AR"。
- ❌ **不进 99 也不进任何 AR**：spec 没有明确 SC 或 FR 承诺的"通用"性能 / 安全 / 兼容性 / 国际化 / 可访问性测试点——这些是抽象方向，不要凭空生成。
- ✅ **极少数仍可进 99 的情形**：真正跨**所有**业务实体且**无法**内联到任何具体业务 AR 的约束（实践中罕见；进 99 前先确认能否分散）。

**反例 1（性能进功能集合）**：性能 / 时间特性的反例（既不进 99 也不内联进业务 AR）见 §7.6。
**反例 2（操作日志聚合）**：把"对每个核心实体完成 新增/编辑/删除 各 1 次，操作日志各产生 3 类事件"合并到 `AR-SATP-99.YYY.001 操作日志覆盖率` ❌——v2.1 改为分散到每条 CRUD 行末段断言 4 要素；100% 抽样 + 回滚不悬挂集中到关联日志 AR。
**反例 3（逻辑删除聚合）**：把"删除任一实体 → del_flag=1，列表不返回"合并到 `AR-SATP-99.YYY.002 逻辑删除语义` ❌——v2.1 改为分散到三个删除 AR（写入）+ 三个列表查询 AR（过滤）。

## 3. 写作规范（直接套用）

**3.1 测试点标题（L3）句式**

```
{角色}登录，{菜单/接口路径}，{操作}，{变体说明}
```

- 角色固定写法：`超级管理员` / `超级管理员/管理员/用户`（多角色覆盖时用斜杠拼写，不用空格）
- 菜单走 UI 时写菜单路径（如 `用户管理-测试集`）；走接口时写接口名（如 `新增CICD任务接口`）
- 变体说明放在最末，覆盖：`查看页面` / `历史数据` / `特殊符号` / `重复` / `为空` / `长度超过128` / `不存在` / `权限范围外` 等
- 一个测试点只覆盖一个变体；不要"X 且 Y"

**3.2 预期结果（L4）写法**

- 必须以 `1.` 开头编号；多条预期用 `\n2. ... \n3. ...` 真实换行隔开
- 每条预期一句话讲清"应该看到什么 / 应该返回什么"，不写操作步骤
- UI 类典型句式：`新增产品名称字段` / `产品名称必填` / `列表展示信息正确`
- 校验失败类典型句式：`不允许创建` / `不允许修改` / 接口断言可写返回码与文案

**3.3 UI 自动化判断标准（L7=是 的判定）**

L7 字段的本项目语义：**通过 Playwright 脚本驱动浏览器，对完整业务流程进行端到端覆盖**（不是单页面元素静态校验）。下游栈：playwrightmode 模板（`tests/test_arNN_*.py::test_TC_NNN_*`）。

**判定原则**：4 条通用 UI-lane 判定标准见 [skill v2 §3.2 "Automation triage rule of thumb"](../../../.claude/skills/kdev-test-points/SKILL.md)（端到端可串入 / DOM 可断言 / 不依赖肉眼 / 不依赖一次性 UI 状态），逐条满足才 L7=是；任一不满足即 L7=否。

**项目特定例外**：与 §3.5"兜底否"列表合用——本项目对"导出报告 / PDF 内容 / 首次部署一次性场景"统一判 L7=否。

**3.4 API 自动化判断标准（L8=是 的判定）**

L8 字段的本项目语义：**通过 pytest 脚本对业务流程涉及的多个接口进行串联覆盖**（如"创建→查询→更新→删除"形成完整业务闭环），而**非单个接口单参的孤立测试**。下游栈：pytest + httpx 业务流程脚本。

**判定原则**：4 条通用 API-lane 判定标准见 [skill v2 §3.2 "Automation triage rule of thumb"](../../../.claude/skills/kdev-test-points/SKILL.md)（接口可观测 / 可串入业务链 / 契约稳定 / 非单接口单参孤立），逐条满足才 L8=是；任一不满足即 L8=否。

**项目特定例外**：单接口单参的孤立用例统一进单测 / 契约测试，不进 API 自动化集合。

**3.5 兜底"否"判定（L7、L8 同时为否）**

凡满足下列任意一项，L7、L8 均判为"否"：

- 需要人工肉眼确认 UI 排版 / 报告样式（如"导出报告展示信息列表"）
- 需要观察 PDF / HTML 报告内容
- 涉及历史数据快照、首次部署后的一次性场景
- 当前测试集环境无法稳定复现（外部依赖、需要特定基础数据）

其余按 §3.3、§3.4 分别判定。

**3.6 一致性硬约束（生成前先固化，生成后逐项核对）**

> **通用硬约束（角色单一 / 单变体 / AR ≤ 8 行 / CRUD 拆 3 / 名称字段 XSS 最小子集 / 编辑非必填字段空必有行 / 标题长度反模式等）见 [skill v2 §3.2](../../../.claude/skills/kdev-test-points/SKILL.md)（必过）。本节只列项目特定阈值 + 与下游栈强耦合的约束。**

- **角色术语来源（v2 Phase 5 修订 — A/B 实测反馈）**：本 MOD **不写死任何默认角色**。所有 spec 的角色术语必须由 spec 自身明示——通常在 spec 的「全局前提」/「Global Preconditions」段含一行 "主角色统一为「X」" 或第一条 User Story 的"作为 X，我希望..."。skill 严格按 spec 抽取（grep `--input`），不再让 MOD 兜底。若 spec 未声明角色，skill 应 stop-and-ask 或要求用户传 `--role` override。本项目历史上 spec 多写 `超级管理员` 或 `平台管理员`，但这是 spec 选择不是 MOD 选择；不同迭代不同 spec 可能写不同角色（如管理员视角的"超级管理员"vs 业务接入视角的"平台管理员"），MOD 不绑死。
- **L3 标题长度阈值 = 200 字符**：本项目下游 xmind / xlsx / 测试用例平台 varchar(500) 中段会被 join 字段挤占，留给标题 ≈ 200 是安全水位。计算口径：含中英文 / 标点 / 全角半角 / `<script>` 等所有字符按 1 计；`<br>` 等 HTML 标记按字面字符计。**通用原则**（splits over abbreviations、反模式禁止压变体说明硬塞）见 skill v2 §3.2 "Title length floor"。
- **TC 全局编号**：转下游测试用例时建议为每行附 `TC-NNN` 全局序号（在 §5 表格右侧加一列，或在标题后加 `[TC-001]`）；它是 md→playwrightmode 自动生成 `tests/test_arNN_*.py::test_TC_NNN_*` 的锚点。
- **L7/L8 与"是否准入"共振**：`准入=是` 行原则上 `L7=是` **或** `L8=是`（即至少一个自动化通道兜住准入冒烟），否则准入会卡在人工。
- **CRUD 三件套的项目业务断言细节**（在 skill v2 §3.2 通用 "CRUD = three ARs" 基础上的本项目额外要求）：
  - **新增 AR**：必含"4 要素操作日志 inline"行（操作人 / 时间 / 目标实体 / 操作类型）；必含"必填字段查看"行作为 §4 schema 的 L3=显示性测试点。
  - **编辑 AR**：必含 "API 直连关联到已删实体 → 拒绝" 行（本项目 FR-002 逻辑删除语义衍生）。
  - **删除 AR**：必含 `del_flag=1 + 物理记录在` 写入断言（FR-002 inline）+ "4 要素操作日志 inline" 行（FR-004 inline）。
- **性能整体不在功能用例集合范围**：详见 §7.6"性能 / 时间特性整体 OOS"主声明段。
- **默认无模块 99 — 跨需求约束分散承载**：详见 §1.5"模块 99 边界"主声明段（FR-004 4 要素、FR-002 逻辑删除、链路冒烟、E2E 通过率等具体分散落点）。
- **L1 AR 编号 spec 原样、严禁自造子段（v2 Phase 6 hard rule）**：文档中 `### AR-…` L3 标题块的数量必须 == spec 真实 AR 数；**严禁**用 `.01 / .02 / .03` 子段把 1 个真实 AR 拆成多个伪 AR。判定流程：生成后 `grep -cE "^### AR-"` 文档 == `grep -cE "^# 需求 AR-"` spec。
- **L2 需求名称完整保留**：L2 必须是 spec 原样的需求名称（如"查看产品线列表"），**禁止**被"测试范围中文摘要"挤占；摘要属于 h4 子层（§1）。
- **h4 测试范围分组永远存在（v2 Phase 6 unified — 统一包装）**：每张 8 列表格前必有 `#### 测试范围 N — {摘要}（spec FRn criteria X / Y / Z）` 包装层，即便 AR 下只有 1 张表也包成 `测试范围 1`——下游解析器永远按「AR → 测试范围 → 表」一种形态处理，无需分支判断。h4 命名规范见 §1.1；判定流程：每个 `### AR-…` 块下挂的 `#### 测试范围` 数 ≥ 1 且与该 AR 下的 8 列表格数严格相等。

### 示例补充（R-NEW54 反例样本）：易误判为异常流的基本流场景

> **目的**：上面 3 个 CRUD AR 范例全是「输入异样 → 拒绝」(`输入异样 → 异常流`) 的模式，下游写非 CRUD 用例（查看 / 筛选 / 列表 / UI 默认态）时容易把"输入形态不规整 / UI 默认态非典型"也贴成异常流。本节给 2 行反例样本，断"输入形态 → 异常流"的过度泛化。**判类型永远看预期结果的成功/失败极性**，不是输入或状态长得不寻常。配套自检规则见 §7「用例类型 ↔ 预期结果 语义一致性 (R-NEW54)」。

| # | 测试点标题 | 预期结果 | 用例类型 | 是否准入 | 是否UI自动化 | 是否API自动化 | 优先级 |
| - | --- | --- | --- | --- | --- | --- | - |
| 1 | 超级管理员登录，用户管理-测试集，列表筛选输入前后含空白且大小写不同的关键字 | 1. 匹配去除前后空白、大小写不敏感后命中目标记录 | **基本流** | 否 | 是 | 是 | 2 |
| 2 | 超级管理员登录，用户管理-测试集，未选择任何行时查看工具栏编辑/删除按钮状态 | 1. 未选中行时编辑、删除入口按未选中状态禁用 | **基本流** | 否 | 是 | 否 | 2 |

**为什么不是异常流（R-NEW54）**：
- **行 1**「输入形态不规整 但 预期成功（命中）」→ 这是验证筛选 normalization 正常工作的 happy-path，属基本流；若误标异常流，等于把"系统正确处理了用户输入"当成了"系统拒绝了非法输入"。
- **行 2**「UI 默认态非典型 但 预期正常（按规定禁用）」→ 这是验证 UX 默认行为的 happy-path（未选行时按钮就应该是禁用态），属基本流；若误标异常流，等于把"按钮按规定灰着"当成了"按钮坏了"。

**真正的异常流必有**：预期含「拒绝 / 不允许 / 阻止 / 校验失败 / 提示错误 / 错误态文案 / 越权拒绝」等**失败极性**词汇——参照本节上面 AR-001.01 #3 (产品名称为空) / #4 (长度超过 128) / #5 (含\<script\>) / #6 (重复) 4 行，它们都满足"输入异样 + 预期拒绝"双条件。

## 5.1 通用模块（99）使用边界（v2.1 修订 — 默认不出 99）

## 7.6 残余风险与未覆盖说明（强制段落）

按以下分类显式列举本期不覆盖项，给评审一个"我们承认没盖什么"的清单——不写就违约。

- **批量操作** — 批量新增/编辑/删除/批量取消关联是否在范围；不在则一句话说明并指明后续 PRD。
- **国际化** — 是否含多语言断言；本期单语言则声明。
- **权限 / 越权** — 是否引入权限码；本期不引入则说明所有登录用户等价于本测试集预设角色。
- **审计字段细化** — `created_by/at`、`updated_by/at` 是否单独立 AR，还是仅做"非空 + 时间戳格式"存在性断言。
- **视觉 / UX 排版** — 标 ❌ 的 UI 二次确认、字段红星等本期是否升 🚩；后续接入 Playwright/playwrightmode 时重评。
- **大文本 / 安全（XSS / SQLi）** — 当前是否仅在个别用例点涉及；如需全面安全测评，建议另立 cso/security-audit 域 AR。
- **外部对接 / 跨系统** — RDM / CQ / 其他平台对接是否本期不在范围。
- **维护性 / 可移植性 / 可恢复性** — GB/T 25010 中本期不评测的子特性显式声明，避免被外部测评机构当作遗漏。
- **性能 / 时间特性整体 OOS（本节为本文件唯一主声明段；§1.5 / §3.6 / §5.1 / §7 一律以一行引用回指本段）**：
  - **规则**：spec 中任何时间特性承诺（如 `SC-004 ≤Ns`、列表分页响应时间、Tab 切换响应、模糊查询响应等）以及任何形式的"≤Ns"断言**整体不进本测试点文档**——性能不属于功能用例设计范畴，与数据规模无关。功能集合中列表查询 AR 的预期结果**只**断言"功能正确性 + 字段返回结构 + 过滤/排序/分页逻辑"，**不**写"≤2 秒"等时间断言。
  - **数据档位均不例外**：产品线列表（≤200 条）/ 项目列表（≤2000 条）/ 版本列表（≤20000 条）/ 任何更高档位 —— **均不单设性能 TC**。
  - **反模式 ❌**：把"项目列表 2000 条 ≤2 秒"写成业务 AR 内的一行（既不单立 99 性能 AR 也不内联到列表查询 AR）——这种"打散内联"看似分散，实际仍是把性能混进功能集合，同样违规。
  - **承诺转出**：SC-004 一类承诺统一在本节标注为 OOS（措辞模板："列表性能 / SC-004：性能测试不属于功能用例设计范畴；由独立 benchmark 集合兜底"），由独立 benchmark / 性能测试集合承担时间特性断言，建议交付前另立专项。
  - **闭环要求**：本项**必须**在本节显式列出以闭合 RTM——否则审视时会被认为"遗漏"而非"主动转出"。

- **md → xlsx**：每行展开成 Excel 一行，列序对齐用例平台模板（标题 / 预期 / 用例类型 / 是否准入 / 是否UI自动化 / 是否API自动化 / 优先级）。
- **md → playwrightmode**：每条 `L7=是` 的测试点对应一个 `tests/test_arNN_xxx.py::test_TC_NNN_xxx` Playwright 函数；每条 `L8=是` 对应一个 `pytest+httpx` 接口断言函数。`L7=L8=否` 的人工核对项暂不接入或后续用例升级时重评。
- **回归校验**：转出后用 `python3 -c "import zipfile, json; ..."` 解 xmind 的 `content.json`，比对节点数与各字段计数应与 md 中的行数 / `§2` 分布吻合。
