# Test Point Designer (ISO/IEC/IEEE 29119-4 × GB/T 25000.51)

Senior QA test analyst working to a dual standard:

- **29119-4** = the workshop. How to design test points — formal techniques + the pipeline `Test Basis → Test Conditions → Coverage Items → Test Cases → Test Procedures`.
- **GB/T 25000.51** (≡ ISO/IEC 25051) = the acceptance checklist. What must be tested + how to declare conformity — three domains × 8 characteristics × 31 sub-characteristics × 符合/部分符合/不符合 verdict.

Without 29119-4 your tests are ad-hoc. Without 25000.51 you only test functions and miss the non-functional half. Use both. Every output must be auditable on both axes.

This skill produces the **测试点 / test-design** artifact. If you need to then render those test points as Playwright-handoff fielded test cases (1:1 mapping, byte-equality contract), hand the output to `kdev-test-cases`.

## 1. Modes (declare first, gate the rest)

The volume of ceremony depends on **why** the user wants tests. Don't apply third-party-certification rigor to a feature spec, and don't ship a bare API checklist when an audit is on the line.

| Mode | When | Main file emits | Audit companion (`<stem>-audit.md`) |
|---|---|---|---|
| `feature-spec-lite` (auto when `--example` is a fielded-template, see §1.1) | internal PRD/spec **and** user provided a fielded-template example (e.g. SP15 / xmind / SOP 测试点 — the project's own house style) | 范围与参数表 + §6.5 TC (example columns intact, **no extra columns**) + RTM + Risk + 自检清单 | **NOT generated by default** — opt-in via `--audit`. If requested: §6.1 Header, §6.2 Quality Matrix (in-scope rows only), §6.3 COND, §6.4 CI, §6.7 Coverage Summary, §6.8 Conformity light, §6.11 Risk-detail |
| `feature-spec` | internal PRD/spec; software being built; no docs published yet; no fielded-template example | Header, Quality Matrix (in-scope rows), Conditions, CIs, TC core, Coverage Summary, RTM, Risk | not generated by default — opt-in via `--audit` |
| `api-contract` | one or more API endpoints with request/response schema | Header, Conditions, CIs, TC core, Coverage Summary, RTM | not generated by default — opt-in via `--audit`; if requested: Quality Matrix collapsed to in-scope rows only; TC-DOC skipped; Conformity light |
| `full-conformity` | RUSP / COTS / 25051 软件产品测评 / 第三方测评机构验收 | All §6.1–§6.11 in [references/output-templates.md](references/output-templates.md) | none (everything inline in main; `--audit` flag has no effect) |

**Pick the mode in this order**: explicit `--mode` flag > user keywords (`测评`/`RUSP`/`COTS`/`25051`/`认证` → `full-conformity`; `接口`/`API`/single-endpoint paste → `api-contract`; fielded-template example detected → `feature-spec-lite`; otherwise `feature-spec`). When ambiguous, ask once.

`--lite` further compresses any mode by writing only Header + Conditions + TC core + RTM. Use when the user wants a sketch.

**Audit companion default = OFF**: across all non-`full-conformity` modes, the audit companion is **not generated** unless the user passes `--audit`. Rationale: in real internal pipelines (typically: fielded-template example → kdev-test-cases renderer → Playwright tests), the 25051 audit ceremony (8×31 matrix, COND/CI tables, conformity verdicts, bidirectional RTM) is dead weight — the main file's short RTM + Risk + self-check already closes the loop. The audit material is meant for third-party certification scenarios; if you need that, either use `--audit` explicitly or use `full-conformity` mode (which inlines everything). Do **not** generate the companion file proactively "to be safe" — users who want it will say so.

> **Out of scope for this skill**: rendering an already-existing 测试点 .md as Playwright-friendly fielded test cases (用例编号 / 用例名称 / fielded code blocks). That is a pure 1:1 renderer with a byte-equality contract — qualitatively different from test design — and lives in `kdev-test-cases`. If `--input` resolves to a markdown file that already contains `### AR-[A-Z]+-\d{2}\.\d{3}\.\d{3}` headers with numbered table rows shaped `| # | 测试点标题 | 预期结果 | ...`, stop and tell the user: "This looks like an upstream 测试点 .md — hand it to `kdev-test-cases` for fielded rendering, not to this skill."

### 1.1 Fielded-template example auto-detection

An `--example` is fielded-template style when **any** of the following match — in which case `feature-spec` auto-promotes to `feature-spec-lite` (main file stays template-clean; audit companion remains opt-in per §1 default):

- The example contains a row matching `AR-[A-Z]+-\d{2}\.\d{3}\.\d{3}` (three-segment AR编号 — common in SATP-style numbering), **or**
- The example header has columns `测试点标题 | 预期结果 | 用例类型 | 是否准入` and total column count ≤ 9, **or**
- The example file path or content references known fielded-template markers (e.g. `SOP_测试点`, `SP15`, `xmind`, `playwrightmode`, or the project's own equivalents).

When detected, **prefer the example's own self-check / RTM / 残余风险 sections if present** rather than emitting fresh ones — they are the project's house style.

> **Bundled default**: when `--example` is omitted, the skill defaults to its own bundled `assets/SOP_测试点MODv2.md` (see §2). That file matches the markers above (`SOP_测试点`, three-segment AR编号, ≤9-column fielded schema), so the **default run auto-promotes to `feature-spec-lite`**. This is intended: the skill ships ready-to-use in the project's house style with zero arguments. Override with `--example <other>` or opt out with `--example none`.

## 2. Arguments

```
/kdev-test-points [--input <path>]
                     [--example <path>]
                     [--output <path>]
                     [--mode feature-spec|api-contract|full-conformity]
                     [--lite]
                     [--audit]
                     [--ar-prefix <prefix>]
                     [--role <term>]
                     [--allow-derived-ar <编号列表>]
                     [free-form prompt]
```

- `--input <path>` — spec / requirements / API contract / source code. **v2 Phase 5**: this is also the authoritative source for AR-编号 prefix (grep'd from spec headers) and role term (grep'd from spec's 全局前提 / first user-story actor). May be repeated.
- `--example <path>` — a template the user wants the **rendering layer** to follow (e.g., fielded code blocks, internal xmind/xlsx-aligned format, or the project's MOD). See §3. **v2 Phase 5**: example provides rendering schema (column count, table shape, RTM format) but **NOT** AR-prefix or role — those come from `--input` only.
  - **Default when omitted (bundled example)**: if the user does **not** pass `--example`, default to the skill-bundled template `assets/SOP_测试点MODv2.md` — **resolve this path relative to this SKILL.md's own directory**, never as an absolute or project-relative path, so the default travels with the skill to any install location. If the user passes any `--example <path>`, **the user's value always wins** — do not merge, layer, or fall back to the bundled default. Because the bundled default is itself a fielded-template (matches §1.1 markers), the effective default mode becomes `feature-spec-lite` (see §1.1). To run with **no** rendering example at all, the user must pass `--example none` (treat the literal `none` as "no example" and proceed in `feature-spec`).
- `--ar-prefix <prefix>` (v2 Phase 5 new) — explicit override for the AR-编号 prefix when `--input` spec lacks AR-NNN headers. Use this instead of letting the skill fabricate a default. Example: `--ar-prefix AR-PRL-FUN-`.
- `--role <term>` (v2 Phase 5 new) — explicit override for the role term when `--input` spec lacks an explicit role declaration. Use this instead of letting the skill default. Example: `--role 平台管理员`.
- `--allow-derived-ar <编号列表>` (v2 Phase 5 third hard rule — R-NEW51 / G-011) — explicit override to allow output AR count to **exceed** `--input` spec's physical AR count. Use this **only** when spec preamble explicitly authorizes the derived ARs (e.g. "拆条说明：CRUD 各为独立需求"), and **always** cite the spec preamble line that grants the authorization in the trailing `模式与生成元数据` section. Example: `--allow-derived-ar=AR-PRL-FUN-01.003.001,AR-PRL-FUN-01.004.001,AR-PRL-FUN-01.005.001`. **Default behavior** (no flag): if `N_output > N_spec`, the skill **stops and asks** the user to either (a) reduce output AR count to `N_spec`, or (b) pass this flag with explicit spec-preamble citation. **Do NOT** silently补 ARs from preamble hints / prototype handlers / inferred BR IDs / symmetric权限点 naming.
- `--output <path>` — where to Write the result. If absent, return inline.
- `--mode` — explicit mode override. See §1.
- `--lite` — compressed output.
- `--audit` — **opt-in** to generate the `<stem>-audit.md` companion (full 25051 audit material — Header / 8×31 in-scope matrix / COND / CI / Coverage Summary / Conformity light / bidirectional RTM / Risk-detail). Default: companion is **not** generated for `feature-spec-lite` / `feature-spec` / `api-contract` modes. Has no effect on `full-conformity` (always inline). Use this when handing the output to a third-party assessor, internal compliance review, or whenever the user explicitly asks for "审计材料" / "25051 矩阵" / "符合性评价表".

If only a free-form prompt is given (no flags), treat the prompt body as the test basis and infer the mode.

## 3. Template override rule (read this — it's the bug we're fixing)

When the user gives `--example <template>`, **the example replaces only the §6.5 Test Case rendering layer**. The main file always emits Header (light) + Risk + short RTM + self-check; the heavier audit scaffolding (Quality Matrix / COND / CI / Coverage Summary / Conformity Evaluation / bidirectional RTM) is **opt-in via `--audit`** for the first three modes, and inline for `full-conformity`.

**Default placement by mode (audit OFF by default):**

| Mode | Without `--audit` (default) | With `--audit` |
|---|---|---|
| `feature-spec-lite` | main file = template main only (范围与参数 + §6.5 TC + RTM short + Risk + self-check). **No companion file.** | also emit companion `<stem>-audit.md` (heavy ceremony) |
| `feature-spec` | main file with Header + in-scope Quality Matrix + COND + CI + TC + Coverage Summary + RTM + Risk **inlined as appendix** (single file) | also emit companion `<stem>-audit.md` separating the audit appendix |
| `api-contract` | main file with Header + COND + CI + TC + Coverage Summary + RTM (single file) | also emit companion `<stem>-audit.md` with Quality Matrix + Conformity light |
| `full-conformity` | inline in main (single auditable document, §6.1–§6.11 in one file) | unchanged — flag has no effect |

Why audit-OFF by default for the first three modes: fielded-template examples (e.g. SP15, SOP 测试点) feed directly into downstream renderers (`kdev-test-cases` → Playwright tests). Internal-pipeline users delete the companion immediately ~100% of the time — it's dead weight for their workflow. Conformity ceremony is appropriate for third-party assessors, internal compliance reviews, or when the user is preparing to defend the test design upstream. **Do not generate the companion proactively** "in case the user wants it" — that's the failure mode this default is preventing. If the user wants it, they'll pass `--audit` or ask for "审计材料" / "25051 矩阵" / etc.

### 3.1 Column contract — do not extend the example's row schema

When the example's §6.5 table has `cols ≤ 9` (typical fielded-template schema, e.g. SP15, is 7–8 cols: `# | 标题 | 预期 | 类型 | 准入 | UI自动化 | API自动化 | 优先级` ± `标记`):

- **Do NOT** add `覆盖CI` / `质量子特性` / `测试技术` as new right-side columns. Doing so breaks downstream conversion scripts (e.g. md→xmind/xlsx) that depend on column count.
- **Without `--audit` (default)**: do not surface CIs / Sub-char / Domain / Technique anywhere in the main file. They live implicitly in the section headers (the AR title naturally cites the FR(s) it implements) and the short RTM (FR↔AR↔key-test-point). That's enough closure for the internal pipeline.
- **With `--audit`**: surface them in the companion file `<stem>-audit.md` as an "AR-CI 映射表" — one row per AR (not per TC), referencing TC ranges. Do **not** retrofit them into the main §6.5 table.
- **Only when the user explicitly asks for a single-file deliverable that exposes CIs**: use per-AR header annotation `### AR-XXX-NN.NNN.ZZZ — 标题（覆盖 CI-001..CI-005｜功能完备性 / 正确性｜EP+DT）`. Title prefix on each TC row (`[CI-EP-01,功能正确性,软件] ...`) is a last-resort fallback.

When the example's `cols ≥ 10` (Excel matrix style), the schema already has room — extending it with the 4 invariant columns is fine. See [references/template-override.md](references/template-override.md) for the per-family rules.

### 3.2 Hard constraints (lock these before generating, verify after)

- **AR 编号空间锁定 — `--input` spec is the ONLY authoritative source (v2 Phase 5 hard rule)**：grep **the `--input` spec** for `AR-[A-Z]+-\d` patterns (e.g. spec headers like `# 需求 AR-PRL-FUN-01.001.00：...` or section markers like `### AR-...`). The prefix you find in the spec is the **only** legal prefix for the output. **Never** take the prefix from `--example` / project's MOD / skill defaults — those are downstream rendering aids, not numbering authorities. **Never** invent a new prefix. **If the spec contains no AR-prefix pattern**, stop and ask the user to either (a) annotate the spec with an explicit AR编号 namespace, or (b) pass `--ar-prefix <prefix>` as an explicit override — do **not** fabricate, do **not** silently inherit from `--example`. Rationale: A/B testing on real spec (KDevSec/Iteration-one-v0.1/产品线管理需求) found that example-locked prefix produced `AR-SATP-` output even though the spec used `AR-PRL-FUN-` — a regression vs the spec's declared namespace. F3 feedback: "测试点编号不应固定，应根据需求文档中的需求编号来决定".
- **模块号语义锁定**：if the project MOD declares module-number conventions (e.g. an "entity-type to module-number" mapping table), use it as **a reference** for matching the spec's AR-编号 中间段 to its semantic. This is a **lookup table only** — it does not override the spec's own AR编号. Do **not** flatten every FR into a single module just because it's convenient.
- **角色单一锁定 — `--input` spec is the ONLY authoritative source (v2 Phase 5 hard rule)**：grep the **`--input` spec** for an explicit role declaration. Conventional spec locations to look in (in priority order): (1) a "全局前提" / "Global Preconditions" / "Assumptions" section with a line like "主角色统一为「X」" / "Role: X" / "As a X, I want to..."; (2) the first User Story's actor clause ("作为 X，我希望..."). The term you find in the spec is the **only** legal role term for the whole output. **Never** take the role from `--example` / project's MOD / skill defaults — even if the MOD declares a "project default role", the spec is authoritative. **If the spec contains no explicit role term**, stop and ask the user to either (a) annotate the spec with an explicit role, or (b) pass `--role <term>` as explicit override — do **not** fabricate, do **not** silently inherit from `--example`. Rationale: A/B testing on real spec found v2 (Phase 1-4) wrongly emitted `超级管理员` (MOD default) when the spec's 全局前提 明示 `平台管理员` — Rule 7 "Surface Conflicts" must be reinterpreted with spec at the top of the priority chain, not MOD. F1 feedback: "角色应当由需求文档中提供，skill 和 mode 中不能约定用例使用什么角色". Do not switch role mid-file unless the spec mandates per-role variants.
- **AR 数量上界锁定 — `--input` spec is the ONLY authoritative source (v2 Phase 5 third hard rule — R-NEW51 / G-011)**：before generating, **count `--input` spec's physical AR headers** via `grep -cE '^# 需求 AR-|^### AR-' <spec>` (adjust pattern to match the spec's actual header style; v1.md SP15-style specs use `^# 需求 AR-...`, fielded-template specs use `^### AR-...`); record as `N_spec`. **The output 测试点 .md AR count `N_output` MUST NOT exceed `N_spec`**. If `N_output > N_spec` and `--allow-derived-ar` was **not** passed → **stop and ask** the user, emitting a comparison table (spec ARs listed by编号 + title, vs proposed output ARs) and offering two paths: (a) reduce `N_output` to `N_spec` (drop the推导 ARs and surface them in residual-risk section as "spec 未详写 X / Y / Z，本迭代 OOS，下一迭代 spec 落条后再补"), or (b) pass `--allow-derived-ar=<编号列表>` with a citation of the spec preamble line that explicitly authorizes the derived ARs. **Do NOT** silently补 ARs from any of the following sources (the 6-item forbidden actions checklist):
  1. **禁止把 spec preamble 文档结构声明当作"补 AR 授权"** — phrases like 「拆条说明：CRUD 各为独立需求 / RDM 同步单独成条」or「各自独立需求」or「单独成条」are文档元数据 describing the spec's intended organization, NOT criteria授权. They say "I will eventually split into N条" not "you can write those N条 for me".
  2. **禁止把高保真原型 (index.html / mockup / 截图 / wireframe) 的字段 / 按钮 / JS handler 当作字段级行为权威源** — prototype is a reference implementation for developers, NOT a requirements contract. Functions like `openPLForm('edit')` / `savePL(mode='edit')` / `deletePL(id)` / `rdmSync('productline')` show what the system might do, but only spec criteria are testable contracts.
  3. **禁止引入推断 BR-XXX 业务规则编号** (e.g. `BR-PRODUCTLINE-SYNC-001`) — if spec doesn't declare a BR-编号, don't invent one; inventing BR-编号 in test points pollutes the下游 development实现 (developers may treat the编号 as authoritative and code to it).
  4. **禁止"对称命名"推断权限点** — if spec only declares `productline:list` / `productline:create`, do **not** infer `productline:edit` / `productline:delete` / `productline:sync` by symmetric naming. 权限点 naming is the backend developer's job; test-side推断 contaminates the backend namespace and creates"phantom对称性"that production may not honor.
  5. **禁止把 spec 的「假设与依赖 / 待澄清项 / 范围外 / OOS」段当作 spec criteria 用** — these sections provide context, boundaries, and unresolved items, but they are NOT acceptance criteria. They tell you what's outside scope or pending, not what to test.
  6. **禁止用注脚（"注：依据 X + Y + Z 推导"小字）给非 spec AR 背书** — even with honest 标注 ("本 AR 依据 preamble + 原型推导"), the AR / TC still appears as first-class content with equal visual weight to spec-derived AR / TC. Readers (and downstream renderers like `kdev-test-cases`) ignore the注脚 and treat all ARs as authoritative. fail-loud 适用 "I did not test X" (OOS confession), NOT "I wrote requirements you didn't write, but I添加 a footnote" (越界 + 注脚).

  **How to handle spec gap (replace推导 with proper out-of-scope handling)**:
  - **Residual-risk section's first item**: explicitly list "spec 未详写 X / Y / Z，本迭代 OOS，下一迭代 spec 落条后再补"
  - **Surface to PM / 需求方 as a hard-agenda item** for the next requirements review: "补齐 X / Y / Z 验收标准文本"
  - **Re-run this skill after spec is updated** with真落条 detail — that re-run is the合规 "覆盖完整版"
  - **Do NOT** preemptively write the missing ARs and claim "we have everything covered" — that's the越界 pattern this rule prevents

  Rationale: A/B testing on KDevSec/Iteration-one-v0.1/产品线管理需求 (2 spec ARs) found that v1 emitted 5 ARs / 47 TCs by补充 编辑 / 删除 / RDM 同步 from spec preamble's "拆条说明" + index.html prototype handlers + inferred `BR-PRODUCTLINE-SYNC-001` + symmetric `productline:edit/delete/sync` 权限点 — even with footnote标注, the 5-AR output was treated as first-class spec coverage by downstream readers, contaminating the development namespace. v2 (Phase 5 first 2 hard rules) correctly emitted 2 ARs / 37 TCs, OOS'ing the 3 missing ARs explicitly. **R-NEW51 (sourced from .kdev/memory/改进建议.md) is the source-of-truth; G-011 (.kdev/memory/踩坑日志.md) is the实证 evidence; this hard rule is the skill-layer enforcement.** Step-by-step: see [踩坑日志.md G-011](../../.kdev/memory/踩坑日志.md) for the越界 event chain.
- **AR 内行数 ≤ 8 — split via `#### 测试范围` h4 groups, NOT via sub-AR**：if any AR table would emit ≥ 9 rows, **keep the single `### AR-` header intact** (so `N_output` stays equal to `N_spec` and the AR-count gate above is satisfied) and split the rows into multiple `#### 测试范围` h4 sub-tables under that AR, each ≤ 8 rows. **Naming format**: `#### 测试范围一：[主题概括]（[用例类型构成]）` — Chinese numeral (一 / 二 / 三 / ...), full-width colon `：` after the index, parenthetical suffix listing the 用例类型 composition of that group (e.g. `（基本流）` when all rows in the group are 基本流; `（基本流/异常流）` when mixed; `（异常流）` when all异常). The `#` column within an AR continues to **number consecutively across all 测试范围 groups** (e.g. 测试范围一 rows 1–6 → 测试范围二 rows 7–12 → 测试范围三 rows 13–15), so downstream `kdev-test-cases` can use it as a stable TC-编号 anchor without collision. Optionally add a one-line note under the `### AR-` header documenting the split, e.g. `> 单 AR 行数超 8，按「测试范围」分组保持每表 ≤ 8 行；# 列在本 AR 内连续编号（下游 TC 编号锚点）`. **Do NOT split into `.001` / `.002` / `.003` sub-ARs** — that would inflate `### AR-` count and violate the AR-count gate (R-NEW51 / G-011). Target ≈ 5 rows per 测试范围 group.
- **Title length floor (new in v2 — promoted from project MODs)**: each test-point 标题 must fit within the downstream platform's varchar/cell limit (typical industry default = 200–500 chars; the project MOD declares the exact threshold). When a title would overflow, **split it into multiple TCs by variant** ("single variant per row" + "split rather than abbreviate" — see also Exception flow completeness floor below). **Anti-patterns** that look like length-fixes but actually break the schema: abbreviating menu names, dropping the role prefix, compressing the variant clause into ambiguous wording. Reason: downstream xmind/xlsx/playwrightmode importers cap at varchar(500); titles that overflow either truncate silently or break the import; titles that hide multi-variant behavior produce un-debuggable failures. Reviewers also cannot validate a 200+ char title at a glance — keeping titles short is also a readability floor.
- **One-FR-one-AR — do NOT merge cross-FR common test points**: even if three modules each have a "delete-precondition" check, they belong in three separate ARs (one per business module), not a single shared "通用删除前置" AR. Duplication is preferable to losing per-FR traceability.
- **Cross-cutting constraints disperse rather than aggregate**: prefer inlining a cross-entity constraint (audit-log fields, soft-delete semantics, end-to-end smoke chains, pass-rate guarantees) into the relevant business AR rows (e.g. each CRUD row, each list-query AR, the terminal AR of a chain), rather than collecting them into a virtual "catch-all" module. Project-specific anchors (which FR/SC each constraint maps to, which module number is the catch-all if any, exact inline wording) belong in the project's MOD, not in this skill.
- **CRUD = three ARs (create / edit / delete), not one lumped AR**: split each entity's CRUD basics into three independent ARs by verb. Don't pack "create + edit + delete" together. Generic per-verb floors (apply universally; project-specific assertions like audit-log fields, soft-delete persistence semantics, dictionary checks belong in the project's MOD):
  - **Create AR** — required-field display + happy-path create + one independent "empty" exception row per required field + length-bound row + special-symbol row (e.g. `<script>` — name-bearing entities) + reference-non-existent rejection via API.
  - **Edit AR** — modify primary fields + **modify-optional-field-to-empty → accept + NULL persisted** (mandatory floor, easily missed) + modify required to empty → reject + modify to uniqueness conflict → reject + special-symbol row + API direct-call to relate-to-deleted-entity → reject.
  - **Delete AR** — happy-path delete (project MOD specifies whether logical or physical) + one independent exception row per delete precondition (has-children / has-related-entities) + API direct-call to delete-non-existent → reject.
- **No abstract non-functional ARs**: do **not** invent ARs for security / compatibility / i18n / a11y / maintainability / portability **unless** the spec contains an explicit SC/FR clause for that direction. When in doubt, omit. (Performance is treated separately — see next bullet.)
- **Performance / time-behaviour belongs in a dedicated benchmark suite, not a functional suite**: response-time / latency / throughput assertions are not part of functional test-point design. Do not emit dedicated performance TCs in a functional document, and do not inline `≤Ns` assertions into list-query AR rows. Functional list-query ARs assert correctness, field-return structure, and filter/sort/paging logic only. Time-behaviour claims from the spec must be declared OOS in the residual-risk section with an explicit benchmark-handoff note. The reason: mixing time assertions into functional rows produces flaky tests, conflates two test populations, and forces functional CI to bear performance noise. Project-specific spec anchors (which SC/FR ID, which data-volume tiers) belong in the project's MOD.
- **Name-field XSS minimum subset**: every name-field-bearing create / edit AR emits one row `name contains special characters \<script\>` as the minimum XSS subset (异常流 — reject or escape-and-store-safely). Full XSS / SQLi sweep belongs to a separate security-audit pass; this row is the floor, not the ceiling.
- **Exception flow completeness floor per AR**: for each business AR, the exception-flow rows must independently cover every validation declared in the spec for that FR — empty/required, length over-cap, uniqueness conflict, non-existent reference, format-illegal, state-illegal, dictionary-out-of-range, **special character `<script>`**, etc. Each as its own row. Do not consolidate ("名称为空 + 长度超长") into one row.
- **Every non-performance quantitative claim in the spec has a TC**: claims like `≥N records`, `100% coverage`, `violation-rate = 100%`, `trigger-rate = 100%` must each surface at least one corresponding TC, placed inline next to the functional AR that owns the claim — not aggregated into a generic "metrics AR". **Performance / time-behaviour claims (`≤Ns`) are excluded** — see the dedicated performance rule above; they go to a benchmark suite, not a functional TC.
- **Automation triage rule of thumb (new in v2 — promoted from project MODs)**: when the schema asks each TC to declare "UI-automatable? Y/N" + "API-automatable? Y/N" (e.g. SP15-style L7/L8 fields), apply these 4 universal criteria *independently per lane*. A TC qualifies as **UI-automatable** when **all four** hold: (a) the TC sits on a real business-flow chain (login → menu → action → verify), pluggable into an end-to-end runner; (b) the expected result is observable via DOM (visibility / text / interaction response) without human visual judgment; (c) it does not rely on look-and-feel judgement (layout aesthetics, report styling, PDF rendering); (d) it does not rely on un-reproducible UI state (historical snapshots, external system timing, one-off bootstrap scenes). A TC qualifies as **API-automatable** when **all four** hold: (a) the function is observable via backend call (return value / DB side-effect / state transition is assertable); (b) the TC can be wired into a pytest business-flow script as one link in a multi-call chain with real data dependencies (not an isolated single-endpoint-single-param check — those belong in unit/contract tests); (c) the contract is stable, not reliant on flaky third-party responses or temp mocks; (d) it's not "isolated single-call coverage only". **Both NO** when the TC requires human visual inspection / PDF or HTML report rendering / historical snapshot / un-reproducible env. Project-specific anchors (the actual L7/L8 field names, the downstream test framework, the project's default UI/API ratio, project-specific exclusions) belong in the project's MOD.
- **测试点是交付产物 —— 正文不夹评审/修订过程留痕（KDevSec 实地反馈 2026-06 沉淀）**：测试点是给下游流水线 / 评审方 / 客户读的**交付物**，不是"怎么改出来的"审计日志。TC 预期单元格与各节正文只写**最终测试设计**（被验证行为 / 验收标准）。**禁止**夹带评审与修订的过程痕迹：评审意见回指（「评#11 研发反馈：超长按截断」）、改判注解（「与 v3.0「阻止保存」预期 surface 改判」）、待决标注（「待 spec/研发明确，本 vN.Y 保留 vN.0 预期」）、评审闭环自检清单（「评#4→删 TC-005 ✓」）。评审改了某 TC 的预期 → 把**结论**直接写进预期（如「UI 截断至 N 字符 + 保存成功」），而**为什么改 / 怎么改**留到项目 changelog / 工程记忆，不进测试点正文。理由：过程痕迹对下游是噪声、泄露内部评审流程、且**会过期**——决策已定后还留着的「待研发明确」会悄悄变成事实错误。
- **标题/用例名称的「变体子句」陈述被验证的内容/条件，不写 UI 操作**：标题句式 `{角色}{菜单}{操作}{变体说明}` 中，{操作} 槽是**被测的业务动作**（新增/编辑/删除/查询）；{变体说明} 子句必须陈述**被验证的条件/数据/状态**（「描述为 1001 字符」「数据范围=全部」「名称含 `<script>`」），**不写**为达到该条件而做的底层 UI 操作（「在描述框输入 1001 字符」「点击…尝试粘贴 65 字符」）——这些操作放 测试步骤。理由：名称要让人一眼看出**验什么**，按键级的**怎么做**会污染名称、且把名称绑死到某个具体 UI。同一测试范围内变体措辞保持一致（如所有 BVA 行写「{字段}为 N 字符（合法上界/超上界）」）。
- **输入-约束类（超长/边界/格式/越权）须 UI + 接口双 lane**：输入约束类测试点（长度超界、边界、格式非法、越权），当两条 lane 都可观测时，**「是否UI自动化」与「是否API自动化」两个标志都标 是**——两条 lane 验的是**不同的事**、互不替代。UI lane 验前端处理（如 maxlength 截断）；API lane **独立验证：前端被绕过时后端自己是否守约束**（直连提交超界/非法值 → 后端约束或拒绝、绝不持久化违规值）。这是上方 automation-triage 规则在约束类场景的精化：把约束当「纯 UI 行为」而关掉 API lane 是 **defense-in-depth 缺口**——最常见的缺陷正是后端信任前端。预期里带一条 API 维度断言（如「API 直连提交超界值 → 后端约束至 ≤上界，不持久化超长内容」）。

## 5. Choosing techniques

Two routing axes that overlap: **input shape** (29119-4) and **quality sub-characteristic** (25000.51). Apply both and union the technique set.

**By input shape:**
1. State machine / lifecycle / workflow → State Transition (mandatory: 0-switch + invalid)
2. Multi-condition business rule → Decision Table
3. Numeric/length/date with bounds → EP + BVA together
4. ≥4 parameters × ≥3 values → Pairwise
5. Structured strings (regex, JSON schema, URL, grammar) → Syntax Testing + EP on field values
6. End-to-end journey → Scenario Testing
7. Source code with safety-criticality → MC/DC or Branch
8. **Always layer**: Error Guessing + a domain checklist (OWASP / WCAG / etc.)

**By sub-characteristic** — for any sub-characteristic marked "在范围" in §6.2, use the technique floor in [references/quality-characteristics.md](references/quality-characteristics.md). Don't generate placeholder cases for sub-characteristics marked out of scope; record the reason in §6.2 and you're done.

## 6. Density heuristics (so the output scale matches the input scale)

A spec with N functional requirements should not produce 5N cases (under-tested) or 50N cases (bloat). Aim for:

- **≥1 TC per FR or per acceptance scenario** (functional completeness floor)
- **CRUD as 3 ARs by default**: one entity's CRUD basics emit ≈ 3 ARs (新增 / 编辑 / 删除) before counting uniqueness / list / 人员 etc. Don't lump them.
- **≥4 BVA cases per bounded numeric or length field** — extreme-low / valid / invalid-type / upper-bound (e.g. for 漏洞数 must include `=0`, `=-1`, `=1.5`, `=非数字`, `=int 上界`)
- **Name-field XSS floor**: every 新增/编辑 AR for a name-bearing entity emits 1 row `名称含特殊符号 \<script\>` (异常流, 拒绝或转义安全)
- **Edit-clears-optional floor**: every 编辑 AR emits 1 row "修改非必填字段为空 → 接受 + NULL/空字符串持久化" (often missed)
- **≥1 negative TC per uniqueness constraint, foreign-key constraint, state-machine transition, or precondition** (error-path floor)
- **≥1 TC per in-scope sub-characteristic in §6.2** (quality conformity floor)
- **≥1 negative TC per *non-performance* quantitative claim in product description / user documentation** (claim-falsifiability floor — covers `≥N条`, `100% 覆盖`, `违规率=100%`, `触发率=100%`, etc.). **Performance/time-behaviour claims (`≤Ns`) are excluded** — see next bullet.
- **Performance OOS for functional suites**: do not emit any performance TC at any data scale; do not inline `≤Ns` assertions into list-query AR rows; declare time-behaviour claims OOS in residual-risk with a benchmark-handoff note. Functional list-query ARs assert correctness, structure, and filter logic only. (See the §3.2 long-form rule.)
- **State-machine coverage**: 0-switch (every adjacent pair) + ≥1 invalid transition + double-confirm dialog confirm/cancel both branches if the spec mandates it
- **Decision-table coverage**: every rule row including the impossible/null-edge combinations (e.g., `parent_id IS NULL` on uniqueness)

Use the floor as a budget, not a target. For a 23-FR spec expect roughly 100–120 cases (≈ 5 cases × 20–25 AR after CRUD-split), weighted toward FRs with state machines, uniqueness, or numeric bounds. **Per-table ceiling = 8 rows per `#### 测试范围` h4 sub-table** — when an AR would emit 9+ rows, split into `#### 测试范围一 / 二 / 三 ...` h4 groups under the single `### AR-` header (see §3.2 for naming format and rationale). Do **NOT** split into `.001` / `.002` sub-AR — that would inflate `### AR-` count and violate the AR-count gate (R-NEW51 / G-011). If your output is far below or above this band, double-check before emitting — likely you skipped FRs or fabricated filler.
