## 0. 元结论（先读这一段）

**今天高速演进，但代码侧的跨插件接缝整体是健康的——漂移风险主要不在代码、在文档治理。**

1. **核心契约干净**：events.jsonl 唯一所有者 = kdev-core，kdev-memory **零** events 引用、**无双账**；kdev-hud 纯只读、**无空壳读不存在的事件**；reviewer callee 寄生 caller flow 的 verdict 入账链路字段 100% 对齐；4 员工 node-table 同构、callee 用 dispatch-table 是设计内的两形态、非漂移；coding/design-flow 死链=0、死代码=0。**这些是真接缝，且对得上。**

## 1. 给主控拍板的【新 Q/G 候选】（其余都是已登记 defer 或纯文档回写）

> 只有这 3 条是**新增、未被现有 Q/G 覆盖**、需要主控拍板的。

### G 候选 1 — 时间戳 ID 双认遗漏，召回静默漏召新踩坑【🔴 真 bug】
> kdev-memory 记录 ID 时间戳化（Q-020）后，`hooks/lib/trigger-match.py:236` 的踩坑召回扫描器正则仍是 `^##\s+(G-\d+)[：:]`，**只认旧顺序号 G-NNN，对新时间戳形 `## G 20260613-101432-ly: …` 静默 MISS**。踩坑召回是"防重踩"的核心通道，新坑的 triggers 行永远不被收集 → 用户再撞同类坑时 recall 静默不召。`distill.py:59-65` 同作者已正确双认（`G(?:-\d+|\s+{_TS})`），trigger-match 漏改 = 双认改造在多个 hook 各自维护正则导致的半修。**根因建议**：抽 `step_id.parse_record_id` 做单一解析真相源，消除 distill/trigger-match/step_completeness 各自正则漂移。
> （`trigger-match.py:270` 的 Step 扫描器靠 `[\w.-]+` 宽松通配"碰巧"兼容时间戳形——当前能跑，但与同文件 G 路径策略自相矛盾，属脆弱巧合，一并处置。）

### Q 候选 2 — reviewer 回函 schema 与 handoff CLI reader 契约混用【🔴 未测的跨插件断点】
> handoff 机制现已"一机三用"：(a) intra-flow 主循环↔业务 agent、(b) P-B 跨员工、(c) reviewer 发函回函。前两用共用 CLI `handoff-write/read` + 同一 `<node_id>.handoff.json` schema（强制 `node_id/employee/status/summary` 四必填键，`flow_state.py:453`），落位、调用与产出逐一对得上（健康）。但**第三用（reviewer 回函）**：reviewer 用裸 Write 产 `handoffs/reviewer/<gate>.handoff.json` = `{verdict,scores,counts,revisions,by,request_id}`，**缺全部四个 CLI 必填键**；而 `reviewer-orchestrator.md:35` / `SKILL.md:291` / `gate-decision-logic.md` 都让 caller 用 **CLI `handoff-read`** 取该文件的 verdict——真走 CLI 会 `FlowStateError: missing required key 'node_id'`，回函读不出来。且 `test_reviewer_wiring.py` 不覆盖该读取路径（假绿）。**需拍板**：reviewer 回函改走 CLI `handoff-write`（verdict 塞 `gate_input`，status=done）统一 schema，**还是**显式降级为"裸文件交接"并把文档里的 CLI 动词 `handoff-read` 改成普通 `Read`——二选一并补一条 wiring 测试钉死读取方式。

### G 候选 3 — callee 想往 caller events.jsonl 留痕但无写入通道【🟡 小】
> `reviewer-orchestrator.md:55` 写"发现两能力相反结论 → 在 caller `events.jsonl` 留痕（标元评审异常）"，但 reviewer 是 callee、**不碰 kdev-core 状态机**（同文件 line 35 明示），core 也无"meta-review anomaly"事件类型——这条 agent 指令**无落地手段、悬空**，跑到仲裁冲突分支会卡。最小修：改成写进 reviewer 本就产出的 `<gate>.arbitration.md` / 回函结构化字段，由 caller 决定是否转事件；或给 core 加 anomaly 事件类型 + 约定 caller 代写。需定一条 callee→events 的 anomaly 契约。

## 4. 逐接缝详情（10 个 targeted angle）

### 接缝1 · scope 机制生死【高危】

### 接缝2 · events.jsonl 所有权 + 双账【高危】
**判定 = 无双账、所有权干净**。全插件 grep：写入者只有 kdev-core（`events.py:25` append_event 唯一入口，行内 actor），读取者只有 kdev-hud（`datasource.py`）。kdev-memory **零** events.jsonl/append_event/kdev_core.events 引用。完全符合"加料层原则/不造第二本账"。**唯一 gap**：设计承诺的"rollup 从 events 派生"未落地，Step 仍 100% 手写 markdown（→ D5 🟡，Phase B/C defer）；记忆底座 §6 把"派生 rollup"当已定能力写，配合 DOC1 §6 stale 会放大"以为已实现"的误导。

### 接缝3 · 记录 ID 体系一致性
**判定 = ID 回链结构上不通（且设计本就不要求机器通）+ 残留一处真 bug**。`origin`（"决策日志#Q-012"）是只存不解析的人读自由文本，kdev-core/hud 全程无 split/parse/resolve；反向 memory→features/<slug> 链无任何约定（⚪，设计如此，§2.2 mint 规约本就标"待优化"；将来 HUD/CQO 要程序化跳转需先定 grammar）。`mint_record_id` 实现与 P-C2 §11 逐条一致（who 缺省省后缀/同秒 flock atomic .N/本地时区），111 测试绿（⚪）。**但** `trigger-match.py:236` G 召回正则漏双认（→ G 候选1 🔴），Step 扫描器脆弱巧合兼容（🟡）。
