# 兜底记录 + 三层防线 — design

> 状态：**P1 已实施 0.19.6 + P2 缺口①②④ 已实施 0.19.7（2026-07-07）**；P2 升格工具化 + P3 待续（见 §8）。作者会话：2026-07-07。
> 起源：诊断"记忆积累断档"发现——`.kdev/memory/` 跨天 3 个 `WARN-未记录-*.md` 堆着没人补，"兜底"名不副实。根因：实时落盘靠**主会话主动 dispatch step-recorder**（行为、会漏），SessionEnd hook 只产 WARN 提醒不真记。
> 相关：G-004（版本 stale，本方案落地依赖刷 marketplace 激活 0.19.x JSONL 主账）。

## 1. 问题

- **实时落盘 = 主会话自觉动作**：做完 step-worthy 单元 → 写 YAML summary → dispatch `kdev-step-recorder`。dispatch 靠 CLAUDE.md 铁规驱动，**主会话会漏**（实测跨天 3 场未落盘）。
- **现状"兜底"退化成"提醒"**：SessionEnd hook 是纯脚本、无 LLM，写不出合格 Step（`append_step` 有硬 gate：拒过泛 title、key_facts 必填），只能产 WARN 便签让下会话补——而下会话也没人补，等于没兜住。
- **根本约束**：合格 Step 的灵魂（title / key_decisions / key_facts）需 LLM 读 transcript 提炼；hook 只能机械捕获 commits/files/porcelain。

## 2. 设计原则

1. **兜底=防丢，只在"LLM 缺席的丢失关口"上**：kdev 仅两处——**SessionEnd（会话终结）+ PreCompact（压缩截断上下文）**。其余 hook 各归其位（见 §3）。
2. **LLM 在场→促 dispatch，不代记**：Stop 等 LLM 在场的关口，让主会话自己记（质量最高），hook 只强化提醒。
3. **机械层保证"不丢"，LLM 层补"质量"**：hook 机械落**降级 Step**（骨架不丢）；下会话 LLM **升格**成规范 Step。
4. **兜底收敛、共享机制**：两关口共用一个 `append_fallback_step` helper，别各写各的（避免碎片 + 维护地狱）。

## 3. 三层防线（整体架构）

```
① 进行中（LLM 在场）   Stop 促 dispatch + 主会话铁规自觉        ← 首选，质量最高
② 丢失关口（LLM 缺席） SessionEnd + PreCompact → append_fallback_step  ← 保底不丢（降级 Step）
③ 恢复时（LLM 回来）   SessionStart 检测 auto-fallback → 主会话升格   ← 补质量
        全程：PostToolUse(commit-tracker) → pending-commits.json 供料
```

各 hook 角色（不是每个都兜底）：

| Hook | LLM 在场 | 角色 | 本方案是否改 |
|------|:---:|------|:---:|
| SessionEnd | ❌ | 丢失关口·机械兜底 | ✅ 接入 helper |
| PreCompact | ❌ | 丢失关口·机械兜底 | ✅ 接入 helper（复用已有未落盘判定+porcelain） |
| Stop | ✅ | 促 dispatch | ⚪ 可选强化提醒 |
| PostToolUse | ✅ | 供料（已在攒 commits） | ⚪ 不改 |
| SessionStart | ✅ | 升格触发点 | ✅ brief 检测 auto-fallback |
| UserPromptSubmit | ✅ | 读侧·召回 | ⚪ 无关 |

## 4. 核心机制

### 4.1 `append_fallback_step`（新 helper，`hooks/lib/step_log.py` 或新 `fallback_step.py`）

机械组装一条降级 Step record，**宽松写入、不经 `append_step` 严格 gate**（机械 title 必然过不了 gate 1），append 到 JSONL 主账。SessionEnd + PreCompact 共用。

**供料** = drain `pending-commits.json`（`pending_commits.read`）：
- `commits[].sha / .subject / .ts` —— 降级 title 的种子 + commit_shas
- `transcript_path` + `since_offset` —— **留给升格**：LLM 读这段 transcript 提炼语义
- `since_step_id / since_ts` —— 增量边界

**降级 Step record schema**（在正式 Step schema 上加标记、放宽必填）：
```json
{
  "record_id": "<mint>",
  "title": "[待升格·session-end] fix(x): y 等 3 个提交",   // commit subject 种子；无 commit 时 "[待升格] <files 摘要>"
  "date": "<today>",
  "about": "project",                     // 机械默认；可从 conventional prefix 猜（fix→bugfix/ feat→feature/）
  "status": "auto-fallback",              // 新档
  "key_facts": {"tools_invoked_count": <commits+1 或占位>, "errors_hit": 0},
  "fallback": {
    "source": "session-end | pre-compact",
    "commit_shas": ["abc1234", ...],
    "files_touched": ["..."],             // git porcelain
    "transcript_path": "/.../s.jsonl",
    "since_offset": 640,
    "fallback_at": "<ts>"
  },
  "schema_version": "<current>"
}
```

### 4.2 `auto-fallback` status 新档

`hooks/lib/status_schema.py`：加入合法枚举（现 `open|scored|voided-faded|voided-r-<NNN>` → +`auto-fallback`）。**消费方须认这档**：

| 消费方 | 对 auto-fallback 的处理 |
|--------|------------------------|
| `step_completeness` | **不算"已落盘合格 Step"**——否则"是否需兜底"判定被自己产的降级 Step 骗过 |
| SessionEnd / PreCompact 的"是否需兜底"判定 | "今日无**合格** Step"（排除 auto-fallback）才兜；但**重复兜底**由"pending 是否已 drain"闸住（drain 后 pending 清空，第二关口无料可兜） |
| `daily_render` | 标注 "⚠️ 待升格"，不当正常 Step 渲染进日总结 |
| `distill`（`collect_entries` / `is_misalignment_step` / `is_skill_feedback_high`） | **过滤 auto-fallback**，不进任何数据集切片（骨架不是训练料） |
| `step_dualread` | 认（投影时保留标记） |

### 4.3 升格（append-only，SessionStart 触发）

`step_log` 无 update-by-id → **不 in-place 改**，走 append-only（符合 JSONL 账本语义）：

1. `session-start-brief` 检测 JSONL 今日有 `auto-fallback` Step → brief **P0 提示**"N 条降级 Step 待升格"。
2. 主会话读降级 Step 的 `fallback` 块（commit_shas/files + `transcript_path`+`since_offset`）→ 读那段 transcript → 提炼 title/decisions/key_facts。
3. 走正常 `append_step`（**过严格 gate**）append 一条正式 Step，加 `supersedes: <fallback_record_id>`；把降级 Step 标 `voided-faded`（或新 `voided-superseded`）。
4. 两条都在账本里（append-only），dual-read/蒸馏认 supersedes 关系，只取正式条。

## 5. 判定逻辑要点（防两个坑）

- **别被自己骗**：兜底判定用"今日无**合格** Step"（排除 auto-fallback），否则第一次兜底后 hook 以为"已有 Step"再不提醒升格。
- **别重复兜底**：SessionEnd 和 PreCompact 可能同场先后触发。用 `pending-commits.json` **drain 语义**闸住——`append_fallback_step` 成功后清空 pending；第二关口 read 到空 pending → 无料可兜 → 跳过。（纯 Edit 无 commit 的 step-worthy 工作 pending 为空，仍靠 porcelain 判"有变更"兜一条 files-only 降级 Step；此时用一个 `.last-fallback` 时间戳去重。）

## 6. 改动面清单（定案后）

- 新 `hooks/lib/fallback_step.py`（或 step_log 内）：`append_fallback_step(source, root)` + drain pending + porcelain。
- `hooks/lib/status_schema.py`：+`auto-fallback` 枚举。
- `hooks/session-end-check.py`：WARN 之外，调 `append_fallback_step("session-end")`（保留 WARN 作人读信号，或降级为"已兜底"提示）。
- `hooks/pre-compact-check.py`：checkpoint 之外，调 `append_fallback_step("pre-compact")`（复用现有未落盘判定 + porcelain）。
- `hooks/session-start-brief.py`：检测 auto-fallback Step → P0 升格提示。
- `hooks/lib/step_completeness.py` / `daily_render` / `distill.py`（collect + 三筛）/ `step_dualread.py`：认 auto-fallback（隔离/过滤/标注）。
- 测试：helper 组装、drain 去重、双关口不重复兜底、蒸馏过滤、completeness 不误判、升格 supersedes。

## 7. 风险与权衡

- **降级 Step 质量低**：升格前只有骨架。缓解：`status` 隔离 + 蒸馏过滤 + commit subject 让 title 半可读。
- **升格仍需 LLM**：若长期没人升格，降级 Step 低质留存——但**至少不丢**（骨架 + transcript_path 让事后升格容易），严格优于现状 WARN（没人补则语义+骨架全丢）。
- **改动面广**（status 新档触达 5+ 消费方）：靠共享 helper + 单一 status 枚举收敛。
- **不推荐的替代**：SessionEnd spawn headless LLM 直记规范 Step —— token 约束（[skill-creator-run-eval 教训](../../../.kdev/memory/踩坑日志.md)：CC subscription token 不能喂 headless run_loop）+ SessionEnd 后进程生命周期不确定，脆弱，仅作探索项。
