## TL;DR

**总体结论**：python-security-coding skill v0.1.0 **整体可用、召回率优秀（96.9%）、无硬误报**。SKILL.md 关键词映射表 + 8 个 reference 文件结构合理，子代理在显式调用时能稳定挖出预埋漏洞，自主发现路径在父级会话/有"安全审查"上下文时也能触发。**主要短板在 reference 内容深度上：四类常见但具体的反模式未明确收录**，导致即便走完 8 项清单仍可能漏过；另 §3.2.5 弱哈希规则因缺"非密码学场景例外"在 md5-checksum 类用法上有误报风险。

**最严重问题 Top 3**：

1. **(P0) `references/02-security-features.md` §3.2.5 加密哈希算法 缺"非密码学场景例外"** —— md5/sha1 用作完整性校验、缓存键、非对抗性指纹时是合规标准做法，但当前规则文字"禁止使用弱加密哈希算法（如 MD2、MD4、MD5、RIPEMD-160、SHA-1）"无任何条件限定，对 d04-md5-checksum.py 这类用法会产生误报。
2. **(P1) 4 类常见反模式未在任何 reference 内单独成条** ——
   - 用户枚举（reset_start 区分 200/404 暴露注册邮箱）
   - 重放攻击（webhook 无 timestamp / nonce 校验）
   - 速率限制（暴力请求 / 重置 / 登录）
   - 媒体处理特有风险（PIL decompression bomb / EXIF GPS 泄露 / SVG XSS）
3. **(P1) `references/06-error-handling.md` "关键操作审计日志"未独立成条** —— 现有 4 条规则集中在异常处理；但 B6 fixture 的"关键金融操作（DELETE / UPDATE balance）无任何 logger 调用"未被子代理列为独立 finding（仅在修复建议里隐式提到）。

## A 段 — 触发可靠性

### 评测方法修订

设计阶段假设"子代理通过 description 字段自动触发 skill"。Task 1 探针发现：**fresh general-purpose subagent 不在 system prompt 内看到本地 plugin skill 列表**（自省返回 `NO_SKILLS_VISIBLE`），但 Skill 工具仍可显式调用（`INVOKE_OK`）。

A 段因此 reframe 为：**"给 subagent 一个 implicit security-review 提示，看它是否会自主想到并 invoke skill"**。Prompt 模板：

```
You are a security reviewer. Below is a Python file. Review it thoroughly
for any concerns. You may use any tools or skills you have access to that
might help. Report findings.
<贴 fixture>
```

### 结果

| Fixture | 自动 invoke skill? | Reference 命中合理? | 报告了真实漏洞 / 误报? | 备注 |
|---|---|---|---|---|
| `a01-vulnerable.py` | ✓ | ✓ 01/02/03/06/08 | 全部预埋（SQL 拼接 ×2 + eval）+ 多个附加（auth/headers/error）；无误报 | 完美 |
| `a02-clean.py` | ✓ | ✓ 01/02/03/06/08 | 正确把参数化查询识别为安全；无硬误报；建议偏激进（多个架构加固项） | 通过 |
| `a03-borderline.py` | ✓ | ✓ 01/02/03/04/06/08 | **正确**把 `yaml.safe_load` 与 `subprocess shell=False+allowlist` 列入 "What's done correctly"；H3 "should use stdlib instead of subprocess" 是 §3.1.1 规则文字的合规提示，**非误报** | 完美 |

**A 段汇总**：3/3 自主发现 skill；3/3 reference 选择合理；0 个硬误报。**Skill 可发现性 = 优秀**。

### 启示

- description 字段对 fresh subagent 不直接生效。但只要 prompt 包含"security review"语义提示，subagent 自己会通过 Skill 工具列表 / 关键词联想找到 `python-security-coding`。
- 这说明 description trigger 的真实消费者是**父级 Claude 会话**（系统提示有 skill 列表 + description）。subagent 场景下需依赖"语义上下文 + skill 工具"双轨。

### 唯一漏报详情（b06-error.py）

预埋项 #4 "**关键操作缺审计日志**"（账户 delete / 余额 update 等敏感动作无 `app.logger.*`）。subagent 报告了 traceback 暴露 ×2 + 异常吞噬 + 多个附加发现，但未把"缺日志"列为独立 finding。修复建议中提到了 `logging.exception(...)` 但属于异常路径，**不等价于业务关键操作的审计日志**。

根因：`references/06-error-handling.md` 第 8 行规则"应记录抛出的异常"措辞偏向"异常发生时记录"，未明确"关键写操作 / 删除 / 权限变更等成功路径也需记录审计日志"。

### 附加发现（共性）

8/8 子代理都给出了若干**预埋外的合理发现**（auth、headers、conn 泄漏、CSRF、HSTS、CORS 收紧、版本上报等）。这些不是误报、不计入分母，但说明 skill 引导下的审查偏"全面/激进"风格 —— 对生产代码是好事，对评分本身是噪音。

### B 段已知噪声

## D 段 — 对抗 / 误报

主 session inline 模拟（按 SKILL.md 关键词映射 → Read 对应 reference → 按规则文字判定）。

| Fixture | 触发关键词 | Read 的 reference | 实际判定 | 结论 |
|---|---|---|---|---|
| `d01-subprocess-safe.py` | `subprocess` `shell=` | 01 §3.1.1 命令操作 | 规则核心是"避免拼接外部数据，并对执行的命令进行白名单限制"。代码用参数列表 + 服务端 dict 白名单，不触发"漏洞"判定；但 §3.1.1 第一条"避免直接调用系统命令…推荐使用系统 API 操作"会促使 subagent 建议"用 stdlib 替代"（A3 实测确认） | **不误报**（合理边界提醒） |
| `d02-yaml-safe-load.py` | `yaml.load`（子串匹配）| 01 §3.1.2 反序列化 | 适用场景明确写 `yaml.load(..., Loader=yaml.Loader)`，**显式排除 `safe_load`** | **不误报** |
| `d03-random-non-crypto.py` | `random.choice` | 02 §3.2.6 随机数 | 适用场景明确写"生成会话 ID / token / 一次性密码 / Salt / IV / **非业务展示性**的随机数"。NPC 选择属于业务展示，**显式排除** | **不误报** |
| `d04-md5-checksum.py` | `hashlib` `md5` | 02 §3.2.5 加密哈希 | 规则文字"禁止使用弱加密哈希算法（如 MD2、MD4、MD5、RIPEMD-160、SHA-1）"**无任何条件限定**；适用场景写"`hashlib.md5/sha1/...` / `hashlib.pbkdf2_hmac` / 自实现 KDF；Salt 生成；密码哈希存储"。完整性校验场景**未被排除**，按规则字面会触发警告 | **可能误报** |

**D 段汇总**：3/4 不误报，1/4（md5-checksum）依规则字面会被错误标记为漏洞。

### d04 误报根因

`references/02-security-features.md` §3.2.5 第 142 行：
> 禁止使用弱加密哈希算法（如 MD2、MD4、MD5、RIPEMD-160、SHA-1）。

对比 §3.2.6 随机数（第 198 行）：
> 适用场景：生成会话 ID / token / 一次性密码 / Salt / IV / **非业务展示性**的随机数。

§3.2.6 显式区分"密码学场景 vs 业务展示场景"，§3.2.5 没有。**修法**：在 §3.2.5 适用场景或规则后追加 "非密码学场景（完整性校验、非对抗性指纹、缓存键、bloom filter 等）使用 md5/sha1 不在禁用范围"。

## 改进建议（按严重度排序）

### P0 — 影响正确性，应近期修复

**P0-1** | `references/02-security-features.md` §3.2.5 第 142 行规则补"非密码学场景例外"
- 现状：`禁止使用弱加密哈希算法（如 MD2、MD4、MD5、RIPEMD-160、SHA-1）。`
- 建议：在规则后或适用场景内追加：`非密码学场景（文件完整性校验、非对抗性指纹、缓存键、bloom filter 等）使用 md5/sha1 不在禁用范围；但任何安全敏感用途（密码存储、签名、token 生成、抗碰撞）必须使用 sha256+ 或专用 KDF。`
- 影响：避免 d04-md5-checksum.py 类用法被误报；让 skill 能区分"算法弱 vs 用途弱"

### P1 — 覆盖盲点，应中期补齐

**P1-1** | `references/02-security-features.md` §3.2.1 访问控制 加"用户枚举攻击"条款
- 现状：§3.2.1 仅讲数据库访问控制 + 文件权限
- 建议：新增子项"账户存在性枚举防护：**密码重置 / 注册 / 登录** 等接口在用户存在与不存在时应返回**结构相同**的响应，避免通过状态码 / 响应字段差异暴露账户是否注册。"
- 影响：补 C1 reset_start 盲点

**P1-2** | 新增 reference `09-protocol-safety.md` 或扩展 §3.4 加"webhook 协议安全"
- 内容：HMAC 签名 (sha256+) / 时序安全比较 (`hmac.compare_digest`) / **timestamp 防重放** / **nonce 幂等** / **rate limit**
- 影响：补 C3 三个盲点；补 OAuth callback / SSE 等同类协议

**P1-3** | 新增 reference `10-media-safety.md` 或扩展 §3.4 加"媒体处理安全"
- 内容：PIL `MAX_IMAGE_PIXELS` 强制 raise / `Image.verify()` 早期解码校验 / EXIF 元数据移除 / SVG 不可作为头像格式 / polyglot 检测
- 影响：补 C2 三个盲点；图片/视频处理是常见的攻击面

**P1-4** | `references/06-error-handling.md` 加"关键操作审计日志"作为独立条款
- 现状：4 条规则全部围绕异常处理；"应记录抛出的异常"被理解为异常路径日志
- 建议：新增第 5 条"**关键写操作（删除 / 权限变更 / 资金 / 认证状态变更）的成功路径必须记录审计日志**，便于事后追溯。"
- 影响：解决 b06-error.py 唯一漏报；这是 SOX/GDPR 等合规也会关注的项

### P2 — 锦上添花

**P2-1** | SKILL.md 关键词映射表加 `secrets.token_*` / `hmac.compare_digest` 作为正例标记
- 当前关键词只触发 reference Read，没区分"出现这个关键词 = 警告" vs "出现这个关键词 = 安全实践"。可在映射表里给"正例关键词"打 ✓ 标，让 subagent 在看到这些时不警告。

**P2-2** | SKILL.md 完成清单加附加项
- 在 8 项 gate 后追加 2 项可选项："9. 速率限制 / 输入大小限制" 和 "10. 审计日志覆盖关键写操作"
- 这两项是 C1/C2/C3 共同盲点

**P2-3** | `references/01-input-validation.md` §3.1.1 命令安全的"推荐 stdlib 替代"语气可放缓
- 现状：A3 实测中 subagent 把"使用 stdlib 替代 subprocess"作为高严重度 finding 报出。语气可改为"如可行，优先使用 stdlib API；当确需调用外部命令时…白名单 + 参数列表 + shell=False"。
- 影响：减少 d01-subprocess-safe.py 类正确用法被推到"高严重度"
