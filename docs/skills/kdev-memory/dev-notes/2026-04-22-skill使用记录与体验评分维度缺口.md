# kdev-memory Step schema 缺口：skill 使用记录 + 体验评分维度

**提出日期**：2026-04-22
**提出语境**：token-statistics 项目 Step 11（.kdev/ 文档清理）评分时用户反馈暴露
**相关 skill**：kdev-memory（v0.4.0 及以后）
**优先级建议**：P1（不阻塞当前使用，但未覆盖时用户侧就会出现"评分维度塞不下"的信号，本次已出现一次）

---

## 背景事件

2026-04-22 token-statistics 项目 Step 11 完成后，用户采集评分时反馈：

> "评分 4.5，感受比较流畅。**不过这个过程好像没用到任何 skill，感受不明显**"

这句话的后半句说明：kdev-memory 当前的 Step schema 把"skill 使用体验"塞到了"一句话评价"的自由文本字段里，用户不得不夹带着写，不够顺手。

---

## 现状盘点（基于 token-statistics 项目）

### 1. Schema 层（CLAUDE.md v0.4.0 模板）

Step 标题模板：

```
## Step N: <工作单元名 — 触发 skill 或事件>
```

"触发 skill 或事件"只是**标题级**提示语（可写可不写），不是**独立结构化字段**。

"执行事实"段记录的客观字段：

```markdown
### 执行事实（模型客观陈列）
- 工具调用次数：约 <量级>
- 报错 / 命令失败次数：<精确整数>
- 绕路次数：<精确整数>
- 本步 token 消耗感：<低/中/高/偏高>
```

**无 "使用的 skill" 字段**。

"用户评分"段：

```markdown
### 用户评分
- 完成时间：YYYY-MM-DD HH:MM
- 顺畅度：N/5
- 用户评价：<原话或概述>
- 🚨 关键 GAP：<如有>
```

**维度只有"顺畅度"一个**，无"skill 体验感"。

### 2. 实际记录现状

| 时期 | skill 记录方式 |
|------|--------|
| Sprint 0 + iter 1~3（Step 0~15）| 详细，每步"执行"段都显式写"调用 XXX skill"——当时项目主题就是做 skill 实战评估，skill 信息是数据采集重点 |
| iter 4+（Step 1~11 及 M-5~M-8 meta 回补）| **散落在自由文本里**，有时提（如 Step M-7 提到 subagent-driven + gstack-qa），有时完全不提（Step 11 用户反馈"没用到 skill"即是信号） |

结论：**skill 使用是隐含在自由文本里，不是强制结构化字段**。一旦 Step 是"纯对话驱动 / 账本维护 / 目录整理"这类天然无 skill 覆盖的任务，评分就变成"顺畅度"单维度打分，无法分拆"skill 贡献 vs 主控能力 vs 任务本身"。

### 3. 问题暴露点

Step 11 评价"没用到 skill 感受不明显"被用户塞进"一句话评价"字段。这信号本质是**评分维度缺失**，不是对执行质量的批评。当前 schema 把两件事挤在同一个字段里：

- "整体节奏顺畅度"（1-5）
- "skill 具体对本任务的贡献是否到位"（0-5 或 N/A）

schema 缺第二个维度的容器，用户只能自己手写信号，等着每日汇总 / R-NNN 流程去捞。

## 相关历史信号（不止本次）

翻执行日志时发现同类信号在 iter 3 起就零散出现：

- **Step 13（迭代 3）**：用户评价 "frontend-design 的 UI 原型设计顺利；**bmad-create-architecture 感受不强**"——对特定 skill 的体验评价，塞在自由文本里
- **Step 14（Sprint 0 扫尾）**：提到 "bmad-create-architecture 感受不强的可能原因"分析——本应是"skill 体验感 2/5 + 说明"的结构化数据，实际是自由叙事段
- **Step M-7（迭代 7 灵码集成 meta 回补）**：用户"跳流程"——整个迭代 skill 覆盖率极低——也是在事后才被感知，没有字段当场捕获

如果 schema 有"skill 体验感 N/5"维度 + "N/A（未触发 skill）"选项，这些信号一出现就能结构化捕获并在每日汇总 / R-NNN 阈值判定时直接参与决策，不用等模型主控事后挖。

## 改进候选

### 方案 A（轻，只改事实层）

`### 执行事实` 段新增一行：

```markdown
- 使用的 skill：<列表 | 无 | N/A（对话驱动）>
```

**优点**：零评分维度改动；只补齐客观事实。
**缺点**：不解决"评分维度单一"。skill 体验话题仍塞在一句话评价里。

### 方案 B（中，补评分维度）

`### 用户评分` 段新增一行：

```markdown
- skill 体验感：N/5 | N/A（本 Step 未触发 skill）
```

顺畅度（1-5）答"我对整体节奏的感受"；skill 体验感（1-5）答"skill 具体对本任务的贡献是否到位"。N/A 时不参与统计，避免"没法打分"的尴尬。

模型自评段同步补该维度。评分差异分析规则同步扩：

- 顺畅度差值 ≥ 2 → 原规则
- skill 体验感差值 ≥ 2 → 新规则（升 R-NNN，专门记"skill 设计 vs 使用体感偏离"）
- 顺畅度 OK 但 skill 体验感 ≤ 2/5 → 专门信号（skill 对该 Step 无贡献或负贡献）

**优点**：直击本次盲区；评分矩阵变二维，每日汇总和 R-NNN 阈值分析更精准；skill 质量观测通道跟 Step 评分流程直连。
**缺点**：采集成本 +1 项；模型自评段也要对应加维度；老 Step 历史数据不强求补，统计口径要处理"有维度前 / 后"切分。

### 方案 C（方案 A + B 合并）

事实层 + 主观层双加。最完整。

### 方案 D（零改 schema，自律约定）

CLAUDE.md 加一句"评价里遇到 skill 体验话题时写 `skill=<内容>`"，不改字段只改写作约定。**轻但易漂移**，本质是"把坑继续留在用户心智里"。

## 倾向

> **2026-04-22 session 决策**（用户反馈）：**方案 A（事实层加 skill 名单 + 评分
> 维持单一）**，**不走方案 B 的分两个评分**。理由：
>
> - 用户对"skill 体验"的评价本身就在用同一把尺（"这 skill 帮到我没"）——
>   和顺畅度是同一种判断，分两个数字是过度设计
> - 两个数让用户每步做两次判断，采集负担 +1，ROI 不划算
> - 单一顺畅度配合事实层的 skill 名单：模型读日志时能看到"这步用了
>   X/Y/Z skill，顺畅度 N/5"，skill 的正/负贡献自然融进顺畅度里
> - 分析 skill 质量时靠"跨 Step 聚合+skill 名单+顺畅度"统计即可，不需要
>   独立维度
>
> 所以最终采纳方案 A 及其微扩展（见下）。

原分析保留作历史对照（方案 B 的 pros 是"结构化捕获 skill 信号"，但付出的
采集成本 > 收益——用户判断）：

1. 本次反馈本质是评分维度缺失 ❌（修正：本质是事实层缺 skill 追踪，评分
   层不缺维度）
2. skill 体验感 1-5 + N/A 可让纯对话驱动 Step 显式标 N/A ❌（修正：只要
   事实层 skill 列表为空，"无 skill"信号已清晰表达，不需要 N/A 占位）
3. 同类 skill 体验信号结构化捕获 ✅（保留：但靠事实层就够）
4. 方法论级调整 ⚠️（保留：但范围缩小为只改事实段）

### 方案 A 的最终形态（session 决策版）

`### 执行事实` 段新增**一行**：

```markdown
- 使用的 skill：[bmad-create-architecture, subagent-driven]  | 无 | N/A（对话驱动）
```

用户评分段**保持原样**（只有顺畅度 1-5 + 一句话评价 + 🚨 关键 GAP），**不**
加 skill 体验感维度。

### 分析路径

跨 Step 统计时按 skill 名单聚合：

```
skill=X 的所有 Step → 平均顺畅度 N1/5
skill=Y 的所有 Step → 平均顺畅度 N2/5
无 skill 的所有 Step → 平均顺畅度 N3/5
```

跨几个 skill 对比 → 找低分 skill 的共性问题。不需要独立 skill 体验维度。
这个分析可以做成 kdev-memory 的新 skill 召唤词（如"分析 skill 体感 / skill
质量盘点"），不侵入 Step schema。

## 潜在顾虑

1. **评分认知负担**：从"一个数字"到"两个数字 + 可选 N/A"——用户是否嫌繁？可通过"N/A 默认值 + 一键 skip"交互缓解，但主控采分脚本暂无 UI，完全靠口语交互时"skill 体验感几分？"会是多一轮问答
2. **老 Step 历史数据**：v0.4.0 之前的 Step 没这维度。能否当场补？基本不能——skill 体验感是主观感受，褪色后比顺畅度还不可靠。只能统计分两段
3. **"N/A"滥用风险**：如果 Step 明明用了 subagent-driven 但用户图省事打 N/A，就漏掉了有效信号。SKILL.md 应明确"触发过任一 skill 即不允许 N/A"作为自律硬规
