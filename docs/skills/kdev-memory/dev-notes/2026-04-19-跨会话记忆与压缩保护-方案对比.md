# 跨会话记忆与压缩保护 —— 方案对比

> 日期：2026-04-19
> 背景：kdev-memory@0.2.0 已补齐 Stop/PostToolUse/SessionEnd 三层 hook。
> 用户进一步提出两个需求：
> 1. **压缩前自动记录**，避免压缩后丢失关键信号
> 2. **跨会话时把之前的记忆带到新会话**，或至少提醒 Claude 去读
>
> 本文先横向调研 6 家主流记忆框架的做法，再给我们插件三档候选方案，最后给出推荐。

---

## 一、问题定义

### 1.1 需求 A：压缩前保护

Claude Code 超过 token 阈值会**自动压缩**会话（或用户 `/compact` 手动触发），压缩后的会话只保留摘要 + 近期消息，早期细节就丢了。

对 `.kdev/` 工程记忆而言：
- 如果本会话的 Step/决策/踩坑尚未**实时落盘**到 `.kdev/` 文件，压缩后就永远没了——Claude 再也回忆不起来那些细节
- 目前 Stop hook 在"用户自然 idle"时软提醒都触达不到，压缩场景下更是没机会

**关键缺口**：需要在压缩**之前**触发一个写盘动作。

### 1.2 需求 B：跨会话记忆注入

新开一个会话：
- Claude 不知道 `.kdev/` 存在（除非 CLAUDE.md 有触发规则段明确提到）
- 即便知道存在，也不会自动去读——除非用户第一句话就说"先看下昨天做到哪了"
- 实际场景：用户开新会话往往直接扔任务（"帮我修这个 bug"），Claude 不会绕路去补课

**关键缺口**：新会话启动时应该自动把 `.kdev/` 的现状摘要推给 Claude，让它"开局就知道有记忆"。

### 1.3 Claude Code 原生 Hook 能力

两个需求都能靠原生 hook 解决：

| Hook | 触发时机 | 能拿到的输入 | 输出效果 |
|---|---|---|---|
| `PreCompact` | 压缩**之前**触发，Claude 仍有完整上下文 | `trigger`（auto/manual）、manual 时的 `custom_instructions` | stdout 注入到下一轮 / exit 2 阻塞 / 脚本自己写文件 |
| `SessionStart` | 新会话启动时触发 | `source`（startup / resume / clear / compact） | stdout 被 Claude 作为初始上下文读到；JSON 输出 `hookSpecificOutput.additionalContext` 可结构化注入 |

两者都走**文件标准输出**或**JSON 结构化输出**两种路子。下面的对比就围绕各家怎么用这两个能力展开。

## 二、六家记忆框架对比

### 2.1 横向总览

| 方案 | Stars | PreCompact 做法 | SessionStart 做法 | 存储形态 |
|---|---|---|---|---|
| **claude-mem** | 45k | 无专门 PreCompact；PostToolUse 持续采集 + Stop 生成摘要 + SessionEnd 持久化 | 从 SQLite 加载上下文，Progressive Disclosure 分层 | **SQLite + FTS5 + Chroma 向量库** |
| **claude-memory-compiler** | 125 | PreCompact/SessionEnd → `flush.py` 调用 Claude Agent SDK 提取"有价值信息"写 `daily/YYYY-MM-DD.md` | 注入编译后的 `index.md`（无 RAG，靠索引） | 纯 Markdown 目录 |
| **claude-reflect** | 881 | 无专门 PreCompact；PostToolUse + regex 捕获纠正 | `session_start` 注入纠正历史 | CLAUDE.md 同步写入 |
| **OMC (oh-my-claudecode)** | 25k | PreCompact checkpoint 防丢失 | 加载 `state/ + notepad.md + project-memory.json` | 三层 notepad + 6 类 JSON 状态 |
| **ECC (everything-claude-code)** | 143k | 无专门 PreCompact；Continuous Learning 自动提取模式 | SessionStart 从 SQLite State Store 加载 | SQLite State Store |
| **KDev 自有设计**（仓库 `docs/01-design/2026-04-08-02`） | — | **PreCompact 写 `.kdev/checkpoints/checkpoint-TS.json`**（questions-log + gotchas + session-state 三份全量快照） | **`hookSpecificOutput.additionalContext` 注入** global-prefs + corrections + questions + gotchas + session-state，各段 `<kdev-memory-*>` 标签包裹 | 纯 Markdown + JSON，无数据库 |

### 2.2 PreCompact 策略分两派

**派别 1：不做专门 PreCompact**（claude-mem、claude-reflect、ECC）
- 依赖"持续采集"：PostToolUse 每次都捕获一点，累积起来
- 优点：不用专门处理压缩事件
- 缺点：采集是"事中"的，压缩前最后一刻的新信息仍可能丢

**派别 2：做 PreCompact 写盘**（claude-memory-compiler、OMC、KDev 自有设计）
- 压缩事件到来时直接把关键状态 dump 到文件
- 优点：兜底强，压缩后仍能从文件回读
- 缺点：要额外脚本，要处理 timestamp 命名

我们的 `.kdev/` **天然是"文件落盘"范式**，走派别 2 一致性更好。

### 2.3 SessionStart 策略分两档

**极简档**（claude-reflect）：只注入纠正历史到 CLAUDE.md，不做额外摘要
**全量档**（OMC、KDev 自有设计）：结构化注入多段（preferences / corrections / questions / gotchas / state），用 `<tag>` 分类

各家的一致做法是**用 `hookSpecificOutput.additionalContext` 结构化注入**，而不是裸 stdout。

### 2.4 我们不走的几条路

| 路线 | 代表 | 为什么不走 |
|---|---|---|
| SQLite + 向量库 | claude-mem | 核心定位背离（我们是"文件落盘 + 人可读"） |
| 调用 LLM 做压缩摘要 | claude-memory-compiler | 引入 API key、成本、延迟；跟 `.kdev/` 实时落盘定位冲突 |
| 三层 notepad + 6 类 JSON 状态 | OMC | 碎片化、记忆文件膨胀；我们已有 `.kdev/` 自洽结构 |
| regex 捕获纠正 | claude-reflect | 属于独立的 correction-detector 能力，单独插件更合适 |
| Continuous Learning 自动提取 | ECC | 需要 LLM 二次调用；不如让下游 skill 作者复盘时提炼 |

## 三、Claude Code Hook 技术要点

### 3.1 `additionalContext` 注入机制

SessionStart hook 的标准输出格式：

```json
{
  "continue": true,
  "hookSpecificOutput": {
    "hookEventName": "SessionStart",
    "additionalContext": "<kdev-memory-brief>\n...摘要内容...\n</kdev-memory-brief>"
  },
  "suppressOutput": true
}
```

- `additionalContext` 会被 Claude 当作初始上下文读到
- `<tag>` 包裹有助于 Claude 结构化理解
- `suppressOutput: true` 避免 stdout 的调试信息污染终端

### 3.2 stdin hang 问题（OMC issue #240）

hook 脚本从 stdin 读 JSON 时，如果上游没有关闭 stdin，会永久 hang 住。所有引用该 issue 的设计都加了 **5 秒超时**。

我们当前 Stop hook 的 `[ ! -t 0 ] && INPUT=$(cat)` 写法已经**隐含了这个风险**（当 stdin 是管道但对方不 EOF 时 `cat` 会 hang）。**0.3.0 时一起修**。

### 3.3 Exit 码语义

| Exit code | Stop / SessionStart / PreCompact | PostToolUse |
|---|---|---|
| `0` + stdout | 软提醒（注入上下文） | 软提醒 |
| `2` + stderr | Stop 阻塞 / SessionStart 阻止加载 / PreCompact 阻止压缩 | 拒绝工具调用 |
| 其他非零 | 普通错误（软失败） | 普通错误 |

**PreCompact 用 exit 2 阻塞压缩** 是个危险选项——会导致用户想压缩时压不掉。**不推荐**，应该只用 exit 0 + 写盘。

### 3.4 SessionStart 的 `source` 区分

- `source: "startup"`：全新会话，需要完整摘要
- `source: "resume"`：`claude --continue` 恢复，Claude 其实已有上下文，再注入全量是冗余
- `source: "clear"`：用户 `/clear` 后重开，类似 startup
- `source: "compact"`：刚压缩完重启，应当提示"checkpoint 在哪"而不是重复所有内容

**精细化**：不同 source 注入不同粒度，可以省 token。

## 四、对我们插件的三档候选方案

### 方案 A：极简档

**PreCompact**：只在"执行日志今天空 + 工作区有变更"时，写一个 `.kdev/WARN-压缩前-YYYY-MM-DD-HHMMSS.md`（和现有 SessionEnd WARN 机制同形态，只是文件名前缀不同）。

**SessionStart**：注入 3-5 行摘要：
```
<kdev-memory-brief>
项目有 .kdev/ 工程记忆。
- 有无 WARN-*.md：有 2 个（列路径）
- 今日执行日志：空
- 最近 Step：Step 23（2026-04-17）
</kdev-memory-brief>
```

- ✅ 实现量最小（~30 行脚本）
- ✅ 复用 WARN 机制，概念最少
- ❌ 压缩场景下只写 WARN，当天已有条目的会话**不额外备份**——压缩后如果 `.kdev/执行日志.md` 之后被误覆盖，没 checkpoint 可回
- ❌ SessionStart 摘要太简，不足以让 Claude 形成"项目全景"

### 方案 B：中等档（推荐）

**PreCompact**：双重保险
1. **总是写 checkpoint**：把今天 `.kdev/执行日志.md`、`.kdev/当前状态.md` 的当前内容 + 工作区 `git status` 快照拼成 `.kdev/checkpoints/压缩前-YYYY-MM-DD-HHMMSS.md`（纯文本复制，不做 JSON dump）
2. **额外检查**：如果执行日志今天空 + 工作区有变更，checkpoint 里加一个显眼的 "⚠️ 未落盘" 区块

**SessionStart**：根据 `source` 注入不同粒度
- `startup` / `clear`：8-12 行摘要（见下示例）
- `resume`：2-3 行（Claude 已有上下文，只提"有无待处理 WARN"）
- `compact`：3-5 行（提醒 checkpoint 在哪）

```
<kdev-memory-brief>
项目有 .kdev/ 工程记忆。当前状态（2026-04-19）：

⚠️ 待处理：
- .kdev/WARN-未记录-2026-04-17.md（上次会话兜底警告）
- .kdev/checkpoints/压缩前-2026-04-17-143012.md（上次压缩快照）

今日进度：
- 执行日志：空
- 每日汇总：未生成
- 最近 Step：Step 23（"实现采集器核心循环"，2026-04-17）

待解决决策 / 踩坑：
- Q-008（采集频率定为 5 分钟，待验证）
- G-012（pnpm workspace 依赖）

建议：如需细节，Read .kdev/当前状态.md 和 .kdev/每日汇总/2026-04-17.md
</kdev-memory-brief>
```

- ✅ 压缩 checkpoint 是"原件复制"，压缩后可完整回读
- ✅ SessionStart 让 Claude 一开局就知道项目全景，不用等用户提
- ✅ source 区分避免 resume 时重复注入
- ⚠️ 脚本量中等（~120 行）；需要处理中文文件名、timestamp、source 分支
- ⚠️ `.kdev/checkpoints/` 会累积文件，要定期清理（加一个"保留 7 天"的 retention 逻辑）

### 方案 C：完整档（KDev 自有设计风格）

**PreCompact**：写 `.kdev/checkpoints/checkpoint-TS.json`，内含 questions-log / gotchas / session-state / 执行日志 / 踩坑日志 / 决策日志 / 每日汇总当天 / 当前状态 的**全量快照**（JSON 格式）

**SessionStart**：注入多段结构化上下文
```xml
<kdev-memory-global>
# 用户偏好 (全局)
...
</kdev-memory-global>

<kdev-memory-corrections>
# 纠正历史 (全局)
...
</kdev-memory-corrections>

<kdev-memory-questions>
# 待解决问题
...
</kdev-memory-questions>

<kdev-memory-gotchas>
# 踩坑记录
...
</kdev-memory-gotchas>

<kdev-session-restore>
# 会话状态恢复
当前任务: xxx
阶段: yyy
</kdev-session-restore>

<kdev-daily-pending>
今日有未完成的工作项...
</kdev-daily-pending>
```

- ✅ 功能最完备；和 kdev-agent 自有设计文档对齐
- ✅ 全局 `~/.kdev/user-preferences.md` 层让跨项目偏好也能加载
- ❌ **相当重**：脚本 ~300 行、引入 `~/.kdev/` 全局目录概念、涉及 JSON 化 dump
- ❌ 和当前插件"文件落盘 + 人可读"定位冲突：JSON 快照人不好读
- ❌ SessionStart 注入 6 段 tag 容易污染上下文（每段都是"必读"信号）
- ❌ 需要额外处理"纠正捕获"才能填满 corrections 区块——否则该区块永远空

## 五、三档方案逐项评估

| 维度 | 方案 A 极简 | 方案 B 中等 | 方案 C 完整 |
|---|---|---|---|
| **实现量（行）** | ~30 | ~120 | ~300+ |
| **新增脚本** | pre-compact-check.sh + session-start-brief.sh | 同上 + checkpoint 写盘逻辑 + source 分支 | 同上 + 全局偏好加载 + JSON dump + 多段 tag 生成 |
| **新增目录/文件概念** | 0（复用 WARN） | 1（`.kdev/checkpoints/`） | 2（`.kdev/checkpoints/`、`~/.kdev/`） |
| **压缩后可回读** | ❌ 只在风险场景写 WARN | ✅ 总是写 checkpoint | ✅ 写 JSON + 可程序化解析 |
| **SessionStart 覆盖度** | 仅报最关键信号 | 含 Step/Q/G 编号和标题 | 含全局偏好 + 纠正历史 + 多段分类 |
| **source 精细化** | 无 | 三档（startup / resume / compact） | 三档 + 每档内不同 tag 组合 |
| **对 `.kdev/` 定位的契合度** | ✅ 零冲突 | ✅ checkpoint 文件也是可读 md | ⚠️ JSON 快照人不好读 |
| **维护成本** | 低 | 中 | 高 |
| **Token 注入压力** | ~50 tokens | ~200 tokens | ~500+ tokens |
| **是否需要纠正捕获配套** | 否 | 否 | 是（否则 corrections 段永远空） |

## 六、推荐：方案 B 中等档

### 6.1 理由

1. **和当前 0.2.0 的"四层防线"叙事自然衔接**：Stop → Strict → PostToolUse → SessionEnd → **+ PreCompact（压缩兜底）+ SessionStart（开局注入）= 六层防线**
2. **压缩场景硬需求被覆盖**：checkpoint 总是写盘，压缩后可完整回读
3. **Token 压力可控**：SessionStart 按 source 精细化后平均 ~200 tokens
4. **不引入 `~/.kdev/` 全局层**：保持单项目内聚，跨项目偏好等真需要了再单独做
5. **checkpoint 是纯 Markdown 复制**：和 `.kdev/` 其他文件形态一致，人可读，不需要工具解码

### 6.2 和 0.2.0 的关系

两种选择：

- **并入 0.2.0**（推荐）：6 层防线一次成型，叙事更完整。反正 0.2.0 还没提交
- **拆成 0.3.0**：如果觉得"先让 0.2.0 稳一阵再加新能力"更保守

**建议并入 0.2.0**。逻辑上四层→六层只差两个 hook，分两版发行的价值不高，反而增加用户理解负担。

### 6.3 实现清单（如果 go）

| 文件 | 变更 |
|---|---|
| `hooks/pre-compact-check.sh` | 新增，写 `.kdev/checkpoints/压缩前-TS.md` + 软提醒 |
| `hooks/session-start-brief.sh` | 新增，按 source 分档注入 `<kdev-memory-brief>` |
| `hooks/hooks.json` | 注册 `PreCompact` 和 `SessionStart` |
| `hooks/lib/checkpoint.sh` | 新增公共库，`.kdev/checkpoints/` retention（保留最近 7 天） |
| `hooks/stop-check.sh` / `session-end-check.sh` / `post-write-check.sh` | 顺手修 stdin hang 风险（加 `read -t 1` 超时或用 python3 加 timeout） |
| `skills/kdev-memory/SKILL.md` | 触发规则段加"看到 checkpoint 文件按日期判断是否仍相关" |
| `README.md` | Hook 章节"四层防线"→"六层防线" |
| `CHANGELOG.md` | 0.2.0 条目追加两段 |

## 七、遗留问题与未来路线

### 7.1 本次不做但值得记录的

| 能力 | 为什么不做 | 何时考虑 |
|---|---|---|
| 纠正捕获（regex 检测"no, use X"等） | 属于独立 concern，单独插件 `kdev-correction` 更合适 | v0.4+ 或单独插件 |
| 全局 `~/.kdev/` 用户偏好 | 跨项目需求信号弱，YAGNI | 有明确跨项目偏好场景时 |
| LLM 辅助压缩摘要（memory-compiler 路线） | 引入 API key / 成本 / 延迟；违背实时落盘定位 | 永不做，属下游 skill 作者工作 |
| `.kdev/` 语义搜索 | 过度工程 | 永不做，用户 grep 足够 |
| SQLite 后端 | 核心定位背离 | 永不做 |

### 7.2 checkpoint retention 策略

`.kdev/checkpoints/` 会随时间积累。建议默认保留策略：
- 保留最近 7 天的 checkpoint
- 超期的移动到 `.kdev/checkpoints/归档/YYYY-MM/`（按月归档）
- 或者直接删除——因为 checkpoint 本质是"压缩前的保险"，过了那一刻价值衰减

**倾向第二种：超期直接删**。如果用户想保存某个 checkpoint，自己 mv 出来或 rename 即可。

### 7.3 一个边缘问题

`PreCompact` hook 在 `auto` 触发时，Claude 不一定预期到它会被触发。此时 stdout 软提醒"已保存 checkpoint"是否有用？

- 理论上 Claude 在压缩**之前**会读到这个 stdout，一轮内的工作可能还在进行
- 实践上压缩往往在"轮次之间"，stdout 提醒和 Stop hook 一样面临"会不会被下轮读到"的问题

**结论**：PreCompact 的 stdout 提醒是 bonus，主要价值在"写盘"，不指望 Claude 当场看到提醒。
