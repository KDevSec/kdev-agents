# 2026-04-24 Step 粒度：从「phase = Step」硬规则到「自然停顿点」三信号的认知演进

**提出日期**：2026-04-24
**提出语境**：kdev-memory v0.6.0 scope 对话中，同一 session 内连续两轮 user 修正推翻 AI 的初版粒度方案
**相关 skill**：kdev-memory（v0.6.0 及以后）
**方法论价值**：展示**"AI 提议结构化定义 → user 反问行为化定义"**这组对立的完整演化过程，作为未来类似场景的推理模板

---

## 背景

v0.6.0 scope 讨论"事实层加「使用的 skill」字段"时，用户顺带抛出一个新问题：

> "当 CC 列了很多 Todos 时，是否应该记录下来这些 todos 和简要的任务描述，否则后面可能会遇到压缩就没了，或者中间打断了？"

AI 的初步回应把这当成"TodoWrite 数据持久化"问题来答（→ 详见 `2026-04-24-todowrite跨会话接力缺口.md`）。但用户接下来连抛两轮澄清，让 AI 逐步发现真正的问题不是"todos 要不要存"，而是**"Step 边界在哪"这个更底层的粒度定义**。

---

## 三轮认知演进的完整轨迹

### Round 0：AI 初版——把 TodoWrite 当独立问题

AI 给出三档方案：不动 / body 约定同步 / hook 自动 dump。推荐方案 ②（约定级）。

### Round 1：用户澄清 1——todos 不是独立单位，phase = Step

用户原话：

> "有个大的需求，拆分了 20 个 todo，分为 4 个 phase 来完成。这时候可能每个 phase 停下来打个分比较合适，不适合每个 todo 停下来，也不是所有 phase 完成了再停下来。"

AI 认知更新：

- TodoWrite 不是独立持久化问题，是 **Step 粒度定义**问题
- 层级关系：大需求（20 todos）→ 4 phases → 4 Steps；todos 是 Step 的子任务分解
- Step 完成闸门触发 = phase 完成时，不是每个 todo 完成时

AI 新方案：**"phase = Step"硬规则 + todos 跟随记**。

### Round 2：用户澄清 2——不是机械映射，看运行时间和人干预

用户原话：

> "但是有的时候，可能两个 phase 合并到一起评分，这个不是特别的硬性规定，主要是看运行的时间和中间是否需要人的干预。"

AI 第二次认知更新：

- "phase = Step"是**结构化定义**（按需求分解拓扑）
- 真正的依据是**行为定义**（按协作节奏）：时长 + 人干预需求
- 两个短 phase + 中间无干预 → 合成 1 Step；一个长 phase + 中间有决策点 → 切成 2 Steps

AI 重写方案：**三个停顿信号**（时长 / 干预 / 验收），三种 phase 映射合法，三种反模式。

### Round 3：用户校准量级 + 行数警告

用户原话：

> "量级是 30min-1h 吧，另外会增加多少 SKILL.md 的行数，咱们现在行数已经不少了"

AI 第三次认知更新：

- 时长阈值：AI 猜的 10min → 真实 30min-1h（3x 偏差）
- 行数警告：**SKILL.md 主文每加一行都是每次触发加载成本**，不能又把它加肥
- 方案重构：细则全放 references（按需 Read），SKILL.md 主文只留 anchor

最终落地 SKILL.md 主文净 +1 行（287 → 286），references +44 行。

## 形成的最终指引（已落 references/六类记录-schema.md §3）

**Step 粒度 = 自然停顿点**

三个停顿信号（任一命中 → 触发 Step 完成闸门）：

1. **时长信号**：模型连续独立执行较长（量级约 **30min–1h** / 工具调用约 **30 次以上**）
2. **干预信号**：中间需要用户决策（升 Q-NNN）、拍板取舍、确认产出物
3. **验收信号**：有可 review 的产出（代码 / 文档 / eval 结果）且用户会对它有感受

和 phase / todo 的映射（三种都合法）：

- 1 phase → 1 Step（默认：phase 本就是按直觉做出的自然停顿划分）
- 2 phase → 1 Step（两 phase 都短 + 中间无干预 → 合并打分）
- 1 phase → N Steps（phase 内部有明确决策点或验收点 → 切开）

反模式：

- 每 todo 一 Step → 评分疲劳
- 整需求一 Step → 感受褪色、信号失真
- 机械按 phase 切 → phase 可能只是思维组织，未必是停顿点

## 方法论信号提取

这次翻转提取出 4 条可复用的推理模板：

### 1. 结构信号 vs 行为信号的本能偏向

**AI 的偏向**：倾向用"结构信号"（phase 数、todo 数、commit 数）作为判断依据——因为它：

- **可数、可形式化**：方便写代码 / 写约束
- **有明确边界**：不用判断"是不是"，只要看"有没有"
- **看起来权威**：像工程决策而不是玄学

**真实依据往往是"行为信号"**：时长、干预、感受——因为它对应用户"啥时候想看一眼"的朴素诉求。

**应对**：AI 给出初版粒度定义时，先问"这个定义是按结构切还是按行为切？如果按结构，行为边界在哪？"

### 2. AI 硬规则 vs user 弹性判断的永恒对立

类似模式在方法论设计里会反复出现：

| AI 倾向 | user 修正 | 源场景 |
|---|---|---|
| "phase = Step"硬规则 | "有时两个 phase 合并" | v0.6.0 Step 粒度 |
| "+22 行 = 成本"静态分析 | "每次触发省 tokens 是长期收益" | iter-7 P0-1 闸门 |
| "阈值设 10min" | "量级是 30min-1h" | v0.6.0 时长信号 |
| "每次 todo 变都落盘" | "约定级就够，不要 hook" | v0.6.0 TodoWrite 方案 |

**根因**：AI 推理倾向**提前收敛到形式化规则**，user 倾向**保留弹性给真实场景**。AI 的规则化倾向本质是对训练数据里"有规则比无规则更专业"模式的过拟合。

**应对**：AI 给出方案时应主动标注"这是硬规则还是弹性判断"，并问 user "要不要弹性？弹性的边界在哪？"

### 3. 数字阈值不要凭感觉写

AI 猜"10 min"作为时长信号量级，user 校准到"30min-1h"——3x 偏差。这不是 AI 的单次失误，而是**AI 对真实工作时长的基准校准长期失真**（训练数据里写的多是"5 分钟"、"15 分钟"这类粗粒度，没真实工程时长的分布）。

**应对**：

- 量级数字优先让 user 给，AI 不要主动给出
- 如果必须 AI 给，明确标注"这是量级估算，误差可能 3x 起"
- 写进文档时措辞用"约 X / 参考值"而不是硬阈值

### 4. "是否要做"和"怎么做"要分开问

本次 session user 抛出 TodoWrite 问题时，AI 立刻跳到"怎么做"（三档方案）。但正确的第一问是"**是否要做**"——真实痛点频率多高？低频痛点不值得高频代价。

**应对**：AI 收到新需求时先问三件事：

1. 真实痛点频率（按天 / 周 / 月）
2. 现有机制是否已经部分覆盖
3. 最轻的可观察方案是什么

**然后再**进入方案横比。

| 维度 | iter-7 翻转 | 本次翻转 |
|---|---|---|
| 触发时机 | 跨 session（初版结论已 commit 后 user 反问） | 同一 session 内连续 3 轮 |
