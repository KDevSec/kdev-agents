## 问题定义

kdev-memory 的编号体系（Q-NNN / G-NNN / R-NNN / Step N）是基于**文件追加**的。多会话同项目同时推进会出现：

1. **场景 A**：一人同时开两个终端 / 两个 IDE 窗口，分别跑同项目的不同任务
2. **场景 B**：主会话 + 多个 subagent 并行，每个 subagent 都可能独立推进 Step
3. **场景 C**：团队协作（多人一项目）—— 本插件目前**不覆盖**这个场景

三种场景共同表现：

- 会话 A 算出"下一个 Step 是 20"
- 会话 B 也算出"下一个 Step 是 20"
- 两会话各自追加 → 出现 **双 Step 20**（或 git merge 冲突 / silent 覆盖）

Q/G/R 编号类似。

## 6 方案横比

| 方案 | 方案要点 | 代码改动 | 语义改动 | 优点 | 缺点 |
|---|---|---|---|---|---|
| **I. 硬分片** | Q-A023 / Q-B045（session 号前缀） | 大 | 大 | 零冲突 | 改现有编号语法；跨会话引用脑补负担；破坏 v0.5.x 编号契约 |
| **II. 中缀时间戳** | Step 20-14:35（加时分戳消除冲突） | 中 | 中 | 不改编号本体 | 引用链变长；和现有 Step N 语法轻度不兼容 |
| **III. 换单调 ID** | `Q-20260424-01`（日期+当日序号）替 Q-NNN | 很大 | 很大 | 大幅降低冲突概率 | 全面重写 Q/G/R/Step；老项目 migration 风险；违反 append-only 接口契约 |
| **IV. hook 级 assignment** | 子会话写 `Step ?-desc` 作 placeholder；Stop hook 分配真实编号 | 中-大 | 中 | 保留现有编号美学 | Stop hook 要做 id 分配逻辑；编号分配时机延后，实时落盘语义受损 |
| **V. git 原生 + 文档约定** | 默认串行单会话；必须并发时各自分支，合并时冲突 rename 为正式流程 | 零 | 零（文档） | 零代码；符合 kdev-memory 精神 | 用户要理解约束；"一人多终端"场景可能实际不开分支 |
| **VI. 文件锁** | `.kdev/memory/.lock-<session>` 独占写 | 小 | 小 | 简单 | 锁遗留清理难；违反轻量哲学 |

## 2026-04-24 session 决策：方案 V + 未来 V.5

### 为什么不走方案 I / III

- **破坏 append-only 接口契约**：v0.3.0 起约定"编号不回收、不改名"；换语法会让所有老文件失去可读性
- **迁移风险大**：长项目有几百条 Q/G/R 条目，迁移容易出错
- **解决低频问题付高频代价**

### 为什么不走方案 IV

- Stop hook 做 id 分配会把"实时落盘"语义打破——`Step ?-desc` 是 **placeholder 状态**，不是真正的 Step 数据
- id 分配逻辑涉及原子性保证，纯 bash / python 无锁的 hook 实现容易出 race condition
- 属于**用代码解决本该文档约定的问题**——违反 kdev-memory"hook 只戳一下、decision 给 skill"的精神

### 为什么不走方案 VI

- 锁文件在 session 崩溃后遗留，清理机制要再写代码
- 小代码、大心智负担（用户要理解为什么"这里不能改"）
- 效果不如方案 V 直接

### 方案 V（选中）：git 原生 + 文档约定

**约定文档**（v0.6.1 或 v0.7.x 落 references/切档与归档.md 或新建 references/多会话并发.md）：

> ### 多会话同项目编号冲突处理
>
> **默认场景**：本插件设计假设**单会话单项目**。绝大多数开发场景（一人一项目）天然满足。
>
> **例外场景**：必须并发多会话时（如主会话 + 长跑 subagent、或一人跨多终端），按以下流程：
>
> 1. 各自开 git branch（推荐命名 `kdev-session-<label>`）
> 2. 本会话内追加 Q/G/R/Step 条目时使用本地编号
> 3. 合并时，git merge 的冲突标记会显示双方都新增了相同编号
> 4. **合并策略**：保留两方全部条目，把后合并方的编号 +1（或按时间戳排序重编号），并在 R-NNN 记一次（方便未来 skill 作者看冲突模式）
> 5. 合并完 `.kdev/memory/` 内部的交叉引用（"见 Q-007"这种）要人工复核——虽然跨项目场景极少

**Stop hook 扩展（v0.7.x 候选，不阻塞 v0.6.0）**：

加一个 **冲突检测**功能——扫主文件，发现同编号两条 → 红字提醒"编号 NN 出现 2 次，请手动处理"。这是 hook 适合干的事（戳一下注意），不做 assignment 避免越界。

## 实际频率判定

问自己：**真实用量中，多会话同项目同时推进的频率是多少？**

- **一人一项目**：≥ 95% 场景——天然无冲突
- **一人多终端同项目**：~3%——但实际上用户很少真的在两个终端都做**需要落盘**的动作（通常一个终端看、一个做）
- **subagent 并行写**：~2%——subagent 通常只做"单个 phase 内的子任务"，不独立推进 Step

按这个频率，方案 V 的"文档约定"足够。如果 v0.7.x 观察期间出现实际冲突事件 ≥ 3 次/月，再考虑升级。

---

## 不纳入 v0.6.0 的理由

v0.6.0 scope 已经包含：

- 使用的 skill 字段
- Step 粒度指引（三停顿信号 + 三映射）
- todos 跟随记约定

**再塞多会话冲突处理会让 scope 失焦**。且本问题的最佳答案是"文档约定"——不需要搭 v0.6.0 的车（跟 schema 字段和 Step 粒度是不同主题）。

## 方法论信号

1. **低频问题的第一反应不是写代码**：问"频率多少？最轻的方案是什么？代码成本是每次还是一次性？"
2. **append-only 接口契约的惯性价值**：破坏一次就毁所有跨版本引用，不轻易动
3. **hook 要做的是"戳一下"而不是"解决"**：编号冲突的 detect 适合 hook（提醒人类），assignment 不适合（越界做决策）
4. **文档约定 + git 原生 = 很多"协作冲突"的最便宜解法**：git 本身就是多人协作工具，重新发明锁机制 / 分片机制几乎总是亏
