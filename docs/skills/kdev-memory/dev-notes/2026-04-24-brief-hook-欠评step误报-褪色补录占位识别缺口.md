# brief hook 欠评 Step 误报：褪色补录占位未被识别为已销账

> 2026-04-24 在 token-statistics 项目新会话启动时发现。SessionStart brief 把 Step M-5/M-6/M-7/M-8（迭代 5/6/7/8 meta 回补条目）持续报为"欠评 Step"，要求用户核对补齐。但这些条目已在 04-22 kdev-memory v0.4.0 漂移审计修复时显式标注为"褪色补录，保留占位不强求补"，属于**已销账**状态，不应再反复报。

## 现象

SessionStart hook 注入的 `<kdev-memory-brief>` 含：

```
⚠️ 待处理：
- 缺失的过去每日汇总（跨天会话遗漏）：2026-04-23
- 发现 1 条欠评 / 半残 Step（最近 21 条中）：
  · Step 11-12（2026-04-12）: 用户评分段「完成时间」为 '—'
  · 新会话第一件事：向用户核对评分后补齐，或明确销账
```

实际核对结果：

- 报"Step 11-12" **编号错**：当前执行日志里 Step 11 是 `.kdev/ 根目录清理`（04-22 20:38，4.5/5 评分完整），Step 12 是 `collector v1.3.2`（04-22 22:19，3.5/5 评分完整），两者均无欠评
- 报"2026-04-12" **日期错**：token-statistics 项目在 2026-04-12 的活动属于 Sprint 0 迭代 4，当时的 Step 编号是 Step 5/6/7，不是 11/12
- 真正"完成时间：—"的条目是 **Step M-5 / M-6 / M-7 / M-8**（迭代 5/6/7/8 meta 回补），各自条目里已显式写明：

```markdown
### 用户评分
- 完成时间：—
- 顺畅度：—/5（**褪色补录**，距今 6~7 天；保留占位不强求补）
- 用户评价：—
- 🚨 关键 GAP：iter 5 主控全程未问用户"整体顺畅度"——R-010 首条证据链

### 评分差异分析
- 无原生评分 → 不计入差值；本条目见证 iter 5~8 subagent-driven 零评分系统性失守
```

并且有 > 🔴 回补声明：... 当时 iter 5 subagent-driven 批次零评分。内容基于 state.md + 账本 + git log 客观重构，**非原生当场采集**。

## 根因

brief hook 的欠评检测逻辑估计是：

1. `grep` 最近 N 条 Step 的"完成时间：—"或"顺畅度：—"
2. 命中即报"欠评 Step"
3. 同时给出编号 + 日期作为用户参考

但这个逻辑漏了三个关键信号：

### 1. 没识别"褪色补录占位"语义
条目里明写 `**褪色补录**，保留占位不强求补` + 回补声明 `非原生当场采集` —— 这是已销账状态。但 hook 只看"完成时间：—"字面匹配。

### 2. 没识别"回补 meta"标题前缀
Step M-5 / M-6 / M-7 / M-8 标题是 `## Step M-5（回补 meta）: ...`，编号含 `M-` 前缀就是回补标识。但 hook 把它识别成普通 Step 编号"11-12"（可能是字符串切片 bug？值得核查）。

### 3. 日期字段没读对
报"Step 11-12（2026-04-12）"，实际 M-5 日期是 2026-04-15~04-16，M-6 是 04-17~04-18，M-7 是 04-19，M-8 是 04-20 —— 没有一个是 04-12。hook 的日期生成逻辑和 Step 编号匹配逻辑可能脱钩。

## 影响

- 用户每次新会话启动都会看到重复的"待处理欠评"警告，注意力被无效消耗
- 按 CLAUDE.md 规则"⚠️条目立刻处理，不越过摘要做用户新任务"，模型会被拉去"核对评分"，但核对完发现是已销账占位 —— 浪费一轮工具调用和用户时间
- 长期下去用户会对 brief 的预警信号脱敏（"狼来了"效应），真正的欠评 Step（如本次 Step 19/20 确实需要评分）可能被忽略

## 建议的改进方向

### 方案 A（最小改动）：显式销账标记字段
在 Step 条目里加一行 `状态：已销账（褪色补录占位）` 或在 triggers 下一行加 `锁定：已销账-褪色补录`。hook 检测时 grep 这个显式标记，匹配即跳过。

Pros：改动小，对现有褪色条目加一行即可；语义明确。
Cons：需要在 kdev-memory skill schema 里正式定义销账标记字段（当前 schema 没有）。

### 方案 B（启发式）：识别现有文字信号
hook 在报"欠评"前，对命中 Step 再次 grep：
- 标题是否含 `回补 meta` / `M-N` 编号前缀
- 评分段是否含 `褪色补录` / `保留占位` / `不强求补`
- 差异分析段是否含 `无原生评分 → 不计入差值`

任一命中即跳过。

Pros：不改 schema，只改 hook 逻辑。
Cons：启发式可能误判；未来如果有人写其他类型的"占位"也需要跟进加关键词。

### 方案 C（结构化）：引入状态机
Step 条目 frontmatter 加 `status: open | scored | voided-faded | voided-r-nnn`。hook 只报 status=open 的。

Pros：最严格，状态明确。
Cons：需要批量迁移所有历史 Step（300+ 条），工程量大。

### 建议选方案 B（启发式）作为 v0.4.1 修订
理由：
- 方案 A 虽然明确但需要用户补手工标记；token-statistics 已经 23+ Step 条目，人工上加标签成本高
- 方案 C 是长期方向但改动太大，不适合 patch-level fix
- 方案 B 只改 hook 一个文件，且文字信号已经在现有条目里（褪色补录/保留占位/非原生当场采集都是 04-22 审计修复时加的标准措辞），可靠性够用

实现点：hook 脚本在"grep 完成时间：—"命中后，在该 Step 条目范围内再 grep 以下任一文字（任一命中即 skip）：

```
褪色补录
保留占位不强求补
非原生当场采集
无原生评分
不计入差值
## Step M-
```

另外需要修的次要 bug：
- 编号匹配逻辑可能把 `Step M-5 ~ Step M-8` 误合成 "Step 11-12"（怀疑是字符串切片 + 数字合并的 off-by-one，值得单独查 hook 代码）
- 日期字段生成逻辑与 Step 编号脱钩（报 04-12 对不上任何 M-N Step）—— 怀疑是从其他历史条目拿的日期，需要核查生成逻辑

## 观察：brief hook 的信噪比

本次发现印证一个更深的问题：brief hook 当前是"字面 grep 驱动"（完成时间：— / 每日汇总缺失 / unresolved_gotchas 列表等），没有 trust-level 分层。

更好的设计可能是：

- **P0 硬阻塞**（必须处理）：新 Step 当场评分段为空、WARN-未记录-\* 文件存在
- **P1 需核对**（向用户确认）：跨天每日汇总缺失
- **P2 仅报告**（不要求动作）：褪色补录占位、历史 Growth backlog、已销账的改进建议

当前 brief 把所有"有问题迹象"都等同为 P0/P1，导致真正 P0 的信号被稀释。可作为 kdev-memory v0.5.0 的设计输入。
