# kdev-code-graph：CRG 融合评估 + 目标定位结论（决策记录）

> 日期：2026-05-22
> 性质：决策记录（ADR 风格）。回答三件事——这个 plugin 的目标到底是什么、要不要融合 code-review-graph(CRG)、build 成本怎么管。
> 触发：用户提出"把 CRG 作为 UA 的输入以减少 token 消耗"的想法，经一轮分析后收敛。

---

## 一、结论速览（TL;DR）

1. **目标定位**：kdev-code-graph = **UA 通用代码图谱（地基）+ 安全规范追溯（增值）**。用户确认采用 **C（通用做地基、安全做增值）**，且 **B（dashboard / onboarding / 知识蒸馏也是真目标）**。
2. **要不要融合 CRG**：**不融合。** 在"要 UA 富资产"的前提下，CRG 一分钱都省不下来（理由见第三节）。
3. **成本怎么管**：只有三个杠杆，且都与 CRG 无关——**改 API 计费 + 吃 UA 增量更新 + 砍掉不用的 UA 阶段**。全量 build 贵是 B 路线的内在代价，靠增量摊平。
4. **真正的缺口在功能不在成本**：安全规则↔代码的 linker **没实现**（已用代码核实）；文档同步 / 需求追溯 / 变更分析等也需逐一核实落地状态（B 阶段处理）。

## 二、目标定位

### 2.1 这个 plugin 实际是什么

权威来源是 [实施计划 v2](2026-05-10-实施计划-v2.md) 的 Goal：

> 把 kdev-code-graph 改造成「装上 Understand-Anything(UA) 上游 + 安全特化覆盖层」的 plugin，实现**需求追溯 / 爆炸半径 / 文档同步 / 知识蒸馏 + 安全规范追溯**五大场景。

拆开看"谁提供能力"：

| 场景 | 提供方 |
|---|---|
| 需求追溯 | UA 原生（document 节点 + documents 边） |
| 爆炸半径 | UA 原生（/understand-diff） |
| 文档同步 | UA 原生 + kdev doc-sync skill |
| 知识蒸馏 / onboarding | UA 原生（/understand-onboard / tour） |
| **安全规范追溯** | **kdev 唯一的增量**（ingestor + trace/impact） |

> 注意：早期 [PRD v1.1](2026-04-27-产品需求文档.md) 描述的是一个"自研 SQLite + Tree-sitter + MCP Server"的产品，**与现行实现已完全不同**。现行实现以 v2 实施计划为准——不重写引擎，套 UA + 安全覆盖层。PRD 仅作历史参考。

### 2.2 用户的目标与定位的对齐

用户自述的 5 个目标：(1) 全库图谱给大模型快速定位函数；(2) 给人看模块关系；(3) 需求↔代码追溯、验证需求完整实现；(4) 变更时追溯文档同步 + 测试覆盖；(5) 减少 LLM 编码 token。

- 目标 1–5 **几乎全部是 UA 的原生能力**。
- 经确认，用户**也要** dashboard / onboarding / 知识蒸馏（选择 B）。
- 安全规范追溯作为**增值层**保留（选择 C）。

## 三、为什么不融合 CRG

### 3.1 成本分两种，CRG 对两种都使不上力

| 成本类型 | CRG 能否帮 | 原因 |
|---|---|---|
| 请求次数（coding plan 配额） | ❌ 不需要它 | 改用 API 计费后该约束消失 |
| token 量（API 计费） | ❌ 帮不了 | 见下 |

### 3.2 决定性事实：UA 的结构提取本就免费

UA 的 file-analyzer 是**两阶段**（见 UA `agents/file-analyzer.md`）：

- **Phase 1 结构提取**：跑 bundled `extract-structure.mjs`，用 tree-sitter，**确定性、不调 LLM、不花 token**。
- **Phase 2 语义层**：LLM 生成 summary / tags / complexity / 语义边——**token 全花在这**。

而 **CRG 能产出的就是 Phase 1 那种结构**——也就是 UA 已经免费拿到的部分。UA 真正烧 token 的 Phase 2 语义层，CRG 产不出来。

> **净结论：CRG 能给的 UA 已免费有了；UA 真正花钱的 CRG 给不了。融合还要额外搭 SQLite↔JSON 转换层，投入为负。**

### 3.3 "轻量替代路线"为何也被否决

曾考虑：用 CRG 结构图 + 只对安全/文档相关代码做定向 LLM 关联，绕开全库语义分析。但该路线会**牺牲掉 dashboard / onboarding / 每函数摘要**——而用户选择 B，明确要这些。故该路线出局。

### 3.4 请求次数 vs build 时间的关系（备查）

- UA build 的时间主导项是 **LLM 请求延迟**（每次数秒~数十秒），不是本地解析（毫秒级）。
- 因此**请求次数 ↓ → build 时间 ↓**，基本成正比（file-analyzer 已有最多 5 并行，砍并行段省得少、砍串行段省得多）。
- 单次请求受**上下文窗口**硬约束，**不能无限合并**；且质量比窗口更早成为瓶颈（批太大→注意力稀释/输出截断）。所以"全库语义分析"前提下，file-analyzer 这段请求数有不可压缩的下界。

## 四、成本三杠杆（B 路线下能做的全部）

1. **改 API 计费** —— 解决 coding plan 配额被打空。（用户已定）
2. **吃 UA 增量更新**（最重要）—— UA 有 Phase 0 增量检测（fingerprint + meta.json gitCommitHash）。**全量 build 只在第一次付全价，之后只重分析改动文件。** 这是把"build 贵"摊平的正解。
3. **砍掉不用的阶段** —— tour（onboarding）、layers（模块视图）要留；**domain-analyzer**（业务领域）不用可关，graph-reviewer 可酌情跳（牺牲质量校验）。常数级节省。

> 天花板就到这。接受"全量 build 贵、靠增量摊平"是 B 路线的内在代价。

## 五、真正的待办：未实现/待核实的功能

成本问题已有答案；plugin 当前的真实短板在**功能落地**，留待 B 阶段设计：

| 功能 | 当前状态 | 证据 / 备注 |
|---|---|---|
| 安全规则 → 代码 linker | ❌ **确认未实现** | ingestor 只 `upsert_node`，从不 `upsert_edge`（[security_rules.py:94](../../../plugins/kdev-code-graph/ingestor/kdev_ingestor/security_rules.py#L94)、[cli.py:38](../../../plugins/kdev-code-graph/ingestor/kdev_ingestor/cli.py#L38)）。规则节点是孤岛，trace/impact 跑出来为空。 |
| 文档 ↔ 代码同步（doc-sync） | ⚠️ **需核实** | 依赖 UA 是否可靠产出 `documents` 边；kdev doc-sync skill 只查询不创建。 |
| 需求追溯 | ⚠️ **需核实** | UA 原生 documents 边支撑，但完整性/准确性未验证。 |
| 变更分析（impact） | ⚠️ **需核实** | UA /understand-diff 提供爆炸半径，kdev 在其上叠安全过滤——过滤依赖上面那条缺失的 linker 边。 |

### 关键设计线索（B 阶段起点）

> **"文档↔代码 linker" 与 "规则↔代码 linker" 本质是同一个机制**——都是"把一段语义描述（文档段落/需求/安全规则）关联到具体代码节点"。可统一成一个**定向语义关联器**，一份实现同时服务通用（需求追溯/文档同步）与安全（规则追溯）。其成本按**文档数/规则数**计，**不碰全库**，与第三节的 token 大头无关。
