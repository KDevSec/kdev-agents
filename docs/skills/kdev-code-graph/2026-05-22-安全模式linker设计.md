## 1. 背景与目标

kdev-code-graph 当前的安全增量是**半成品**：ingestor 只把安全规则灌成图里的孤立 `concept` 节点（[security_rules.py:94](../../../plugins/kdev-code-graph/ingestor/kdev_ingestor/security_rules.py#L94)），从不创建任何边（[cli.py:38](../../../plugins/kdev-code-graph/ingestor/kdev_ingestor/cli.py#L38) 只调 `upsert_node`）。后果：`kdev-codegraph-trace` / `kdev-codegraph-impact` 跑出来恒为空，安全追溯实际不可用。

**本 spec 的目标**：实现一个**零 LLM、确定性**的"安全规则→代码"关联器，把自带可识别 API 模式的安全规则连到使用了这些 API 的代码节点，让 trace/impact 有真边可查。

这是"关联器家族"的**成员 1（PatternLinker）**。成员 2（行为级需求文档→代码的 SemanticLinker，基于 LLM）是未来独立 spec，本设计为其预留接口。

---

## 2. 范围

### 2.1 In scope
- 从安全规则 markdown 提取可识别的 API 模式 token（运行时来自规则"适用场景"段）。
- 扫描项目源码，找出使用这些 token 的位置。
- 把命中映射到图谱的 function/file 节点，产出"规则→代码"的 `related` 边。
- 折进 `/kdev-codegraph-build` 流程。
- 小幅更新 `kdev-codegraph-trace` skill，使其报告 `related`（疑似涉及）边。

### 2.2 Out of scope（明确不做）
- 行为级需求/文档→代码语义关联（成员 2，未来）。
- 判断"代码是否*正确*实现了规则"（语义活，非模式匹配能力范围）。
- AST 级精确匹配（v1 用文本扫描，AST 留作后续增强）。
- 无 API 特征的纯实践类规则（错误处理/代码质量等）的关联——这些规则**诚实留空**，等成员 2。

## 3. 架构

### 3.1 关联器家族接口

在 ingestor 内新增 linker 抽象，定义统一契约：

```python
# kdev_ingestor/linkers/base.py
class Linker(Protocol):
    def link(
        self,
        rules: list[SecurityRule],
        graph: KnowledgeGraph,
        source_root: Path,
    ) -> list[dict]:        # 返回 edge dict 列表，纯函数，不落盘
        ...
```

- **成员 1**：`PatternLinker`（本 spec 实现）
- **成员 2**：`SemanticLinker`（未来，签名一致，内部接 LLM；`source_root` 对它可选）

编排层（CLI）：选 linker → `link()` 拿到边 → 逐条 `upsert_edge` 落盘。`upsert_edge` 及其校验（节点存在性、类型白名单、weight、direction、幂等索引）已存在于 [graph_io.py:155](../../../plugins/kdev-code-graph/ingestor/kdev_ingestor/graph_io.py#L155)，直接复用。

### 3.2 模块清单

| 模块 | 职责 | 新建/改 |
|---|---|---|
| `security_rules.py` | 扩展：解析规则时一并提取 `patterns` | 改 |
| `linkers/base.py` | `Linker` 协议 + 公共 helper（命中→节点映射） | 新 |
| `linkers/pattern_linker.py` | `PatternLinker` 实现 | 新 |

## 4. 数据流

```
规则 md ──parse──► SecurityRule{..., patterns: ["cmd_exec", "dyn_eval", ...]}
                                    │
项目源码树 ──扫描(token, 词边界)──► 命中列表 [(rel_path, line_no, token), ...]
                                    │  按 filePath + lineRange 映射
图谱节点(function 带 lineRange / file 带 filePath) ◄┘
                                    │
                    upsert_edge: kdev-sec:rule:<id> ──related──► <code_node_id>
```

## 5. 边编码（决策 A）

模式命中的诚实语义是"此代码**落在该规则适用范围内 / 疑似触碰**"，不是"实现了"。因此：

| 关系 | UA 边类型 | direction | weight | 识别 |
|---|---|---|---|---|
| 规则涉及某段代码（模式命中） | **`related`** | `forward` | `0.6` | source 节点带 `kdev:security_rule` |

- 保留 `documents`(backward)="代码正确实现了规则" 给成员 2（语义）。
- `tested_by` 维持 UA 原生，不在本 linker 职责内。
- weight `0.6` 体现文本扫描的不确定性（可能命中注释/字符串）。
- UA 边 schema 不允许自定义字段（zod strip），故所有 kdev 语义只能靠"边类型 + 端点节点的 tag"编码——这里靠 source 端的 `kdev:security_rule` tag 区分。

边对象形状：
```json
{"source": "kdev-sec:rule:3.1.1", "target": "function:app/api.py:run_cmd",
 "type": "related", "direction": "forward", "weight": 0.6}
```

## 6. 模式提取（security_rules.py 扩展）

### 6.1 信号来源（优先级）
1. **主**：`### 适用场景` 段里**反引号包裹的 token**——结构最干净、噪音最低。
   - 形如："代码中出现 `cmd_exec` / `shell_run` / `dyn_eval`" → 提取 `["cmd_exec","shell_run","dyn_eval"]`
   - （真实规则文件里这些是命令执行 / 动态求值 / 反序列化类 API 名）
2. v1 仅取上述反引号 token。`### 反例` 代码块的标识符解析作为**后续增强**，不在 v1。

### 6.2 规则
- token 字符集：允许 `[A-Za-z_][A-Za-z0-9_.]*`（含点号，以匹配带模块前缀的调用如 `mod.func`）。
- 一条规则可有 0..N 个 patterns。0 个 → 该规则不参与 linking（诚实留空）。
- `SecurityRule` dataclass 增加字段 `patterns: list[str]`。
- `rule_to_node` 不变（节点仍不带 patterns；patterns 只在 linking 阶段用，不写进图谱节点）。

## 7. 匹配机制

- v1：**文本扫描 + 词边界**。对每个 token 构造 `\b<re.escape(token)>\b`（点号转义后仍按字面匹配）。
- 扫描范围：`source_root` 下的源码文件。**复用图谱已有的文件清单**——只扫图谱里存在 `file:<path>` 节点的文件（天然跳过 .understandignore 排除的文件，也避免扫 .venv/node_modules）。
- 命中 → `(rel_path, line_no, token)`。

### 7.1 命中 → 节点映射（决策 B：函数级优先，文件级兜底）
1. 找该 `filePath` 下所有 `function:` 节点，若命中 `line_no` 落在某 function 的 `lineRange:[start,end]` 内 → 连到该 function 节点。
2. 落不进任何 function 区间（或该文件无 function 节点）→ 连到 `file:<path>` 节点。
3. 同一 (rule, node) 多次命中 → 一条边（`upsert_edge` triple 幂等）。

> **实现首步必须验证**：真实 UA 输出的 function 节点确实带 `lineRange`（样例 fixture 带）。若真实字段名不同（如 `line`/`startLine`），在 helper 里做兼容。

## 8. CLI 与 build 集成（决策 C：折进 build）

### 8.1 新增 CLI 子命令
```
kdev-ingest link --rules-dir <dir> --graph <path> --source-root <dir>
```
- 默认 linker = pattern。
- 输出：`linked N security edge(s) into <graph>`。

### 8.2 build 流程（kdev-codegraph-build/SKILL.md）
原 `understand → inject` 改为 **`understand → inject → link`** 三步连跑：
```bash
python3 "$INGESTOR" inject --rules-dir "$KDEV_RULES_DIR" --graph "$GRAPH"
python3 "$INGESTOR" link   --rules-dir "$KDEV_RULES_DIR" --graph "$GRAPH" --source-root .
```
build 报告表格增加一行：`安全关联边(related)：<N>`。

## 9. 错误处理

| 现象 | 应对 |
|---|---|
| `source_root` 不存在 | 报错 exit 2 |
| 所有规则都无 patterns | 正常退出，报 `linked 0 security edge(s)`（不是错误） |
| 命中文件不在图谱（被 ignore） | 跳过该命中，不产悬空边 |
| function 节点无 lineRange 字段 | helper 兼容；都拿不到 → 退文件级 |
| 重跑 | upsert 幂等，不堆重复边 |

## 11. 下游 skill 更新

### kdev-codegraph-trace（小改）
- 模式 1（规范→代码）：除现有 `documents`/`tested_by`，**新增报告 `related` 边**，标注为"⚠️ 疑似涉及（模式命中，weight=0.6，需人工确认是否违规）"。
- 模式 2（代码→规范）：代码节点的 `related` 邻居中带 `kdev:security_rule` 的 → 列为"该代码涉及的安全规则"。
- 措辞强调：模式命中 ≠ 一定违规，是"该规则适用于此处，请人工核对正反例"。

### kdev-codegraph-impact（不改）
Step 2 本就是"找邻居中带 `kdev:security_rule` tag 的节点"，`related` 边天然满足，无需改动。
