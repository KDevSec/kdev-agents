## 1. 背景与目标

PatternLinker（成员 1）已落地，解决安全规则↔代码的"模式命中"关联。本设计实现**成员 2 `SemanticLinker`**：解决**"行为级 spec / 需求文档 → 代码"** 的语义关联——这是 PatternLinker 抓不到的部分（spec 描述行为，不点名 API token），也是用户长期目标 3（需求追溯）与 4（文档同步）的真正落地基础。

UA 自动产的 `documents` 边是浅启发式（只在文档按文件名提到代码时才连），无法支撑行为级追溯。本设计填补这一空白。

**同时合并并取代** 现有的 `kdev-codegraph-doc-sync` skill（它依赖 `documents` 边而 UA 不充分产出，跑出来基本是空报告，功能由本 skill 完整覆盖且加强）。

- **跨文档关联**：每条 intent 独立判定

## 3. 架构

### 3.1 三阶段（脚本 + LLM agent 分层，对齐 UA 的"script + analyzer agent"模式）

```
/kdev-codegraph-spec-link
   │
   ├─① ingestor: spec-link-prepare  ← stdlib only，本地确定性
   │   读图谱（document 节点）
   │   读 source_root 下对应 markdown 文件
   │   B+ 粒度抽 intent
   │   B 策略候选检索（KeywordRetriever）
   │   写 .understand-anything/intermediate/spec-link-intents.json
   │
   ├─② skill 主控驱动 subagent  ← LLM 工作（每 intent 一个 sonnet subagent，并发上限 5）
   │   per intent 派 subagent（prompt 模板见 §7）
   │   subagent 返回严格 JSON verdict
   │   主控汇总写 .understand-anything/intermediate/spec-link-verdicts.json
   │
   └─③ ingestor: spec-link-finalize  ← stdlib only
       读 verdicts.json
       清掉上一次 owned 的 documents 边（按 graph.extras.kdev_spec_link.owned_edge_triples）
       upsert 新 documents 边（direction=backward，weight=confidence）
       更新 graph.extras.kdev_spec_link 元数据
       生成统一两维报告（写文件 + 打印摘要）
```

```json
{
  "schema_version": 1,
  "generated_at": "2026-05-22T12:34:56+08:00",
  "graph_path": ".understand-anything/knowledge-graph.json",
  "source_root": ".",
  "doc_count": 3,
  "intent_count": 25,
  "intents": [
    {
      "intent_id": "docs/prd.md#F1.1",

      "intent_title": "F1.1 多语言 AST 解析",
      "intent_text": "支持 Python, JS/TS, Go, Rust, Java, ...",
      "candidates": [
        {"node_id": "function:src/parsers/ts.ts:parseTs", "summary": "Parses TypeScript using tree-sitter"},
        ...
      ]
    }
  ]

```

### 4.2 `spec-link-verdicts.json`（subagent 汇总产出）
```json
{
  "schema_version": 1,
  "generated_at": "2026-05-22T12:40:11+08:00",

    {
      "intent_id": "docs/prd.md#F1.1",
      "status": "implemented",
      "linked": [
        {"target_node_id": "function:src/parsers/ts.ts:parseTs", "confidence": 0.85, "reason": "..."},
        ...
      ]
    }
  ]
}
```

`status` ∈ `{implemented, partial, not_found, error}`。

### 4.3 `graph.extras.kdev_spec_link`（finalize 写入图谱顶层 extras）
```json
{
  "schema_version": 1,
  "last_run_at": "2026-05-22T12:40:30+08:00",
  "intents_processed": 25,
  "implemented_count": 18,
  "partial_count": 3,
  "not_found_count": 4,
  "error_count": 0,
  "owned_edge_triples": [
    {"source": "function:...", "target": "document:docs/prd.md", "type": "documents"}

```

`owned_edge_triples` 是 finalize 实现幂等重写的关键——下次跑 finalize 时先按这个清旧边，不污染 UA 自己产的 `documents` 边。

## 5. Intent 抽取（B+ 粒度）

### 5.1 输入
所有 graph 节点中满足：`type == "document"` 且 `filePath.endswith(".md")`。

### 5.2 解析规则（按优先级）
1. 读取每份 doc 的 markdown 内容（`source_root / filePath`）
2. **结构化 FR 表格检测**（高优先）：扫描所有 markdown 表格，若**首列 header 含 "ID" 或首数据行单元格匹配 `^[A-Z][A-Za-z]?\d+(\.\d+)+$`**（如 `F1.1`、`G2.3`、`NF1.1.2`）→ 每行一条 intent：
   - `intent_kind = "fr_row"`
   - `intent_id = <doc_relpath>#<first_cell>`
   - `intent_title = <first_cell> + " " + <second_cell>`（若有第二列）
   - `intent_text = ` 整行文本拼接（去 markdown 表格符号）
3. **Section 级**（默认）：未被 FR 表格吃掉的 `##` 或 `###` 标题各产一条 intent：
   - `intent_kind = "section"`
   - `intent_id = <doc_relpath>#<heading-slug>`（中文标题保留，仅替换空白和特殊字符）
   - `intent_title = ` 标题文本
   - `intent_text = ` 该 section 主体（截至下一个同级或上级标题），含子 section 文本，去 markdown 标记，截断到 800 字符
4. 无 H2/H3 标题的 doc → 整篇产 1 条 intent（`intent_kind = "doc"`，title = filename）

### 5.3 去重
`intent_id` 全局唯一。重复 → 后者覆盖前者并 warn。

### 5.4 测试 fixture（用于 Task 1）
新建 `tests/fixtures/sample-docs/`：
- `simple.md`：3 个 H2 section，纯叙述 → 3 条 section intent
- `with-fr-table.md`：1 个 H2 包含 FR 表（首列 `F1.1`/`F1.2`）+ 1 个普通 H2 → 2 条 fr_row + 1 条 section
- `no-headings.md`：无标题 → 1 条 doc intent

## 6. 候选检索（B 策略 + future C 接口）

### 6.1 协议
```python
class CandidateRetriever(Protocol):
    def retrieve(
        self,
        intent_title: str,
        intent_text: str,
        graph: KnowledgeGraph,
        top_k: int = 30,
    ) -> list[dict]:
        """Return [{"node_id": str, "summary": str}, ...] of length ≤ top_k."""
        ...
```

### 6.2 v1 实现 `KeywordRetriever`
```python
TOKEN_RE = re.compile(r"\w+", re.UNICODE)  # 包含 Unicode 字符，中文连续 token 自然成串

def tokens(text: str) -> set[str]:
    return {t.lower() for t in TOKEN_RE.findall(text) if len(t) >= 2}
```

- 对 `intent_title + " " + intent_text` 抽 token 集 `Q`
- 对每个 `function` / `class` 节点：抽 `name + " " + summary` 的 token 集 `D`
- score = `|Q ∩ D|`
- 按 score 降序，score>0 取 top_k
- **退化策略**：所有候选 score=0 → 取 `function`/`class` 节点前 top_k 个（保底，让 subagent 能看到东西判 "not_found"）；图谱总函数+类节点 ≤ top_k → 直接全表

### 6.3 已知 v1 限制
- **跨语言不充分**：中文 intent → 英文代码摘要时，无 token 重叠的真正匹配会被漏掉。EmbeddingRetriever（C）将解决这点；v1 接受此缺口，报告中标注"未发现 ≠ 未实现"
- **同义词不识别**：`登录` vs `认证`/`signin`/`auth` 之间无关联；同上由 C 解决

### 6.4 EmbeddingRetriever（future stub）
```python
class EmbeddingRetriever:
    """检测 graph 是否带 embedding；有则用 cosine 相似度。v1 不实现，留接口位。"""
```
v1 阶段不放代码，但在 `candidates.py` 文件末尾留注释说明扩展点。

## 7. LLM 判定（skill + subagent）

### 7.1 编排
SKILL.md 主控读 `intents.json`，对每条 intent **派一个 sonnet subagent**，**并发上限 5**（与 UA file-analyzer 对齐）。汇总后写 `verdicts.json`。

### 7.2 Subagent prompt 模板（写在 SKILL.md 里）

```
You are a spec-to-code semantic linker.

## Task
Given a spec intent and a list of candidate code nodes, determine which candidates implement the intent. Output strict JSON.

## INPUT

intent_id: {intent_id}
intent_title: {intent_title}
intent_text: {intent_text}

candidates (target_node_id MUST be chosen verbatim from this list — do NOT invent ids):
[
  {"node_id": "<id>", "summary": "<summary>"},
  ...
]

## OUTPUT — strict JSON only, no prose:

{
  "intent_id": "<echo input>",
  "status": "implemented" | "partial" | "not_found" | "error",
  "linked": [
    {"target_node_id": "<node_id from candidates>", "confidence": <0.0..1.0>, "reason": "<1 sentence, language matches intent_text>"}
  ]
}

## Rules
- target_node_id MUST be from the candidates list exactly. Do not invent ids. Do not modify ids.
- status semantics:
  - "implemented": one or more candidates fully implement the intent (confidence ≥ 0.7 for at least one linked entry)
  - "partial": some candidates partially address it, none fully (max confidence in [0.4, 0.7))
  - "not_found": no candidate plausibly implements; linked = [].
- linked may be empty only when status is "not_found" or "error".
- reason: ONE sentence, same language as intent_text (中文 intent → 中文 reason).
- If anything blocks judgment, set status="error" and put diagnostic in reason of a single fake entry — DO NOT crash.
- Return ONLY the JSON object. No code fences, no commentary.
```

### 7.3 主控如何汇总
- 派 5 个并发 subagent，等所有完成
- 严格 JSON 解析每条 verdict；解析失败 → 标 `status="error"` 自动兜底
- 汇总写 `verdicts.json`，然后调 `kdev-ingest spec-link-finalize`

### 7.4 Subagent 模型选择
**sonnet**（per CLAUDE.md 编码执行类）。判定任务确定性高、范围窄，sonnet 够用。

## 8. 边编码 + extras 元数据

### 8.1 documents 边
| 关系 | UA 边类型 | direction | weight | source / target |
|---|---|---|---|---|
| 代码实现某 spec intent | `documents` | `backward` | `confidence`（LLM 0..1） | source = code node id；target = document node id |

- direction=backward 表"代码 ← 文档 documents 关系反向 = 文档被代码实现"，对齐 v2 实施计划的语义映射
- weight = LLM 给的 confidence，**不做阈值裁剪**，让下游报告分段
- 同一 (source, target, "documents") triple 在 upsert 时按 type+source+target 去重（已有索引机制）

### 8.2 owned_edge_triples（幂等关键）
- finalize 跑之前：读 `graph.extras.kdev_spec_link.owned_edge_triples` → 这些边都是上次自己写的 → 全部从 graph.edges 删除（按 triple 匹配）
- 然后写新 verdicts 产出的边
- 写完后更新 `owned_edge_triples` = 本次写入的所有 triple
- 好处：**不污染 UA 自己产的 documents 边**（不在 owned 列表里的 documents 边 finalize 不动）

### 8.3 finalize 跳过条件
- verdict status = `not_found` 或 `error` → 不产边
- linked 条目 confidence 为非数字 / 不在 [0,1] → 跳过该条
- target_node_id 不在 graph._node_id_to_index → 跳过（防 hallucination 漏网）

### 9.1 文件位置
- 同时打到对话流（摘要）+ 写到 `.understand-anything/reports/spec-link-<ISO_timestamp>.md`（详情，方便回看）

### 9.2 结构
```markdown
# Spec ↔ Code 对齐审计报告
> 生成时间: 2026-05-22T12:40:30+08:00

| 实现状态 | 数量 | 占比 |
|---|---|---|
| ✅ 有实现 | 18 | 72% |
| ⚠️ 部分/疑似 | 3 | 12% |
| ❌ 未发现实现 | 4 | 16% |
| 🔥 LLM 判定错误 | 0 | 0% |

| 同步状态 | 数量 |
|---|---|
| ✅ 新鲜 | 15 |
| ⚠️ 漂移(>30d) | 5 |
| 🔍 缺文档(代码无 doc 归属) | 12 文件 |

## ❌ 未发现实现（优先处理）
| Intent | 文档位置 | 候选数 | LLM 判断 |
|---|---|---|---|
| F1.3 测试文件识别 | docs/prd.md | 30 | "无候选满足 confidence ≥ 0.4" |

## ⚠️ 部分/疑似实现
| Intent | 关联节点 | confidence | reason |
|---|---|---|---|

## ⚠️ 漂移（代码比文档新 >30 天）
| Intent | 文档时间 | 代码最新时间 | 漂移天数 |
|---|---|---|---|

## ✅ 有实现（折叠）
<details>
N 条已实现 intent，详情见图谱 edges。
</details>

## 🔍 缺文档（按文件聚合）
| 文件 | 函数数 | 建议 |
|---|---|---|
| src/billing/refund.py | 5 | 补 refund 流程 doc |

## ⚠️ 重要注意事项
- **未发现实现 ≠ 未实现**：可能是 v1 候选检索的跨语言局限（中文 intent ↔ 英文代码摘要无 token 重叠），请人工核对
- 低 confidence (< 0.5) 项视为疑似，需人工确认
- 涉及 `kdev:security_rule` 的文档漂移阈值收紧到 14 天（与原 doc-sync 一致）
```

### 9.3 实现要点
- "缺文档" 聚合：扫 graph，找 `type=="function"` 且**没有任何 `documents` 边指向自己**的节点，按 file 分组
- "漂移" 计算：用 `git log -1 --format=%ct -- <path>` 拿 mtime（doc 和关联 code 取最大），差值 > 阈值则 ⚠️
  - finalize 调 `git log`：子进程调用，git 不可用时降级（不报漂移，标注"git 不可用"）
  - 涉及 security_rule 的文档（target node 带 `kdev:security_rule` tag）阈值 14 天，其他 30 天
- 报告路径：`reports/` 目录不存在则创建

## 10. 新鲜度 + 前置面板

### 10.1 通用前置面板模板（spec-link + build 都用）
```markdown
## 即将执行: /<skill-name>

### 这次会做什么
1. <步骤 1 一句话>
2. <步骤 2 一句话>
3. ...

### 成本提示 ⚠️
- 估计派出 ~<N> 个 LLM subagent / 约 <M> 次 LLM 请求
- 估算耗时: <X> 分钟
- **不建议在按"请求次数"计费的 coding plan 上跑** —— 建议切换到 API 计费

### 这次会产生
- <落盘文件 1>
- <边/节点/报告 等图谱产物>

### 跑完后你可以做的事
- `/kdev-codegraph-trace` —— 安全规则 ↔ 代码双向追溯
- `/kdev-codegraph-impact` —— 变更安全爆炸半径分析
- `/kdev-codegraph-spec-link` —— spec ↔ 代码对齐审计（也烧 LLM，按需跑）

### 继续? [y/n]
```

### 10.2 spec-link 的两段式面板
- **面板 1（skill 开始）**：先打"即将跑 spec-link"的大致说明 + 兄弟 skill 列表 + 成本警告 + `[y/n]`
  - 此时还没跑 prepare，N、M、X 都是预估区间，措辞为"取决于 doc 数和 intent 粒度，通常 10-50 个 subagent"
- **面板 2（prepare 完成、judge 开始前）**：根据 intents.json 的精确 intent_count，打**确定数字**的面板 + 第二次 `[y/n]`
  - 用户 n → 跳过 judge，直接拿现有 owned edges 出报告（明确标"用的是过期数据"）
  - 用户 y → 进 judge 阶段

### 10.3 `--yes` flag（跳过确认）
skill 支持 `--yes` 跳过两次确认，给自动化用。

### 10.4 build SKILL 的前置面板（附加改动）
在 `kdev-codegraph-build/SKILL.md` 顶部（Step 1 之前）加同样模板的面板：
- 步骤：UA `/understand` 全库语义分析 → ingest 灌规则 → link 模式匹配
- 成本：UA 派几十~上百个 file-analyzer subagent，**特别强调 coding plan 配额风险**
- 产物：knowledge-graph.json + 安全节点 + related 边
- 后续可用 skill：trace / impact / spec-link
- `[y/n]` 才进入 Step 1

## 11. 失败 / 降级

| 现象 | 应对 |
|---|---|
| 图谱无 `document` 节点 | prepare 报告 "no document nodes; 可能 UA 未把 docs 纳入，建议 `/understand --full`"，写空 intents.json |
| 某 doc 文件读不到（路径错） | 跳过该 doc，记日志，不中断整体 |
| FR 表格识别失败 | 退化为 section 级 |
| subagent 返回非 JSON 或字段缺 | 主控记 `status="error"`，finalize 跳过 |
| subagent target_node_id 不在 candidates / 不在图谱 | finalize 跳过该边，记 warn |
| git 不可用（无 git 仓库 / git 命令缺） | 漂移段标 "git 不可用，跳过漂移检查"，其他维度正常 |
| 重跑 finalize（幂等） | 按 owned_edge_triples 清旧，重写新 |
| 用户在面板 2 选 n | 跳过 judge，用现有 owned edges 出"过期"报告，明确标注 |

## 15. 风险与回滚

| 风险 | 概率 | 缓解 |
|---|---|---|
| 中文 spec / 英文代码跨语言召回低 → 大量"未发现实现"误报 | 中 | 报告显著标注；EmbeddingRetriever 留 future hook；可通过加 doc 中英对照部分缓解 |
| FR 表格格式多样导致识别漏 | 中 | v1 只认带 ID 列 / 首列规则模式；命中失败退化 section；后续根据实际样例增强 |
| subagent JSON 输出不严格 | 中 | prompt 强调"only JSON"，主控解析失败标 error 兜底 |
| 用户没注意成本面板直接 y | 低 | 面板做得人友好、显眼；提供 `--yes` 同时也提供"再次确认" |
| owned_edge_triples 越来越长 / 图谱 extras 膨胀 | 低 | 每次 finalize 全量重写该字段，不累积 |

**回滚**：本设计无破坏性改动（doc-sync 删除可从 git 恢复；新加 skill 可禁用）。如发现质量不达预期，禁用 spec-link，doc-sync 从 git revert 找回即可。
