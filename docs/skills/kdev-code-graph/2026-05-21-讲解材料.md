# kdev-code-graph 讲解材料

> 用途：对外/对内讲解 kdev-code-graph plugin 的定位、能力、使用方式与演示脚本。
> 版本对应：plugin **v0.2.0**（4 个 skill + ingestor）—— 历史快照。
>
> ⚠️ **本文档是 2026-05-21 的快照,自此后能力已演进**：v0.3.0 起 `doc-sync` 已被 `spec-link` 取代（spec ↔ code LLM 判定 + 两维报告），新增 PatternLinker（安全规则 ↔ 代码模式关联）。讲解时如复用，请把本文中"doc-sync"出现处替换成"spec-link"语义，并参考 CHANGELOG 与最新 spec 文档。

## 一、它是什么 / 主要作用

一句话：**给项目建一张"代码知识图谱"，并在上面叠一层安全规范网络，让"安全规范 ↔ 代码 ↔ 文档"三者可以互相追溯。**

它**不重造轮子**，核心图谱引擎完全复用上游开源项目 [Understand-Anything](https://github.com/Lum1104/Understand-Anything)（简称 UA）。kdev-code-graph 只做两件增量的事：

1. **安全覆盖层** —— 把 `kdev-secure-coding` 的安全规范（如"3.1.1 命令操作安全"）灌进 UA 的图谱，编码成图里的节点，并与真实代码节点连起来。
2. **4 个面向安全场景的 skill** —— 在图谱上做查询、追溯、影响分析。

设计原则（讲解时强调的"克制"）：

- **不重写代码图谱引擎** —— 完全复用 UA 上游。
- **对 UA 零修改** —— 安全节点借用 UA 自带的 `concept` 节点类型 + `kdev:*` 标签来编码，不动 UA 的 schema。
- **Contract test 当护栏** —— UA 上游升级若改了节点/边白名单，`tests/contract/` 会自动报警。

### 数据流示意

```
        kdev-secure-coding 规范.md
                  │  ingestor 灌入
                  ▼
   UA /understand ──► knowledge-graph.json ◄── 安全规则节点 (concept + kdev:* tag)
                  │
      ┌───────────┼───────────┬──────────────┐
      ▼           ▼           ▼              ▼
    build       trace       impact        doc-sync
   (建图)    (规范↔代码)  (变更爆炸半径)  (文档代码同步)
```

## 二、4 个能力（讲解主线）

| Skill | 解决什么问题 | 典型提问 |
|---|---|---|
| **/kdev-codegraph-build** | 建图 + 灌安全规范，是其它 3 个的前置 | "建图谱 / 灌安全规范" |
| **/kdev-codegraph-trace** | 规范 ↔ 代码**双向**追溯 | "3.1.1 这条规范在哪实现？""这段代码涉及哪些安全规范？" |
| **/kdev-codegraph-impact** | 变更的**安全爆炸半径** | "改这段代码会影响哪些规范？该跑哪些回归？" |
| **/kdev-codegraph-doc-sync** | 文档与代码**同步审计** | "哪些 PRD 没实现？哪些文档过期了？" |

### 底层 ingestor

ingestor 是一个 stdlib-only（零依赖）的 Python 工具，通过 `${CLAUDE_PLUGIN_ROOT}/ingestor/run.py` 直接调用，**不需要 venv，也不需要 pip install**。它只有两个命令，skill 会自动调用，一般无需手敲：

- `inject` —— 把规范 markdown 灌成图节点（`concept` 类型 + `kdev:*` 标签）。
- `list-tags` —— 核对图里灌了哪些 `kdev:*` 标签。

## 三、如何使用

### 前置环境

- Node.js ≥ 22（UA 上游需要）
- Python ≥ 3.11（ingestor 零依赖，stdlib only）
- 已安装 UA 插件

### 安装

> ⚠️ `/plugin` 命令只在 Claude Code CLI / 桌面应用 / Web App 可用，**VSCode 扩展里不可用**。

```
/plugin marketplace add KDevSec/kdev-agents
/plugin marketplace add Lum1104/Understand-Anything
/plugin install kdev-code-graph        # UA 会作为 dependency 自动连带装
```

也可本地 clone 后跑一行脚本（macOS / Linux / Windows）：

```bash
git clone https://github.com/KDevSec/kdev-agents.git
cd kdev-agents
./scripts/setup-kdev-codegraph.sh   # macOS / Linux
.\scripts\setup-kdev-codegraph.ps1  # Windows PowerShell
```

### 使用流程（全部走自然语言，自动路由到 skill）

1. **第一步永远是建图**：说"给这个项目建代码图谱"→ 触发 `/kdev-codegraph-build`
   - 内部：先调 UA 的 `/understand` 建基础图 → 再用 ingestor 把安全规范灌进去。
   - 产物落到 `.understand-anything/knowledge-graph.json`。
2. **建好后直接问业务问题**，自动路由：
   - "3.1.1 在哪实现" → trace
   - PR 合并前 "这次改动有什么安全风险" → impact（内部调 UA `/understand-diff` 拿 L1/L2/L3 爆炸半径，再过滤出带安全标签的节点）
   - "盘点哪些文档过期了" → doc-sync

## 四、效果怎么展示（演示建议）

核心卖点是 **"四种可视化报告表格"** —— 每个 skill 跑完都输出 markdown 表格，演示时直接投这些表最直观。

### ① build 的产出表（证明图建起来了）

| 指标 | 值 |
|---|---|
| 总节点数 | N |
| UA 节点 | N |
| kdev 安全规则节点 | N |
| 总边数 | N |
| 图谱文件 | `.understand-anything/knowledge-graph.json` |

### ② trace 报告（最有"哇"感的双向追溯）

```markdown
# 安全规范 3.1.1「命令操作安全」追溯报告
## 直接关联的代码（N 处）
| 文件 | 函数/类 | 关系 | 信心 |
| app/web/api.py | execute_command | 实现 | weight=0.9 |
## 测试覆盖（M 处）
## 未覆盖警告
```

### ③ impact 报告（PR 评审场景，分级最有说服力）

```markdown
## 受影响的安全规范（按严重度倒序）
| 规则 ID | 规则名 | 受影响代码 | 严重度 | 建议 |
| 3.2.1 | 加密算法选择 | crypto/aes.py:encrypt | high | 跑回归 |
| 3.1.1 | 命令操作安全 | api.py:execute | high | 人工复审 |
## 建议测试（基于 tested_by 边自动推荐）
```

严重度分级规则：

| 信号 | 严重度 |
|---|---|
| L1 节点关联 `kdev:severity:high` 规则 | **high** |
| L1 关联 `medium` 或 L2 关联 `high` | **medium** |
| 仅 L3 语义关联 | **low** |

### ④ doc-sync 报告（四级状态最直观）

| 状态 | 含义 |
|---|---|
| ✅ 同步 | 文档时间戳 ≥ 关联代码时间戳；存在 `documents` 边 |
| ⚠️ 需更新 | 关联代码比文档新，漂移超阈值（默认 30 天） |
| ❌ 缺实现 | 文档存在但无关联代码 |
| 🔍 缺文档 | 代码有 file/function 但无 document 节点对其负责 |

> 协同点：若文档关联到带 `kdev:security_rule` 的节点，漂移阈值从 30 天收紧到 **14 天**。

## 五、5 分钟演示脚本

1. **建图**：选一个有安全敏感代码的 Python 项目，现场跑 build，投出统计表 → "图建好了，注入了 N 条安全规范"。
2. **trace**：问"命令注入这条规范在哪实现" → 投出追溯表 → 强调 **"它只报图谱里真实存在的边，不瞎猜"**（这是它的可信底线，反复出现在每个 skill 的"不要做"里）。
3. **impact**：改一行代码，问"这次改动的安全影响" → 投出 high/medium/low 分级表 → 强调 **"不是看 git 行号，而是顺着调用链找间接影响"**。
4. **doc-sync**：投四级状态摘要 → 强调"涉及安全规范的文档，漂移阈值会自动收紧"。

---

## 六、一句话总结（收尾用）

> kdev-code-graph 把"安全规范"从一份静态文档，变成了图谱里**可查询、可追溯、可在变更时自动算影响**的活节点 —— 它不重造图谱引擎，只在成熟的 Understand-Anything 上叠了一层零侵入的安全视角。
