# kdev-review 评审Skill设计

> 设计日期：2026-05-09
> 目标：AI Coding试点项目经验蒸馏——评审决策指南体系

---

## 一、背景与目标

### 1.1 问题来源

从kdev-agents试点项目35个会话的分析中，发现：
- 评审事件80次（失败率约42%）
- AI倾向与用户真实需求存在系统性偏差
- 评审决策过程需要蒸馏成可复用规则

### 1.2 目标输出

形成一套完整的AI评审决策指南体系：
- **评审检查清单** - 每次评审必查维度
- **决策模板库** - 常见场景的推理模板
- **偏差预警规则** - AI倾向 vs 用户真实需求
- **评审流程skill** - 完整的SKILL.md定义

## 三、评分门控机制

### 3.1 评分维度

| 维度 | 评分标准 | 权重 |
|------|----------|------|
| **完整性** | 是否覆盖所有必要内容？有无缺失章节？ | 20% |
| **一致性** | 内部是否矛盾？术语是否统一？编号是否连续？ | 20% |
| **可实施性** | 是否足够具体可执行？还是抽象描述？ | 25% |
| **合理性** | 逻辑是否成立？方案是否可行？ | 20% |
| **文档质量** | 格式、表达、可读性 | 15% |

**综合分 = Σ(维度分 × 权重)**

### 3.2 阈值与门控

| 综合分 | 处理 |
|--------|------|
| **≥ 4.0** | ✅ 通过，输出给用户 |
| **3.0-3.9** | 🟡 轻修，AI自动修正后重新评审（最多2轮） |
| **< 3.0** | 🔴 重修，列出问题清单，等待改进后重新评审 |

**循环规则：**
- 每轮评审生成：评分表 + 问题清单 + 建议方向
- 低分时不输出给用户，AI按建议改进后触发新一轮
- 上限：最多循环3轮，仍未过阈值则输出"评审受阻"报告

### 3.3 评分表模板

```markdown
## 综合评分表

| 维度 | 本轮分数 | 上轮分数 | 变化 | 说明 |
|------|----------|----------|------|------|
| 完整性 | 4.2/5 | - | - | 缺少降级策略细节 |
| 一致性 | 3.8/5 | - | - | Layer2/Layer4职责重叠 |
| 可实施性 | 3.5/5 | - | - | 切换机制未定义 |
| 合理性 | 4.0/5 | - | - | 架构逻辑成立 |
| 文档质量 | 4.5/5 | - | - | 编号有跳节 |

**综合分：3.88/5 → 🟡 轻修**
```

## 四、偏差预警规则

从项目dev-notes中蒸馏的6条核心偏差：

### 4.1 规则清单

| # | 偏差类型 | AI倾向 | 真实依据 | 预警触发 |
|---|----------|--------|----------|----------|
| 1 | **结构信号 vs 行为信号** | 用数量/边界/阶段名判断 | 时长/干预/感受 | AI给出粒度定义时 |
| 2 | **硬规则 vs 弹性判断** | 提前收敛到形式化规则 | 保留弹性给真实场景 | AI给出方案时 |
| 3 | **数字阈值凭感觉写** | 猜测阈值（偏差3x） | 用户校准后的值 | AI主动给出数字时 |
| 4 | **跳过"是否要做"** | 立刻答"怎么做" | 先问痛点频率 | AI立刻进入方案对比时 |
| 5 | **技术栈惯性带入** | 用自己熟悉的工具 | 项目主栈生态 | AI建议工具/库时 |
| 6 | **过度形式化评审** | 列大量维度/复杂权重 | 用户判断负担ROI | 评分体系超过3维度时 |

### 4.2 自问模板

每条偏差对应的AI自问：

```
1. "这个定义是按结构切还是按行为切？行为边界在哪？"
2. "这是硬规则还是弹性？要不要弹性？边界在哪？"
3. "这是量级估算，误差可能3x起。用户校准过吗？"
4. "真实痛点频率多高？低频痛点不值得高频代价。"
5. "这和项目主技术栈匹配吗？"
6. "用户判断负担ROI划算吗？能精简吗？"
```

## 五、决策模板库

8个常见场景的推理模板：

| # | 模板名称 | 触发场景 | 推理步骤 |
|---|----------|----------|----------|
| T1 | 职责边界重叠判断 | 两个模块声称做同一件事 | 列职责→问顺序→区分视角 |
| T2 | 切换机制定义 | 多路径并列无选择逻辑 | 问参数/推断/混合→补充判断 |
| T3 | 外部依赖关系澄清 | 多框架管同一件事 | 列选项→明确选择→补充接口 |
| T4 | 工具链匹配检查 | AI建议工具有额外依赖 | 对照主栈→问运行时→统一 |
| T5 | 门控配置完整性 | 有门控/阈值配置 | 检查onFailure/timeout/retry |
| T6 | 数据流定义 | 触发→注入无数据流图 | 问时机/内容/规则/格式 |
| T7 | 降级策略细化 | 提到降级无具体内容 | 定义最小可行大纲 |
| T8 | 端到端示例补充 | 分层设计多缺流程 | 构建Happy Path→验证 |

## 六、评审流程

### 6.1 四步循环

```
Step 1: 预检查
  - 判断文档类型
  - 匹配评审维度清单
  - 快速扫描标题/编号/章节

Step 2: 深度评审
  - 逐维度打分
  - 每问题记录：位置+原文+分析+建议
  - 应用偏差预警和决策模板

Step 3: 门控判定
  if 综合分 >= 4.0: 通过输出
  elif 综合分 >= 3.0 and 循环<2: 自动修正→重评
  elif 循环<3: 等改进→重评
  else: 输出受阻报告

Step 4: 输出报告
  - 综合评分表
  - 问题清单（🔴🟡🟢优先级）
  - 改进建议
  - 版本对比（如有上轮）
```

### 6.2 报告格式

```markdown
# XX文档 评审报告 v{N}

> 评审日期：YYYY-MM-DD
> 评审轮次：第N轮
> 综合评分：{分数}/5 → {状态}

## 综合评分表
[5维度评分表]

## 问题清单

### 🔴 高优先
| # | 位置 | 问题 | 建议方向 |

### 🟡 中优先
...

### 🟢 低优先
...

## 本轮改进建议
[具体可执行的改进方向]

## 上轮→本轮对比
[分数变化、已修复问题、新增问题]
```

## 七、多模型评审机制

### 7.1 双维度配置

**两个独立维度：**

| 维度 | 说明 | 举例 |
|------|------|------|
| **模型** | 选择哪个AI执行 | opus, sonnet, qwen, gemini |
| **Skill** | 选择评审角色/关注点 | gstack-qa, gstack-design-review |

### 7.2 配置结构

```json
{
  "default_combination": [
    {"model": "opus", "skill": "gstack-qa"},
    {"model": "sonnet", "skill": "gstack-design-review"}
  ],
  "threshold": 4.0,
  "max_loops": 3,
  
  "models": {
    "opus": {"weight": 0.4},
    "sonnet": {"weight": 0.3},
    "qwen": {"weight": 0.2}
  },
  
  "skills": {
    "gstack-qa": {
      "focus": "测试覆盖、边界情况、质量保障",
      "dimensions": ["可实施性"],
      "output_format": "评审发现清单"
    },
    "gstack-design-review": {
      "focus": "UX、原型质量、用户体验",
      "dimensions": ["合理性", "文档质量"],
      "output_format": "评审发现清单"
    },
    "gstack-plan-eng-review": {
      "focus": "需求清晰度、技术方案合理性",
      "dimensions": ["完整性", "一致性"],
      "output_format": "评审发现清单"
    }
  }
}
```

### 7.3 执行流程

```
1. 遍历 default_combination
2. 每个组合：模型 + skill → 并行执行
3. 调用模型，传入skill的prompt模板
4. 收集结果到 .review/artifacts/<model>-<timestamp>.md
5. 汇总各skill产出，计算综合分
```

### 7.4 综合报告结构

```markdown
# XX文档 多模型评审综合报告

> 参与模型+skill：opus+gstack-qa, sonnet+gstack-design-review
> 综合评分：4.1/5

## 各组合评分
| 组合 | 分数 | 关注点 |
|------|------|--------|
| opus+gstack-qa | 4.0/5 | 测试覆盖、质量保障 |
| sonnet+gstack-design-review | 4.2/5 | UX、用户体验 |

## 共识优点
[所有组合认可的内容]

## 共识问题
[所有组合指出的问题]

## 各组合独特视角
### opus+gstack-qa 建议
### sonnet+gstack-design-review 建议

## 冲突处理
[分歧标注 + 裁决理由]

## 行动清单
| # | 优先级 | 行动项 | 来源 |
```

## 八、触发与使用

### 8.1 触发方式

```bash
# 单模型评审（默认）
/review "评审XX设计文档"

# 多模型深度评审
/review-deep "深度评审XX设计"

# 指定组合
/review-deep --models opus,qwen --skills gstack-qa "评审XX"
```

### 8.2 适用场景

| 场景 | 推荐方式 |
|------|----------|
| 配置文件、小变更 | 单模型快速评审 |
| 重要设计文档 | 多模型深度评审 |
| 安全相关代码 | opus + kdev-secure-coding |
| UX/原型 | gemini + gstack-design-review |
