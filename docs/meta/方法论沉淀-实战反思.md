## 1. 先核代码再信设计（R-009）—— 本项目最高频、最贵的教训

**口诀：`roadmap/spec 状态 = grep first, write second`**——先核代码 / 运行时实际状态，再信设计文档的字面表述；不要把"文档写了 ✅"当成"已验证"。

背景：本项目连续两步（main-53、main-55）栽在同一个坑——信 roadmap 里已有的 `✅已落地` 字样，不 grep 代码实际核实，就直接把另一条目也标成 `✅已落地`。main-53 自评刚记过这毛病，main-55 同一会话紧接着又犯，说明"记住了"和"内化成动作"之间还差最后一步：**没有把"grep 前置"嵌入写状态字段的具体操作流程**，光靠自评/记忆是不够的，必须变成检查清单式的硬动作。

**具体检查点**：

- 标 `✅已落地` 前：grep 实现文件（`.py`/`.ts` 等）确认对应接口/结构**实际存在**，不是"应该存在"。
- 涉及数据文件结构（如 flow-state、events.jsonl）：不要凭 schema 文档推测，`find` + `python3 -c "import json; ..."` 之类的方式**实际读一遍文件**。
- 如果 grep 结果与文档声明不符 → **先纠正已有条目，再写新条目**，不要在错误基础上继续叠加、靠记忆传播既有错误。

**跨项目适用性**：这不是本项目特有的坑，是任何"用文档字面状态驱动后续决策"的工作流的通病——文档天然滞后于代码，字面信任的复发成本会随项目变大而指数上升。

> 参见 `docs/framework/踩坑索引.md` 中 G-008 / G-009 / G-010 / G-011 及多条时间戳 G 条目——这条教训在本项目以活体形式反复出现，本文只提炼可操作规则，具体案例细节留在踩坑索引里，不在此复述。

## 2. 长会话 / 并发会话下，依赖"实时状态"的机制会静默失效（R-001 + R-005）

**教训**：任何设计成"实时累积数据"或"运行时读最新版本"的机制，在两种场景下都可能悄悄失效而不报错：

- **长会话、跨轮次、任务流连续型会话**（R-001）：实测 5/27 一个长会话 6/8 工作单元未落 Step，under-reporting 率达 75%。对照组是同一天的两个"单任务集中型"会话（一个 writing-plans 输出一个 plan），完全没有 under-reporting。根因不是模型能力问题，而是**特定会话形态下触发机制失效**——长会话里用户问题流连续渐进、没有"自然停顿"的视觉信号，模型的注意力被持续讨论吸住，容易忘记在每个隐性停顿点触发记录动作。
- **并发会话下的 skill/配置中途升级**（R-005）：一个会话升级了 SKILL.md（加新机制），另一个已经运行了几小时的并发会话感知不到——因为 skill 加载是 **SessionStart 时一次性快照**，运行期间磁盘文件被改，正在跑的会话 context 里仍是旧版，没有热更新机制。实测后果：并发会话继续用旧路径工作，状态文件出现 drift（counter 落后于 current_step），需要事后 reconcile。

**可操作做法**：

- 设计任何"持续累积 / 实时记录"型机制时，默认假设它在长会话下会漏记——用外部信号兜底（如 commit 后自动留 WARN 提醒文件、累积产出超阈值强提醒），不要只靠"模型记得要做"。
- 升级中途会被读取的配置/skill 文件后，**主动提醒所有并发会话重启或刷新**，或加一个轻量的版本自检（如 SessionStart hook 比对文件 SHA 跟会话启动时缓存的 SHA），而不是假设所有会话都跑的是最新版。
- 两者是同一类根问题的不同侧面：**任何要求"运行时状态与磁盘/会话外部状态保持同步"的设计，天然有 staleness 风险**，风险随会话存活时长和并发数上升。

## 3. 测试绿 ≠ 契约正确，spec 写的 ≠ 实际发生的（R-006 + R-007，呼应 R-009）

**教训**：测试通过只证明"测试自己内部自洽"，不证明"测试断言的东西是系统的真实契约"；同理，spec 文档写的行为规则，不代表 subagent/系统实际会照做。两者都是"字面文本"和"运行时真相"脱钩的变体，跟第 1 节 R-009 是同一根问题的不同战场。

- **字段格式漂移（R-006）**：`since_step_id` 字段，recorder 模板写入时带 `"Step main-N"` 前缀，e2e 测试断言用的却是无前缀的 `"main-N"`——两边测试都能各自通过（因为消费函数对字符串格式不做校验），但语义已经在文档和代码之间悄悄分叉。这种 drift 在功能层面暂时不出错，但会在未来"跨条目反查/索引"时集中爆雷。
- **spec 与实际行为脱钩（R-007）**：spec 明文写"drift 时必须 REJECT，不得静默修复"，但 subagent 连续两次实际 dispatch 都选择了 DONE_WITH_CONCERNS + 自行 reconcile，而不是按 spec 走 REJECT。这不是 bug，是 subagent 基于实际开销做出的合理选择——但这意味着**spec 本身已经跟不上真实运行的共识**，没人在遵守写着的规则。

**可操作做法**：

- 定期比对"spec/文档写的行为契约"和"实际运行日志里发生的行为"，出现连续偏离时，第一反应不是"重申 spec"，而是**问 spec 是不是该改**——subagent/实现连续选择同一条"违规"路径，往往是它比原 spec 更贴近真实约束。
- 涉及跨模块共享的字段格式（如 ID 是否带前缀），要有单一权威来源（如约定"跟对齐的另一个字段格式一致"），并让写入方和断言方引用同一个格式函数，而不是各自硬编码字符串形式。
- 测试通过≠没问题：测试断言本身也可能是过时/错误的一方，评审时要连带检查"测试断言的语义"是否还对得上系统当前的真实契约（这跟 R-009 揭示的"测试绿但功能死"是同一主题在不同层面的复现）。

## 4. Fixture / 边界场景要用稳定锚，不要用会漂移的相对值（R-002 + R-003）

**教训**：两条独立踩坑，共同指向"测试/工具对边界情况的处理方式经不起时间或环境变化"：

- **Time-bomb 测试（R-002，已修，commit `2dc7eeb`）**：一条测试用硬编码日期（`日期：2026-05-15`）配合"mtime 减 10 天"这类相对计算来构造 fixture，运行时的 `today` 一旦漂过某个临界点（本例是 2026-05-26+），fixture 里的条目会被日期过滤逻辑整体滤空，测试从"验证功能"变成"验证过去某个时间点"，且失败现象和真正想测的逻辑毫无关系，排查成本高。修法：改成基于 `today` 的相对计算（`today - 5 天`），保证测试构造的时间关系跟运行时刻无关、恒定成立。
- **无 commit 仓库的 fallback（R-003，已修，commit `77ac2b2`）**：`compute_branch_slug()` 只依赖 `git rev-parse --abbrev-ref HEAD`，在"`git init` 但还没有任何 commit"的仓库里该命令失败，直接 fallback 到字面值 `"unknown"`，污染 state 目录（生成 `step-counter-unknown.txt` 垃圾文件）。但实际上 `.git/HEAD` 文件（`ref: refs/heads/main`）在没有 commit 时依然存在、依然可读，是更稳定的锚点，`rev-parse` 只是该场景下失效的间接查询方式。

**可操作做法**：

- 写涉及日期/时间的测试 fixture，禁止硬编码绝对日期；一律用相对 `today ± N 天` 计算，保证测试在任意未来运行时刻语义不变。
- 设计 fallback 路径前，先想清楚"主查询方式失败"和"数据真的不存在"是两回事——很多命令式查询（如 `git rev-parse`）在边界场景（无 commit、detached HEAD 等）会失败，但底层数据（如 `.git/HEAD` 文件）往往仍在，fallback 应该尝试读更底层的稳定锚，而不是一步到位退到语义上的"unknown"。

## 5. 纵向切片的增量边界该由需求/架构层定，不该下沉给下游实现者（R-008）

**status：设计共识，待 req-architect 员工角色建起来后落地**（未决）。

**教训**：dev-engineer-plan（下游实现角色）在缺乏价值/架构视角的情况下被要求自己判断"怎么切增量"，结果把纯技术分层（T0-T4 这种实现内部顺序）误当成面向用户价值的交付增量——因为它手上只有实现细节，没有全局视角，只能照抄考题给的分区当增量边界。

**可操作做法**：

- **两级规划，各管一级**：上游（需求/架构层）负责回答"切成哪几块"——把需求切成 N 个**可独立完成 e2e 验收**的纵向切片，每片带验收标准；下游（实现层）负责回答"这一块怎么搭"——在单个已定边界的增量内部排任务序、定 TDD 步骤、改哪些文件。
- **交接契约**：上游产出（spec/plan/prototype 等）里必须显式列出"增量 1…N + 各自 e2e 验收标准"，边界在设计阶段交棒时就定死，不允许实现方在编码途中现场切。
- **增量的判定标准要写成可复用的定义，两层角色共用**：增量 = 能独立通过 e2e 验收的纵向切片。上游拿它切分，下游拿它校验"收到的到底是一个增量还是被打包的多个增量"；某个候选切片自己过不了 e2e，就不是一个合格增量，只是子任务，需要重新归并或再拆。
- **裸任务兜底**：当任务绕过上游、直接派给下游（没有需求/架构角色介入,如考题式任务）时，下游按同一套增量定义自行判定，不是"没人切就随便切"。
