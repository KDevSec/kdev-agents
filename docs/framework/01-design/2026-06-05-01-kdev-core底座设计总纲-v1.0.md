## 0. 定位（一句话）

底座 = **从现有 flows 抽共性的 lean 编排底座**（Python/markdown 原生，**借 OMC 范本不 fork**）；当前 **L2 协同**，**完整框架、渐进实现**。底座**只记账 + 流转，不做判断本身**（评审/E2E/决策由 flow 侧 AI 或人执行，结果喂回底座）。

---

## 1. 路线（Q-007）：③ 抽共性渐进

| 选项 | 否决理由 |
|---|---|
| ① 采用 OMC 当底座 | TS 3.3MB 外来运行时，fork+维护+license，跟 Python/markdown 哲学冲突，违从低起步 |
| ② 自建完整 kdev-core | 无视已有实现、从 v0.1 设计重画引擎，工作量大 |
| **③ 抽共性渐进（采纳）** | **从 design-flow/coding-flow 抽 R1–R7 公共层；照 OMC 范本但不 fork；L3+/跨IDE 再重评采用 OMC** |

---

## 2. 现实基础（C-1 摘要）

`design-flow`（需求架构师雏形）/`coding-flow`（开发工程师雏形）**不是原子能力，是数字员工雏形**，各自造了 flow-state/节点机/闸门/Auto Mode，**无共同底座**。底座 = 抽出它们的 **R1–R7 公共编排层**：

| | 职责 | design-flow | coding-flow |
|---|---|---|---|
| R1 | 状态持久化+resume | ✅ `lib/flow_state.py`（金种子）| ❌ 无 |
| R2 | 多节点状态机 | 4 段 | 13 节点（prompt）|
| R3 | 评审闸门 | review×3 iter | Gate-A/B/C + E2E |
| R4 | 自主模式 | ❌ | ✅ Auto Mode |
| R5/R6/R7 | 派单/产物/调度 | 轻 | 二档+三件套+串能力 |

## 4. 核心层设计（实施重点）

### 4.1 R1 状态 — flow-state（泛化 `flow_state.py` + 借 OMC）

**Schema**（`.kdev/flows/<flow>/<slug>/flow-state.json`）：
```json
{
  "flow": "design-flow", "slug": "user-auth", "display_name": "...",
  "status": "in_progress",        // in_progress | completed | aborted
  "active": true,                 // [OMC] stale guard：crash 写 false 而非删，resume 据此判
  "current_node": "n3-prototype", // 节点 id（字符串）
  "created_at": "...", "updated_at": "...",
  "config": { "review_mode": "ai", "auto_mode": false },
  "history": [ /* GateResult 流水，见 4.3 */ ],
  "_meta": { "written_at": "...", "step_id": "..." }   // [OMC] 写时注入、读时剥离
}
```
**API**（`kdev_core/flow_state.py`）：`init / read / write / advance / record_gate`。
**实现要点**：tempfile + `os.replace` 原子写（搬金种子）；不存在/损坏抛 `FlowStateError`；**Python 函数非 class**（OMC 自己把 StateManager 标了 @deprecated 转函数，验证此方向）。

### 4.2 R2 节点机 — 通用 n 节点状态机（借 OMC Team Pipeline）

各 flow 声明**节点表**（不写死段数，design 4 / coding 13）：
```yaml
nodes:
  - { id: n1-sr, name: SR 需求, kind: action, gate: g-sr }
  - { id: n4-design, name: 概要+详设, kind: action, gate: g-design }
# kind: action（产出后过 gate）| gate（纯判断点，无产物）
```
**advance 三段式**（[OMC] `transitions.ts`）：
1. **邻接表**查 `ALLOWED[from]→[to]` 合法性
2. **守卫** `guard(state)→str|null`（null 过 / 非 null 拒因，**与边解耦**）
3. **不可变变更 + 原子写 + 追加 `phase_history`**

**有界回流 + 幂等豁免**：gate 回流（gate→同 action）须**显式标"回流"= retry++**（非幂等重入）；retry 溢出 → 强制 advance 到 terminal-fail，禁无限回流。

### 4.3 R3 三类 gate — 统一抽象（核心难点，已验证覆盖两 flow）

| kind | 谁判（默认/可编排） | verdict | 不过 |
|---|---|---|---|
| **review** 评审重试 | ai（默认）/ both / human | PASS/FAIL | 改→重试(max N) |
| **decision** 路线判断 | **ai（默认）/ human（编排可接管）** | 分支选择 | 选分支即流转 |
| **acceptance** 验收 | test（跑 E2E）/ +human 复核 | PASS/FAIL | 回流 fix |

**GateResult**（flow 执行完喂回底座）：
```json
{ "gate": "g-sr", "kind": "review", "node": "n1-sr",
  "request_id": "uuid-…",   // [OMC] approval 锁：approval 须携匹配 uuid，防跨轮串扰
  "iter": 2, "verdict": "PASS", "by": "ai",
  "issues": [], "revisions": [], "ts": "..." }
```
**解析**（`record_gate` 三类统一）：review/acceptance → PASS 走 on_pass / FAIL 且 iter<max 回流 / **iter≥max → escalate 转人工 BLOCKED**（[OMC 避坑] 不学它 force-accept 退化接受）；decision → 按 verdict 选 branches[choice].next。
**双重计数**（[OMC]）：本 gate 重试 + 本 flow 总 gate 调用。
**结果用 structured JSON verdict**（[OMC 避坑] 不学它正则扫文本）。

### 4.4 谁来判 + Auto Mode 停靠（[OMC] pause flag + awaiting_human TTL）

「谁来判」是**编排可覆盖旋钮**（默认 AI，人按需接管）。**Auto Mode 只在 `decider:human` 的 gate 停靠**（写 `awaiting_human`+TTL：review 超时自动续 / decision 等人）；其余 AI 自己过。哪里停由编排决定。

## 5. phase / node 两层 + kdev-memory 对接

| 维度 | 是什么 | 何时有 | 谁拥有 |
|---|---|---|---|
| **`phase`（普适）** | 工作性质（10 值）| 恒在 | kdev-memory 内置 |
| **`node`（编排层）** | 流程位置 | **编排存在时才有** | 底座 node-table（唯一来源）|

**node 三来源优先级**：① 底座 node-table 的 `current_node`（kdev-core+flow 在跑）> ② 项目 `.kdev/memory/config.yaml` > ③ 无 node（只 phase 兜底）。**无 kdev-memory 内置通用 node**（已核实未实现；node 一直是编排/配置驱动）。

**node 标签 + 跨层联查**：底座把 `current_node` 暴露给 kdev-memory，记忆条目据此打 node 标签 → "D2 节点的所有信号" 可按 node + scope 联查。**底座（kdev-core）自身不依赖 kdev-memory 运行时**（可单装）；但记忆层是三层模型的一部分，**kdev-staff 硬依赖 kdev-memory**（详见 [记忆架构 §3/§5](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md)）。

## 6. 编排可配 — SOP 三层覆盖

R2 节点表 + R3 gates = **一个 flow 的 SOP，可编排**：

| 层 | 谁编排 | 位置 | 时机 |
|---|---|---|---|
| L0 默认 | flow 作者 | flow 内 `node-table.yml` | ship |
| L1 用户 | 用户 | `.kdev/flows/<flow>/orchestration.yml` | 安装/项目初始化 |
| L2 自主（未来 L4） | 数字员工自己 | 同上 | 运行/自演进 |

底座加载 **有效 SOP = merge(L0, L1, [L2])**。每个 flow 配**编排指南**（能改什么/怎么改/风险）。L2 自主编排 = 持续升级飞轮反哺 SOP 本身，**当前 L2 不实施，预留挂钩**（`source: default|user|self`，self 变更需人确认）。

## 7. OMC 吸取（Top 8 好经验 + 2 避坑，落实 Q-007 借范本）

**借**：① R2 邻接表+守卫分离 ② 有界回流+强制failed+幂等豁免 ③ R3 request_id 锁 ④ R1 active/_meta stale guard ⑤ 双重计数+awaiting_human TTL ⑥ mode=JSON状态文件+TTL+Stop circuit-breaker（防死循环）⑦ R5 tier env override+provider inherit ⑧ HUD statusline stdin 结构化。
