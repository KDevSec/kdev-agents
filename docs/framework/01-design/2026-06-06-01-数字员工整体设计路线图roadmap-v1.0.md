### 1.5.4 排序考量：feature-first 存储 + events.jsonl 提前（HUD 与 P-A 的共同前置 · ✅ Q-013 已拍板）

**决策**：把 **kdev-core 的 feature-first 存储迁移 + events.jsonl 实现**（原列阶段3 框架层）**提前为一个独立底座块（P-Core-FF），置于 P-A 之前**。一次落地解锁三件：① P-A 接最终契约不返工 ② kdev-hud 数据源就绪、可越过阶段3 其余早做 ③ 阶段3 框架层提前消化一部分。

### 1.5.6 编排执行层选型：先 B（主会话派单）后 A（Workflow 引擎）（决策 · 2026-06-12）

### 4.2 边界（保持骨架薄）

- **记忆**：flat 单 scope（[记忆架构 §4](./2026-06-10-05-KDev数字员工-记忆底座-合稿-v1.0.md) flat 态），不上 scope/JSONL。
- **评审**：开发工程师 solo，gate 由 **AI 内联判**（R3 review-kind，decider:ai 默认，跑 UED §10 自检清单）——**不引入评审专家**（阶段3）、不引入跨员工协作。**竞赛分由外部 benchmark grader 出**，不是我们的 flow。
- **dogfood 任务**：**UED 6.0 改造 benchmark**（RuoYi-Vue3-FastAPI 视觉改造）。题目 + materials 在 `测试题目/`：[EXAM-PROMPT.md](../../../测试题目/EXAM-PROMPT.md) + `01-pic/`（3 原型）+ `02-ued6.0/`（UED 硬约束 + design-tokens + tailwind preset + css）。后端不允许改，纯前端改造。
- **两 pass**（用户定）：
  - **Pass 1 验底座**：砍成小切片（T0 全局主题 + T1 登录页，或 T0 + 用户管理标杆页），**不追分**，目的把 kdev-core R1/R2/R3 + 开发工程师端到端抠通。
  - **Pass 2 真 benchmark**：底座稳后，**整题原样不改**跑全量（T0–T4 全菜单），拿可比竞赛分（题目要求"原样发给每个参赛智能体"，砍题即不可比）。
- **TDD 适配**：视觉改造主体**非 TDD 可驱动** → R3 acceptance 走 **build 通过 + UED §10 自检 + 视觉 diff + 功能冒烟**（登录→首页→用户管理 CRUD）；coding-flow 接底座时需一条「前端/视觉」验证支路，不假设处处 TDD 红绿。

### 4.3 验收标准（tracer-bullet 级，不镀金）

**A. 底座机制类**（证明 R1/R2/R3 真在驱动，不是 prompt 演戏）
1. **R1 真驱动**：flow-state.json 全程结构化记 `current_node` + `history[GateResult]`；中途 kill → `resume` 从上次 node 正确续（非从头）。
2. **R2 真流转**：13 节点按邻接表推进；至少一次 gate 不过的**有界回流**（retry++ 记状态、可查），有上限不死循环。
3. **R3 三类各跑通一次**：review(代码/质量) · decision(Gate-A/B 选分支) · acceptance(E2E) 各至少触发一次正确流转；其中至少一次 **FAIL→修→重评**闭环；GateResult 是结构化 JSON（带 request_id/iter/verdict），下游不靠正则解析。
4. **escalate 不 force-accept**：构造一次评审超限(≥max)，验证转 BLOCKED 升人工、非自动降级接受。

**B. 端到端交付类**（证明它是能干活的员工，不只机制 demo）

5. **真任务端到端**：环境对齐→实施计划→（视觉改造主体非 TDD）→**build 通过 + 功能冒烟**（登录→首页→用户管理 CRUD）跑完，产出可运行前端 + RuoYi 既有功能不破。
6. **L2 画像成立**：员工自跑完整 flow，主控只在 gate 确认/决策，不替员工写代码。

**C. 工程质量类**

7. **底座 R1/R2/R3 单测齐全且过** + 接 coding-flow 后的集成 smoke（参考 X3 full-lifecycle+resume 模式）。

**UED 合规（已确认启用，dogfood 是前端任务）**：**UED §10 自检清单**作为一道 R3 review 闸——禁裸 hex / 8px 网格 / 按钮圆角 2px / 表头 44px / chart palette 定序 / 字体白名单 等，**客观可机检**（grep + 人工抽查）。Pass2 全量时还叠加题目自带的"遍历全菜单逐页截图"评审。

**明确非目标（防镀金）**：不要评审专家(阶段3)、scope/JSONL 记忆(阶段2)、CQO(阶段4)、多员工协作、HUD 完整体(阶段3)；dogfood 求打通不求规模。

## 6. X3 参考映射（reference-only）

X3 最大价值 = **反向验证 kdev-core 设计方向**：X3 赶工自造 runtime 踩的坑，恰好是 lean 底座要修的点（同一问题两次相遇）。

**搬（成熟设计，照搬当参考）**：12 评审节点表（R/D/T/F + 阻断/告警 + D2 并行）· 10 评审员 checklist（代码评审最细）· 冲突仲裁 3 步 SOP · COMPLETE.md 聚合 manifest · 轻组长 4 职责分离 · agent frontmatter + lint 规则 · smoke 测试模式 · kdev-start-feature 入口结构。

**别学（赶工/技术债/与 canonical 冲突）**：

| X3 做法 | 为何别学 / 底座正解 |
|---|---|
| 状态用 markdown + regex + 全量覆写 + class | R1 必须 **JSON + 原子写 + 函数非 class**（并发会撞）|
| **gate 逻辑埋 prompt 文本**，retry 计数只在 events.log 文字 | R3 必须**结构化 GateResult**，否则没法程序化查重试轮次、resume 全靠正则猜 |
| events.log 单文件 + tab 分隔 + 全量读 | 操作层必须 **per-scope JSONL**（跨 feature 无法 grep、加字段破格式）|
| `AGENT_GROUP`/`AGENT_TO_LEAD` 双份硬编码路由表 | **单一数据源**（已有漂移风险）|
| on-blocked hook 触发条件过宽 + stdout 注入 hack | hook framework **标准化 Blocked 信号** |
| handoffs 只有 COMPLETE.md 无中间 checkpoint | 中间节点失败靠文本推断 → 需结构化 checkpoint |

| 风险 | 缓解 |
