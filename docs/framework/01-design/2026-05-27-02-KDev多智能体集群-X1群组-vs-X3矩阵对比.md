## 0. 一句话目标

**X1（群组原版）**：组长是路由器（PgM），所有跨组通信都过组长 + 主控员。心智模型严格层级。
**X3（矩阵式）**：组长是顾问（PM），通信走扁平，组长只在「阶段聚合 + 异常应急 + 标准维护」激活。

## 2. 方案 X1：群组原版（严格层级）

### 2.1 架构图

```
            ┌────────────────┐
            │    主控员       │
            └───────┬────────┘
       ┌──────┬─────┴─────┬──────┐
       ▼      ▼            ▼      ▼
   ┌──────┐ ┌──────┐  ┌──────┐ ┌──────┐
   │需求  │ │开发  │  │测试  │ │审查  │
   │组长  │ │组长  │  │组长  │ │组长  │
   └──┬───┘ └──┬───┘  └──┬───┘ └──┬───┘
      │        │         │        │
   ┌──┴──┐ ┌──┴──┐  ┌──┴──┐ ┌──┴──┐
   │ 5员 │ │ 6员 │  │ 4员 │ │10员 │
   └─────┘ └─────┘  └─────┘ └─────┘

  组长 = PgM（路由器）——所有通信必经组长
```

### 2.2 通信硬规

🔴 **铁规 1**：用户只跟主控员对话
🔴 **铁规 2**：主控员只跟 4 个组长对话（**不直接派组员**）
🔴 **铁规 3**：组长只跟本组组员对话
🔴 **铁规 4**：跨组通信必须经"组长→主控员→另一组长"三跳
🔴 **铁规 5**：组员之间不互通

### 2.3 通信示例：TDD 想要代码评审

```
TDD 实现员 → 开发组长
            ↓
        开发组长 → 主控员
                  ↓
              主控员 → 审查组长
                        ↓
                    审查组长 → 代码评审员 + 质量评审员（并行）
                              ↓
                          回流：评审员 → 审查组长 → 主控员 → 开发组长 → TDD 实现员
```

**跳数：7 跳**（往返计 14 跳）

### 2.4 优劣

| 优势 | 劣势 |
|---|---|
| ✅ 跟用户最初心智模型 100% 一致 | ❌ 跨组通信 7 跳延迟 |
| ✅ 审查标准统一（审查组长统筹）| ❌ token 成本高（组长每通信都过）|
| ✅ 主控员复杂度低（4 组长接口）| ❌ 组长挂 = 整组瘫痪（单点故障）|
| ✅ 审计 / 汇报链清晰 | ❌ 审查组长成瓶颈（并发评审排队）|

## 3. 方案 X3：矩阵式（轻组长 / 组长当顾问）

### 3.1 架构图

```
                ┌────────────────┐
                │    主控员       │
                └───────┬────────┘
       ┌───────────┬────┴────┬───────────┐
       │           │         │           │
   ┌───▼──────┐ ┌──▼─────┐ ┌─▼──────┐ ┌─▼──────┐
   │需求组长  │ │开发组长│ │测试组长│ │审查组长│  ← 不路由
   │（顾问）  │ │（顾问）│ │（顾问）│ │（meta）│     只做 meta 4 件事
   └────┬─────┘ └───┬────┘ └───┬────┘ └───┬────┘
        │           │          │          │
        └──━━━━━ 后台监听 events.log ━━━━━┛
        │           │          │
        │ 阶段聚合 / blocked 事件介入
        ▼           ▼          ▼

   主控员直接派工作 agent（快路径）↓
   ┌──────────────────────────────────────────────┐
   │ 需求组 5员 │ 开发组 6员 │ 测试组 4员 │评审池10│
   └──────────────────────────────────────────────┘
       ↑                                      ↑
       └────── 工作 agent 直接派评审员 ────────┘
              （不经任何组长）

  组长 = PM（顾问）——通信走扁平
```

### 3.2 组长的 4 meta 职责（不做路由）

| 职责 | 干什么 | 何时激活 |
|---|---|---|
| **① 监督** | 后台监听 events.log，对本组所有 agent 的产物抽查 | 后台被动 |
| **② 标准** | 维护本组的 system prompt 模板和评审标准（落在 `plugins/<group>-flow/standards/`）| 静态资产，不参与运行时 |
| **③ 聚合** | 阶段完成时出本组总结报告 | 每阶段 1 次 |
| **④ 应急** | 组员 BLOCKED 时介入决策（hook 自动触发，见 §3.5）| 异常时 |

### 3.3 通信规则

| 规则 | 实现 |
|---|---|
| 用户 ↔ 主控员 | 当前 session 对话 |
| 主控员派工作 agent | `Agent({subagent_type, run_in_background})` 直接派（**快路径**）|
| 主控员调组长 | `Agent({subagent_type: "需求组长"})` 仅在阶段聚合 / 异常时（**慢路径**）|
| 工作 agent 派评审员 | `Agent({subagent_type: "代码评审员"})` 直接派 |
| 工作 agent 呼救组长 | events.log 写 `blocked` 事件 → hook 触发组长介入（见 §3.5）|
| 跨组通信 | `.kdev/handoffs/` 文件总线（同 X2）|

### 3.4 3 个典型场景

#### A：日常派单 — TDD 想要代码评审

```
TDD 实现员 ──┬─→ Agent("代码评审员")  ┐
             └─→ Agent("质量评审员")   │ 并行
                                       ┘
            ←── 评审结果返回 ──
            ↓
        TDD 实现员决定继续 / 修复

（开发组长在背景静默监听 events.log，
 不参与通信）
```
**跳数：2 跳**

#### B：阶段完成 — reqs 全部跑完

```
方案设计师（reqs 最后一员）→ 写 .kdev/handoffs/reqs/COMPLETE.md
                            ↓
            主控员监测到 COMPLETE → 调 需求组长 做阶段聚合
                            ↓
                  需求组长 → 读 events.log（reqs 全部条目）
                          → 读 .kdev/handoffs/reqs/*
                          → 出 reqs-summary.md
                          → 报告主控员
                            ↓
            主控员 → 决定进 dev 阶段（或要求 reqs 返工）
```

#### C：异常处理 — TDD 实现员 BLOCKED

```
TDD 实现员 → 连续 3 次失败 → events.log 写 "blocked" 事件
                            ↓
                blocked hook 触发 → 自动派 Agent("开发组长")
                            ↓
                    开发组长 → 评估根因 → 决策：
                        ├── 重派 TDD 实现员（升 opus）
                        ├── 改派 实施计划师 重做计划
                        ├── 通知主控员（需要用户介入）
                        └── 标污染样本 + 继续
```

### 3.5 已拍板细节决策（2026-05-27）

| # | 决策点 | 拍板结果 |
|---|---|---|
| 1 | 组长激活的「阶段节点」 | ✅ **4 个**：reqs/dev/test 各组完成时本组长聚合；最终审查组长终审 |
| 2 | 组员 BLOCKED 呼叫组长方式 | ✅ **方案 b（hook 自动）**：events.log 写 `blocked` 事件，hook 监听自动派组长。**理由**：完全声明式，agent prompt 不污染 |
| 3 | 组长 meta 标准存储位置 | ✅ `plugins/<group>-flow/standards/`（评审 checklist + system prompt 模板）|
| 4 | 主控员快/慢路径判断逻辑 | ✅ **先按建议规则**（单 step→快，阶段结束/blocked→慢），**详细评审节点接入清单见 §5** |

### 3.6 优劣

| 优势 | 劣势 |
|---|---|
| ✅ 跨组通信 2 跳（同 X2 一样快）| ❌ 组长职责需要重新定义（PM 不是 PgM）|
| ✅ 组长存在，心智模型基本一致（95%）| ❌ 主控员要懂何时走快路径 vs 慢路径（多一层判断）|
| ✅ token 成本低（组长仅阶段+异常激活）| ❌ 一定程度违背"严格层级"的纯粹性 |
| ✅ 失败容错好（组长不在通信关键路径）| |
| ✅ 审查并行性好（评审池共享）| |
| ✅ 审查标准统一（组长 meta 维护）| |
| ✅ 异常处理自动化（hook 触发）| |

### 5.5 评审接入规则

🔴 **阻断（必过）**：评审不过则当前 step 不能 COMPLETE，回流到该工作 agent 修复，循环最多 3 次，第 4 次自动 BLOCKED 触发组长介入（§3.5 决策 #2 b 方案）

🟡 **告警（可过）**：评审不过写到 events.log + state.md 的 warnings 字段，但不阻断流转，主控员看到告警可以提示用户

⚪ **抽查**：评审员主动从 handoffs 文件读取产物做合规性检查，不阻断主流程

### 5.6 评审员调用规约（X3 扁平）

```
工作 agent 在 step 产物完成后，立即在 system prompt 硬规里执行：

  if (产物属于 R2/R4/R5/D2/D4/T1/F1/F2 任一节点):
      Agent({
        subagent_type: "<对应评审员>",
        prompt: "评审 .kdev/handoffs/<group>/<artifact>，
                 评审标准见 plugins/<group>-flow/standards/<reviewer>.md",
        run_in_background: false   // 阻断节点必须等评审返回
      })

      if (评审不过):
          if (重试次数 < 3): 修复后重派评审
          else: events.log 写 'blocked' → hook 触发组长
```

- [ ] 实施计划完成（D1）是阻断还是告警？

### 8.6 实测对比指标

| # | 指标 | 度量方式 | 期望差异 |
|---|---|---|---|
| 1 | 端到端耗时 | events.log 首末时间戳 | X3 < X1（少 7 跳）|
| 2 | 总 token 消耗 | 每 agent 自报 + 主控总账 | X3 < X1（组长 context 省）|
