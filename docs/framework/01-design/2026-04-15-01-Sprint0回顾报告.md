## 五、KDev 技术形态建议

### 5.1 决策：Skill + 薄 CLI 双形态并存

基于 Sprint 0 实战观察，**纯 Skill 包**方案不足：

| 能力 | 纯 Skill 方案局限 | CLI 补足 |
|------|-------------------|---------|
| state machine 编排 | Skill 间无法共享状态 / 跨会话持久化 | CLI 维护 `.kdev/state.md` + session-state |
| 强制节点 gate（M3） | Skill 无法拦截 LLM"自行跳过"的行为 | CLI 包一层"必须回答问题才能 next" |
| 原子能力调度（M1） | Skill 是被 LLM 调用的单元，不主导流程 | CLI 主导原子节点拼装 |
| Token 预算监控 | Skill 看不到 session 上下文 | CLI 挂 Hook（参考 OMC）采集 |
| 跨项目 memory（Level 5） | Skill 全局存储能力弱 | CLI 写 `~/.kdev/` |

### 5.2 形态细节

```
KDev 技术栈（Sprint 1 目标）
├── kdev CLI（Node.js + TypeScript）
│   ├── 命令：kdev start / ir / sr / prototype / review / ...（14 个）
│   ├── 状态机：读写 .kdev/state.md
│   ├── 记录自动化：决策 → questions-log / 踩坑 → gotchas / 步骤 → journal
│   └── 原子节点调度：组合 skill 原子能力 + 本地模板
├── kdev Skills（.claude/skills/）
│   ├── 仅承载"人受众节点"的交互（问答 / 核对 UX）
│   └── 本身不做编排，由 CLI 调用
└── 外部 skill 原子能力库（.kdev/skill-atoms/）
    ├── 从 gstack / bmad / superpowers 抽取的原子封装
    └── 纯函数接口：输入 → 输出，不含 onboarding / 次要节点
```

### 5.3 不引入的东西（Sprint 1 明确排除）

- ❌ OMC / Compound 整体集成 — 先手写 Hook，验证后再考虑
- ❌ knowledge-graph / learnings.jsonl 自动化（Memory Level 1/2）— 先跑通 Level 3
- ❌ kdev 本身的"skill 包发布"给 Claude Code Marketplace — 自用稳定前不做

## 六、Sprint 1 第一批要实现的 KDev 功能

> 以"封口 P0 痛点 + 产出最小可用 CLI"为目标。每项都对应已有证据链。

### 6.1 必交付（Sprint 1 出口条件）

1. **`kdev` CLI 骨架（Node.js/TypeScript）**
   - 命令：`kdev start` / `kdev state` / `kdev next`
   - 读写 `.kdev/state.md`（维持 Sprint 0 已验证格式）
   - 入口自动读 `.kdev/conventions.md` + 展示"当前位置 + 下一步"（封 G-001）

2. **强制节点 gate（封 M3）**
   - 关键节点到达必须 emit "跑 / 降级 / 跳过" 三选一决策
   - LLM 只能提议（带选项 + 推荐 + 理由），不能沉默跳过
   - 实现：`.kdev/state.md` 加 `pending_decision` 字段，未填则 `kdev next` 拒绝前进

3. **skill 能力对照表 v1（封 M1 第一步）**
   - 列出 10+ 个外部 skill 的原子能力
   - 标注：吸收（抽出为本地节点）/ 丢弃（引导流等）/ 改写（需裁剪）
   - 产出：`docs/01-design/2026-XX-XX-skill-能力对照表.md`

4. **节点受众标注 + 受众化 UX（封 M2）**
   - 14 个命令每个标注 `audience: human | agent | mixed`
   - `audience: human` 节点带决策 gate；`audience: agent` 节点跳评分，只采集自查报告
   - 落到 CLI 节点元信息

5. **文档留白式多决策核对（封 M4）**
   - 新增 `kdev decide <topic>` 命令：LLM 生成 markdown 决策档（选项 + 推荐 + 理由 + 留白），用户编辑后 `kdev decide --collect` 自动写入 questions-log
   - 模板已在 kdev-v3.0.1-改进建议 §建议 4 定义，可直接用

6. **INIT 时自动生成 `.kdev/conventions.md` 模板（封纰漏 #6）**
   - 含 §1 AR 编号 / §2 CSV schema / §5 commit 格式 / §6.3 踩坑即时落盘 / §7 垂直切片 / §8 验收定义 / §9 评分即时化 / §13 E2E 分层
   - 每迭代启动读一次（G-010 后的铁规）

7. **E7-ACCEPT 两条通路模板（封 M6）**
   - dogfooding 指令模板（看"像不像"）
   - PRD 核对脚本（扫 FR/NFR 全集对照 AR 实现）
   - 输出统一的验收报告模板

### 6.2 应交付（若 Sprint 1 有余力）

8. **commit 粒度检查器** — 自检 commit 是否单一事项，不单一需 body 说明理由（封 M9）
9. **AR CSV schema lint** — `kdev lint ar` 对照 conventions §1/§2 自动校验（G-010 预防）
10. **Token 预算记账** — 节点完成时估算 token 消耗 → journal（封纰漏 #1）

### 6.3 不做（Sprint 1 明确排除 / 留给 Sprint 2+）

- ❌ knowledge-graph 自动化
- ❌ learnings.jsonl 抽取机制
- ❌ 跨项目 memory sync
- ❌ skill 包发布到 Marketplace
- ❌ kdevsec 开发（按 v3.0.1 规划留给"1→N 验证"阶段）
