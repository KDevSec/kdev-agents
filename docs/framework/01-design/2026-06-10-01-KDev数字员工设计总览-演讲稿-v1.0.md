## 0. 一句话

> **KDev 不是一个工具集，是一家 6 人 AI 团队。** 用「能力 → 数字员工 → 持续升级」逐层组装出能端到端交付软件的数字员工；跑在一套 lean 编排底座（kdev-core）上、靠分层记忆（kdev-memory）越用越强。**今天它已经第一次真在底座驱动下端到端跑通了一个真实改造任务。**

---

## 1. 概念：能力 → 数字员工 → 持续升级

### 1.1 三段主干

| 段 | 定义 | 关键区分 |
|---|---|---|
| **① 能力** | 通用 Agent ＋ Skill／规范／模板 ＝ 专用 agent（某类活的专家）| 能力是 **building block**，不是数字员工 |
| **② 数字员工** | 多个「能力」＋ 编排能力 → 自主调度、端到端交付 | 数字员工 = 多能力 **+ 编排** |
| **③ 持续升级** | ＋ 专有知识库（持续注入）→ 反哺能力 → 越用越强（复利飞轮）| 知识库是升级引擎，不挂级别 |

**能力四件套**：通用 Agent（推理＋工具/MCP＋模型选型）＋ Skill（know-how）＋ 规范（质量约束）＋ 模板（产出 schema）。

### 1.2 金字塔（自下而上逐层组装）

```
              ▲ 数字员工（编排 + 多能力 → 端到端交付）
          ▲ 能力（= 专用 agent · 某类活的专家）
      ▲ 底层技能：Skill ＋ 规范 ＋ 模板 ＋ 知识库
  ▲ 通用 Agent（底座 · 推理 + 工具/MCP + 模型选型）
```

### 1.3 复利飞轮（③ 持续升级）

干活（需求→验收）→ 过程沉淀进专有知识库 → 知识反哺能力（下次更准更快）→ 越用越强 → 回到干活。**升级版**：飞轮不止反哺「能力」，未来还反哺「SOP 本身」（自演进，见 §5）。

### 1.4 自主度阶梯（单个员工的成熟度，类比自动驾驶 L1–L5）

| 级 | 名 | 人/机分工 | 长出的机制 | 质量把关 |
|---|---|---|---|---|
| L1 | 助手 | 人驱动、执行单步 | 能力起步 | 自评审 |
| **L2** | **协同** | **员工做完整任务、人编排确认** | **＋编排 ＋记忆** | **记忆增强自评审** |
| L3 | 自主 | 端到端自跑、人管异常 | ＋协作 ＋第三方评审 | 独立第三方评审 |
| L4 | 自治 | 自监督 / 自演进 | ＋元监督 ＋自演进 ＋跨 IDE | 元监督 |

> 知识库 / MCP **不在**此轴——它们属「能力组成」与「③ 持续升级」。**当前在 L2 协同**。

## 2. 数字员工：6 人 AI 团队

### 2.1 组织架构

```
                         用户
                          │（只跟 CEO 对话）
                  ┌───────▼────────┐
                  │  CEO 主控员     │ 对话 / 派单 / 决策 / 升级裁决
                  └───────┬────────┘
          ┌───────────────┴───────────────┐
   ┌──────▼──────┐                ┌────────▼──────────────────────┐
   │ CQO 元监督   │←── 元监督 ───→ │ 3 交付员工（每个 = 一个数字员工）│ ＋ 评审专家（收敛把关）
   │ 流程合规·阻断 │                └ 需求架构师 / 开发工程师 / 测试工程师 ┘
   └─────────────┘                   └ 编排间直接协作（不经 CEO）· 产出 → 评审专家收敛评审 ┘
```

- **员工 = N+1 个平级能力**（含 1 个编排能力）；工作 agent 是员工伸出去的手脚。
- **员工间直接协作**：编排能力之间直接发函，不必经 CEO 中转。
- **评审专家是一个资深质检官**，收敛把关 3 个交付员工的产出。

### 2.2 28 agent（起步态）

`28 agent = 1 CEO ＋ 1 CQO ＋ 4 编排 ＋ 15 业务能力（需求5＋开发6＋测试4）＋ 7 评审能力`

> **7 必要核心评审**（对齐 R→D→T→F，宣传期可落地）：需求评审、方案评审、代码评审、质量评审、安全评审、测试评审、终审聚合。
> **L4 完整态 = 39 agent**（评审专家拆回 18 视角）；起步态把评审 19→8 聚合，升级时再拆。

### 2.3 评审治理（CQO 把关）

`R 需求 → D 开发 → T 测试 → F 终审`（4 阶段 · 7 核心评审闸）。**双重通过门**：总分 ≥ 阈值（默认 80，关键 85）**且**阻断性 = 0 才放行；3 次不过 → 升 CEO 交用户拍板。**评审给建议、员工自主判断**。

## 3. 整体架构：3 个 plugin 各司其职

| Plugin | 一句话定位 | 服务对象 | 现状 |
|---|---|---|---|
| **kdev-team**（员工层，原 kdev-staff）| 6 人 AI 团队业务实现 | 用户的软件交付任务 | 阶段2 P-0 落地：开发工程师 7 canonical agent |
| **kdev-core** | 数字员工编排底座（R1–R7）| kdev-team（未来其他 plugin）| 核心三件 R1/R2/R3 定稿、66 测试全绿 |
| **kdev-memory** | 用户工程记忆 + 生态范式定义 | 用户本人（跨会话续航）| 在用（v0.13.0），git 托管已激活 |

## 4. kdev-core：lean 编排底座（R1 / R2 / R3）

> **定位**：从现有 flows 抽共性的 lean 编排底座（Python/markdown 原生，借 OMC 范本不 fork）。**铁律：底座只记账 + 流转，不做判断本身**——评审/E2E/决策由 flow 侧 AI 或人执行，结果喂回底座。

| 件 | 是什么 | 卖点 |
|---|---|---|
| **R1 · flow-state 状态机** | 每个 flow 一份 `flow-state.json`：原子写（tmp + os.replace）＋ history 流水 ＋ active/_meta stale guard | **白捡 resume**（断点续跑，断电/kill 后从上次 node 续）|
| **R2 · 通用 n 节点机** | 节点表由 flow 自己声明（不写死段数）；底座只提供流转引擎：顺进（gate PASS）/ 回流（gate FAIL）+ 有界回流 + 幂等豁免 | 同一底座跑任意业务 SOP |
| **R3 · 三类 gate 统一抽象** | 核心难点 = 卖点，见下表 | 一套 `record_gate` 解析三类 gate |

### 4.1 R3 三类 gate

| kind | 谁评（默认/可编排）| verdict | 不过怎么办 |
|---|---|---|---|
| **review** 评审重试 | ai（默认）/ both / human | PASS / FAIL | 改 → 重试（max N），iter≥max → **escalate 转人工**（不 force-accept）|
| **decision** 路线判断 | ai 默认 / human 可接管 | 分支选择 | 选分支即流转（无重试）|
| **acceptance** 验收 | test 跑 E2E / +human 复核 | PASS / FAIL | 回流 fix 节点 |

**GateResult 结构化 JSON**（带 `request_id` / `iter` / `verdict` / `by`）——下游不靠正则解析（胜 OMC 避坑①②）。

### 4.2 韧性与人机协同

- **retry 断路**：同 gate 连续 max_retries（默认 3）不过 → 升级人工 / CEO，不无限打转。
- **reviewer 锁定 + tier 升级**：重试时同一 reviewer 不漂移；复杂度涨了升模型 tier（sonnet→opus）。
- **decider 旋钮**：每类 gate「谁来判」默认 AI，编排可改 human 接管——人有最终决定权、默认不挡路。
- **Auto Mode 停靠**：自主推进只在 `decider: human` 的 gate 停，其余 AI 自己过。

## 5. SOP 可编排 + 自演进

R2 节点表 + R3 gates ＝ **一个 flow 的 SOP，可编排**。三层覆盖：

| 层 | 谁编排 | 位置 | 改什么 |
|---|---|---|---|
| **L0 默认** | flow 作者（我们）| flow 内 `node-table.yml` | 默认 SOP（节点表 + gates）|
| **L1 用户** | 用户 | `.kdev/flows/<flow>/orchestration.yml` | 开关 gate · decider ai↔human · retry 数 · 跳过/重排节点 |
| **L2 自主**（未来 L4）| 数字员工自己 | 同上 | 基于经验提议 SOP 优化（自演进，`source: self` 需人确认）|

底座加载的**有效 SOP ＝ merge(L0, L1, [L2])**。**自演进** = 复利飞轮从「反哺能力」延伸到「反哺 SOP 本身」——从人编排 SOP 走向员工自优化 SOP（L3–L4，当前只预留挂钩）。

## 6. 记忆：三层 × 多 scope

> kdev-team **硬依赖** kdev-memory（叙事层）；kdev-memory 单装仍干净（默认单 scope）。

| 层 | 形态 | 谁用 | 落位 |
|---|---|---|---|
| **操作层** | JSONL（events，scoped）| 业务员工高频读写「做了啥/要做啥」| `.kdev/events/<scope>.jsonl`（kdev-core）|
| **叙事/蒸馏层** | markdown（4 段 Step / 决策 / 踩坑）| 人类续航 + 训练数据 + CEO 会话续航 | kdev-memory（scope-aware）|
| **任务状态** | JSON | agent 操作「当前在哪/下一步」| `flow-state.json`（kdev-core）|

**关键**：markdown Step 不再每动作每员工写一次，而是**从 JSONL events 周期性 rollup 派生**——agent 全程只碰便宜的 scoped JSONL，重 markdown 一阶段聚合一次。

**scope 拓扑**：`shared/`（项目级，全员共享决策/踩坑/CEO 续航）+ `staff/<员工>/`（per-员工私有 rollup + 评审经验 `subject:review`）。**git 托管**：`.kdev/` = nested repo，自举 hook（SessionStart clone/pull、SessionEnd commit/push）。

### 7.1 里程碑 M0 ✅ — 底座立起来了

底座核心三件 **R1+R2+R3 = 66 测试全绿、main 已 push** + **git 托管已激活**（`.kdev/` = nested repo，已 push 记忆仓 `KDevSec/kdev-agents-memory`）。`plugins/kdev-core/` = 完整 lean 底座核心。

### 7.2 里程碑 M1 ★✅ — 集群史上首次端到端实测

开发工程师 coding-flow 经 kdev-core 底座（薄 CLI 驱动 R1/R2/R3）**端到端跑通 UED6 改造 Pass1**（RuoYi-Vue3-FastAPI 视觉改造，T0 全局主题 + T1 登录页）。**底座真在驱动**（非 prompt 演戏）的硬证据：

- **13+ 节点经 ~25 次独立 CLI 进程流转**，flow-state.json 全程持久续接（**resume-across-process**）；
- **9 gate 覆盖 R3 三类**（review / decision / acceptance 全跑）；
- 一次 **FAIL（完成验证）→ reflow → 修 → 重评 PASS** 活体闭环；
- `complete` 终结后 resume 正确拒续；
- 6 业务 Agent 在对应节点派单；真产出 `build:prod` RC=0 + 登录页对版原型。

### 7.3 阶段2 P-0 ✅ — 员工集中 kdev-team

新建 `plugins/kdev-team/`（v0.2.0 + marketplace 注册）：**开发工程师 7 个 canonical agent**（orchestrator / env / plan / frontend / e2e / deploy / sec，真 CC frontmatter，可 `Agent({subagent_type})` 直派）+ `staff.yml` 花名册 + `orchestration/dev-engineer.node-table.yml`（**16 节点 / 9 gate**）。kdev-coding-flow SKILL 回归纯方法论（编排指令归 `dev-engineer-orchestrator` agent）。测试 **kdev-team 11 + kdev-core 76 绿**。
