## 0. 心智模型：6 人 AI 公司（核心隐喻）

### 0.1 一句话目标

**KDev 不是一个工具集，是一家 6 人 AI 公司。** 用户雇了 6 名员工——1 个 CEO 跟你对话 / 1 个 CQO 元监督质量 / 4 个业务员工干活——每个员工自带多种能力，协作完成软件交付。

### 0.2 关键认知

| 不是 | 是 |
|---|---|
| ❌「4 个部门 / 4 个组」 | ✅「4 个员工，每人多种能力」 |
| ❌「组长管下属」 | ✅「员工通过编排能力调用自己的其他能力」 |
| ❌「员工 = 本体 + 能力（两层）」 | ✅「员工 = N+1 个平级能力，含 1 个编排能力」 |
| ❌「工作 agent 是独立的人」 | ✅「工作 agent 是员工伸出去的手脚」 |
| ❌「跨组通信」 | ✅「员工间协作」 |
| ❌「评审专家 N 评审能力 = N 个人」 | ✅「评审专家是一个资深质检官，他有 18 种评审视角」 |
| ❌「员工间协作必经 CEO 中转」 | ✅「员工的编排能力之间直接协作」 |
| ❌「评审通过/不通过 = 二元结果」 | ✅「评审 = 百分制评分 + 问题分级 + 双重通过条件」 |
| ❌「评审不过 → 强制返工」 | ✅「评审专家给建议，员工自主判断哪些值得修；3 次循环后用户拍板」 |
| ❌「评审有强制项 = 用户不能关」 | ✅「评审 = 13 默认开 + 5 默认关，用户编排自由」 |

## 3. 通信规则（员工间协作）

### 3.1 通信硬规

🔴 **硬规 1**：用户只跟 CEO 对话
🔴 **硬规 2**：员工的编排能力之间**直接发函协作**（不经 CEO）
🔴 **硬规 3**：CEO 仅在 4 种场景介入：**阶段聚合 / 异常升级 / 用户问询 / 评审 3 次循环后升级**
🔴 **硬规 4**：业务能力 / 评审能力**不直接对外**，只跟自家员工的编排能力对话
🔴 **硬规 5**：跨员工协作**必须走员工的编排能力**（业务能力不能直接外联其他员工的能力）
🔴 **硬规 6**：CQO **只读不写**，纯监督，只对 CEO 汇报
🔴 **硬规 7**：员工 BLOCKED 时**自动激活该员工编排应急**（具体触发机制见整体架构 doc）

### 3.2 编排能力的 4 meta 职责

| 职责 | 干什么 | 何时激活 |
|---|---|---|
| ① 监督 | 后台监听本员工的能力产物抽查 | 后台被动 |
| ② 标准 | 维护本员工的能力 prompt 模板和评审标准 | 静态资产 |
| ③ 聚合 | 该员工阶段完成时出员工总结报告 | 每阶段 1 次 |
| ④ 应急 | 能力 BLOCKED 时介入决策 | 异常时（详见 §3.4）|

### 3.3 通信路径速查

| 通信类型 | 路径 | 跳数 |
|---|---|---|
| 用户 ↔ CEO | 主对话 | 0 |
| CEO 派员工 | `Agent({subagent_type: "需求架构师-编排"})` | 1 |
| 员工内部调用能力 | 编排 → 自家能力（内嵌派单） | 1 |
| **员工间协作**（如开发请评审）| 员工 A 编排 → 员工 B 编排 → 能力 → 回流 | 3 跳 |
| 异常升级 | 员工编排 → CQO 或 CEO | 1 |
| CQO 抽查 | CQO 读运行时文件（被动监督）| 0 |

### 3.4 BLOCKED 应急流程（概念性）

```
业务能力 → 连续 3 次失败 → 触发"blocked"信号
                          ↓
                自动激活该员工的编排能力
                          ↓
                编排 → 评估根因 → 决策：
                    ├── 重派该能力（升级模型 / 调参）
                    ├── 改派其他能力重做计划
                    ├── 发函给 CEO（需要用户介入）
                    ├── 跨员工求援
                    └── 标污染样本 + 继续
```

**注**：具体"blocked 信号"如何写入运行时（events.log / state.md）、"自动激活"的 hook 实现 → 见整体架构 doc。

### 3.5 通信示例：开发工程师请评审专家代码评审

```
开发工程师/编排（TDD 实现能力 COMPLETE）
    ↓ 直接发函（员工间，不经 CEO）
评审专家/编排
    ↓ 根据本项目流程编排，并行调用对应评审能力
    ↓ 代码评审 + 质量评审（+ 可观测性，如果用户在编排中打开）
    ↓ 每能力按 standards 模板出百分制评分表
    ↓ 评审专家/编排 汇总 + 仲裁评审员冲突
    ↓ 回函给开发工程师/编排
    
（CEO 不参与日常协作；CQO 后台抽查这次协作是否合规）
```

**单程：3 跳 / 往返：6 跳**

## 4. CQO 元监督定位

### 4.1 角色辨别：CQO ≠ CEO ≠ 评审专家

| 角色 | 关注点 | 比喻 | 典型问题 |
|---|---|---|---|
| **评审专家** | 内容质量 | Code Reviewer | "这段 SR 写清楚了吗？这段代码有 bug 吗？" |
| **CQO** | 流程合规 | ISO 审计师 / 总监理 | "R2 节点该过 SR 评审，跑了吗？评审给的评分能复现吗？" |
| **CEO** | 决策派单 | 总裁 | "现在进 dev 阶段还是返工 reqs？评审循环 3 次没过，怎么办？" |

**核心：CQO 同时监督评审专家**——评审走过场 / 漏评，CQO 能发现并升级到 CEO。

### 4.2 职责清单（7 项）

#### ✅ 做

| # | 职责 |
|---|---|
| 1 | 流程完整性检查 —— 所有节点都跑了吗 |
| 2 | 能力跳过检测 —— 应执行的能力有没有被遗漏 |
| 3 | 评审到位度检查 —— 评审专家给的评分是否完整、问题分级是否合理、维度评分是否极端 |
| 4 | BLOCKED 闭环检查 —— blocked 事件后员工编排是否介入、是否恢复 |
| 5 | 重试次数检查 —— 评审循环到第 3 次时 CQO 先告警 |
| 6 | 产物体检 —— 员工产物文件是否齐全、关键 section 是否为空 |
| 7 | 元评审评审专家（终审前 + 用户触发）—— 抽查评审专家的评审质量是否退化 |

#### ❌ 不做

- 不评审内容质量（评审专家的职责）
- 不参与决策派单（CEO 的职责）
- 不写代码 / 不出产物（纯监督）
- 不直接命令员工（只升级到 CEO）

### 4.3 触发时机（5 种模式）

| 模式 | 何时启动 | 默认 |
|---|---|---|
| a. 能力 COMPLETE 后被动抽查 | 每个能力写 COMPLETE 时 | ✅ 开 |
| b. 员工阶段聚合时 | 员工编排出阶段聚合报告时 | ✅ 开 |
| c. 终审前总检 | 最终阶段聚合前 | ✅ 开（最后一道闸）|
| d. 常驻后台监听 | 每条事件都抽查 | ❌ 关（省 token）【改为默认开，先跑通流程】|
| e. 用户主动触发 | 用户说"做个 QA 检查"时 | ✅ 开（兜底）|

### 4.4 输出物三档

🔴 **阻断**：流程严重遗漏 / 评审完全跳过 → CEO 必须处理才能继续
🟡 **告警**：评审走过场 / 重试 3 次 → CEO 决策
⚪ **正常**：合规通过 → 留痕

**CQO 有阻断权**——流程合规是硬要求。

### 4.5 通信规则

| 规则 | 实现 |
|---|---|
| CEO 派 CQO | `Agent({subagent_type: "CQO"})` |
| CQO 读数据 | 直接读员工产物 / 评审记录 / 流程编排文件 / 评审经验记忆（具体路径见整体架构 doc）|
| CQO 报告 CEO | 写 CQO 报告 + 通知 CEO |
| CQO 越权 | ❌ 不直接命令员工 |
| CQO 监督 CEO | ❌ 否（CEO 决策由用户监督）|

### 5.2 评审能力分类（2 档，用户编排自由）

🟡 **默认开（13 项，可关）**：13 评审默认启用。用户在流程编排中可关闭，但 CEO 会对关掉**核心评审**（代码 / 安全 / 终审聚合等）给软警告 + 要求填关闭理由。

⚪ **默认关（5 项，可开）**：5 评审默认不启用。用户在流程编排中按项目类型决定开启。

具体清单见 §2.3 员工 4 评审专家。

### 5.3 评审接入硬规（员工协作语义）

🔴 **旧（群组思维）**：开发工程师内的 TDD 实现员 → 直接派审查组的代码评审员

🔴 **新（员工思维）**：
```
开发工程师/编排（产物 COMPLETE）
    ↓ 直接发函（员工间协作，不经 CEO）
评审专家/编排
    ↓ 根据本项目流程编排，并行调用对应评审能力
    ↓ 每能力按 standards 模板出评分表
    ↓ 评审专家/编排 汇总 + 仲裁
    ↓ 回函给请评审的员工/编排
```

**关键差异**：评审不是"插入流水线的工序"，而是"开发工程师请评审专家同事来 review"——两个**独立员工之间的协作**。

### 5.4 评审循环 + CEO 升级路径（评分机制）

#### 5.4.1 百分制评分双重通过条件

每个评审能力按 standards 模板出评分表，含：

- **总分**：3-5 维度评分（每维度 20-25 分），合计 100 分
- **问题分级**：🔴 阻断 / 🟡 重要 / ⚪ 轻微 三档
- **通过判定（双重条件 AND）**：
  ```
  通过 = 总分 ≥ 阈值（默认 80）AND 🔴 阻断性问题 = 0
  ```

#### 5.4.2 评审循环

```
评审 → 评分 → 判定
  ├── ✅ 通过 → 下一阶段
  └── ❌ 不通过
        ↓
      重试次数 < 3
        ├── 是 → 评审专家给"修订建议清单"（按 🔴/🟡/⚪ 分级）
        │       → 员工编排自主判断"哪些值得修 / 哪些 tech debt 化"
        │       → 修订完成 → 增量评（只评修订部分）→ 重试次数 +1
        └── 否（== 3）→ 升级 CEO
                ↓
              CEO 跟用户对话：
                "节点 <id> 评审已修订 3 次仍不达 <阈值> 分
                 当前评分: <N>/100 / 剩余: 🔴 X 个 / 🟡 Y 个 / ⚪ Z 个
                 请拍板：
                 (a) 接受当前评分，记 tech debt，进入下一阶段
                 (b) 继续修订（第 4 次循环）
                 (c) 推翻当前产物，重做"
              → 用户拍板 → 记录决策路径
```

#### 5.4.3 默认阈值（standards 可覆盖）

| 类型 | 默认阈值 |
|---|---|
| 大部分评审 | **80**/100 |
| 关键评审（代码 / 安全 / 方案 / 架构）| **85**/100 |
| 创意类评审（原型）| **75**/100 |
| 高风险评审（上线就绪度）| **90**/100 |
| 终审聚合 | 不评分（决策聚合）|

### 5.5 评审模板 standards 规约

每个评审能力配套一份 standards 模板（markdown），含：

- 评审目的
- 评审对象（哪个产物文件）
- 评分维度（3-5 项 + 每项 checklist）
- 通过阈值（默认 80，可覆盖）
- 问题分级标准（什么算 🔴 / 🟡 / ⚪）
- 评分表输出 schema

**目录组织**（待整体架构 doc 确定具体路径）：
```
<staff plugin>/standards/评审专家/
├── 通用评分模板.md           ← 100 分制 schema + 双重条件 + 循环规则
├── 需求方向评审.md
├── SR需求评审.md
├── AR需求评审.md
├── ...
└── 终审聚合.md
```

### 5.6 评审员冲突仲裁

如代码评审 vs 质量评审给出矛盾意见 → 评审专家/编排仲裁；编排也犹豫 → 升级 CEO + 标记元评审异常给 CQO。

## 6. 跟整体架构的接口（占位 / 待整体架构 doc 决定）

本节列出**员工能力依赖的外部基础设施**，具体实现等整体架构 doc 落地：

### 6.1 评审经验蒸馏接口

- **需求**：每次评审后用户可给反馈（评得对/错/松/紧）→ 落盘成持久记忆 → 下次评审 recall
- **接口契约**（待整体架构定）：
  - 评审能力 prompt 顶部 `MUST recall <评审经验存储路径>` 过滤 `subject: review:<能力名>`
  - 蒸馏导出按评审能力分组（dataset-review-by-capability）
- **关键决策**：评审经验存哪？复用 kdev-memory / 独立 kdev-staff 目录 / 抽到 kdev-core multi-namespace —— **见整体架构 doc**

### 6.2 yml 流程编排接口

- **需求**：CEO 首次对话引导用户生成 yml 配置文件，含：
  - 18 评审能力的 default_on / default_off 编排
  - 评分阈值覆盖
  - CQO 触发模式 a-e 开关
  - 人类介入 4 通道开关
- **接口契约**：每次员工编排启动前 read yml 决定本节点该跑哪些评审
- **关键决策**：yml schema 完整定义 + CEO 校验软警告机制 —— **见整体架构 doc**

### 6.3 hook 系统接口

- **需求**：
  - BLOCKED hook（events.log 写 blocked → 自动激活员工编排）
  - 评审循环 hook（评分 < 阈值 → 触发员工编排修订）
  - 低分预警 hook（评分通过但 < 高阈值 → CEO 提示用户）
  - SessionStart / PreCompact hook（恢复 / 压缩前快照）
- **关键决策**：hook framework 自建 / 复用 kdev-memory hook / 抽到 kdev-core —— **见整体架构 doc**

### 6.4 人类介入 4 通道接口

- 通道 ①：阶段聚合时 CEO 摘要本阶段评审分布
- 通道 ②：用户在评分表下加 review-feedback section
- 通道 ③：CQO 终审前元抽查
- 通道 ④：低分预警（可选默认关）
- **关键决策**：通道实现机制（UI / hook / CEO prompt）—— **见整体架构 doc**

### 6.5 协作总线接口

员工产物 / CEO 状态 / 事件流 / 流程编排 / 评审记录 / 评审经验等运行时文件的目录布局 —— **见整体架构 doc**。
