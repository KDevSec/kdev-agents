## 0. 一句话

数字员工从「单兵」长成「组织」，本质是从 **单进程**（一个 Claude Code 进程、员工是进程内的 subagent 集群）走向 **多进程异构**（每个员工是一个独立的 agent 进程、可跑不同编码 Agent / 模型，由一个研发经理跨进程调度）。三步走：

1. **第一步** —— **单进程** 数字员工团队（一个 CC 进程，**当前聚焦：尚未开发完、未完全验证**）；
2. **第二步** —— **异构多进程研发团队**：**Hermes 研发经理**跨进程调度 + 每个员工独立编码 Agent 进程（CC / Codex / CodeBuddy 可切换）+ **MemOS** 补记忆 + 对接企业研发平台（禅道 / GitLab / Jenkins）；
3. **第三步** —— **组织化大规划**：百人百智 / 选用育留备退 / 价值经营治理（待专项展开）。

## 1. 现状与起点：第一步 = 单进程数字员工团队（当前聚焦）

我们已经设计并在建一套**数字员工技术底座**，不是 PPT 构想，是已端到端跑通真实任务的系统——但**第一步本身尚未做完、未完全验证**，**当前聚焦就是把它做完、验证透**。

> **形态本质 = 单进程**：整套系统跑在**一个 Claude Code 进程**里；**每个员工本身由多个 agent 组成（1 编排 + N 能力 agent），4 业务员工共 26 个 agent**，都是这个进程内的 subagent，共享一个运行时 / 上下文。**注意：CC 只是运行底座，模型可换**（经 Anthropic 兼容端点接 GLM / DeepSeek / Kimi 等国产或本地模型），并非锁死 Anthropic 模型。详见 §6。

**两个底座**
- **编排底座 `kdev-core`** —— 让员工"按流程跑起来"：状态记录 + 节点流转 + 结构化评审 gate，铁律"只记账 + 流转，不替员工做判断"。
- **记忆底座 `kdev-memory`** —— 工程记忆制度：叙事 / 步骤 / 决策 / 踩坑 / 自评他评，多 scope（shared / staff），Git 托管跨会话续航（**初级记忆**，员工运行时记忆是短板，第二步用 MemOS 补）。

**6 角色团队（= CEO + CQO + 4 业务员工，均为第一步设计）**
- ✅ **4 业务员工已建**（需求架构师 → 开发工程师 → 测试工程师 → 评审专家，覆盖"需求→开发→测试→评审"全链）：合计 **22 项业务/评审能力 + 4 编排 = 26 个 agent**（`kdev-team` v0.8.0）。
- ⏳ **CEO（主控员）** 现由主会话承担、**CQO（监督员 / 质量监督）** 尚未建为独立实体 —— 二者是第一步团队的固有角色，只是实现排序靠后（第二步分别承接为"研发经理"与"QA 质量工程师"，见 §7.2）。

**第一步的"成色"（诚实校准）**
- ✅ **会干活** —— 单员工端到端跑通真实改造任务（自主度 L2 协同，M1）。
- ✅ **会协作（进程内 subagent 级）** —— 多员工分工 + 跨员工 handoff + 评审 + 发函硬规（M2/M3）。⚠️ 这是**单进程内的 subagent 协作**，**不是多进程组织**（多进程要到第二步）。
- ⚠️ **记忆初级** —— 工程记忆是 md / 召回，员工运行时长期记忆是短板。
- ❌ **第一步尾巴未完** —— CEO/CQO 角色未建、自演进 / 跨 IDE 未做、全量验证未跑。

> 🔴 **当前聚焦 = 收尾并验证第一步**：把"单进程数字员工团队"真正落稳，再迈向第二步的多进程异构。

## 2. 目标愿景：一个完整的数字员工 + 一支可经营的队伍

终点对齐数字员工目标蓝图——单体**类真人**，且能被组织**成建制经营**。

**"我是谁 - 从哪来 - 到哪去" = 单体生命周期三视角**
- **我是谁（角色管理）**：经营力 / 业务力 / 专业力 + 9 维度（价值贡献度 / 场景覆盖度 / 流程渗透度 / 能力成熟度 / 系统体验度 / 数据完备度 / 模型适配度 / 智算应用度 / 安全可信度）。
- **从哪来（周期管理）**：HR + IR 双线，**选 - 用 - 育 - 留 - 备 - 退** 生命周期，"百人百智"规模化复制。
- **到哪去（运用实例）**：单个数字员工（如"软件代码助理小智"）在真实业务里持续产出、量化价值。

**"新环境 5 保障"** = AI 架构 / 流程 / 能力 / 组织 / 投入保障。

> 终点定位：**从"一个 CC 进程里的单 Agent 团队"，到"多个独立异构 Agent 进程组成、可被组织选育留并算清价值贡献的数字员工队伍"。**

落地路线沿两根轴往右上爬；其中**横轴（组织化）的本质，是从「单进程」走向「多进程异构」**：

## 4. 落地路线：3 步（单进程 → 多进程异构）

```
第一步｜单进程（当前聚焦·进行中）             第二步｜多进程异构研发团队（CC 与 Hermes 联合）
┌───────────────────────────┐            ┌────────── 研发经理 = Hermes 进程（Kanban 跨进程调度）──────────┐
│   Claude Code（1 个进程）      │            │   按任务路由到各员工进程（不同活可分给 CC 或 Codex）           │
│  ┌────┬────┬────┬────┐      │  ── 第二步 ─►│  ┌────────┐┌────────┐┌────────┐┌────────┐┌────┐        │
│  │需求│开发│测试│评审│      │  单进程→多进程 │  │系统设计师││开发工程师││测试工程师││评审专家││ QA │ …各为独立 │
│  └────┴────┴────┴────┘      │            │  └────────┘└────────┘└────────┘└────────┘└────┘ 编码Agent进程│
│  员工＝进程内 subagent 集群     │            │   记忆：MemOS │ 平台：禅道/GitLab/Jenkins/Gerrit（经 MCP）   │
└───────────────────────────┘            └──────────────────────────────────────────────────────────┘
                                          第三步｜组织化大规划（百人百智 / 选用育留备退 / 价值治理）— 待展开
```

### 第一步 · 单进程数字员工团队（🔴 当前聚焦，进行中）
一个 CC 进程；**每个员工由多个 agent 组成（编排 + 能力），4 业务员工共 26 个 agent**，均为进程内 subagent。**已建**：kdev-core 编排 + kdev-memory 初级记忆 + 4 业务员工 + kdev-hud 观测 + L2→L3 协作（M0–M3）。**待补**：CEO/CQO 角色、自演进、跨 IDE、全量验证。**本质**：强内聚（单一 CC 进程）；模型可换（GLM/DeepSeek），锁的是 harness 软件、不是模型。

### 第二步 · 异构多进程研发团队（CC 与 Hermes 联合）
**每个员工升格为独立的编码 Agent 进程**（CC / Codex / CodeBuddy 可切换可配置），由 **Hermes 研发经理**跨进程调度，**MemOS** 补员工记忆，对接企业研发平台。**这就是"Claude Code 与 Hermes / OpenClaw 联合使用"的落地形态。**（详见 §7.2）

### 第三步 · 组织化大规划（待专项展开）
从研发团队扩到**数字员工组织**：百人百智规模复制 + 选用育留备退生命周期 + 经营力/业务力/专业力价值度量 + 跨域扩展。（大规划后续专项展开，§7.3 占位）

### 第二步 vs 第三步（并排看）

| 维度 | **第二步：异构多进程研发团队** | **第三步：组织化大规划** |
|---|---|---|
| 一句话 | 研发域内：多进程异构、研发经理调度 | 跨研发域：成建制经营一支数字员工队伍 |
| 进程/规模 | 一个团队、多员工进程（异构编码 Agent）| 多团队 / 多域 / 百人百智 |
| 核心 | Hermes 经理 + 异构编码 Agent 池 + MemOS + 平台对接 | 选用育留备退 + 价值经营度量 + 治理 |
| 状态 | 🔨 第一步收尾后聚焦（落地方案见 §7.2）| ⏳ 待专项展开（§7.3 占位）|

## 6. 底座选型说明（三类底座 + 单/多进程）

第二步起，按**角色**分三类底座，各用所长：

| 底座 | 选型 | 职责 | 说明 |
|---|---|---|---|
| **经理（调度）底座** | **Hermes Agent**（推荐）/ OpenClaw（备选）| 跨进程派活 / 路由 / 并发 / 熔断 + 平台对接 + 统一账本 | 选型结论见下 + 附录 B.3 |
| **研发（执行）底座** | **异构编码 Agent 池**：Claude Code / Codex CLI / CodeBuddy CLI | 各研发岗员工独立进程，写代码 | **可切换、可配置**：经理按任务把不同活路由给不同编码 Agent |
| **记忆底座** | **MemOS**（openmem.net）| 员工持久记忆：向量 + BM25 双引擎召回、结构化自动捕获、多 agent 隔离、token 省 ~72% | 有现成 OpenClaw 插件；Hermes/CC 待适配。⚠️ 偏云端，内网私有化需核其开源/自托管路径。**勿与同名笔记应用 usememos/memos 混淆** |

**单进程 → 多进程（核心）**：第一步全团队 = 一个 CC 进程（员工＝进程内 subagent）；第二步**每个员工 = 一个独立的编码 Agent 进程**（独立上下文、可配不同模型），Hermes 经理跨进程调度。这才是"单 Agent → 多 Agent"的真身。

**经理底座选型结论（OpenClaw vs Hermes，调研详见附录 B.3）**：推荐 **Hermes Agent** ——① Kanban dispatcher 是**为"经理派活"工程化设计**（6 阶段调度循环 + 并发限额 + 熔断 + crash 恢复）；② **安全基线更优**（OpenClaw 2026-Q1 暴 512 漏洞/8 critical、13.5 万实例公网裸奔，内网不可接受）；③ 私有化更轻（SQLite 单机 / MIT / 离线可行）。**代价（诚实）**：把 CC/Codex CLI 当跨进程 worker 派单，Hermes/OpenClaw **都未铺好开箱路径**，需自建（spawn + workspace + 回传，约 1–2 周）；Hermes **缺 RBAC**，内网靠网络隔离 + skill 白名单外补。

**与 Claude Code 关系**：CC 是成熟运行底座（agentic loop / 工具 / subagent / 技能开箱），**模型可换**（GLM/DeepSeek/Kimi）。第二步引入 Hermes 经理 + 多编码 Agent，是用各底座所长，不是推翻 CC。

## 7. 落地方案

> 每条按"**基于什么现状 + 怎么做 + 业界对标 + 落到哪 + 可替换点**"展开，确保有支撑、可执行。

### 7.1 第一步收尾（🔴 当前聚焦）
- **基于现状**：4 业务员工已建（M0–M3 已验证），CEO/CQO/自演进/跨IDE 未落地，全量验证未跑。
- **怎么做**：① 把 **CEO（主控员）** 沉淀为独立实体（第二步承接为"研发经理 / Hermes"）；② 建 **CQO（监督员）**（第二步承接为"QA 质量工程师"）；③ 跑**全量验证**把单进程团队证稳。
- **落到**：`kdev-team` + `kdev-core` / `kdev-hud`。

### 7.2 第二步 · 异构多进程研发团队（核心）

**架构**：
```
研发经理 = Hermes 进程（Kanban 跨进程调度：派活/路由/并发/熔断 + 平台对接 + 统一账本）
   │ 按任务路由（不同活分给不同编码 Agent）
   ├─ 系统设计师  ＝ 独立 CC/Codex 进程   （承接 需求架构师）
   ├─ 开发工程师  ＝ 独立 CC/Codex 进程
   ├─ 测试工程师  ＝ 独立 CC/Codex 进程
   ├─ 评审专家    ＝ 独立 CC/Codex 进程
   ├─ QA 质量工程师 ＝ 独立进程            （承接 CQO 质量监督）
   └─（知识库等其他岗待定）
        ▼ 各进程经 MCP 对接
企业研发平台：禅道(需求) · BUG管理 · GitLab(代码) · Jenkins/Gerrit(CICD)
        ▼ 各进程接
记忆：MemOS（员工持久记忆）
```

**五个要素**：

| # | 要素 | 怎么做 | 业界对标 / 现成 | 落到 / 可替换点 |
|---|---|---|---|---|
| 1 | **研发经理（调度）** | Hermes Kanban dispatcher 跨进程派活 / 路由 / 并发 / 熔断 | Hermes（推荐）/ OpenClaw（备选）| 经理底座，附录 B.3；OpenClaw 可替 |
| 2 | **异构编码 Agent 池（执行）** | CC / Codex CLI / CodeBuddy CLI **可切换可配置**，经理按任务路由 | —— | 每个员工 = 独立进程；编码 Agent 可换 |
| 3 | **员工花名册** | 系统设计师 / 开发 / 测试 / 评审 / QA（承接现有 4 业务员工 + CQO）+ 知识库 | 承接现有设计 | `kdev-team` 员工进程化 |
| 4 | **员工记忆 MemOS** | 向量 + BM25 召回 / 结构化自动捕获 / 多 agent 隔离 | MemOS（openmem.net；OpenClaw 插件现成）| ⚠️ 偏云端→核内网自托管；后端可换 |
| 5 | **企业平台对接** | 经 MCP 接禅道(需求)/BUG/GitLab(代码)/Jenkins·Gerrit(CICD) | GitLab 借 Composio；禅道/Gerrit 需定制 MCP | 新增 MCP server |

**核心工作量（诚实）**：① **Hermes → 异构编码 Agent 跨进程 worker 派单适配**（官方未铺路，需自建 spawn + workspace + 回传，约 1–2 周）；② 企业平台 MCP（禅道/Gerrit 定制、GitLab 借 Composio）；③ **MemOS 内网自托管核实** + Hermes/CC 集成；④ RBAC 外补（网络隔离 + skill 白名单）。

### 7.3 第三步 · 组织化大规划（占位 · 待专项展开）
从研发团队扩到**数字员工组织**：
- **百人百智**：多团队 / 多岗位域规模复制（共享底座 = 数字员工工厂）。
- **选用育留备退**：agent 生命周期治理，对标 Workday Agent System of Record（注册/授权/监控/下线，详见附录 B.4）。
- **价值经营度量**：经营力 / 业务力 / 专业力 + 价值贡献度 / ROI 看板（呼应 Gartner"40% 项目因 ROI 不清被砍"）。
- **跨域 / 跨平台互联**：A2A 协议等异构 Agent 互联标准（详见附录 B.4）。

> 本步为**大规划**，本文先占位，后续专项展开落地方案。

## 附录 B：对比分析（行业调研沉淀）

> 作为正文判断的佐证。结论：**"单进程 → 多进程异构组织"叙事与行业主线同向；底座选型（CC 运行 + Hermes 经理 + 异构编码 Agent + MemOS 记忆）有据可依。**

### B.1 数字员工概念与市场速览（2025-2026）
- **演进**：RPA → AI+RPA → **AI Agent 型数字员工**（感知-推理-规划-执行闭环）。
- **单 → 多 agent 数字劳动力**是主线：[Salesforce Agentforce](https://markets.financialcontent.com/stocks/article/tokenring-2026-1-15-the-copilot-era-is-dead-how-salesforce-agentforce-sparked-the-autonomous-business-revolution) 自称"数字劳动力平台"。
- **异构 Agent 联合是 2026 趋势**：Coze 3.0 已作 meta-orchestrator 调 CC + OpenClaw 等。
- **自主度 [ASDLC L1-L5](https://asdlc.io/concepts/levels-of-autonomy/)**，L3 = 生产可持续上限——印证 L2 起步、L4 defer 稳妥。
- **⚠️ [Gartner：40%+ agentic 项目 2027 前因 ROI/治理被砍](https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027)**——治理/价值度量是生死线（第三步）。

### B.2 运行底座：Claude Code vs 自建 runtime
| 维度 | Claude Code（研发执行底座之一）| 自建 runtime（Hermes/OpenClaw 等）|
|---|---|---|
| 工程成本 / 上线速度 | 低 / 快（agentic 能力开箱，数月跑通 M1–M3）| 高 / 慢（自造引擎）|
| 模型选择 | **可换**：经兼容端点接 GLM/DeepSeek/Kimi/Bedrock/Vertex | 自由：本地/私有/国产/混搭 |
| 私有化 | 模型/数据路径可走国产/自托管；CC harness 软件依赖 Anthropic 分发 | 可完全私有化离线 |

### B.3 经理底座：OpenClaw vs Hermes Agent（第二步选型）
| 维度 | OpenClaw | Hermes Agent | 胜 |
|---|---|---|---|
| 调度架构 | 控制面 + 社区 Kit 拼 | **原生 Kanban dispatcher**（6 阶段循环 + 并发限额 + 熔断 + crash 恢复）| **Hermes** |
| 安全/内网 | 🔴 Q1 2026 暴 512 漏洞/8 critical、13.5 万实例公网裸奔 | 暂无已知 CVE（仅约 4 个月，样本小）| **Hermes** |
| 私有化 | Docker，需加固 | SQLite 单机 / MIT / 离线可行 | **Hermes** |
| 治理 RBAC | Mission Control 审批流较成熟 | **缺 RBAC**（官方承认），需外补 | OpenClaw |
| 异构 CLI worker 派单 | 未铺路 | 未铺路（worker lane "not yet paved"，需自建）| 平 |
| 模型灵活 | Ollama/vLLM/兼容端点 | 接 GLM/DeepSeek/Kimi/Ollama，提供商更长 | 平 |

> **结论**：内网研发场景**推荐 Hermes 作研发经理底座**（调度架构 + 安全 + 私有化三胜）；代价 = 异构 CLI worker 派单需自建（1–2 周）、RBAC 需外补。OpenClaw 仅在"已有 Mission Control 运维 + 愿做安全加固 + 重审批流"时更优。来源：[Hermes Kanban](https://hermes-agent.nousresearch.com/docs/user-guide/features/kanban) · [OpenClaw 安全危机](https://dev.to/waxell/the-openclaw-security-crisis-135000-exposed-ai-agents-and-the-runtime-governance-gap-e26) · [Composio GitLab×Hermes](https://composio.dev/toolkits/gitlab/framework/hermes-agent)。

### B.4 业界 Agent 治理平台对照（第三步治理层旁证）
| 平台 | 生命周期治理 | 互联 / A2A | 价值度量 |
|---|---|---|---|
| **Workday** Agent System of Record | register→configure→activate→monitor→deactivate；权限 mirror 人类 | — | Analytics Hub：ROI/成本 |
| **ServiceNow** AI Agent Orchestration | 正式 6 阶段 + 运行时 kill switch + role masking | 跨 30+ 平台发现 | Value Dashboard |
| **IBM** watsonx Orchestrate | Agent Control Plane（预览）| **原生 A2A + AgentCard** | 评测 + 运行时成本 |
| **Salesforce** Agentforce | Activate/Deactivate + 双执行身份 | 无跨平台发现 | Agentforce Analytics |

> A2A（Google → Linux Foundation，[150+ 机构生产用](https://www.prnewswire.com/news-releases/a2a-protocol-surpasses-150-organizations-lands-in-major-cloud-platforms-and-sees-enterprise-production-use-in-first-year-302737641.html)）正成跨平台互联事实标准；选用育留备退（含"退/kill switch"）是企业刚需——均为第三步素材。
