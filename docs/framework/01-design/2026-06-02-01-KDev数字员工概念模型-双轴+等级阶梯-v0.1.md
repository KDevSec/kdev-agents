## 0. 核心命题（主干）

> **能力 → 数字员工 → 持续升级**
>
> ① **能力** ＝ 通用 Agent ＋ Skill／规范／模板 ＝ 专用 agent（某类活专家）
> ② **数字员工** ＝ 多个「能力」＋ 编排能力 ＝ 自主调度、端到端交付
> ③ **持续升级** ＝ ＋ 专有知识库（持续注入）→ 反哺能力 → 越用越强（复利）

**关键区分**：「能力」≠「数字员工」。能力是**building block**（专用 agent）；数字员工是**多能力 + 编排**的端到端交付单元。

---

## 1. 总纲

### 1.1 三段主干 + 金字塔

数字员工是**自下而上逐层组装**起来的（金字塔，顶窄底宽）：

```
                    ▲ 数字员工（编排 + 多能力 → 端到端交付）
                ▲ 能力（= 专用 agent · 某类活的专家）
            ▲ 底层技能：Skill ＋ 规范 ＋ 模板 ＋ 知识库
        ▲ 通用 Agent（底座 · 推理 + 工具/MCP + 模型选型）
```

底座 → 底层技能 → 能力 → 数字员工，再叠加 ③ 持续升级的复利飞轮。

### 1.2 两个补充维度（不在主干上）

- **自主度**（单个数字员工的成熟度）：L1 助手 → L4 自治（§4），类比自动驾驶 L1–L5。
- **组织**（多个数字员工组队）：6 人 AI 团队 ＝ 28 agent（§3）。
- **平台底座**（横跨所有员工）：kdev-staff / kdev-core / kdev-memory ＋ 治理测评（§6）。

## 2. ① 能力 ＝ 通用 Agent ＋ Skill／规范／模板

| 组件 | 是什么 | 例 |
|---|---|---|
| **通用 Agent** | 通用大脑：推理 ＋ 工具/MCP ＋ 模型选型（Opus/Sonnet） | Claude 底座 subagent |
| **Skill** | 专业 know-how | TDD 实现 / 需求澄清 |
| **规范** | 质量约束/标准 | 编码安全规范 / 评分标准 |
| **模板** | 产出结构 schema | 实施计划 / 评分表 |

> 四件组合 ＝ **专用 agent**（测试用例 agent、安全编码 agent、需求澄清 agent…）。
> **连接器/MCP** 是通用 Agent 的工具能力，不单列；**知识库**不在能力核心里 → 见 ③ 持续升级。

---

## 3. ② 数字员工 ＝ 多能力 ＋ 编排 → 6 人 AI 团队（28 agent）

> 多个「能力」由编排能力组装成端到端交付的数字员工；多个数字员工组成一支 **6 人 AI 团队**。详见 [v1.5](./2026-05-28-01-KDev数字员工架构-员工能力专项-v1.5.md)。

### 3.1 组织架构

```
                         用户
                          │（只跟 CEO 对话）
                  ┌───────▼────────┐
                  │  CEO 主控员     │ 对话 / 派单 / 决策 / 升级裁决
                  └───────┬────────┘
          ┌───────────────┴───────────────┐
   ┌──────▼──────┐                ┌────────▼────────────────────────┐
   │ CQO 元监督   │←── 元监督 ───→ │ 3 交付员工（每个 = 一个数字员工）│  ＋  评审专家（收敛把关·7 核心评审）
   │ 流程合规·阻断 │                └─ 需求架构师5 / 开发工程师6 / 测试工程师4 ─┘
   └─────────────┘                   └── 编排间直接协作（不经 CEO）· 产出 → 评审专家收敛评审 ──┘
```

- **评审专家收敛把关**：3 个交付员工（需求/开发/测试）产出 → **评审专家收敛汇总评审**（起步 7 核心；L4 完整 18 视角，见 v1.5），是质量收敛点，与 3 个交付员工区分。

### 3.2 28 agent 拆解

`28 agent = 1 CEO ＋ 1 CQO ＋ 4 编排 ＋ 15 业务能力（需求5＋开发6＋测试4）＋ 7 评审能力`

> **7 必要核心评审**（对齐 R→D→T→F，宣传期可落地）：需求评审、方案评审、代码评审、质量评审、安全评审、测试评审、终审聚合。
> 其余评审（原型 / 可观测 / 文档 / 合规 / 可访问性 / 上线就绪 / 架构细分等，**完整 18 视角见 [v1.5](./2026-05-28-01-KDev数字员工架构-员工能力专项-v1.5.md)**）按需扩展。

### 3.3 评审治理（CQO 把关）

`R 需求 → D 开发 → T 测试 → F 终审`（4 阶段 · 7 核心评审闸）。**双重通过门**：总分 ≥ 阈值（默认 80，关键 85）**且**阻断性 = 0 才放行；3 次不过 → 升 CEO 交用户拍板。

## 4. ② 自主度分级（L1 助手 → L4 自治）

| 级 | 名 | 人/机分工 | 按级长出的机制 | 质量把关 |
|---|---|---|---|---|
| L1 | 助手 | 人驱动、执行单步 | 能力起步 | 自评审 |
| L2 | 协同 | 员工做完整任务、人编排确认 | ＋编排 ＋记忆 | 记忆增强自评审 |
| L3 | 自主 | 端到端自跑、人管异常 | ＋协作 ＋第三方评审 | 独立第三方评审 |
| L4 | 自治 | 自监督 / 自演进 | ＋元监督 ＋自演进 ＋跨 IDE | 元监督 |

> 此轴只长**自主/可靠/治理**类机制。**知识库 / MCP 不在此轴**——它们属「能力组成」与「③ 持续升级」。

---

## 5. ③ 持续升级 · 复利飞轮

```
① 数字员工干活（需求→验收）
        ↓
② 过程沉淀进 专有知识库（持续注入）
        ↓
③ 知识反哺能力（下次更准更快）
        ↓
④ 越用越强（复利上升）── 回到 ①
```

- **知识库是持续升级引擎，不是静态组成、不挂级别**；形态 md·wiki 或 RAG 皆可、不分高低。
- 它在金字塔里作为「底层技能」资产存在，并通过飞轮**持续注入、反哺增长**——越用越强。
