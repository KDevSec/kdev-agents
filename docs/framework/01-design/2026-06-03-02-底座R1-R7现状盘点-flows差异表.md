## 0. 一句话结论

**两个 flow 实现成熟度严重不对称**：
- **design-flow 有 Python `lib/flow_state.py`**——R1/R2/R3 的状态 schema 已是**原子写代码**（slug / stage / iter / review_mode / history）。
- **coding-flow 全在 SKILL.md prompt**——13 节点 / 3 Gate / Auto Mode 都是提示词，**零持久化、不能 resume**。

→ 底座的第一桶金 = **把 design-flow 的 `flow_state.py` 泛化成 flow-agnostic 的 R1/R2/R3 模块，让 coding-flow 也用上**（它现在最缺的就是持久化 + resume）。

## 1. 两个 flow 的实现方式

| | design-flow（需求架构师雏形） | coding-flow（开发工程师雏形） |
|---|---|---|
| 状态持久化 | ✅ `lib/flow_state.py`（Python，atomic write，`.kdev/design-flow/<slug>/flow-state.json`） | ❌ 无，全靠 prompt + worktree |
| 节点结构 | 4 段（SR→AR→原型→详设），stage 存在 state | 13 节点 SOP，全在 prompt |
| 闸门 | 评审重试闸门（ai/both/human × 3 iter，`references/review-gate-prompt.md`）| 3 判断 Gate（A 关联/B 复杂/C E2E）+ Per-Increment E2E Gate |
| 自主模式 | ❌ 无（交互式按 review_mode） | ✅ Auto Mode（完全自主） |
| 测试 | ✅ `tests/test_flow_state.py` | evals（benchmark），无 state 单测 |

## 2. R1–R7 差异表

| | 职责 | design-flow 现状 | coding-flow 现状 | 共性（可抽） | 差异（需泛化） |
|---|---|---|---|---|---|
| **R1** | 状态持久化 + resume | ✅ `flow_state.py`：slug / status / created/updated / atomic 写 / read 抛 FlowStateError | ❌ 无持久化 | **design-flow 的 flow_state.py = 抽取基底** | coding-flow 要新增 resume；path 从 `design-flow/<slug>` 泛化为 `<flow>/<slug>` |
| **R2** | 多节点状态机 | `current_stage 1..4` 存 state；节点逻辑在 prompt | `13 节点`全在 prompt，无 state | 「当前节点 + 进度」入 state 的模式 | **节点数不同（4 vs 13）→ 底座要做"通用 n 节点状态机"，节点表由各 flow 声明** |
| **R3** | 评审闸门 | **评审重试型**：review_mode(ai/both/human) + current_iter(1..3) + history[{stage,iter,verdict,reviewer}] + gate prompt（VERDICT PASS/FAIL + UNCHECKED + ISSUES + REVISIONS）| **判断/验收型**：Gate-A/B/C（人工决策）+ Per-Increment E2E Gate（100% PASS 才推进）| iter/retry + verdict + history 的记录结构 | **⚠️ R3 是 3 个子型**：①评审重试 ②人工判断决策 ③E2E 验收。底座 R3 要统一抽象这三类 gate |
| **R4** | 自主模式 | ❌ 无 | ✅ Auto Mode（跑到人工 Gate 才停；LSP 未装不阻塞等自适应）| — | **只 coding-flow 有 → 从它抽 Auto Mode 引擎，design-flow 可选启用** |
| **R5** | 派单分工 | 轻（sonnet/opus 提及） | ✅ CLAUDE.md 二档 + subagent 派单（节点 6b）| 二档分工约定 | coding-flow 更完整，抽它 |
| **R6** | 产物 / 交接 | `references/output-merge-rules.md`（产物合并规则）| 三件套 + worktree + sop-execution-log | 「阶段产出落盘 + 交接」 | 产物路径/格式不同，底座给通用 handoff 约定 |
| **R7** | 调度能力插件 | 串 spec-kit / frontend-design（`/skill`）| 串 secure-coding（节点10）/ spec-kit 等 | 都用 Claude Code 原生 `/skill` 调用 | 基本无需底座新建，登记调用关系即可 |

---

## 3. 关键发现

1. **R1 有现成金种子**：`flow_state.py`（86 行，atomic write + FlowStateError + 单测）直接泛化即可，**coding-flow 接上就白捡 resume 能力**——这是最高 ROI 的第一步。
2. **R3 不是一回事，是三类 gate**：评审重试（design-flow）/ 人工判断决策（coding Gate-A/B）/ E2E 验收（coding Gate-C）。底座 R3 抽象必须能装下三类，否则两个 flow 接不进来。这是 R3 设计的核心难点。
3. **R2 节点数不同（4 vs 13）**：底座做"通用 n 节点状态机 + 各 flow 声明节点表"，不能写死段数。
4. **R4 只 coding-flow 有**：Auto Mode 引擎从它抽，design-flow 可选启用（它现在是交互式）。
5. **R5/R7 基本现成**：二档派单 + `/skill` 调用都是约定层，底座登记即可，工作量小。

## 4. 抽取优先级（对应详细设计 §7 C-3~C-6）

| 步 | 抽什么 | 依据 | 优先级 |
|---|---|---|---|
| **C-3 / R1** | 泛化 `flow_state.py` → `kdev-core` flow-state（flow-agnostic，path `<flow>/<slug>`）| 金种子现成 | 🔴 最高 |
| **C-4 / R2** | 通用 n 节点状态机 + 节点表声明接口 | 4 vs 13 需泛化 | 🔴 高 |
| **C-5 / R3** | 三类 gate 统一抽象（评审重试 / 判断 / 验收）| 最难、最关键 | 🔴 高（设计先行）|
| **C-6 改造** | design-flow / coding-flow 改调公共层；coding-flow 白捡 R1 resume | dogfood 验证 | 🟡 |
| C-7 / R4 | Auto Mode 引擎（从 coding-flow 抽）| 按需 | 🟡 |
| C-7 / R5,R6,R7 | 二档派单 / handoff / skill 调度登记 | 约定层 | ⚪ 低 |

> **下一步建议**：先把 **R3 三类 gate 的统一抽象**设计出来（最难、卡 C-4/C-5），再动 R1 泛化（最易、ROI 最高）。
