## 0. 一句话

数字员工记忆 = **三层（操作 / 叙事 / 任务状态）× 多 scope（CEO-shared + per-业务员工）**；kdev-staff **依赖** kdev-memory（叙事层），业务员工高频操作走轻量 JSONL，CEO 会话续航走 kdev-memory 叙事层——token 与续航各得其所。

---

## 1. 背景：为什么不能直接套现有 kdev-memory

- **kdev-memory 是"单流"设计**：单 `执行日志.md` + 单 `main-N` counter + 单 `当前状态.md`，为"一个用户 + 一个 Claude"线性工作设计。
- **数字员工是多 agent 并行**：需求架构师写 SR 时评审专家在 review，每个员工要记**自己的** Step 流，一条合并日志塞不下多员工并行。
- **全用 kdev-memory 重 markdown → token 爆**：每条 Step 是 4 段（执行事实 / 模型自评 / 用户评分 / 评分差异）~500-600 token，是给人读 + 蒸馏设计的；agent 知道"做了啥"不需要这些。`N 员工 × M 动作 × 600` 不可承受。

**结论**：既不收紧 kdev-memory（多用户协作仍要它），也不照搬——**扩 scope（多流）+ 分层（轻操作 / 重叙事）**。

---

## 2. 依赖关系（反转旧决策）

```
kdev-staff ──硬依赖──▶ kdev-memory   （拿叙事/续航底料 + 决策/踩坑共享）
kdev-memory                           （仍可单装：默认单 scope，行为不变）
```

- **kdev-staff 硬依赖 kdev-memory**：不再各起炉灶复制执行记忆。
- **kdev-memory 单装仍干净**：无 staff 时只有默认 1 个 scope，UX 与现状一致。
- **本条取代**整体架构 §3.3「kdev-staff 不依赖 kdev-memory / 可选」——是有意识的反转，理由见 §1（执行记忆是真共享，分两套反而把最该共用的复制两份）。

## 3. 三层记忆模型

| 层 | 形态 | 谁用 | 频率/成本 | 落位 |
|---|---|---|---|---|
| **操作层** | **JSONL**（events，scoped）| 业务员工读写"做了啥/要做啥" | 高频 / 1 行 ~30 token，grep·jq 过滤 | `.kdev/events/<scope>.jsonl`（由 02 §1.6 的 events.log 泛化成 per-scope，kdev-core）|
| **叙事/蒸馏层** | **markdown**（4 段 Step / 决策 / 踩坑）| 人类续航 + 训练数据 + CEO 会话续航 | 低频 / 重，**派生 rollup** | kdev-memory（scope-aware）|
| **任务状态** | JSON | agent 操作"当前在哪/下一步" | 实时 | `flow-state.json`（kdev-core，见整体架构 §1.2 / 底座总纲 §4.1）|

> **配置（非记忆）**：CEO 顶层约束（质量门槛 / 评审编排 / 合规基线）在 `flow-config.yml`，不进任何记忆层，别混。

**关键**：markdown Step **不再每动作每员工写一次**，而是**从 JSONL events 周期性 rollup 派生**（阶段完成 / 每日 / 按需）。agent 全程只碰便宜的 scoped JSONL；重 markdown 一个阶段才聚合一次。

## 4. scope 拓扑（统一多用户 / 多员工 / 多分支）

多用户、多员工、多分支本质都是 **"独立工作流 + 共享层"** —— 用统一的 scope 抽象：

**单装 kdev-memory（默认 1 scope，= 现状扁平态，UX 不变）**：
```
.kdev/memory/                ← scope=default（扁平）
├── 执行日志.md · 决策日志.md · 踩坑日志.md · skill-feedback.md · 当前状态.md
├── state/ · checkpoints/ · dataset/
```

**装了 staff（多 scope）—— 完整目录（全路径，含 kdev-core 与 kdev-memory 两摊）**：
```
.kdev/                                   ← 独立记忆仓 nested repo（kdev-sync.yml 自举）
├── memory/                              ← kdev-memory 管（叙事/经验层）
│   ├── shared/                          ← 项目级 scope（= 单装扁平内容归此）
│   │   ├── 决策日志.md · 踩坑日志.md(默认) · skill-feedback.md
│   │   ├── 当前状态.md（CEO 续航 + TODO 工作计划）
│   │   ├── 执行日志.md（项目时间线 rollup）· CQO 合规基线
│   ├── staff/<员工>/                    ← per-员工 scope
│   │   ├── ceo/ · 需求架构师/ · 开发工程师/ · 测试工程师/
│   │   ├── 评审专家/   → 评审经验（subject:review:<能力>, F-NNN）
│   │   ├── cqo/        → 审计报告
│   │   └── （各员工自己的执行 rollup）
│   ├── state/          ← ❌ hook plumbing（counter/去重/pending）本机不托管
│   ├── checkpoints/    ← ❌ PreCompact 快照 本机不托管
│   └── dataset/        ← 🟡 蒸馏产物 按需导出
├── events/<scope>.jsonl                 ← kdev-core 操作层（做了啥，per-scope）
├── flows/<flow>/<slug>/flow-state.json  ← kdev-core 任务态
├── handoffs/<员工>/                     ← kdev-core 员工产物
└── hud.md                               ← ❌ kdev-core 运行时 本机不托管
```

> **扁平 → scoped 迁移**：当前 dogfood 仓是**扁平态**（`执行日志.md` 等直接在 `.kdev/memory/` 下，= 默认/单装 scope）。staff 化时：扁平内容归入 `memory/shared/`，再长出 `memory/staff/<员工>/` 子 scope，并出现 `.kdev/{events,flows,handoffs}/`。**§5.3 master 表里的 `/staff/<员工>/` 即 `.kdev/memory/staff/<员工>/` 的简写。**

- **scope = 工作流身份**：协作者 / 员工 / 分支，统一处理；外加一个 `shared/` 共享层。
- **counter 泛化**：kdev-memory 已有 per-分支 counter（`Step main-N` / `Step <branch-slug>-N`，worktree 并发）——把 scope 从"分支"推广到"员工"是同台机器的自然延伸（`Step 开发工程师-N`），非另造。
- **借** CrewAI MemoryScope（路径前缀隔离 + shared 合并），落在**引擎级**（不止评审）。

## 5. 各员工记忆归属

| 员工 | 特性 | 会话/叙事 | 操作 | 其它 |
|---|---|---|---|---|
| **CEO（人机接口）** | 低频、续航 critical | **kdev-memory 叙事层 · shared scope**（续航/pending/checkpoint/brief）| 派单事件 JSONL（ceo scope）| 顶层约束 → flow-config |
| 需求架构师 / 开发 / 测试 | 高频、机器 | per-员工 rollup（按需）| **JSONL 操作 per-员工 scope** | 任务态 → flow-state |
| 评审专家 | 高频 + 经验飞轮 | 同上 | JSONL 操作 | **评审经验 = 评审专家的员工记忆**：`/staff/评审专家/` scope + subject:review:<能力> 子路由 + F-NNN 通道 |
| CQO | 监督 | 合规基线（shared）| 读 `.kdev/events/<scope>.jsonl`（per-scope 全读）审计 | — |

### 5.1 CEO 会话记忆 = kdev-memory 的老本行

CEO 是唯一跟人对话的员工（低频、续航 critical），它的"会话记忆"**不是存整段 transcript**（那是 harness/压缩管的），是 **distilled 续航**，正好由 kdev-memory 现成机制承载：

- **当前状态.md** frontmatter（`phase` / `pending_decisions` / `current_step`）= "到哪了 / 有啥待拍板"
- **决策日志 Q/G** = 对话里跟用户拍过的板
- **checkpoints/压缩前-\*.md**（PreCompact hook）= 上下文压缩前会话快照
- **SessionStart `<brief>` 注入** = CEO 醒来召回"昨天聊到哪"

CEO 代表项目跟人对接 → 它的会话续航是**项目级 / shared scope**（不是私有 CEO silo），CEO 是 shared 叙事的管家。

### 5.2 评审经验 = 评审专家的员工记忆（`/staff/评审专家/`）

评审经验（评得对/松/紧、评分调整建议）是**评审专家这个员工自己的记忆**——下次评审要 recall 自己的历史校准。所以它落在 **评审专家的 per-员工 scope `/staff/评审专家/`**，不是飘在扁平通道：

- **scope**：`/staff/评审专家/`（评审专家私有 + recall 时自动并 `/shared` 通用基线）
- **子路由**：`subject:review:<能力>`（区分是哪个评审能力的经验）
- **通道**：F-NNN（不占 kdev-memory 的 R-NNN「改进建议」编号）
- **蒸馏**：`dataset-review-by-capability`（复用 kdev-memory 蒸馏管道）

其它员工拿的是评审**结果**（走 handoff），不是评审的校准记忆——校准记忆只在评审专家 scope 内。CQO 元抽查时 `recall(scope="/staff/评审专家", subject:review:<能力>)` 读取。

| # | 记忆类型 | 谁产 | store · 层 | scope | git |
|---|---|---|---|---|---|

| 16 | 蒸馏 dataset（by-subject / by-capability）| `/distill` | kdev-memory · 派生产物 | — | 🟡 按需导出（不默认托管）|

> **`state/`（#14）≠ 任务态（#2）**：`state/` 是 kdev-memory hook 的内部记账（计数/去重/提醒），非语义、本机不托管；flow-state（#2）是 kdev-core 编排的语义任务态（到哪了/下一步），托管以便接力续作。两者层级与 git 待遇相反，不可合并。

### 5.4 TODO / 工作计划持久化（防压缩丢失）

主会话（CEO）的 TodoWrite 列表 = **CEO 级工作计划**（区别于 per-flow 的 flow-state），现状只活在上下文里、**压缩会丢**。

- **落点**：kdev-memory 叙事层 **shared scope**（随 CEO 会话续航，写入 `当前状态.md` 的工作计划段或专门 todos 快照）。
- **触发**：TodoWrite 变更时 / PreCompact hook → 落盘；**SessionStart 随 brief 恢复**。
- **效果**：上下文压缩、换会话、换机器都不丢"要做啥/做到哪"。

## 6. token 优化：两层表示 + 派生 rollup

| | 不优化（全 markdown）| 优化（JSONL 操作 + 派生）|
|---|---|---|
| 写 | N 员工 × M 动作 × ~600 token | N × M × ~30（JSONL）+ 几次 rollup |
| agent 召回 | 读整本 markdown | 读自己 scope 的 JSONL 切片（node 过滤）|

- **agent 召回** = `recall(scope="/staff/dev-engineer", node="D2")` 只捞相关 JSONL 行，不碰重 markdown。
- **markdown rollup** = 阶段/每日/按需，从 events 聚合，一次性付费。

---

## 7. 不重复 / 归属边界（防双记浪费）

- **每事件只在自己 scope 的 JSONL 记一次**；markdown 是派生，不双写。
- **评审反馈路由**（互斥 subject 空间）：
  - 对**评审能力**的反馈 → 评审专家员工记忆 `/staff/评审专家/` scope，`subject:review:<能力>`（F-NNN）
  - 对**外部 skill/工具**的反馈 → kdev-memory，`subject:plugin:X`
- **决策 / 踩坑**：项目级 → `shared/`；某员工特有 → 其 scope。

## 9. git 托管方案（已定 · Q-009）

记忆与**代码分支彻底解耦**，走**独立记忆仓（nested repo）+ 自举 hook**。

### 9.1 三件套

| 件 | 内容 |
|---|---|
| **代码仓**（KDevSec/kdev-agents）| `.gitignore` 保持 `/.kdev/`（彻底无视，不 track 不钉指针）+ tracked 的 `kdev-sync.yml`（只记"记忆仓地址"，随 clone 到任何机器，静态不刷历史）|
| **记忆仓**（独立 nested repo，自己 remote）| `.kdev/` 内容；per-scope 文件 + Step-ID 前缀（`<branch/scope>-N`）防并发撞 |
| **自举 hook**（kdev-memory ship）| SessionStart：无 `.kdev/.git` → 从 `kdev-sync.yml` clone；有 → `pull`。SessionEnd/rollup → commit + push |

### 9.2 为什么 nested repo（而非 submodule / track-in-main / 单独分支）

| 方案 | 否决理由 |
|---|---|
| 单独 kdev-sync 分支 | 切到 main 记忆不在场 + 切换复杂 |
| track 进 main | 记忆变代码分支内容 → 切分支/worktree 割裂、未提交挡 checkout、刷历史 |
| submodule | 父仓钉 per-commit 指针 → 切分支记忆跟着变 + 拿旧 SHA + 刷 main 历史 |
| **nested repo（取）** | 父仓彻底忽略 → 切分支/worktree 不碰记忆；自举 hook 给 submodule 的"自动带下来"便利，却**永远 pull 最新**、零指针污染 |

### 9.3 跨机 / worktree

- **新机器自举**：`git clone 代码仓` 带来 `kdev-sync.yml` → 首跑 SessionStart hook 读地址自动 clone 记忆仓 → 零手动。
- **多 worktree**：symlink 指向同一 canonical `.kdev/` → 记忆一份不割裂；Step-ID 前缀防并发编号撞。

### 9.4 🔴 托管提醒（kdev-memory 强制行为）

- **触发**：① 建立工程记忆时 ② 检测到 `.kdev/` 未托管 git（无 `.kdev/.git` 且 `kdev-sync.yml` 无远端）。
- **动作**：**中文提醒用户，强烈建议设为 nested repo 独立同步**（否则跨机 / 团队 / 换机将丢记忆）。
- **话术（中文示意）**："检测到工程记忆尚未 git 托管。强烈建议建立独立记忆仓（nested repo）并在 `kdev-sync.yml` 登记，以便跨机器 / 团队同步、换机不丢记忆。是否现在初始化？"
- **强烈建议、非强制**：用户可拒（仅本地，自负丢失风险）。

### 9.5 隐私待确认

记忆含用户 verbatim 原话 → 推到**团队共享**记忆仓 = 原话进团队仓（个人仓无碍）。用户留意；必要时团队仓做脱敏或限私有。
