## 0. 一句话

底座 = **完整框架、渐进实现**：核心是「**状态 + 可编排节点机 + 三类闸门**」，框架上预留派单/产物/事件/HUD，**对接 kdev-memory 不依赖**，重型评审/CQO/自演进 defer。**node 是编排层产物（不是中心契约）；phase 才是普适基础。**

---

## 1. 底座组成框架（四层）

| 层 | 块 | 状态 |
|---|---|---|
| **① 核心层**（lean，先建） | **R1 状态** flow-state（操作态：current_node + gate 流水 + status + resume）<br>**R2 编排-结构** node-table（唯一节点词表）+ gates（可编排 config，默认/用户/自主三层）<br>**R3 闸门** 三类 gate（review/decision/acceptance）统一抽象 + 谁来判旋钮 | 🔴 先建 |
| **② 框架层**（骨架完整，渐进填充） | **R5 派单**（二档分工 helper）<br>**R6 产物总线**（handoff 约定）<br>**事件流** events.log（给 HUD/CQO 打底）<br>**HUD**（进度可见：statusLine / hud.md） | 🟡 框架在、渐进填 |
| **③ 对接层**（不内建，留接口） | **记忆接口**：底座暴露 `current_node` → kdev-memory `phase_node` 引用；跨 store 按 node 联合查询；**不依赖 kdev-memory 运行时**<br>**能力调度 R7**：flow prompt 按 `current_node` 调 `/skill`，**不在底座** | 🟡 接口 |
| **④ Defer**（L3-L4） | 评审素材记忆 / flow-config-评审 / CQO 监督 / 知识库注入反哺 / **L2 自主编排的执行** | ⚪ 后置 |

> **核心边界（[lean §0]）**：底座**只记账 + 流转**，**不做判断本身**（评审/E2E/决策都在 flow 侧由 AI prompt 或人执行，结果喂回底座）。

---

## 2. 三块边界（状态 / 编排 / 记忆）

| 块 | 谁管 | 存哪 | 边界 |
|---|---|---|---|
| **状态（操作态）** | 底座自存 | `.kdev/flows/<flow>/<slug>/flow-state.json` | "这次跑到哪 / gate 过没过 / 能否 resume"；临时，流程结束归档 |
| **编排** | 底座持有**结构**，flow 负责**执行** | node-table + gates（config）| 结构进底座（可编排）；"每节点调哪个能力"留 flow prompt |
| **记忆（经验态）** | kdev-memory（独立插件，复用）| `.kdev/memory/`（独立 store）| "跨流程/会话学到啥"（踩坑/决策/评审反馈）；持久 |

**两个 store 各管各、互不依赖**：底座不依赖 kdev-memory 运行时（守 [Q-007]）；kdev-memory 可单装。**只在 L3 评审经验沉淀时对接**（评审素材 → kdev-memory，那时再设计）。

## 3. phase（普适） / node（编排层） 两层元数据

**node 不是中心契约——它是编排层产物**，编排/配置定了才有。普适基础是 `phase`。

| 维度 | 是什么 | 何时有 | 谁拥有 |
|---|---|---|---|
| **`phase`（coarse·普适）** | 工作性质（需求/设计/开发/测试/调试/评审/安全扫描/部署/文档/其他，10 值）| **恒在** | kdev-memory 内置（草案 2026-05-30-01，**待实现**）|
| **`node`（fine·编排层）** | 流程位置（在 spec 哪个节点）| **编排存在时才有** | 底座 node-table（唯一来源）|

### 3.1 node 三来源优先级

| 优先级 | node 从哪来 | 条件 |
|---|---|---|
| 1 | **底座 node-table** 的 `current_node` | kdev-core 在 + flow 在跑 |
| 2 | **项目 `.kdev/memory/config.yaml`** | 单装 memory + 项目配了（例：v1.5 §5.1）|
| 3 | **无 node**，只有 `phase` 兜底 | 都没有 |

> **无"kdev-memory 内置通用 node"这一档**（已核实：插件未实现 phase_node；草案里内置的只有 coarse `phase`，node 一直是项目 config 驱动）。严格 **node = 编排/配置产物**，无配就 phase 兜底。

### 3.2 跨 store 用 node 联合查询（都装时）

```
   phase（普适·恒定维度）──── 跨记录蒸馏切片
   node （编排层·都装时三方共用）
        ├─ 底座 flow-state.current_node（活态）
        ├─ kdev-memory phase_node（历史："在哪个节点踩的坑"）
        └─ flow prompt（按 current_node 执行）
```

底座是 node 唯一来源；kdev-memory `phase_node` 引用之 → "D2 节点的所有信号"可跨 store 联查。

## 4. 跟现有的关系

| 现有 | 关系 |
|---|---|
| **kdev-memory** | 独立 store，底座不依赖；都装时 `phase_node` 引底座 node；单装用项目 config 或只 phase |
| **13 能力插件** | flow prompt 按 node 调它们（R7，不在底座）|
| **design-flow / coding-flow** | 改造为：声明 node-table + 调底座状态/闸门；coding-flow 白捡 resume |
| **kdev-core v0.1 详细设计** | schema 复用（state/events/hook/plugin.json），重心已挪到 R2/R4 编排引擎 |
