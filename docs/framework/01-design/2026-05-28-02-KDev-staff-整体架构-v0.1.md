### 0.2 三个 plugin 一句话定位

| Plugin | 一句话定位 | 服务对象 |
|---|---|---|
| **kdev-staff** | 6 人 AI 公司业务实现 | 用户的软件交付任务 |
| **kdev-core** | 业务运行时基础设施 | kdev-staff（未来其他 plugin）|
| **kdev-memory** | 用户工程记忆 + 范式定义 | 用户本人（跨会话续航）|

### 0.3 关键认知

| 不是 | 是 |
|---|---|
| ❌ kdev-memory 是 KDev 生态的"记忆中枢" | ✅ kdev-memory 只管用户的工程过程记忆；评审素材是 kdev-core 的事 |
| ❌ kdev-core 是 oh-my-claudecode 那种 everything framework | ✅ kdev-core 只做 4 件事：state/HUD/hook/评审素材 |
| ❌ kdev-staff 要"注册到 kdev-memory" 才能存评审 | ✅ kdev-staff 调用 kdev-core 的评审素材接口；kdev-memory 无运行时依赖 |
| ❌ kdev-memory 要升级 multi-namespace | ✅ kdev-memory v0.10 不动；范式（markdown / R-NNN / verbatim / subject）作为生态共识被 kdev-core 借鉴 |
| ❌ OMC TypeScript 代码直接 fork | ✅ OMC 仅作**架构参考**（Notepad 生命周期 / 6 类 JSON state / Team Pipeline）|

#### 1.1.2 `.kdev/events.log` —— 事件流

JSONL 格式，全员高频 append，CQO 元监督依赖：

```jsonl
{"ts":"2026-05-28T14:33:12Z","actor":"开发工程师/编排","event":"capability_started","capability":"TDD实现","node":"D2","retry":0}
{"ts":"2026-05-28T14:55:08Z","actor":"开发工程师/编排","event":"capability_completed","capability":"TDD实现","node":"D2","handoff":".kdev/handoffs/开发工程师/D2-iteration1/"}
{"ts":"2026-05-28T14:55:09Z","actor":"开发工程师/编排","event":"协作请求","to":"评审专家/编排","intent":"review","subject":["代码评审","质量评审"]}
{"ts":"2026-05-28T15:02:33Z","actor":"评审专家/代码评审","event":"review_scored","score":84,"blocking":0,"path":".kdev/reviews/D2-代码评审/评分-1.md"}
{"ts":"2026-05-28T15:08:11Z","actor":"评审专家/编排","event":"review_aggregated","passed":true,"to":"开发工程师/编排"}
```

**事件类型 v0.1 清单**：

| 事件 | 触发者 | 关键字段 |
|---|---|---|
| `capability_started` | 员工编排 | capability / node / retry |
| `capability_completed` | 员工编排 | handoff |
| `协作请求` | 员工编排 | to / intent / subject |
| `review_scored` | 评审能力 | score / blocking / path |
| `review_aggregated` | 评审专家编排 | passed / to |
| `blocked` | 业务能力（连续 3 次失败）| reason / retry_count |
| `escalated_to_ceo` | 员工编排 / 评审专家编排 | reason |
| `user_decision` | CEO 转用户拍板后 | choice (a/b/c) |
| `cqo_audit` | CQO | severity (🔴/🟡/⚪) / target |
| `flow_config_updated` | CEO | yml_diff |

### 1.3 hook framework

#### 1.3.1 hook 类型清单（v0.1 收敛 7 种）

| Hook 名 | 触发 | 实现者 | 干什么 |
|---|---|---|---|
| `SessionStart` | 会话开始 | kdev-core | 读 state.md + events.log 恢复上下文；CEO 检查 flow-config.yml 是否存在（无则引导）|
| `PreCompact` | 上下文压缩前 | kdev-core | 写 `.kdev/checkpoints/压缩前-<ts>.md` 快照 |
| `UserPromptSubmit` | 用户每次发言 | kdev-core | CEO 检查用户意图（评审反馈 / 命令 / 普通对话）|
| `Blocked` | 业务能力连续 3 次失败 → events.log 写 `blocked` | kdev-core | 自动派该员工编排能力应急（见 v1.5 §3.4）|
| `ReviewRetryExhausted` | 评审循环 == 3 次 → events.log 写 `review_retry_exhausted` | kdev-core | 自动升级到 CEO，触发用户拍板对话（见 v1.5 §5.4.2）|
| `LowScoreAlert`（可选）| 评审评分通过但 < 高阈值 | kdev-core | CEO 摘要给用户："要复核吗？"（人类介入通道 ④）|
| `StageComplete` | 阶段所有节点 COMPLETE | kdev-core | CEO 出阶段摘要给用户（人类介入通道 ①）|

#### 1.3.2 hook 注册规约

kdev-core 在 `plugins/kdev-core/hooks.json` 注册所有 hook handler：

```json
{
  "hooks": [
    {
      "name": "SessionStart",
      "handler": "kdev-core/hooks/session-start.sh",
      "ship_with": "kdev-core"
    },
    {
      "name": "Blocked",
      "handler": "kdev-core/hooks/on-blocked.sh",
      "trigger_pattern": "events.log line matching event=blocked",
      "ship_with": "kdev-core"
    }
  ]
}
```

**关键**：kdev-core 不抢 kdev-memory 已有的 hook（SessionStart 由 kdev-memory ship 一份 + kdev-core ship 一份，各自处理自己关心的部分；Claude Code 原生支持多个 hook 同名注册顺序执行）。

#### 1.3.3 OMC 借鉴

OMC 的 Notepad 3 层生命周期（priority 永久 / working 7 天 / manual 永久）→ kdev-core hook framework 中可以引入"hook 触发时机的生命周期管理"概念（v0.2+ 演进）。v0.1 不引入。

### 1.4 评审素材记忆

#### 1.4.1 物理目录

```
.kdev/
└── staff-memory/                    ← kdev-core 管，评审素材专属
    ├── review-memory.md             ← 评审经验（R-NNN）
    ├── flow-config.yml              ← 流程编排
    └── reviews/                     ← 评审记录（按节点+能力分目录）
        ├── R1.5-需求方向评审/
        │   ├── 评分-1.md
        │   └── 评分-2.md
        ├── R2-SR需求评审/
        │   └── 评分-1.md
        ├── D2-代码评审/
        │   ├── 评分-1.md
        │   └── 评分-2.md
        ├── D2-质量评审/
        │   └── 评分-1.md
        └── ...
```

`.kdev/handoffs/<员工名>/` 仍然是员工产物总线（不在 staff-memory/ 下，因为不是评审素材）。
`.kdev/state.md` / `.kdev/events.log` 在 .kdev/ 根目录（任务状态，不在 staff-memory/ 下）。

#### 1.4.2 review-memory.md schema（R-NNN，借用 kdev-memory 范式）

```markdown
# 评审经验记忆

## R-001
- subject: review:SR需求评审
- type: 评分调整建议
- verbatim: "评得太松了。SR 第 3 条用户角色定义'管理员'没明确权限边界"
- 上下文:
  - 节点: R2
  - 评分: 84/100 / 阻断性: 0
- 用户判断:
  - 评分: 75/100
  - 阻断性: 应标 🟡（用户角色权限边界未明）
- 维度: 完整性维度漏检
- 改进建议: standards/评审专家/SR需求评审.md 完整性维度加"用户角色权限边界明确性"
- 日期: 2026-05-28
```

#### 1.4.3 kdev-core 提供的 3 个接口

```
A. recall_review_memory(subject_filter, max_items=10, max_age_days=180) → 召回 R-NNN 条目列表
B. write_review_feedback(R_NNN_payload) → 落盘新 R-NNN
C. distill_review_memory(by_capability=true) → 导出 dataset-review-by-capability/ 切片包
```

评审能力 prompt 调用接口 A 做 recall；CEO 处理用户反馈时调用 B 落盘；用户说"导出评审训练数据"时通过 slash command 调用 C。

#### 1.4.4 范式借用 kdev-memory 一览

| 范式要素 | kdev-memory（工程记忆）| kdev-core（评审素材，v0.1 借用） |
|---|---|---|
| 存储格式 | markdown 主存 | ✅ markdown 主存 |
| 编号体系 | F/G/Q-NNN | ✅ R-NNN（独立编号，不复用 F-NNN）|
| subject 路由 | 三级推断 | ✅ 单级（review:<能力名>，比 kdev-memory 简化）|
| verbatim | 用户原话不可改写 | ✅ verbatim 保留 |
| recall | UserPromptSubmit hook 注入 brief | ⚠️ 评审能力 prompt 顶部硬规 Read（不走 hook，避免污染对话流）|
| 蒸馏 | markdown 切片包 | ✅ dataset-review-by-capability/ markdown 切片包 |

**关键**：kdev-core 借范式，**不依赖 kdev-memory plugin**——即使用户没装 kdev-memory，kdev-core 也能独立工作。

## 2. kdev-staff 在 kdev-core 上的实施

### 2.1 39 agent 注册到 kdev-core

kdev-staff 在 `plugins/kdev-staff/agents/` 下提供 39 个 agent 定义（详见 v1.5 §3.1）。kdev-core 的 plugin 注册机制扫描该目录并注册到 Claude Code 的 subagent 系统：

```json
// plugins/kdev-staff/plugin.json
{
  "name": "kdev-staff",
  "version": "0.1.0",
  "depends_on": ["kdev-core@>=0.1.0"],
  "agents_dir": "agents/",
  "standards_dir": "standards/",
  "hooks_extension": "hooks/"
```

kdev-core 启动时读取所有依赖 plugin 的 plugin.json，把 agents 注册到 Claude Code subagent，把 standards 加载为 prompt 资产，把 hooks_extension 合并到 hook framework。

### 2.3 评分循环 + CEO 升级 + 低分预警

依赖 kdev-core 提供的 3 个 hook：

#### 2.3.1 评审循环（不需要新 hook，纯 prompt 逻辑）

评审专家/编排在 prompt 里硬规：

```
1. recall（kdev-core 接口 A）
2. 并行派评审能力
3. 汇总 + 双重通过判定（总分 ≥ 阈值 AND 阻断性 = 0）
4. 不通过 → 给修订建议 → 员工编排修订 → 重试 +1 → 回 1
5. retry == 3 → events.log 写 review_retry_exhausted
```

#### 2.3.2 ReviewRetryExhausted hook

kdev-core hook framework 监听 events.log，发现 `review_retry_exhausted` 事件 → 自动派 CEO，触发对话流：

```
CEO 跟用户：
"节点 <id> 评审 3 次未通过，当前 84/100，🟡 X / ⚪ Y
 (a) 接受当前评分，记 tech debt
 (b) 继续修订（第 4 次循环）
 (c) 推翻当前产物，重做"
```

用户选择 → CEO 写 events.log: `user_decision` + 推进流程。

#### 2.3.3 LowScoreAlert hook（可选）

flow-config.yml 中 `low_score_alert: enabled` 时启用。kdev-core 监听 `review_scored` 事件：

```
if score ∈ [threshold, low_score_threshold)
  → 触发 CEO 摘要给用户："评分 84 < 90，要复核吗？(y/n/skip)"
```

### 2.4 人类介入 4 通道（hook + HUD 组合）

| 通道 | 实现机制 |
|---|---|
| ① 阶段聚合摘要 | `StageComplete` hook → CEO 出摘要 + 写 `.kdev/hud.md` |
| ② 异步反馈 | 用户在 reviews/<节点>/<能力>-N.md 加 `## review-feedback` section → kdev-core 后台扫描文件变更 → 调用接口 B 落 R-NNN |
| ③ CQO 元抽查 | `StageComplete` hook（terminate 前）→ 派 CQO 读 reviews/ + review-memory.md 出审计报告 |
| ④ 低分预警（可选）| §2.3.3 `LowScoreAlert` hook |

## 3. kdev-memory 共存关系

### 3.1 物理目录隔离

```
.kdev/
├── memory/                ← kdev-memory plugin 管（不动）
│   ├── 执行日志.md
│   ├── 踩坑日志.md
│   ├── 决策日志.md
│   ├── skill-feedback.md
│   └── 当前状态.md
├── staff-memory/          ← kdev-core 管（评审素材）
│   ├── review-memory.md
│   ├── flow-config.yml
│   └── reviews/
├── handoffs/              ← kdev-core 管（员工产物总线）
│   └── <员工名>/
├── state.md               ← kdev-core 管（任务状态）
├── events.log             ← kdev-core 管（事件流）
├── hud.md                 ← kdev-core 管（HUD 实时）
├── flow-config.yml        ← 注：实际在 staff-memory/ 下，此处仅占位
└── checkpoints/           ← kdev-core 管（PreCompact 快照）
```

### 3.2 范式借用（不是依赖）

kdev-core 在评审素材 schema 设计上**借用** kdev-memory 范式：

| 借用要素 | 用法 |
|---|---|
| markdown 主存 | review-memory.md 用 markdown 而非 JSON |
| R-NNN 编号 | 借鉴 F/G/Q-NNN 风格，独立编号系列 |
| verbatim | 用户原话不可改写（核心范式）|
| subject 路由 | 简化为单级 `review:<能力名>` |
| 蒸馏切片包 | dataset-review-by-capability/ markdown 输出 |

**关键**：借用范式 ≠ 引入依赖。即使用户没装 kdev-memory plugin，kdev-core 也能独立工作。

### 3.3 无运行时依赖

| 维度 | kdev-staff 依赖 kdev-core 吗？ | kdev-staff 依赖 kdev-memory 吗？ |
|---|---|---|
| 必需 | ✅ 是（framework）| ❌ 否（独立 plugin）|
| 装机 | 必须装 | 可选 |
| 调用 | Agent / hook / 接口 A-C | 无 |

用户可以**只装 kdev-core + kdev-staff**（不装 kdev-memory）就跑得起来；或者**只装 kdev-memory**（不装 kdev-core + kdev-staff）只用工程记忆功能。两套互不强制。

### 3.4 kdev-memory 受益于 kdev-core 吗？

不强制，但**未来 v0.11+ 可选**：
- kdev-memory 可以注册自己的 hook 到 kdev-core hook framework（统一管理）
- 但 kdev-memory v0.10 自带 hook 也能继续工作

v0.1 阶段：**不做迁移**，两 plugin 完全平行。

## 4. OMC 架构参考点（不引入源码）

**底座路线 = ③ 抽共性渐进（[Q-007]）**：从现有 flows 抽 R1–R7 公共层，**照 OMC 范本但不 fork TS 源码**。历史调研一度提"OMC 当底座"，但 OMC 是 3.3MB TS 编译运行时，与 KDev Python/markdown 哲学冲突、且违从低起步（[Q-004]）——故只**借设计范本**（mode / Team Pipeline / Notepad），不引入源码；L3 自主 / 跨 IDE（v2.x）真要完整多 agent 运行时再**重评采用 OMC**（不 foreclose）。OMC 借鉴点：

### 4.1 OMC Notepad 3 层生命周期 → kdev-core 借鉴方向

| OMC | KDev 借鉴 | 落地时机 |
|---|---|---|
| Notepad priority（永久）| state.md / 评审 standards | ✅ v0.1 |
| Notepad working（7 天自动清理）| events.log / checkpoints/（按 ttl 清理）| 🟡 v0.2 |
| Notepad manual（永不过期）| review-memory.md / handoffs/ | ✅ v0.1 |

### 4.2 OMC 6 类 JSON 状态 → state.md schema 借鉴

OMC 拆 6 个 JSON（session/tasks/quality/agents/notepad/project-memory），KDev 收敛为 1 个 markdown state.md + 1 个 events.log（避免文件碎片化 + markdown 用户可读）。

但 OMC 的"6 类信息分类"思路被 state.md 的 section 结构继承（§1.1.1 已体现：当前阶段 / 员工状态 / warnings 三个 section）。

### 4.3 OMC Team Pipeline 状态机 → kdev-staff 员工编排借鉴

OMC 的 `plan → spec → exec → verify → fix` 状态机思路在 kdev-staff 中演化为：
- `IR (R1) → 需求方向 (R1.5) → SR (R2) → AR (R3) → 原型 (R4) → 方案 (R5) → ...`
- 评分循环 + CEO 升级 = OMC 的"fix 循环"加强版（加百分制 + 双重通过条件 + 用户拍板兜底）

### 4.4 ECC 借鉴（语言规范 / AgentShield）

ECC 的语言规范库已由 `kdev-secure-coding` plugin 在做（python-security-coding 等），不在 kdev-core 范围。AgentShield 远期可考虑独立 plugin（v0.3+ 评估）。

## 5. 落地路径（按 [Q-007] ③ 抽共性渐进）

> 不再"先自建完整 kdev-core 再上 kdev-staff"，改为**从已跑通的 flows 抽公共层、边跑边长**。

| 步 | 内容 | 备注 |
|---|---|---|
| **1 盘点** | 列 R1–R7（状态机 / 节点机 / 评审闸门 / 自主 / 派单 / 产物 / 调度）在 `design-flow`/`coding-flow` 的现有实现 + 差异 | 反推底座该统一什么 |
| **2 抽公共层** | `plugins/kdev-core/`（lean）先做 **R1 状态机持久化 + R2 节点编排 + R3 评审闸门**（L2 最缺的三块）| Python/markdown 原生 |
| **3 改造 flows** | design-flow / coding-flow 改调 kdev-core 公共层，消除各自重复，dogfood 验证 | 真实验证底座 |
| **4 按需补** | R4 自主模式 / R5 派单 / R6 产物 / R7 调度 | 渐进 |
| **5 defer L3** | 评审素材记忆 / flow-config / CQO 监督（详细设计 §4 重型块）等 L3-L4 再上 | 当前 L2 |

> kdev-core v0.1 详细设计（848 行）**不作废**：state.md / events.log / hook / plugin.json schema 仍可复用，重心挪到 R2/R4 编排引擎并以 flows 现状校准。
> kdev-staff（28 agent）实施推后到底座抽出、L3 评审循环上线之后——当前先让 1–2 个数字员工（如开发工程师 = coding-flow）在 lean 底座上跑通。
