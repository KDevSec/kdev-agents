### 0.2 三个 plugin 一句话定位

| Plugin | 一句话定位 | 服务对象 |
|---|---|---|
| **kdev-staff** | 6 人 AI 公司业务实现 | 用户的软件交付任务 |
| **kdev-core** | 数字员工编排底座（R1–R7） | kdev-staff（未来其他 plugin）|
| **kdev-memory** | 用户工程记忆 + 生态范式定义 | 用户本人（跨会话续航）|

### 0.3 架构基准（设计准则）

- **底座只记账 + 流转，不做判断本身**：评审 / E2E / 决策由 flow 侧 AI 或人执行，结果（GateResult）喂回底座。
- **底座是从现有 flows 抽出的公共编排层**：`design-flow` / `coding-flow` 是数字员工雏形，各自实现了 flow-state / 节点机 / 闸门 / Auto Mode，底座抽出它们的 R1–R7 公共层。
- **kdev-staff 硬依赖 kdev-memory**（记忆底料）：记忆分三层——操作层 JSONL（events，per-scope）/ 叙事层 markdown（kdev-memory，scope 多流，派生 rollup）/ 任务状态 flow-state（kdev-core）。详见 [记忆架构-分层+scope 拓扑](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md)。kdev-memory 单装仍干净。
- **评审经验并入 kdev-memory**（`subject:review:<能力>`）：不再单设 kdev-core/staff-memory；kdev-core 仅在 R3 gate 结果时发写信号，不自管评审存储。
- **kdev-core 借 kdev-memory 范式**：markdown 主存 / 编号体系 / verbatim / subject 路由作为生态共识。
- **底座是完整框架、渐进实现**：架构按完整态画（R1–R7 + 四层 + 自主编排），实现按 ROI 分批落地（见底座设计总纲）。

## 1. kdev-core 编排底座（R1–R7）

> 本节给**架构全景**；R1–R7 的完整 schema / API / 实现要点见 [底座设计总纲 §4–§6](./2026-06-05-01-kdev-core底座设计总纲-v1.0.md)。

### 1.1 四层组成

| 层 | 块 |
|---|---|
| **① 核心** | R1 flow-state 状态 · R2 可编排节点机 · R3 三类闸门 |
| **② 框架** | R5 派单 · R6 产物总线 · 事件流 events.log · HUD |
| **③ 对接** | 记忆接口（暴露 `current_node` + R3 gate 结果写信号 → kdev-memory）· R7 能力调度（flow prompt 调 `/skill`）|
| **④ 上层** | CQO 流程监督 · 知识库注入反哺 · L2 自主编排执行（评审经验由 kdev-memory `subject:review` 承接，不在 kdev-core）|

R1/R2/R3 是底座心脏（编排引擎本身）；R5/R6/events/HUD 是骨架；对接层把底座接到 kdev-memory 与能力生态；上层是数字员工的质量与自演进能力。

### 1.2 R1 状态 — per-flow flow-state

每个 flow（= 一个数字员工的一次任务）持有独立状态机文件：

```
.kdev/flows/<flow>/<slug>/flow-state.json
```

```json
{
  "flow": "design-flow", "slug": "user-auth",
  "status": "in_progress",          // in_progress | completed | aborted
  "active": true,                   // stale guard：crash 写 false 而非删，resume 据此判
  "current_node": "n3-prototype",   // 节点 id
  "config": { "review_mode": "ai", "auto_mode": false },
  "history": [ /* GateResult 流水 */ ],
  "_meta": { "written_at": "...", "step_id": "..." }   // 写时注入、读时剥离
}
```

- **per-flow 粒度**：每个员工/flow 独立状态机，并发互不串扰。
- **CEO 聚合视图**：CEO 跨所有在跑 flow 聚合出全局态（阶段 / 员工状态 / warnings），供 HUD 与用户对话；聚合视图是只读派生，权威来源仍是各 flow-state。
- **原子写**：tempfile + `os.replace`（搬 `flow_state.py` 金种子）；损坏抛 `FlowStateError`。

### 1.3 R2 节点机 — 通用 n 节点状态机

各 flow 声明**节点表**（不写死段数：design 4 / coding 13）：

```yaml
nodes:
  - { id: n1-sr, name: SR 需求, kind: action, gate: g-sr }
  - { id: n4-design, name: 概要+详设, kind: action, gate: g-design }
# kind: action（产出后过 gate）| gate（纯判断点，无产物）
```

**advance 三段式**：① 邻接表查 `ALLOWED[from]→[to]` 合法性 → ② 守卫 `guard(state)→str|null`（与边解耦）→ ③ 不可变变更 + 原子写 + 追加 `phase_history`。

**有界回流 + 幂等豁免**：gate 回流须显式标"回流"= retry++；retry 溢出 → 强制 advance 到 terminal-fail，禁无限回流。

### 1.4 R3 三类 gate — 统一抽象

| kind | 谁判（默认 / 可编排） | verdict | 不过 |
|---|---|---|---|
| **review** 评审重试 | ai（默认）/ both / human | PASS/FAIL | 改→重试(max N) |
| **decision** 路线判断 | ai（默认）/ human（编排可接管）| 分支选择 | 选分支即流转 |
| **acceptance** 验收 | test（跑 E2E）/ +human 复核 | PASS/FAIL | 回流 fix |

**GateResult**（flow 执行完喂回底座）携 `request_id`（approval 锁，防跨轮串扰）、`iter`、`verdict`、`by`、`issues/revisions`。

**解析硬规**：review/acceptance → PASS 走 on_pass / FAIL 且 iter<max 回流 / **iter≥max → escalate 转人工 BLOCKED（不 force-accept）**；decision → 按 verdict 选分支。结果用 **structured JSON verdict**（不正则扫文本）。

「谁来判」是**编排可覆盖旋钮**（默认 AI，人按需接管）。Auto Mode 只在 `decider:human` 的 gate 停靠（写 `awaiting_human` + TTL：review 超时自动续 / decision 等人）。

### 1.5 phase / node 两层 + scoped 联查

| 维度 | 是什么 | 何时有 | 谁拥有 |
|---|---|---|---|
| **`phase`（普适）** | 工作性质（10 值）| 恒在 | kdev-memory 内置（叙事层）|
| **`node`（编排层）** | 流程位置 | 编排存在时才有 | 底座 node-table（唯一来源）|

**node 三来源优先级**：① 底座 node-table 的 `current_node`（kdev-core + flow 在跑）> ② 项目 `config.yaml` > ③ 无 node（只 phase 兜底）。

**node 标签 + scoped 联查**：kdev-staff 依赖 kdev-memory；底座把 `current_node` 提供给 kdev-memory，记忆条目据此打 node 标签（配置感知——检测到数字员工才挂 node，否则只 phase）→ "D2 节点的所有信号" 按 node + scope 召回。**这是三层记忆模型的一部分**（操作层 JSONL / 叙事层 markdown / 任务态 flow-state），详见 [记忆架构 §3 / §5](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md)。

### 1.6 events.log — 事件流

JSONL，全员高频 append，CQO 元监督依赖：

```jsonl
{"ts":"...","actor":"开发工程师/编排","event":"capability_started","capability":"TDD实现","node":"D2","retry":0}
{"ts":"...","actor":"开发工程师/编排","event":"协作请求","to":"评审专家/编排","intent":"review","subject":["代码评审","质量评审"]}
{"ts":"...","actor":"评审专家/代码评审","event":"review_scored","score":84,"blocking":0}
{"ts":"...","actor":"评审专家/编排","event":"review_aggregated","passed":true,"to":"开发工程师/编排"}
```

| 事件 | 触发者 | 关键字段 |
|---|---|---|
| `capability_started` / `capability_completed` | 员工编排 | capability / node / retry / handoff |
| `协作请求` | 员工编排 | to / intent / subject |
| `review_scored` | 评审能力 | score / blocking / path |
| `review_aggregated` | 评审专家编排 | passed / to |
| `blocked` | 业务能力（连续 3 次失败）| reason / retry_count |
| `escalated_to_ceo` | 员工 / 评审专家编排 | reason |
| `user_decision` | CEO 转用户拍板后 | choice (a/b/c) |
| `cqo_audit` | CQO | severity / target |

### 1.7 HUD 服务

| 通道 | 实现 | 显示什么 |
|---|---|---|
| CLI statusLine | Claude Code 原生 statusLine（stdin 结构化）| 当前阶段 / 节点 / 员工状态摘要 |
| VSCode markdown preview | `.kdev/hud.md` 动态生成 | 公司架构图 + 当前 node 高亮 |
| 对话流 | CEO 在 chat 输出 markdown 进度条 | 每节点 COMPLETE 时摘要 |

### 1.8 SOP 三层覆盖（可编排 + 自演进）

R2 节点表 + R3 gates = **一个 flow 的 SOP，可编排**：

| 层 | 谁编排 | 位置 |
|---|---|---|
| **L0 默认** | flow 作者 | flow 内 `node-table.yml` |
| **L1 用户** | 用户 | `.kdev/flows/<flow>/orchestration.yml` |
| **L2 自主** | 数字员工自己 | 同上（`source: self`，变更需人确认）|

底座加载 **有效 SOP = merge(L0, L1, L2)**。每个 flow 配编排指南（能改什么 / 怎么改 / 风险）。**L2 自主编排 = 持续升级飞轮反哺 SOP 本身**——数字员工依据历史评审 / 踩坑数据自己调整流程，是数字员工自演进的核心通路。

### 1.9 评审素材记忆

> ## ⚠️ 本节被 [记忆架构-分层+scope 拓扑 §7](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md) 取代
> 评审经验**并入 kdev-memory**（`subject:review:<能力>`，走 **F-NNN 反馈通道**，不占 kdev-memory 的 R-NNN「改进建议」编号）；**不再单设 `.kdev/staff-memory/` 或 review-memory.md**。kdev-core 仅在 R3 gate 结果时发写信号，不自管评审存储。下方原文（staff-memory 独立目录 + 3 接口）保留作过程参考，**以新文档为准**。

评审经验飞轮：每次评审后用户反馈 → 落盘 R-NNN → 下次评审 recall。kdev-core 自管，物理目录：

```
.kdev/staff-memory/
├── review-memory.md          ← 评审经验（R-NNN）
├── flow-config.yml           ← 用户流程编排（L1）
└── reviews/                  ← 评审记录（按 node + 能力分目录）
    ├── R2-SR需求评审/评分-1.md
    ├── D2-代码评审/评分-1.md
    └── ...
```

`review-memory.md` 借 kdev-memory 范式：markdown 主存 / R-NNN 独立编号 / verbatim 原话不可改写 / subject 路由简化为单级 `review:<能力名>` / 蒸馏切片包 `dataset-review-by-capability/`。

kdev-core 提供 3 个接口：`recall_review_memory(subject_filter, max_items, max_age_days)` / `write_review_feedback(payload)` / `distill_review_memory(by_capability)`。评审能力 prompt 顶部硬规 Read 做 recall（不走 hook，避免污染对话流）。

### 1.10 hook framework

| Hook 名 | 触发 | 干什么 |
|---|---|---|
| `SessionStart` | 会话开始 | 读 flow-state + events.log 恢复；CEO 检查 flow-config.yml（无则引导）|
| `PreCompact` | 上下文压缩前 | 写 checkpoints 快照 |
| `UserPromptSubmit` | 用户发言 | CEO 检查意图（评审反馈 / 命令 / 对话）|
| `Blocked` | 业务能力连续 3 次失败 | 自动派该员工编排应急 |
| `ReviewRetryExhausted` | 评审循环 == 3 次 | 升级 CEO，触发用户拍板 |
| `LowScoreAlert`（可选）| 评分通过但 < 高阈值 | CEO 摘要给用户"要复核吗？" |
| `StageComplete` | 阶段全节点 COMPLETE | CEO 出阶段摘要 |

kdev-core 在 `plugins/kdev-core/hooks.json` 注册；不抢 kdev-memory 已有 hook（Claude Code 原生支持多 hook 同名注册顺序执行）。

## 2. kdev-staff 在 kdev-core 上的实施

### 2.1 agent 注册到 kdev-core

kdev-staff 在 `plugins/kdev-staff/agents/` 提供员工 agent 定义（详见员工能力专项）。plugin.json 声明依赖：

```json
{
  "name": "kdev-staff",
  "depends_on": ["kdev-core@>=0.1.0"],
  "agents_dir": "agents/",
  "standards_dir": "standards/",
  "hooks_extension": "hooks/"
}
```

kdev-core 启动时读依赖 plugin 的 plugin.json，把 agents 注册到 Claude Code subagent、standards 加载为 prompt 资产、hooks_extension 合并到 hook framework。

### 2.2 评审流程编排（flow-config.yml = SOP L1）

CEO 首次对话引导用户生成 `.kdev/flows/<flow>/orchestration.yml`（SOP L1 用户层；本节所称 "flow-config.yml" 即此文件，由 kdev-core 管，不在已废弃的 staff-memory/ 下）：

```yaml
project:
  name: <项目名>
  type: web-public | web-internal-tool | api | mobile | cli

review_flow:
  default_on:      # 13 项默认开
    需求方向评审: enabled
    SR需求评审: enabled
    代码评审: enabled
    # ...
  default_off:     # 5 项默认关
    性能评审: disabled
    合规评审: disabled
    # ...
  disable_reasons: {}        # 关核心评审必填

review_scoring:
  default_threshold: 80
  max_retries: 3
  threshold_overrides:
    代码评审: 85
    安全评审: 85
    架构评审: 85
    上线就绪度评审: 90
    原型评审: 75

cqo_mode: { trigger_a: enabled, trigger_b: enabled, trigger_c: enabled, trigger_d: enabled, trigger_e: enabled }

human_in_loop:
  stage_aggregate_summary: enabled   # 通道 ①
  async_feedback_channel: enabled    # 通道 ②
  cqo_meta_audit: enabled            # 通道 ③
  low_score_alert: disabled          # 通道 ④
```

引导流程：CEO 检查 flow-config.yml 是否存在 → 不存在则引导用户选项目类型 + 开关评审 → 关核心评审给软警告 + 要求填理由 → 二次确认 → 写文件 + events.log 写 `flow_config_updated`。

### 2.3 评分循环 + CEO 升级 + 低分预警

评审专家/编排 prompt 硬规：① recall（接口）→ ② 并行派评审能力 → ③ 汇总 + 双重通过判定（总分 ≥ 阈值 AND 阻断性 = 0）→ ④ 不通过给修订建议 → 员工编排修订 → retry +1 → 回 ① → ⑤ retry == 3 → events.log 写 `review_retry_exhausted`。

`ReviewRetryExhausted` hook 监听该事件 → 自动派 CEO → 用户拍板 (a) 接受记 tech debt / (b) 继续修订 / (c) 推翻重做 → CEO 写 `user_decision` 推进。`LowScoreAlert` hook（可选）：评分 ∈ [阈值, 高阈值) → CEO 摘要给用户复核。

### 2.4 人类介入 4 通道

| 通道 | 实现机制 |
|---|---|
| ① 阶段聚合摘要 | `StageComplete` hook → CEO 出摘要 + 写 hud.md |
| ② 异步反馈 | 用户在 `reviews/<节点>/<能力>-N.md` 加 `## review-feedback` → kdev-core 扫文件变更 → 写 R-NNN |
| ③ CQO 元抽查 | `StageComplete` hook（terminate 前）→ 派 CQO 读 reviews/ + kdev-memory `recall(subject:review:<能力>)` 出审计 |
| ④ 低分预警（可选）| §2.3 `LowScoreAlert` hook |

## 3. kdev-memory 共存关系

### 3.1 物理目录隔离

```
.kdev/
├── memory/                ← kdev-memory plugin 管（工程过程记忆）
│   ├── 执行日志.md / 踩坑日志.md / 决策日志.md / skill-feedback.md / 当前状态.md
├── staff-memory/          ← kdev-core 管（评审素材）
│   ├── review-memory.md / flow-config.yml / reviews/
├── flows/                 ← kdev-core 管（per-flow 状态机）
│   └── <flow>/<slug>/flow-state.json
├── handoffs/              ← kdev-core 管（员工产物总线）
│   └── <员工名>/
├── events.log             ← kdev-core 管（事件流）
├── hud.md                 ← kdev-core 管（HUD 实时）
└── checkpoints/           ← kdev-core 管（PreCompact 快照）
```

### 3.2 范式借用（不是依赖）

| 借用要素 | 用法 |
|---|---|
| markdown 主存 | review-memory.md 用 markdown 而非 JSON |
| 编号体系 | R-NNN（借鉴 F/G/Q-NNN 风格，独立编号系列）|
| verbatim | 用户原话不可改写（核心范式）|
| subject 路由 | 简化为单级 `review:<能力名>` |
| 蒸馏切片包 | `dataset-review-by-capability/` markdown 输出 |

### 3.3 无运行时依赖

| 维度 | kdev-staff 依赖 kdev-core？ | kdev-staff 依赖 kdev-memory？ |
|---|---|---|
| 必需 | ✅ 是（底座）| ❌ 否（独立 plugin）|
| 装机 | 必须装 | 可选 |
| 调用 | Agent / hook / 接口 | 无 |

用户可只装 kdev-core + kdev-staff（不装 kdev-memory）跑起来；或只装 kdev-memory 单用工程记忆。两套互不强制。

### 3.4 跨 store 协作

- **数据面**：kdev-memory 的 `phase_node` 引用底座 node-id → 跨 store 联查（§1.5）。
- **hook 面**：kdev-memory 可选注册自己的 hook 到 kdev-core hook framework 统一管理，也可继续自带 hook 独立工作。
