## 第 0 层 · 一句话：底座是什么

> **编排底座 = 让数字员工"按流程跑起来"的统一引擎。**

光有员工不够。员工会干活，但四个问题悬空：**按什么流程干？哪里把关？卡住了怎么办？断了怎么续？**——这四个问题的答案不该每个员工自己造一套（design-flow / coding-flow 当年就是各造各的，这正是底座要抽的共性），而是一套引擎统一回答。

**一条铁律**：底座**只记账 + 流转，不做判断本身**。评审打分、跑测试、选路线，都由员工（AI）或人来做，把**结果**喂回底座；底座只负责记下来、推进到下一步。所以底座很"薄"（lean）、跟用什么大模型无关。

## 第 1 层 · 三件套：每件是什么、起什么作用

| 件 | 大白话 | 起什么作用 |
|---|---|---|
| **R1 · 状态记录** | 每个功能一份**进度档案**：干到哪一步、各道关过没过 | **断点续跑**——中途断电、被打断，再起来从上次那步接着干 |
| **R2 · 流转引擎** | **"下一步去哪"**的引擎。每条流程自己声明节点表（步骤+顺序），引擎只管推进：把关过了 → 下一步；没过 → 退回重做（有次数上限）| **同一底座跑任意流程**——换业务只换节点表，引擎不动 |
| **R3 · 三类关卡** | 所有把关点归成**三类**（评审 / 岔路决策 / 验收），一套逻辑统一处理 | **把关可配置**——开关哪道关、谁来判，改配置即可 |

### 1.1 状态机到底为了什么（设计的根）

四件事，按"离不开什么数据"拆开，**这个切分直接决定了存储设计**：

| 目的 | 离不开的数据 | 数据形态 |
|---|---|---|
| ① **卡节点**：非法跳转直接拒（AI 想跳过评审？节点表上没这条边，拒）| current_node + 节点表 | **控制态**（一小撮字段）|
| ② **有界回流 + 升级**：同一道关连败 3 次转人工，不死循环 | gate_iters 计数 | **控制态** |
| ③ **断点续跑**：崩了从上次那步接着来 | current_node + status + 计数 | **控制态** |
| ④ **留痕**：谁判的、判了啥、走过哪些路（喂 HUD / CQO / 审计）| 判定与流转的流水 | **事实流水**（无限增长）|

> **控制态小而热**（引擎每步都改），**流水长而冷**（只追加、不回读控制）。两者分开存——控制态进状态文件，流水进事件日志。把流水塞进状态文件（早期实现就是这么干的）是状态文件臃肿的根源。

## 第 2 层 · 怎么运转

### 2.1 账记在哪：一个功能一个目录（feature-first）

> 业务关心的 durable 实体是**功能**（"用户认证做到哪了"），不是某条流程。所以**功能当顶层**，一个功能的全部账收在一处：

```
.kdev/features/<slug>/
├── flow-state.json    ← 唯一状态文件：台账 ＋ 当前棒控制态
├── events.jsonl       ← 事实流水：流转 / 关卡判定 / 派单（append-only，行内带 actor）
└── handoffs/<员工>/   ← 交付物：需求架构师的 SR/原型、开发的 src、测试的用例…
```

| 摊 | 比喻 | 一句话 | 谁消费 |
|---|---|---|---|
| **flow-state.json** | GPS：你在哪、下个路口 | **我在哪一步**（控制态，状态机必需）| 引擎 |
| **events.jsonl** | 行车记录仪：开过哪些路 | **发生了啥**（留痕观测）| HUD / CQO / rollup / 审计 |
| **handoffs/** | 后备箱：运的货 | **做出了啥**（员工交接中转）| 下游员工 |

**flow-state.json schema**（台账 + active 单棒控制态）：

```json
{
  "slug": "user-auth",
  "display_name": "用户认证",
  "origin": "决策日志#Q-012",
  "status": "in_progress",
  "stories": [
    {"id": "US-1", "title": "账号登录", "status": "done"},
    {"id": "US-2", "title": "找回密码", "status": "in_progress"}
  ],
  "runs": [
    {"flow": "design", "run": 1, "status": "completed"}
  ],
  "active": {
    "flow": "design", "run": 2,
    "current_node": "n4-plan-review",
    "status": "in_progress",
    "gate_iters": {"g-plan-review": 1},
    "config": {"review_mode": "ai", "auto_mode": false}
  }
}
```

**运转规则三句话**：干活时引擎只动 `active` 那一小块；一棒跑完，把 active 折叠成 `runs[]` 一行摘要、清空 active；回头补活 = 再开一个 run 填进 active。

- `stories[]` = 用户故事清单（可追加），**HUD「需求完成度」的分母**。
- `runs[]` = 台账：跑完的棒次只留摘要，历史不可变。
- `active` = 当前在跑的这一棒（单数），就这么大——M1 的"完结拒续"不变量由 active 生命周期天然保证。

**并行设计（为什么这么分文件）**：

| 维度 | 设计 | 理由 |
|---|---|---|
| **feature 之间** | 分目录，**免锁并行** | 并发单元 = 文件单元，每个文件天然单写者。不学 OMC 全局单文件——引擎每步是独立 CLI 进程（M1 实测一轮 ~25 个进程），两个功能同时推进必撞写冲突（原子替换只保证不写坏、不保证不丢更新）；且单文件爆炸半径大、无限增长、nested-repo 多机同步天天冲突 |
| **feature 之内** | **单棒约束**：active 是单数，同一功能同时只有一棒在跑 | 接力模型下天然成立；唯一例外是"测试左移"，真到那天再加 per-feature 锁或 active 改数组，不为还没发生的并发提前买单 |

### 2.2 slug 生命周期（⚠️ v0.1，待优化）

**核心原则：聊不立项，做才立项。**

```
①聊/探索 ──────立项线──────► ②立项 ────► ③干活（多棒接力）
  无 slug                     mint slug      一路沿用
  记在记忆层                   建 features/<slug>/
```

| 阶段 | 有没有 slug | 记在哪 |
|---|---|---|
| **聊/探索**：聊了很久还没定做啥、方案对比 | ❌ 没有 | kdev-memory 记忆层（决策日志 / 执行日志）——本职 |
| **立项时刻**：用户说"做"、第一棒 flow 起跑 | ✅ **此刻 mint** | 建 `features/<slug>/flow-state.json`，`origin` 字段回链聊的记录 |
| **干活**：design → coding → test 各棒 | 沿用同一个 | features/<slug>/ 下推进 |

**回头补活怎么算**（按**验收边界**判，需求架构师收口时判、CEO 兜底）：

| 情况 | 归属 | 机制 |
|---|---|---|
| 补的是**同一个对外交付物**的遗漏（用户认证缺了"找回密码"）| **同 slug** | 该棒**开新 run**（增量），用户故事追加进 stories[] |
| 其实是**独立的新交付物** | **新 slug** | mint 新的，`relates_to` 关联 |

> ⚠️ **待优化（用户已标记）**：mint 的具体规约（谁执行 mint / 从什么派生 / 防撞与重命名 / 跨 flow 稳定性保证）尚未定稿；当前真代码里 slug 是调用方直接传入、无 mint 规约。后续专项细化。

### 2.3 R2 流转：advance 三段式

每次流转走三段（纯函数，state in → state out）：

1. **邻接表**：目标节点必须在 `adjacency[current_node]` 里——非法跳转直接拒（卡节点的实现）。
2. **守卫**：可选 `guard(state, to_node) → str|null`，非 null 即拒（拒因可读）。
3. **不可变更新**：换 current_node + 追加一条流转记录 → **写进 events.jsonl**（流水，不留在状态文件里）。

**有界回流 + 幂等豁免**：回流（gate→action 重做）显式标记 `reflow=true` 才计数（retry++）；顺进不计数。回流溢出 → 强制流转到 `terminal_fail` 终止节点，**禁无限回流**。

### 2.4 R3 把关：三类关卡，一套解析

| 类 | 干什么 | 谁来判（默认/可配）| 判不过怎么办 |
|---|---|---|---|
| **review 评审** | 检查产出质量（方案/代码/安全…）| ai（默认）/ both / human | 退回改 → 重审（max 3）；还不过 → **escalate 转人工**，绝不降格放行 |
| **decision 岔路** | 分叉口选路（复杂/简单走法）| ai 默认 / human 可接管 | 选哪条走哪条（无重试）|
| **acceptance 验收** | 跑端到端测试、验真功能 | test 自动跑 / +human 复核 | 回流 fix 节点重跑 |

**GateResult 结构化 JSON**（员工判完喂回底座的统一格式）：

```json
{ "gate": "g-plan-review", "kind": "review", "node": "n4-plan-review",
  "request_id": "uuid-…", "iter": 2, "verdict": "PASS", "by": "ai",
  "issues": [], "revisions": [], "ts": "…" }
```

**record_gate 统一解析**（三类共用）：追加流水 + 计数，然后按 kind 分发——review/acceptance：PASS → on_pass、FAIL 且 iter<max → on_reflow 回流、**iter≥max → status=blocked 升人工**；decision：按 verdict 选 branches[choice]。所有目标节点仍过 R2 邻接表校验。

### 2.5 SOP 可编排：三层覆盖

R2 节点表 + R3 关卡配置 = 一条流程的 SOP，**不写死、可编排**：

| 层 | 谁编排 | 改什么 |
|---|---|---|
| **L0 默认** | flow 作者 | 默认 SOP（节点表 + gates）|
| **L1 用户** | 用户（orchestration.yml）| 开关 gate · 谁来判 ai↔human · 重试数 · 跳过/重排节点 |
| **L2 自主**（未来 L4）| 数字员工自己 | 基于经验提议 SOP 优化（`source: self` 需人确认）|

有效 SOP = merge(L0, L1, [L2])。L2 即"复利飞轮反哺 SOP 本身"（自演进），当前只预留挂钩。

## 第 3 层 · 韧性与边界（工程质感）

| 机制 | 设计 | 为什么值得讲 |
|---|---|---|
| **escalate 不降级** | 同关连败 ≥max → `status=blocked` + blocked_reason，**原地等人**、绝不 force-accept | 质量底线：宁可停下，不带病放行（OMC 反例：超限自动接受）|
| **双计数** | `gate_iters`（本关第几轮）+ `gate_calls`（本流程总把关次数）| 前者是 **escalate 的触发器**（控制逻辑必需）；后者是流程级指标，喂 HUD / 审计（"这条流程一共过了几道关、返工率多少"）|
| **结构化判定** | GateResult 是 JSON，不靠正则扫文本 | 下游可程序化查"这关重试了几轮、谁判的"；resume 不靠猜 |
| **断点续跑** | active 控制态 + completed 拒续不变量 | M1 实测硬证据：13 节点经 ~25 次独立进程流转、全程续接 |
| **decider 旋钮** | 每道关"谁来判"默认 AI，编排可改 human | 人有最终决定权、默认不挡路；低自主多挂 human、高自主全 AI——人机分工是**配置**不是硬编码 |
| **Auto Mode 停靠** | 自主推进只在 `decider:human` 的关停，其余 AI 自己过 | 哪里停由编排决定 |

**两套终止机制的分工**（评审代码时发现的易混点，此处定清）：

- **R3 blocked**（评审超限 → 原地等人）= **常规人工升级路径**，gate 驱动的流程走这条；
- **R2 terminal_fail**（机械回流溢出 → 强制终止）= **底层 backstop**，护非 gate 的纯 action 回流。
- 当前 gate 驱动下 terminal_fail 实际到不了（R3 永远以 reflow=false 调 advance）。**简化选项**（砍 terminal_fail 统一走 blocked）待定，不影响当前行为。

## OMC 借鉴边界（一句话说清来源）

**借的是"工艺"**：JSON 状态文件 + 原子写、active/stale guard、邻接表+守卫分离、有界回流、结构化 verdict（避坑：不正则、不 force-accept）、HUD statusline。
**没借的是"结构"**：OMC 全工作区一个 team-state.json、一条 pipeline；**feature-first 翻转（features/<slug>/ + 台账/控制态/流水切分）是自研**，为多功能并行流水线设计。
