## 0. 一句话

数字员工从「单兵」长成「组织」，本质是从 **单 Agent**（一个 Claude Code 平台、内部 subagent 分工出 6 个员工）走向 **多 Agent**（多个异构大 Agent —— CC + OpenClaw + 国产等 —— 互联成受治理的队伍）。三步走：

1. **第一步** —— 建成**单 Agent** 数字员工团队（**当前聚焦：尚未开发完、未完全验证**）；
2. **第二步** —— 单 Agent **纵深**：把单个员工补成"类真人"（记忆 / 知识 / 工具 / 模型 / 安全）；
3. **第三步** —— **多 Agent 横向**：融入异构大 Agent、成建制经营（选用育留备退 + 价值治理）。

## 1. 现状与起点：第一步 = 单 Agent 数字员工团队（当前聚焦）

我们已经设计并在建一套**数字员工技术底座**，不是 PPT 构想，是已端到端跑通真实任务的系统——但**第一步本身尚未做完、未完全验证**，**当前聚焦就是把它做完、验证透**。

> **形态本质 = 单 Agent**：整套系统跑在**一个 Claude Code 平台**上，6 个角色是它内部的 **subagent** 分工。强内聚，但**单运行时 / 单模型 / 单厂商（Claude）锁定**。详见 §6。

**两个底座**
- **编排底座 `kdev-core`** —— 让员工"按流程跑起来"：状态记录 + 节点流转 + 结构化评审 gate，铁律"只记账 + 流转，不替员工做判断"。
- **记忆底座 `kdev-memory`** —— 让员工"记得住、越用越强"：叙事 / 步骤 / 决策 / 踩坑 / 自评他评，多 scope（shared / staff），Git 托管跨会话续航（**初级记忆**，尚未 OS 化）。

**6 角色团队（= CEO + CQO + 4 业务员工，均为第一步设计）**
- ✅ **4 业务员工已建**（需求架构师 → 开发工程师 → 测试工程师 → 评审专家，覆盖"需求→开发→测试→评审"全链）：合计 **22 项业务/评审能力 + 4 编排 = 26 个 agent**（`kdev-team` v0.8.0）。
- ⏳ **CEO（主控员）** 现由主会话承担、**CQO（监督员 / 元监督）** 尚未建为独立实体 —— **二者是第一步团队的固有角色（监督员属团队内质量/合规岗，不是组织化才出现的能力），只是实现排序靠后。**

**第一步的"成色"（诚实校准）**
- ✅ **会干活** —— 单员工端到端跑通真实改造任务（自主度 L2 协同，M1）。
- ✅ **会协作（subagent 级）** —— 多员工分工 + 跨员工 handoff + 评审 + 发函硬规（M2/M3）。⚠️ 注意：这是**单 Agent 内部的 subagent 协作**，**不是多 Agent 组织**（多 Agent 要到第三步）。
- ⚠️ **记忆初级** —— md / wiki 蒸馏 + scope + 召回，尚未 OS 化。
- ❌ **第一步尾巴未完** —— CEO/CQO 角色未建、自演进 / 跨 IDE 未做、全量验证（Pass2 / 完整 dogfood）未跑。

> 🔴 **当前聚焦 = 收尾并验证第一步**：补齐 CEO/CQO、跑通自演进 / 跨 IDE、完成全量验证，把"单 Agent 数字员工团队"真正落稳。memOS / RAG（第二步）是其后的事。

## 2. 目标愿景：一个完整的数字员工 + 一支可经营的队伍

终点对齐数字员工目标蓝图——单体**类真人**，且能被组织**成建制经营**。

**"我是谁 - 从哪来 - 到哪去" = 单体生命周期三视角**
- **我是谁（角色管理）**：经营力 / 业务力 / 专业力 + 9 维度（价值贡献度 / 场景覆盖度 / 流程渗透度 / 能力成熟度 / 系统体验度 / 数据完备度 / 模型适配度 / 智算应用度 / 安全可信度）。
- **从哪来（周期管理）**：HR + IR 双线，**选 - 用 - 育 - 留 - 备 - 退** 生命周期，"百人百智"规模化复制。
- **到哪去（运用实例）**：单个数字员工（如"软件代码助理小智"）在真实业务里持续产出、量化价值。

**"新环境 5 保障"** = AI 架构 / 流程 / 能力 / 组织 / 投入保障。

> 终点定位：**从"一个 CC 平台里的单 Agent 团队"，到"多个异构大 Agent 组成、可被组织选育留并算清价值贡献的数字员工队伍"。**

## 3. 路线为什么是这条：两轴模型 + 单/多 Agent 主轴

落地路线沿两根轴往右上爬；其中**横轴（组织化）的本质，是从「单 Agent」走向「多 Agent」**：

> 关键升级：横轴的"组织化"不是"员工（subagent）变多"——那第一步早就多了——而是**底座从单一 CC 平台，变成多个异构大 Agent 互联**。这比"多 subagent"硬核得多，也正是"组织"区别于"单兵"的本质。

## 4. 落地路线：3 步（单 Agent → 多 Agent）

```
第一步｜单 Agent 形态（当前聚焦·进行中）          第三步｜多 Agent 形态（终态）
┌───────────────────────────┐                  ┌──────────┐ ┌──────────┐ ┌──────────┐
│   Claude Code（1 个大 Agent）  │                  │ CC        │ │ OpenClaw │ │ 国产Agent │
│  ┌────┬────┬────┬────┐      │   ── 第二步 ──►    │ (需求/开发)│ │ (测试)   │ │ (评审)   │
│  │需求│开发│测试│评审│      │   单 Agent 纵深     └────┬─────┘ └────┬─────┘ └────┬─────┘
│  └────┴────┴────┴────┘      │   把个体补成类真人    └──── A2A / MCP / 消息总线 互联 ────┘
│  6 员工＝subagent，单运行时    │   ── 第三步 ──►    异构平台、可异构模型，受统一治理
│  （CEO/CQO 角色待补）          │   融入多 Agent       ＝ 真正的"数字员工组织"
└───────────────────────────┘
```

### 第一步 · 单 Agent 数字员工团队（🔴 当前聚焦，进行中）
一个 CC 平台 + 内部 subagent 员工。**已建**：kdev-core 编排 + kdev-memory 初级记忆 + 4 业务员工（22 能力 + 4 编排）+ kdev-hud 观测 + L2→L3 subagent 协作（M0–M3）。**待补（第一步尾巴）**：CEO/CQO 角色实体、自演进、跨 IDE、全量验证。**本质**：强内聚，但锁定 CC（单运行时 / 单模型 / 单厂商）。

### 第二步 · 单 Agent 纵深：把个体补成"类真人"
**还在 CC 一个平台里**，补单个员工的能力厚度：memOS / 知识库 RAG / 工具系统集成 / 模型适配 / 多模态 / 安全可信（详见 §7.2）。

### 第三步 · 多 Agent 横向：异构组织化 + 经营治理
**引入异构大 Agent**（CC + OpenClaw + Hermes + 国产）互联成队伍：跨 Agent 互联协议（A2A / MCP）+ 异构编排 + 选用育留备退 + 价值经营度量 + 治理（**落地方案三层见 §7.3**）。

### 第二步 vs 第三步（并排看）

| 维度 | **第二步：单 Agent 纵深** | **第三步：多 Agent 横向** |
|---|---|---|
| 一句话 | 还在 CC 一个平台里，把单个员工补强 | 多个异构大 Agent 互联成队伍 |
| Agent 形态 | **单 Agent**（CC + subagent 分工）| **多 Agent**（big-A 异构，员工可跑不同平台/模型）|
| 补什么 | memOS / RAG / 工具系统集成 / 模型适配 / 多模态 / 安全可信 | 跨 Agent 互联（A2A / MCP / 消息总线）+ 选用育留备退 + 价值度量 + 治理 |
| 解决什么痛 | 员工"不够像真人、不够胜任"（记不住 / 查不到 / 接不进系统 / 模型不对路）| **单平台锁定** + 无法成建制经营、无法规模复制（百人百智）|
| 与底座锁定的关系 | 仍受 CC 锁定（仅"模型适配层"开始松绑）| **正是打破 CC 锁定的路径**（异构 Agent 不绑单底座，呼应 §6 两手准备）|
| 对应 9 维度 | 数据完备度 / 模型适配度 / 系统体验度 / 安全可信度 | 价值贡献度 / 场景覆盖度 / 流程渗透度 + 组织治理 |
| 轴向 | 纵轴（个体成熟度）| 横轴（组织化 = 单→多 Agent）|
| 状态 | 🔨 第一步收尾后聚焦 | ⏳ 落地方案三层已给（§7.3），待试点 |

## 6. 底座选型说明（我们站在哪、与开源自建路线的关系）

数字员工要选一个 **agent 运行底座**。市面两条路线：

- **托管路线（我们第一步走的）**：建在 **Claude Code / Claude Agent SDK** 之上——agentic loop、工具、子智能体、技能、钩子、上下文压缩、权限**全开箱**。
- **自建路线**：用开源/自研 runtime（**OpenClaw**、**Hermes Agent**、LangGraph 等）**自己造** agent 引擎，模型后端、记忆、私有化全自控。

**分层看清楚——我们和自建路线不在同一层：**

```
┌─────────────────────────────────────────────────────────────┐
│  数字员工经营层（我们自建·价值所在）                              │
│  kdev-core 编排 · kdev-memory 记忆 · 知识库 · kdev-hud 观测 · 治理 │
├─────────────────────────────────────────────────────────────┤
│  agent 运行底座（第一步＝Claude Code 单 Agent；第三步＝多异构 Agent）│
│  agentic loop · 工具 · 子智能体 · 技能 · 钩子 · 上下文压缩 · 权限    │
└─────────────────────────────────────────────────────────────┘
```

- **为什么第一步这么选**：底座白捡 agentic 能力，是 4 业务员工能在数月内跑通 M1–M3 的红利来源；自建 runtime 要先花数月造引擎才能开始干活。**"开发了很多却与 OpenClaw/Hermes 不是一回事"——因为它们造的是下层引擎，我们造的是上层经营层。**
- **取舍与风险（诚实标注）**：模型锁 Claude（不能直接换国产/本地）· 不支持纯私有化离线（须连 Anthropic）· 编排循环受 Claude 主导 · 记忆 / RAG 无原生须自建。
- **🔴 应对原则 —— 两手准备**：第一步建在 Claude Code 上**快跑**；同时把**关键层（记忆 / 知识 / 编排）做轻耦合、可替换抽象**。**第三步的"多 Agent 异构"正是这条原则的归宿**——把 OpenClaw / Hermes / 国产 Agent 作为异构员工底座融入，CC 不再是唯一底座，集团若要私有化 / 国产模型即可承接。

## 7. 落地方案

> 每条按"**基于什么现状 + 怎么做 + 业界对标 + 落到哪个插件 + 可替换点**"展开，确保有支撑、可执行、符合 §6 两手准备原则。

### 7.1 第一步收尾（🔴 当前聚焦）

- **基于现状**：4 业务员工已建（M0–M3 已验证），roadmap 阶段4（CEO/CQO/自演进/跨IDE）尚未落地，全量验证未跑。
- **怎么做**：① 把 **CEO（主控员）** 沉淀为独立的人机接口/续航/派单实体（现由主会话兼任）；② 建 **CQO（监督员）** 元监督/合规审计角色；③ 自演进、跨 IDE 起步；④ 跑**全量验证**（Pass2 / 完整 dogfood），把"单 Agent 团队"证稳。
- **落到插件**：`kdev-team`（CEO/CQO 角色）+ `kdev-core`/`kdev-hud`（自演进/观测）。

### 7.2 第二步 · 单 Agent 纵深（把个体补成"类真人"）

| # | 方面 | 怎么做 | 业界对标 | 落到 / 可替换点 |
|---|---|---|---|---|
| 1 | **记忆系统 memOS** | 初级记忆 → 结构化 + 向量检索 + 跨员工共享记忆体 | mem0 / Zep / LangMem | `kdev-memory` 演进；存储后端抽象可换 |
| 2 | **知识库 RAG/wiki** | 领域文档沉淀 + chunk/embedding + 检索增强 | LlamaIndex / RAGFlow / Dify | 新增 `kdev-knowledge` + MCP 检索；向量库/embedding 可换 |
| 3 | **工具与业务系统集成** | 从"代码/文档里干活"→"业务系统里干活"（OA/CRM/DB/内部 API、UI 原生交互）| MCP 生态 | 深化 MCP 接业务系统；走标准接口不绑单一供应商 |
| 4 | **模型适配 / 多模型** | 岗位最佳模型选型/路由/微调/评测 | —— | **模型抽象层**（同时是两手准备：未来换国产/私有）|
| 5 | **多模态能力** | 处理图像/音频/文档（简历 PDF、面试录音、设计图）| Claude 多模态可借 | 岗位级多模态流水线；按岗位需求后置 |
| 6 | **安全可信运行时防护** | 输入输出合规、权限最小化、全链路审计、数据安全 | OWASP / 合规规范 | 现有"安全评审能力"扩到运行时防护 |

> 当前聚焦在第一步，第二步以 1、2（记忆/知识）+ 3、4、6（系统集成/模型/安全）为通用核心，5（多模态）按岗位后置。

### 7.3 第三步 · 多 Agent 横向：异构组织化 + 经营治理（落地方案）

> 落地分**三层**：① **异构互联层**（用什么协议把不同平台的大 Agent 接起来）→ ② **异构编排层**（谁主控、异构员工怎么挂）→ ③ **治理层**（选用育留备退 + 价值度量）。**基于现状**：第一步已有统一账本（`events.jsonl` / handoff）+ 跨员工发函硬规 + kdev-hud 观测，是互联与治理的雏形地基。

**① 异构互联层 —— A2A + MCP 两个标准，把异构大 Agent 接起来**

| 协议 | 管什么 | 怎么用 | 成熟度 |
|---|---|---|---|
| **A2A**（Agent-to-Agent）| 员工 ↔ 员工（跨平台）| Google 2025 发布、已捐 Linux Foundation；每个员工发 **AgentCard**（`/.well-known/agent-card.json`）声明能力 + 认证（OAuth2/mTLS/OIDC）+ 任务生命周期状态机；让 CC / OpenClaw / Hermes / 国产 Agent 互发任务、互相发现能力 | 🟢 早期生产（[150+ 机构](https://www.prnewswire.com/news-releases/a2a-protocol-surpasses-150-organizations-lands-in-major-cloud-platforms-and-sees-enterprise-production-use-in-first-year-302737641.html)，含 Salesforce/SAP/ServiceNow）|
| **MCP**（Model Context Protocol）| 员工 ↔ 工具/资源 | 接业务系统工具；多员工挂同一 MCP server ＝ **共享黑板**（共享记忆/知识）| 🟢 生产成熟（万级 server）|

- **分工**：A2A 管员工之间，MCP 管员工接工具/共享记忆知识，两者均在 Linux Foundation、互补。
- **异构身份**：每个员工要有可验证身份与权限——可对标 **Microsoft Entra Agent ID**（2025-11 GA，审计日志带 `agentType` 区分 AI/人、blueprint 一键停用）或 **SPIFFE/SPIRE**（短时 SVID + 工作负载认证，适合私有化）。
- **接到我们现状**：第一步的 `events.jsonl` + handoff + 发函硬规 ＝ 任务契约 + 账本雏形，正好对接 A2A 的 task/artifact 语义（结构化结果、可验证证据，非自然语言）。

**② 异构编排层 —— CC 作主控，把异构员工包成 worker 挂进来**

- **主控（coordinator）**：用 **Claude Agent SDK Managed Agents**（coordinator/worker，roster ≤ 20、独立 session/上下文）或 LangGraph supervisor——与 kdev-core 的"主控编排 + 派单"同构。
- **异构员工作 worker**：CC 既可作 orchestrator 也可作被调 worker（CLI / REST `/v1/agents` / MCP）；OpenClaw、Hermes 均以 **MCP 为集成核心**（OpenClaw 自称 "USB-C for AI"、Hermes 用 MCP bridge），可包成 MCP server / A2A endpoint 纳入 roster。
- **统一网关（Agent Gateway）**：统一认证 / 路由 / 可观测的网关层（业界已有 Istio `agentgateway` 实验）——对应 kdev-hud 观测 + kdev-core 派单的升级方向。
- **角色 ↔ 平台映射示例**：编码岗跑 CC / Codex；测试岗可换 OpenHands / OpenClaw；评审岗可换国产 Agent（私有化）；统一一个 coordinator 持 roster + 统一账本交接观测。

**③ 治理层 —— 选用育留备退 + 价值度量（对标 [Workday Agent System of Record](https://newsroom.workday.com/2025-02-11-The-Next-Generation-of-Workforce-Management-is-Here-Workday-Unveils-New-Agent-System-of-Record)）**

Workday ASOR 把 agent 当员工管（register→configure→activate→monitor→deactivate，三组件 Agent Registry / Agent Gateway / Agent Analytics Hub，权限 mirror 人类）。映射"选用育留备退"：

| 阶段 | 落地动作 | 对标 |
|---|---|---|
| **选** | Agent Registry 注册 + AgentCard 能力声明 + 岗位选型（哪个平台/模型适配）| Registry + Skills 定义 |
| **用** | activate + 授权（权限 mirror 人类 / tool allowlist）+ 派任务 | Agent Gateway 身份/编排 |
| **育** | 反馈 / 评测闭环持续改进（kdev-memory 蒸馏 + 模型他评是雏形）| Feedback loop |
| **留** | 监控（准确率 / 一致性 / 成本）+ 绩效 + 价值度量 | Analytics Hub + AgentOps 可观测 |
| **备** | 后备 / 快速复制（roster 冗余、异构备份、blueprint 重建）| Entra blueprint 重建 |
| **退** | deactivate / 归档（"registration to retirement" / 软删除级联）| Deactivate + 一键停用 |

- **价值度量**：仿 Agent Analytics Hub，在 kdev-hud 上叠 **采纳率 / 自动化率 / 单位成本 / 工时节省 / ROI**，对应经营力·业务力·专业力 + 价值贡献度。可观测可借 LangSmith / Arize Phoenix / AgentOps（OpenTelemetry 链路 + LLM-as-judge 评测 + 成本熔断）。
- **为什么这层是生死线**：[Gartner 预测 40%+ agentic 项目 2027 前因 ROI 不清/治理缺失被砍](https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027)——治理层不是锦上添花，是项目存活的前提。

**⚠️ 关键缺口与试点建议（诚实）**
- **最大缺口**：异构 Agent 的**统一身份与权限治理**（每平台独立 tool allowlist）+ 跨平台 **task state 单一真相**——这两件不解决，"数字员工组织"停在 demo 层；我们的 `events.jsonl` 单一账本是好起点。
- **互通成熟度**：A2A 跨平台虽早期生产可用，但 **CC ↔ OpenClaw 原生对接仍弱**（靠 MCP bridge）；OpenClaw 有安全债（2026-01 审计 512 漏洞/8 critical），融入前须加固。
- **建议路径**：**先在受控内网，用 MCP bridge 把 1 个异构员工试点挂进来**（如测试岗用 OpenHands），跑通"互联 + 治理 + 观测"最小闭环，再按岗位逐步扩，避免一上来全异构。


> 节奏原则（承接技术 roadmap 的 walking-skeleton 思路）：**按真实需求逐级长**——先把第一步做完验证透，再纵深补个体（第二步），最后横向组多 Agent（第三步）。

## 附录 B：对比分析（数字员工概念市场 + 底座路线对比）

> 此附录把行业调研沉淀于此，作为正文判断的佐证。结论：**"单 Agent → 多 Agent 组织"叙事与行业主线同向，不是空想；落地路径合理，唯一战略变量是底座锁定（见 §6），第三步多 Agent 异构正是其解。**

### B.1 数字员工概念与市场速览（2025-2026）

- **演进**：RPA（界面脚本回放）→ AI+RPA（半结构化）→ **AI Agent 型数字员工**（感知-推理-规划-执行闭环，目标导向长程自主）。判定边界：**能否代人执行跨系统流程并达成业务目标**。
- **单 agent → 多 agent 数字劳动力**：2025-2026 主线。[Salesforce Agentforce](https://markets.financialcontent.com/stocks/article/tokenring-2026-1-15-the-copilot-era-is-dead-how-salesforce-agentforce-sparked-the-autonomous-business-revolution) 自称"数字劳动力平台"；多角色 agent"军团"成主流范式。
- **像管真人一样选用育留**：[Workday Agent System of Record](https://newsroom.workday.com/2025-02-11-The-Next-Generation-of-Workforce-Management-is-Here-Workday-Unveils-New-Agent-System-of-Record)（2025-02，agent 入职/授权/监控/下线/ROI），IBM/Oracle/ServiceNow 跟进——**印证第三步方向**。
- **异构 Agent 互联是 2026 趋势**：Coze 3.0 已可作 meta-orchestrator 调用 CC + OpenClaw 等，混合/异构 Agent 架构成主流——**印证第三步"多 Agent"可行**。
- **自主度分级**：[ASDLC L1-L5](https://asdlc.io/concepts/levels-of-autonomy/)，**L3 = 生产可持续上限**；[Anthropic 实测](https://www.anthropic.com/research/measuring-agent-autonomy) Claude Code 单任务自主时长半年翻倍——**印证 L2 起步、L4 defer 稳妥**。
- **⚠️ ROI 警告**：[Gartner 预测 40%+ agentic AI 项目将于 2027 前因 ROI 不清/治理缺失被砍](https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027)——**印证"价值度量+治理"是生死线**。

### B.2 底座路线六维对比：Claude Code（第一步）vs 自建 runtime（OpenClaw / Hermes，第三步可融入）

| 维度 | 建在 Claude Code 上（第一步单 Agent）| 自建 runtime（OpenClaw / Hermes / LangGraph 等）|
|---|---|---|
| **工程成本** | 低：agentic loop / 工具 / 子智能体 / 上下文 / 权限全开箱 | 高：自造引擎与编排，数月工程 |
| **上线速度** | 快：核心数小时可 demo（我们数月跑通 M1–M3）| 慢：框架本身即多月工程 |
| **模型选择** | 锁 Claude（可经 Bedrock/Vertex 间接）| 自由：本地 / 私有 / 国产 / 混搭（Hermes 接 300+ 模型）|
| **记忆 / RAG** | 无原生，须自建（kdev-memory 即此）| 须自建，但完全自控后端 |
| **多智能体编排** | 子智能体开箱，但限**单 Agent 内** | 可跨平台异构编排（多 Agent）|
| **私有化 / 数据不出域** | 不支持纯离线，须连 Anthropic | 可完全私有化离线 |
| **可控性** | 中：行为透明，loop 内核不可换 | 高：逻辑透明、可 replay / 审计 |

- **OpenClaw**：Peter Steinberger 发起，2026 年初 GitHub 史上最多 star 的开源 agent；本地运行 + 100+ 技能 + 持久记忆。[来源](https://en.wikipedia.org/wiki/OpenClaw)
- **Hermes Agent**：Nous Research 开源 agent 框架；118 技能 + 跨会话持久记忆 + 接 300+ 模型。[来源](https://hermes-agent.nousresearch.com/)
- **Claude Code / Agent SDK**：[Agent SDK overview](https://code.claude.com/docs/en/agent-sdk/overview)。

### B.3 业界 Agent 治理平台对照（第三步治理层旁证）

主流企业平台都在建"把 agent 当员工管"的生命周期治理层，印证第三步方向；其中 IBM watsonx 原生支持 A2A，坐实 §7.3 互联层选型。

| 平台 | 定位 | 生命周期治理 | 互联 / A2A | 价值度量 |
|---|---|---|---|---|
| **Workday** Agent System of Record | "数字员工的 HR 平台" | register→configure→activate→monitor→deactivate；Agent Registry / Gateway / Analytics Hub；权限 mirror 人类 | — | Analytics Hub：ROI / 成本 / 影响 |
| **ServiceNow** AI Agent Orchestration | AI Control Tower 跨云治理 | 正式 **6 阶段**（New→Assess→Build→Deploy→Operate→**Retire**）+ 运行时 **kill switch** + role masking + 跨 30+ 平台自动发现 | 经 AI Control Tower 发现外部 agent | Value Dashboard：ROI / 生产力 / 成本规避 |
| **IBM** watsonx Orchestrate | 开发者中心（ADK / CLI，OpenTelemetry 原生）| Agent Control Plane（2026 私有预览，统一身份 + 策略）；`remove` 命令退役 | **原生 A2A（`external_chat/A2A/0.3.0`）+ AgentCard 自动发现** | 评测框架 + 运行时 token / 延迟 / 成本 |
| **Salesforce** Agentforce | 低代码 + Data Cloud | Activate/Deactivate + BotVersion 版本；双执行身份（用户继承 / Agent User 服务账号）；Einstein Trust Layer | 无跨平台发现 | Agentforce Analytics + Audit Dashboard |

> 启示：① **选用育留备退**（尤其"退 / kill switch"）是企业级刚需；② **A2A 正成为跨平台互联事实标准**（IBM 原生支持）；③ **价值度量看板**（ROI / 成本 / 采纳）是每家标配——三点正对应第三步治理层的互联 / 生命周期 / 价值度量。来源：ServiceNow AI Control Tower 文档 · IBM watsonx Orchestrate ADK 文档 · Salesforce Agentforce 文档 · [Workday ASOR](https://newsroom.workday.com/2025-02-11-The-Next-Generation-of-Workforce-Management-is-Here-Workday-Unveils-New-Agent-System-of-Record)。
