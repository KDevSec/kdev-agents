## 0. 一句话

数字员工要"像真人一样胜任并被经营"，得先有**会干活的手**、**会配合的同事**、**记得住的脑**、**查得到的知识**，最后被组织像管真人一样**选育留、算价值**——这就是从**一个能干活的数字员工**，长成**一支可被经营的数字员工组织**的落地路径：**5 步、归为 3 大阶段**。前 2 步已真实跑通，当前聚焦在第 3-4 步（记忆系统 + 知识库），终点是第 5 步的组织化治理（成熟终态）。

## 1. 现状与起点（我们已建成什么）

我们已经设计并建成一套**数字员工技术底座**，不是 PPT 构想，是已端到端跑通真实任务的系统。

**两个底座**
- **编排底座 `kdev-core`** —— 让员工"按流程跑起来"：状态记录 + 节点流转 + 结构化评审 gate，铁律是"只记账 + 流转，不替员工做判断"。
- **记忆底座 `kdev-memory`** —— 让员工"记得住、越用越强"：叙事 / 步骤 / 决策 / 踩坑 / 自评他评，多 scope（shared 共享 / staff 员工私有），Git 托管跨会话续航。

> 这两个底座本身建在 **Claude Code** 这个成熟 agent 运行底座之上（agentic loop / 工具调用 / 子智能体 / 技能机制开箱即用）——**我们不重造 agent 引擎，只在其上建"数字员工经营层"**。底座选型的取舍与风险见 §6。

**已建成的角色（6 人愿景 = CEO + CQO + 4 业务员工）**
- ✅ **4 业务员工已建**（需求架构师 → 开发工程师 → 测试工程师 → 评审专家，覆盖"需求→开发→测试→评审"全链）：合计 **22 项业务/评审能力 + 4 编排 = 26 个 agent**（`kdev-team` v0.8.0）。
- ⏳ **CEO（主控员）**现由主会话承担、**CQO（监督员/元监督）**尚未建为独立实体 —— 这正是走向终态要补的角色缺口（见第 5 步）。

**起点的"成色"（诚实校准）**
- **会干活** ✅ —— 单员工端到端跑通真实改造任务（自主度 L2 协同）。
- **会协作** ✅ —— 多员工分工 + 跨员工交接（handoff）+ 评审专家 + 跨员工发函硬规，已是一个会配合、能互评的小班组。
- **记忆初级** ⚠️ —— 目前靠 md / wiki 形式的蒸馏经验 + scope 分离 + 智能召回，**尚未 OS 化**（结构化检索、组织级共享记忆体仍是升级方向）。
- **无知识库** ❌ —— 还没有领域知识库（RAG / wiki 检索增强）。
- **无组织治理** ❌ —— 还没有"选用育留备退"生命周期管理与价值度量。

> 一句话起点定位：**会干活、会协作的初级数字员工团队，脑子（记忆）还初级、肚子里（知识）还空、上面（组织治理）还没建。**

## 2. 目标愿景：一个完整的数字员工长什么样

终点对齐数字员工目标蓝图——一个**类真人的数字员工**，且能被组织**成建制地经营**。

**"我是谁 - 从哪来 - 到哪去" = 单体生命周期三视角**
- **我是谁（角色管理）**：用经营力 / 业务力 / 专业力 + 9 维度（价值贡献度 / 场景覆盖度 / 流程渗透度 / 能力成熟度 / 系统体验度 / 数据完备度 / 模型适配度 / 智算应用度 / 安全可信度）刻画一个数字员工"成熟到什么程度"。
- **从哪来（周期管理）**：用 HR + IR 双线把数字员工纳入**选 - 用 - 育 - 留 - 备 - 退**生命周期，"百人百智"规模化复制。
- **到哪去（运用实例）**：单个数字员工（如"软件代码助理小智"）在真实业务里持续产出、量化价值。

**"新环境 5 保障"** = AI 架构 / 流程 / 能力 / 组织 / 投入保障，对应安全可信度等底线。

> 终点定位：**从"一个小智"到"一支可被组织选育留、能算清价值贡献的数字员工队伍"。**

## 3. 路线为什么是这条：两轴模型

把"单人 → 组织"硬掰成一条线会拧巴——因为我们**起点就已经是多员工团队**。真正干净的拆法是**两根轴**，落地路线沿对角线往右上爬：

<tr><td rowspan="3"><b>起点（现状）</b></td><td>会干活 ✅</td><td>多员工分工 ✅</td></tr>

## 6. 底座选型说明（我们站在哪、与开源自建路线的差异）

数字员工不是凭空造的，要选一个 **agent 运行底座**。市面两条路线：

- **托管路线（我们走的）**：建在 **Claude Code / Claude Agent SDK** 之上——agentic loop、工具调用、子智能体、技能、钩子、上下文压缩、权限模型**全部开箱**。
- **自建路线**：用开源/自研 runtime（如 **OpenClaw**、**Hermes Agent**、LangGraph、AutoGen 等）**自己造** agent 引擎与编排循环，模型后端、记忆存储、私有化部署全自控。

**分层看清楚——我们和自建路线不在同一层：**

```
┌─────────────────────────────────────────────────────────────┐
│  数字员工经营层（我们自建·价值所在）                              │
│  kdev-core 编排 · kdev-memory 记忆 · 知识库 · kdev-hud 观测 · 治理 │
├─────────────────────────────────────────────────────────────┤
│  agent 运行底座（Claude Code，开箱白捡）                          │
│  agentic loop · 工具 · 子智能体 · 技能 · 钩子 · 上下文压缩 · 权限    │
└─────────────────────────────────────────────────────────────┘
   ⇅ OpenClaw / Hermes 等自建路线 = 自己造"下面这层"，再在上面叠经营层
```

- **为什么这么选**：底座白捡 agentic 能力，是我们 4 业务员工能在数月内跑通 M1–M3 的红利来源；自建 runtime 要先花数月造引擎才能开始干活。**"开发了很多却与 OpenClaw/Hermes 不是一回事"——因为它们造的是下层引擎，我们造的是上层经营层。**
- **取舍与风险（诚实标注）**：模型锁定 Claude（不能直接换国产/本地模型）· 不支持纯私有化离线部署（须连 Anthropic）· 编排循环受 Claude 主导 · 记忆 / RAG 无原生、须自建（kdev-memory 即此）。
- **🔴 应对原则 —— 两手准备**：现在建在 Claude Code 上**快跑**；同时把**关键层（记忆 / 知识 / 编排）做轻耦合、可替换抽象**（存储后端、检索接口、模型调用走标准化接缝），一旦集团要求私有化 / 数据不出域 / 国产模型，经营层能整体迁到自建 runtime，不推倒重来。详见 §7 各步"可替换点"。

## 7. 落地方案（当前聚焦第 3-4 步，第 5 步预研）

> 每条按"**基于什么现状 + 怎么做 + 业界对标 + 落到哪个插件 + 可替换点**"展开，确保有支撑、可执行、且符合 §6 两手准备原则。

### 7.1 第 3 步 · 记忆系统 memOS（🔴 当前聚焦）

- **基于现状**：`kdev-memory` 已有 markdown 主存 + scope（shared/staff）+ 智能召回 + Git 托管续航 + 时间戳记录 ID + 蒸馏管道（`/kdev-memory-distill`），并已规划 JSONL 操作层（roadmap P-C2，Phase A 已落地）。
- **怎么做**：① **结构化索引**——记录从 md 全文走向字段化/可查询（承接 P-C2 JSONL 操作层）；② **向量检索**——加 embedding + 语义召回，补足当前关键词召回的语义盲区；③ **组织级共享记忆层**——把 shared scope 强化为"跨员工共享的组织记忆"，让经验在员工间复用。
- **业界对标**：mem0 / Zep / LangMem —— 跨会话长期记忆 2026 年已进入生产可用与横评阶段。
- **落到插件**：`kdev-memory` 演进（承接 P-C2）。
- **可替换点**：记忆存储后端抽象（markdown / JSONL / 向量库可换）、召回走统一检索接口，**不绑** Claude Code 的 beta memory tool。

### 7.2 第 4 步 · 知识库 RAG（🔴 当前聚焦）

- **基于现状**：当前**无**专门知识库；起点是已有的 wiki（md 蒸馏经验）这点领域沉淀。
- **怎么做**：① **领域文档沉淀**——规范 / SOP / 历史方案 / 踩坑库结构化入库；② **chunk + embedding 索引**；③ **检索增强**——按问题检索 top-k 注入上下文，让员工"查得到、答得准"。Claude Code 无原生 RAG → 经 **MCP 外挂检索工具** 或自建 RAG pipeline 接入。
- **业界对标**：LlamaIndex / RAGFlow / Dify（内置完整 RAG pipeline，含 chunk / rerank）。
- **落到插件**：新增 `kdev-knowledge`（或并入 `kdev-memory` 的知识子域）+ MCP 检索 server。
- **可替换点**：向量库（Chroma / Weaviate / 国产）+ embedding 模型可换；检索经 MCP 标准接口暴露，不绑单一供应商。

### 7.3 第 5 步 · 组织治理 + 价值度量（🟡 终态，预研先行）

- **基于现状**：`kdev-hud` 观测层 v0.1.0（只读消费账本）+ 评审 gate + 跨员工发函硬规 + `events.jsonl` 流水已就绪。
- **怎么做**：① **选用育留备退** = agent 生命周期登记（建立 / 上线 / 授权 / 监控 / 下线），对标 Workday Agent System of Record；② **价值度量看板** = 在 `kdev-hud` 上叠经营力 / 业务力 / 专业力 + 价值贡献度 / ROI 指标；③ **CQO 元监督 + 自演进**（终态角色）。
- **业界对标**：Workday Agent System of Record / IBM 数字劳动力生命周期管理。
- **落到插件**：`kdev-hud` 扩展 + 新建治理 / CQO 层。
- **预研建议**：CQO / 自演进当前 defer；但**价值度量看板可早做**（数据源 `events.jsonl` 已就绪），用于应对"ROI 是否清晰"的拷问（见 §附录 B 的 Gartner 警告）。

## 8. 里程碑与节奏

**已达成（执行力背书）**

| 里程碑 | 内容 | 状态 |
|---|---|---|
| **M0** | 编排底座核心三件（状态 / 节点机 / 三类 gate）+ Git 托管激活 | ✅ |

**后续排序（建议）**

```
当前 ──► 第3步 记忆 memOS 化 ──► 第4步 知识库 RAG/wiki ──► 第5步 组织治理 + 价值度量（终态 L4）
         （已有基础，OS 升级）    （从零建，本期重点）       （CQO/自演进/跨IDE，愿景终态）
```

> 节奏原则（承接技术 roadmap 的 walking-skeleton 思路）：**按真实需求逐级长，不一次性把全部基建压在前面**——先把当前价值密度最高、缺口最痛的记忆 OS + 知识库补齐，再爬向组织治理终态。

## 附录 B：对比分析（数字员工概念市场 + 底座路线对比）

> 此附录把行业调研沉淀于此，作为正文判断的佐证。结论：**我们的"单兵→组织"叙事与行业主线同向，不是空想；落地路径合理，唯一战略变量是底座锁定（见 §6）。**

### B.1 数字员工概念与市场速览（2025-2026）

- **演进**：RPA（界面脚本回放）→ AI+RPA（半结构化）→ **AI Agent 型数字员工**（感知-推理-规划-执行闭环，目标导向长程自主）。判定边界：**能否代人执行跨系统流程并达成业务目标**，仅"答问/建议"不算。
- **单 agent → 多 agent 数字劳动力**：2025-2026 主线。[Salesforce Agentforce](https://markets.financialcontent.com/stocks/article/tokenring-2026-1-15-the-copilot-era-is-dead-how-salesforce-agentforce-sparked-the-autonomous-business-revolution) 自称"数字劳动力平台"；多角色 agent"军团"成主流范式。
- **像管真人一样选用育留**：[Workday Agent System of Record](https://newsroom.workday.com/2025-02-11-The-Next-Generation-of-Workforce-Management-is-Here-Workday-Unveils-New-Agent-System-of-Record)（2025-02，agent 入职/授权/监控/下线/ROI），IBM/Oracle/ServiceNow 跟进——**印证本规划第 5 步方向**。
- **自主度分级**：[ASDLC L1-L5](https://asdlc.io/concepts/levels-of-autonomy/)，**L3 = 生产可持续上限**，L4+ 有认知债务；[Anthropic 实测](https://www.anthropic.com/research/measuring-agent-autonomy) Claude Code 单任务自主时长半年翻倍——**印证本规划 L2 起步、L4 defer 的稳妥**。
- **⚠️ ROI 警告**：[Gartner 预测 40%+ agentic AI 项目将于 2027 前因 ROI 不清/治理缺失被砍](https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027)——**印证"价值度量+治理"必须做、且是生死线**。
- **厂商定位**：国际 Salesforce/Workday = 数字劳动力叙事；中国实在智能最接近"全职能数字员工"，弘玑/来也在 RPA 转 agent 过渡带，字节扣子(Coze)/阿里/腾讯偏平台工具层。

### B.2 底座路线六维对比：Claude Code（我们）vs 自建 runtime（OpenClaw / Hermes 等）

| 维度 | 建在 Claude Code 上（我们的路线）| 自建 runtime（OpenClaw / Hermes / LangGraph 等）|
|---|---|---|
| **工程成本** | 低：agentic loop / 工具 / 子智能体 / 上下文 / 权限全开箱 | 高：自造引擎与编排，数月工程 |
| **上线速度** | 快：核心数小时可 demo（我们数月跑通 M1–M3）| 慢：框架本身即多月工程 |
| **模型选择** | 锁 Claude（可经 Bedrock/Vertex 间接）| 自由：本地 / 私有 / 国产 / 混搭（Hermes 接 300+ 模型）|
| **记忆 / RAG** | 无原生，须自建（kdev-memory 即此）| 须自建，但完全自控后端 |
| **多智能体编排** | 子智能体开箱，但编排逻辑嵌技能 / CLAUDE.md | 可精确 DAG / 状态机编排 |
| **私有化 / 数据不出域** | 不支持纯离线，须连 Anthropic | 可完全私有化离线 |
| **可控性** | 中：行为透明，但 loop 内核不可换 | 高：逻辑透明、可 replay / 审计 |

- **OpenClaw**：Peter Steinberger 发起，2026 年初 GitHub 史上最多 star 的开源 agent；本地运行 + 100+ 技能 + 持久记忆。[来源](https://en.wikipedia.org/wiki/OpenClaw)
- **Hermes Agent**：Nous Research 开源 agent 框架；118 技能 + 跨会话持久记忆 + 接 300+ 模型。[来源](https://hermes-agent.nousresearch.com/)
- **Claude Code / Agent SDK 能力与边界**：[Agent SDK overview](https://code.claude.com/docs/en/agent-sdk/overview)。
