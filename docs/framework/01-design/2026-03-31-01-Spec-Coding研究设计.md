**各阶段 KDev 覆盖映射：**

### 1.2 设计目标

打造一个完整的 Spec Coding 工作流，实现：

- **AI 主导执行**：从规格生成到代码实现的自动化
- **人主导规则**：关键决策点需要人工确认，确保方向正确
- **可追溯纠正**：任何阶段发现偏差都可回溯纠正
- **试点验证**：通过五阶段迭代流程验证融合架构的可行性

### 2.2 各层详细说明

| Layer | 名称 | 核心功能 | 融合来源 |
|-------|------|----------|----------|
| **1** | 规格引擎 | 需求澄清、规格生成、规格评审 | brainstorming + speckit + openspec + gstack |
| **2** | 计划引擎 | 宏观任务分解、微观TDD计划 | speckit Plan + writing-plans |
| **3** | 执行引擎 | Subagent执行、两阶段评审 | subagent-driven-development + TDD + OMC状态管理 |
| **4** | 质量保障 | UT/IT/E2E测试、安全审计 | TDD + gstack:/qa + gstack:/cso |
| **5** | 记忆系统 | 纠正参考、知识提取、持久化 | questions-log + Claudeception |

| `tasks.json` | `status` | TDD 测试通过 + 代码评审通过 | `in_progress → done` | 任务完成条件 |
| `tasks.json` | `status` | 发现阻塞问题 | `in_progress → blocked` | 需要人工介入 |

## §三、Layer 1: 规格引擎设计

### 3.1 三阶段流程

```
用户输入: "/kdev-spec 实现用户登录功能"
                              ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  Stage 1: 需求澄清（superpowers:brainstorming）                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Checklist:                                                                 │
│  1. ✓ 探索项目上下文（检查文件、文档、最近提交）                               │
│  2. ✓ 逐个提问澄清（目的、约束、成功标准）                                    │
│  3. ✓ 提出 2-3 个方案（带权衡和推荐）                                         │
│  4. ✓ 分段展示设计（架构、组件、数据流、错误处理、测试）                        │
│  5. ✓ 用户审批设计（每段确认）                                               │
│                                                                             │
│  输出:                                                                       │
│    - clarified-requirements.md（澄清后的需求）                               │
│    - design-proposal.md（设计提案）                                          │
│    - questions-log.md（问题讨论记录，用于后续纠正）                           │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  HARD-GATE: 用户审批设计方案                                          │   │
│  │  不通过 → 继续讨论直到达成共识                                         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  Stage 2: 规格生成（speckit + openspec）                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  从0到1（新项目）:                                                           │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Constitution → Specify                                              │   │
│  │  输入: brainstorming 输出物                                           │   │
│  │  输出:                                                               │   │
│  │    - constitution.md（项目原则）                                      │   │

### 11.2 纠正闭环机制

**触发条件定义：**

| 触发条件 | 判断标准 | 检测方式 | 回退粒度 |
|----------|----------|----------|----------|
| `reviewReject` | ❌ Rejected（非 ⚠️ Concerns） | reviewer Agent 输出包含 `"status": "rejected"` | 回到 brainstorming 重新讨论 |
| `execConflict` | acceptance.md 定义的行为与代码实际行为不一致 | UT/IT 测试失败 + 错误信息与 acceptance.md 冲突 | 回到 `/kdev-plan` 调整计划 |
| `specMismatch` | 实现缺失规格中的功能 或 实现了规格外的功能 | reviewer Agent 两阶段评审发现差异 | 回到 `/kdev-exec` 让 executor 修复 |

**纠正流程：**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     纠正闭环流程                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  触发 reviewReject（❌ 级别）:                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  1. reviewer Agent 输出: "status": "rejected", "reason": "..."      │   │
│  │  2. 系统自动触发纠正                                                  │   │
│  │  3. 读取 questions-log.md 找到相关决策                               │   │
│  │  4. 回到 Layer 1 Stage 1 (brainstorming)                              │   │
│  │  5. 重新讨论问题并更新决策                                            │   │
│  │  6. 重新生成 specs                                                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  触发 execConflict（验收标准冲突）:                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  1. UT/IT 测试失败                                                   │   │
│  │  2. 错误信息分析：与 acceptance.md 中 Given/When/Then 矛盾            │   │
│  │  3. AskUserQuestion: "发现验收标准与实际冲突，是否调整？"              │   │
│  │  4. 用户选择：                                                       │   │
│  │     - 调整验收标准 → 回到 Layer 2 更新 acceptance.md                 │   │
│  │     - 修复代码 → 回到 Layer 3 修复实现                                │   │
│  │  5. 更新 questions-log.md 记录决策                                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  触发 specMismatch（规格不符）:                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  1. reviewer Agent 两阶段评审：                                       │   │
│  │     - Stage 1 (Spec合规): 发现缺失功能或额外功能                      │   │
│  │     - 输出: "mismatch": { "missing": [...], "extra": [...] }         │   │
│  │  2. 系统自动触发纠正                                                  │   │
│  │  3. 回到 Layer 3 让 executor Agent 修复                               │   │
│  │  4. 修复后重新进入两阶段评审                                          │   │
│  │  5. 最多重试 maxAttempts=3 次                                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 附录B: speckit vs openspec 详细对比

#### B.1 核心理念对比

| 维度 | speckit | openspec |
|------|---------|----------|
| **定位** | GitHub官方，规格驱动开发 | AI专属规格框架 |
| **核心理念** | **规格即执行** - 规格作为可执行产物 | **增量变更** - 轻量级迭代工作流 |
| **适用场景** | 从0到1（新项目启动） | 从1到N（存量项目迭代） |
| **流程风格** | Phase-Gated（门控阶段式） | Fluid（流畅迭代式） |

#### B.2 工作流对比

**speckit 流程（门控阶段式）：**
```
Constitution → Specify → Plan → Tasks → Implement
     ↓            ↓         ↓        ↓        ↓
  [门禁]      [门禁]    [门禁]   [门禁]   [门禁]

每个阶段必须完成并通过门禁才能进入下一阶段
```

**openspec 流程（流畅迭代式）：**
```
/opsx:propose → /opsx:apply → /opsx:archive
      ↓              ↓             ↓
   变更提案      实施应用        归档完成

无强制门禁，随时可以提出变更并快速迭代
```

#### B.3 规格结构对比

**speckit 规格（完整规格）：**
```
.kdev/
├── constitution.md     ← 项目原则（技术栈、质量标准、开发规范）
├── specs/
│   ├── feature-overview.md  ← 功能概览
│   ├── feature-detail.md    ← 功能详情
│   └── acceptance.md        ← 验收标准（GWT 格式）
├── plans/
│   └── implementation.md    ← 实施计划
└── tasks/
    └── tasks.md             ← 任务清单
```

**openspec 规格（增量规格）：**
```
.kdev/
├── changes/                 ← 变更提案目录
│   └── add-dark-mode/
│       ├── proposal.md      ← 变更提案（Delta spec）
│       ├── specs/           ← 需求和场景（仅针对本次变更）
│       ├── design.md        ← 技术方案
│       └── tasks.md         ← 实施清单
└── archive/                 ← 已完成归档
```

#### C.2 brainstorming 技能详解

**流程 CheckList：**
1. ✓ 探索项目上下文 — 检查文件、文档、最近提交
2. ✓ 逐个提问澄清 — 目的、约束、成功标准（一次一个问题）
3. ✓ 提出 2-3 个方案 — 带权衡和推荐
4. ✓ 分段展示设计 — 架构、组件、数据流、错误处理、测试
5. ✓ 用户审批设计 — 每段确认
6. ✓ 写设计文档 — `docs/plans/YYYY-MM-DD-<topic>-design.md`
7. ✓ 调用 writing-plans — 进入实施计划

**HARD-GATE：** 用户审批设计方案后才能继续

**关键原则：**
- 一次只问一个问题
- 多选题优于开放题
- YAGNI 原则：移除不必要的功能
- 探索替代方案：总是提出 2-3 个选项

#### C.3 subagent-driven-development 技能详解

**执行流程：**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  Subagent-Driven Development 流程                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Step 1: 分派实现 Agent                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - 提供完整任务文本 + 上下文                                          │   │
│  │  - Agent 执行：实现 → 测试 → 提交 → 自检                              │   │
│  │  - 状态: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              ↓                                              │
│  Step 2: Spec 合规评审                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - 检查代码是否完全匹配规格                                           │   │
│  │  - 发现缺失功能或额外功能                                             │   │
│  │  - 通过 → 进入代码质量评审                                            │   │
│  │  - 不通过 → 实现 Agent 修复 → 重新评审                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              ↓                                              │
│  Step 3: 代码质量评审                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - 代码风格、可维护性                                                 │   │
│  │  - 性能、安全                                                         │   │
│  │  - 通过 → 标记任务完成                                                │   │
│  │  - 不通过 → 实现 Agent 修复 → 重新评审                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  关键原则：                                                                  │
│    - Fresh subagent per task: 每个任务独立子 Agent，上下文隔离              │
│    - 两阶段评审：先规格合规，再代码质量                                      │
│    - 评审循环：发现问题 → 修复 → 重新评审                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### C.4 TDD 技能详解

**铁律：NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST**

```
RED → GREEN → REFACTOR 循环:

RED 阶段:
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. 写一个失败的测试                                                         │
│  2. 运行测试确认失败（MANDATORY）                                             │
│  3. 确认失败原因是功能缺失，不是语法错误                                        │
└─────────────────────────────────────────────────────────────────────────────┘
                              ↓
GREEN 阶段:
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. 写最少的代码让测试通过                                                    │
│  2. 运行测试确认通过（MANDATORY）                                             │
│  3. 确认其他测试仍然通过                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
                              ↓
REFACTOR 阶段:
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. 清理代码、消除重复                                                       │
│  2. 保持测试通过状态                                                         │
│  3. 不添加新功能                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
                              ↓
                           重复循环
```

**常见错误修正：**

| 错误想法 | 正确做法 |
|----------|----------|
| "太简单不用测试" | 简单代码也会出错，测试只需 30 秒 |
| "事后补测试" | 事后测试无法证明测试有效性 |
| "保留代码作为参考" | 删除代码，从测试开始 |
| "手动测试过了" | 手动测试不是系统化的，无法复现 |

### 附录F: wshobson/agents 三层模型路由详解

#### F.1 路由策略

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    任务复杂度判断                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  简单任务 (Haiku)                                                           │
│  - 格式化、翻译、简单问答                                                     │
│  - 触及 1-2 个文件 + 完整规格                                                │
│  - 适用: explorer Agent                                                     │
│                                                                             │
│  中等任务 (Sonnet)                                                          │
│  - 日常编码、测试、调试                                                      │
│  - 触及多文件 + 集成关注点                                                   │
│  - 适用: executor, tester Agent                                             │
│                                                                             │
│  复杂任务 (Opus)                                                            │
│  - 架构设计、复杂推理、安全审查                                               │
│  - 需要设计判断 + 广泛代码库理解                                             │
│  - 适用: planner, reviewer Agent                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### F.2 成本优化

- **Token 节省**：30-50%
- **策略**：简单任务用 Haiku，复杂任务才用 Opus

#### F.3 路由信号

| 信号 | 路由目标 |
|------|----------|
| 触及 1-2 个文件 + 完整规格 | Haiku |
| 触及多文件 + 集成关注点 | Sonnet |
| 需要设计判断 + 广泛代码库理解 | Opus |

source: extracted
---
