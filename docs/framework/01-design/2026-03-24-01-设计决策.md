## 1. Agent 设计（5 个）

### 确认的 Agent 列表

| Agent | 模型 | 职责 |
|-------|------|------|
| **planner** | Opus | 需求分析、架构设计、任务分解 |
| **executor** | Sonnet | 代码实现、Bug 修复、重构、调试 |
| **reviewer** | Opus | 代码审查、安全检查 |
| **tester** | Sonnet | 测试编写、测试运行、覆盖率分析 |
| **explorer** | Haiku | 代码探索、文档查找、信息收集 |

### 职责边界

| 场景 | 负责 Agent | 说明 |
|------|------------|------|
| 分析需求 | planner | 输出清晰的规范和计划 |
| 编写代码 | executor | 包含 Bug 修复和重构 |
| 编写测试 | tester | TDD 流程中的测试编写 |
| 代码审查 | reviewer | 只读审查，不修改代码 |
| 问题调试 | executor | 调试能力作为 executor 的子能力 |
| 查找信息 | explorer | 快速搜索和定位 |

### Agent 定义格式

```markdown
---
name: agent-name
description: Use when [具体触发条件]
model: opus|sonnet|haiku
---

# Agent Name

## Your Mission
## Capabilities
## Tools Available
## Constraints
```

### Agent 通信协议

定义 Agent 之间数据传递的标准格式：

#### planner → executor（任务分配）

```json
{
  "taskId": "TASK-001",
  "title": "实现用户登录 API",
  "description": "创建 POST /api/login 端点",
  "acceptanceCriteria": [
    "返回 200 状态码和 JWT token",
    "错误凭据返回 401"
  ],
  "dependencies": [],
  "estimatedMinutes": 5,
  "files": ["src/api/login.ts", "tests/api/login.test.ts"],
  "priority": "P1"
}
```

#### reviewer → executor（评审反馈）

```json
{
  "reviewId": "REV-001",
  "taskId": "TASK-001",
  "status": "needs_fix",
  "items": [
    {
      "type": "security",
      "severity": "critical",
      "location": "src/api/login.ts:42",
      "description": "密码未使用 bcrypt 哈希",
      "suggestion": "使用 bcrypt.hash(password, 10)"
    },
    {
      "type": "style",
      "severity": "minor",
      "location": "src/api/login.ts:15",
      "description": "变量名不清晰",
      "suggestion": "将 `d` 改为 `userData`"
    }
  ],
  "summary": "发现 1 个严重问题需要修复"
}
```

#### tester → executor（测试结果）

```json
{
  "testRunId": "TR-001",
  "taskId": "TASK-001",
  "status": "failed",
  "totalTests": 5,
  "passed": 3,
  "failed": 2,
  "coverage": {
    "lines": 75,
    "branches": 60,
    "functions": 80
  },
  "failures": [
    {
      "test": "should return 401 for invalid credentials",
      "error": "Expected 401, got 500",
      "file": "tests/api/login.test.ts:25"
    }
  ]
}
```

## 2. Skill 设计（9 个）

### 确认的 Skill 列表

| Skill | 类型 | 职责 |
|-------|------|------|
| **/kdev-init** | 入口 | 初始化项目，创建 .kdev/ 目录结构 |
| **/kdev-spec** | 阶段 | IR → SR → AR 需求分层，编写规范 |
| **/kdev-plan** | 阶段 | 任务分解（2-5 min 粒度），制定计划 |
| **/kdev-test** | 阶段 | TDD 测试流程，编写失败测试 |
| **/kdev-exec** | 阶段 | 代码实现，让测试通过 |
| **/kdev-debug** | 辅助 | 问题诊断、根因分析、调试支持 |
| **/kdev-review** | 阶段 | 代码审查、三方评审 |
| **/kdev-verify** | 阶段 | 最终验证、门禁检查 |
| **/kdev-flow** | 核心 | 端到端流程编排（状态机驱动） |

### Skill 分类

**阶段 Skill**：对应流程的特定阶段
- /kdev-spec, /kdev-plan, /kdev-test, /kdev-exec, /kdev-review, /kdev-verify

**辅助 Skill**：可在任何阶段调用
- /kdev-debug

**入口 Skill**：初始化和编排
- /kdev-init（初始化）
- /kdev-flow（核心编排）

### /kdev-flow 状态机设计

```
状态转换流程（含错误恢复）:

INIT ──(/kdev-init)──▶ SPEC ──(/kdev-spec)──▶ PLAN ──(/kdev-plan)──▶
  │                      │                       │
  │                      │ HARD-GATE             │ HARD-GATE
  │                      │ 人工确认 ⚠️           │ 可选确认
  │                      ▼                       ▼
  │                   TEST ──(/kdev-test)──▶ EXEC ──(/kdev-exec)──▶
  │                      │                       │
  │                      │ HARD-GATE             │ HARD-GATE
  │                      │ RED 状态              │ 测试通过
  │                      │                       │
  │                      │ ◀───(失败)────────────┤ maxRetry: 3
  │                      │                       │
  │                      ▼                       ▼
  │                   REVIEW ──(/kdev-review)──▶ VERIFY ──(/kdev-verify)──▶ DONE
  │                                              │
  │                                              │ HARD-GATE
  │                                              │ 全部通过
  │                                              │
  │                                              │ ◀───(失败)───┐
  │                                              │              │
  └────────────────────────────────────────── /kdev-debug (随时可调用)
                                                   │
                                                   │ 修复后回到相应阶段
                                                   ▼
                                             [ROLLBACK] 回滚到上一稳定状态

错误恢复路径:
- EXEC 失败 → 回到 TEST（maxRetry: 3）
- REVIEW 不通过 → 回到 PLAN
- VERIFY 失败 → 回到 EXEC
- 超过最大重试次数 → 触发 ROLLBACK
```

### Skill 定义格式

```yaml
---
name: kdev-xxx
description: Use when [具体触发条件]
---

# kdev-xxx

## Purpose
## When to Use
## Steps
## HARD-GATE (如有)
## Verification
```

## 3. 门禁设计

### 门禁配置结构

```json
// .kdev/constitution.json
{
  "version": "1.0.0",
  "project": {
    "name": "project-name",
    "type": "web|api|cli|library"
  },
  "concurrency": {
    "maxSessions": 1,
    "lockStrategy": "file-lock",
    "lockFile": ".kdev/.lock"
  },
  "retry": {
    "maxAttempts": 3,
    "backoff": "exponential",
    "baseDelay": 1000,
    "maxDelay": 30000,
    "timeout": 60000
  },
  "gates": {
    "SPEC": {
      "strict": true,
      "humanConfirmation": true,
      "onFailure": {
        "action": "return_to_spec",
        "maxRetries": 3,
        "requireRootCauseAnalysis": true
      },
      "checks": [
        { "type": "completeness", "required": true },
        { "type": "clarity", "required": true },
        { "type": "testability", "required": true }
      ]
    },
    "PLAN": {
      "strict": true,
      "humanConfirmation": false,
      "onFailure": {
        "action": "return_to_plan",
        "maxRetries": 3
      },
      "checks": [
        { "type": "taskGranularity", "required": true, "maxMinutes": 5 }
      ]
    },
    "TEST": {
      "strict": true,
      "onFailure": {
        "action": "return_to_test",
        "maxRetries": 3
      },
      "checks": [
        { "type": "testsFailing", "required": true }
      ]
    },
    "EXEC": {
      "strict": true,
      "onFailure": {
        "action": "return_to_test",
        "maxRetries": 3
      },
      "checks": [
        { "type": "testsPass", "required": true },
        {
          "type": "incrementalCoverage",
          "required": true,
          "threshold": 80,
          "configurable": true
        },
        {
          "type": "totalCoverage",
          "required": false,
          "monitorOnly": true
        }
      ]
    },
    "REVIEW": {
      "strict": true,
      "humanConfirmation": true,
      "onFailure": {
        "action": "return_to_plan",
        "maxRetries": 2
      },
      "multiModelReview": {
        "enabled": true,
        "models": [],
        "strategy": "consensus",
        "tieBreaker": "human"
      },
      "checks": [
        { "type": "codeReview", "required": true },
        { "type": "securityScan", "required": true }
      ]
    },
    "VERIFY": {
      "strict": true,
      "onFailure": {
        "action": "return_to_exec",
        "maxRetries": 2
      },
      "checks": [
        { "type": "lint", "required": true },
        { "type": "build", "required": true }
      ]
    }
  }
}
```

### 门禁规则

| 规则 | 说明 |
|------|------|
| **严格模式** | 每个门禁必须通过，不可绕过 |
| **可配置阈值** | 覆盖率等指标可在 constitution.json 中配置 |
| **人工确认** | SPEC 和 REVIEW 阶段必须人工确认 |
| **失败处理** | 记录失败原因，根据 onFailure 配置回退 |
| **重试机制** | 支持配置最大重试次数和指数退避 |
| **并发控制** | 文件锁机制，限制单会话操作 |
| **审计日志** | 记录人工确认意见和决策责任 |

### 人工确认审计

人工确认时记录完整的审计信息：

```json
{
  "confirmation": {
    "confirmed": true,
    "phase": "SPEC",
    "confirmer": "user@example.com",
    "confirmedAt": "2026-03-24T10:30:00Z",
    "comments": "需求清晰，但需要注意性能问题",
    "decisions": [
      {
        "item": "SR-002 批量导入",
        "decision": "同意",
        "notes": "建议分批处理，每批 100 条"
      }
    ],
    "sessionId": "kdev-1709123456789"
  }
}
```

### 性能指标定义

| 阶段 | 预期耗时 | 超时阈值 | 告警 |
|------|----------|----------|------|
| SPEC | 30-60 min | 2 hr | ✓ |
| PLAN | 15-30 min | 1 hr | ✓ |
| TEST | 20-40 min | 1 hr | ✓ |
| EXEC | 每 task 5 min | 8 hr | ✓ |
| REVIEW | 30-60 min | 2 hr | ✓ |
| VERIFY | 10-20 min | 30 min | ✓ |

### 门禁检查命令

| 检查类型 | 默认命令 | 可配置 |
|----------|----------|--------|
| 测试通过 | `npm test` | ✓ |
| 覆盖率 | `npm run test:coverage` | ✓ |
| Lint | `npm run lint` | ✓ |
| Build | `npm run build` | ✓ |

## 4. 三方评审设计

### 设计原则

**灵活性优先**：支持用户已接入的任何模型，不限于特定模型。

### 配置方式

```json
{
  "multiModelReview": {
    "enabled": true,
    "models": ["codex", "gemini", "openai", "qwen", "glm"],
    "strategy": "consensus",
    "timeout": 60000
  }
}
```

### 评审策略

| 策略 | 说明 |
|------|------|
| `consensus` | 所有模型都要通过 |
| `majority` | 多数模型通过即可 |
| `any` | 任一模型通过即可 |

### 冲突解决机制

当多个模型意见冲突时：

| tieBreaker | 说明 |
|------------|------|
| `human` | 人工裁决（默认） |
| `claude-final` | 主 Claude 最终决定 |
| `strict-vote` | 严格投票计数 |

```json
{
  "multiModelReview": {
    "enabled": true,
    "models": ["codex", "gemini", "qwen"],
    "strategy": "consensus",
    "tieBreaker": "human",
    "timeout": 60000,
    "retry": {
      "maxAttempts": 2,
      "backoff": "exponential"
    }
  }
}
```

### 评审流程

```
/kdev-review 执行流程:
1. 主 Claude 生成初步评审意见
2. 读取用户配置的模型列表
3. 并行调用各模型进行评审
   - 通过 /ask 方式（如果可用）
   - 或通过 MCP API 方式
4. 合并多个评审结果
5. 根据策略判断是否通过
6. 生成综合评审报告
7. 等待人工确认
```

### 评审报告格式

```markdown
# Code Review Report

## Primary Reviewer (Claude)
- 评审意见...

## Secondary Reviewers
### Codex
- 评审意见...

### Gemini
- 评审意见...

## Consensus
- ✅ 达成共识的问题
- ⚠️ 需要人工判断的分歧

## Action Items
- [ ] 必须修复的问题
- [ ] 建议改进的点
```

## 6. 需求分层模型（IR → SR → AR）

### 概念定义

| 层级 | 全称 | 粒度 | 交付物 |
|------|------|------|--------|
| **IR** | Initial Requirement | 人的原始需求（可能模糊） | IR 文档 |
| **SR** | Story Requirement | 可独立交付的功能模块（1-2 周） | SR CSV |
| **AR** | Acceptance Requirement | 可测试可验收的需求单元（1-3 天） | AR CSV + GWT |

### IR 文档格式

```markdown
# Initial Requirement

## 原始描述
[用户的原始需求描述]

## 澄清问题
- Q1: [问题]
  - A1: [答案]
- Q2: [问题]
  - A2: [答案]

## 假设
- [假设1]
- [假设2]

## 边界
- 包含: [...]
- 不包含: [...]
```

### SR CSV 格式

```csv
id,title,priority,estimated_days,dependencies,ar_count
SR-001,用户登录,P1,3,,2
SR-002,用户注册,P1,2,SR-001,1
SR-003,批量导入,P2,5,SR-001,3
```

### AR CSV 格式

```csv
id,sr_id,title,given,when,then
AR-001,SR-001,登录成功,用户在登录页,输入正确凭据并提交,跳转到首页
AR-002,SR-001,登录失败,用户在登录页,输入错误凭据并提交,显示错误提示
```

## 7. 与外部 Skill 的集成

### 依赖的外部 Skill

| 外部 Skill | 用途 | 调用位置 |
|------------|------|----------|
| `superpowers:brainstorming` | 需求澄清 | /kdev-spec |
| `superpowers:writing-plans` | 计划编写 | /kdev-plan |
| `superpowers:test-driven-development` | TDD 流程 | /kdev-test |
| `superpowers:executing-plans` | 执行计划 | /kdev-exec |
| `superpowers:verification-before-completion` | 验证完成 | /kdev-verify |

### 降级策略

如果外部 Skill 不可用：

| 外部 Skill | 降级方案 |
|------------|----------|
| `brainstorming` | 内置简化版需求澄清流程 |
| `writing-plans` | 内置计划模板 |
| `test-driven-development` | 内置 TDD 检查清单 |
| `executing-plans` | 直接使用 executor Agent |

### 使用方式

```bash
# 完整流程（默认）

# 快速模式（仅 PLAN → EXEC → VERIFY）
/kdev-flow --quick "修复 typo"
```

## 附录 C：回滚机制

### 触发条件

- 超过最大重试次数
- 发现严重设计缺陷
- 用户手动请求回滚

### 回滚策略

```
ROLLBACK 流程:
1. 记录当前状态到 .kdev/sessions/rollback-{timestamp}.json
2. 恢复到上一稳定状态（最近一个 COMPLETED 的阶段）
3. 保留失败原因分析文档
4. 重置相关计数器
```

### 回滚配置

```json
{
  "rollback": {
    "enabled": true,
    "maxHistory": 10,
    "autoBackup": true,
    "backupLocation": ".kdev/backups/"
  }
}
```

### 状态快照

每个阶段完成时自动创建快照：

```
.kdev/backups/
├── snapshot-INIT-2026-03-24T10:00:00Z.json
├── snapshot-SPEC-2026-03-24T10:30:00Z.json
├── snapshot-PLAN-2026-03-24T11:00:00Z.json
└── ...
```
