## 2. 三工作流 + 5 阶段

**三条工作流**：**A** = kdev-core 底座（R1–R7 + 框架）· **B** = kdev-memory 进化（git / scope / JSONL / rollup）· **C** = 员工层（在底座上重建、参考 X3）。

```
阶段0  kdev-core 核心三件 + git 托管前置
        A: R1(白捡 golden seed) → R2 节点机 → R3 结构化三类 gate
        B: Q-009 nested-repo + 自举 hook + kdev-sync.yml（与 A 并行，不阻塞）
          ▼
阶段1  ★ 开发工程师端到端 dogfood —— 史上首次真实实测 ★   [L2 协同]
        C: coding-flow 13 节点接底座（R1 新接/R2 node-table/R3 gate 映射）
        记忆暂用 flat 单 scope；评审由 AI 内联判（decider:ai），不引入评审专家
          ▼  验收: R3 三类各跑通 + 有界回流 + resume 可续 + 真任务出可运行代码
阶段2  记忆 scope/JSONL + 第二个员工（需求架构师，喂上游 SR/AR）
        B: scope 维度 + JSONL 操作层 + per-scope counter
          ▼
阶段3  框架层 + 评审专家 + 跨员工协作              [L2 完整 → L3 起步]
        A4: events/HUD/派单/产物总线
        C: 评审专家（搬 X3 的 10 checklist + 冲突仲裁）+ 直接发函硬规
          ▼
阶段4  CQO 元监督 + 自演进 + 跨 IDE              [L4 自治 · 北极星 · defer]
```

> 自主度阶梯：阶段1 = **L2 协同**（员工跑完整 flow、主控在 gate 确认）→ 阶段3 = L3 起步 → 阶段4 = L4。**L1 不作为起点**——它是单步助手，装不下"端到端跑通一条 flow"，从 L2 起步是承 [Q-006「当前≈L2」](../../../.kdev/memory/决策日志.md) 的自然结论。

## 3. 阶段0 — kdev-core 核心三件 + git 托管前置

### 3.1 A 轨：底座核心三件（纯底座，TDD）

| 件 | 范围 | 来源 | 验收 |
|---|---|---|---|
| **R1 状态** | flow-state.json + 原子写(tempfile+os.replace) + active/_meta stale guard + resume | 泛化 [kdev-design-flow/lib/flow_state.py](../../../plugins/kdev-design-flow/lib/flow_state.py)（🔴 最高 ROI，白捡）| 原子写 / 损坏恢复 / resume 单测过 |
| **R2 节点机** | node-table loader（merge L0+L1 编排）+ advance（邻接表 + 守卫 guard(state)→str\|null）+ 有界回流 + 幂等豁免(retry++) | 借 OMC Team Pipeline 范本，不 fork | 顺进 / 回流 / 幂等豁免 / 完成 单测过 |
| **R3 三类 gate** | GateResult(request_id/iter/verdict 结构化 JSON) + record_gate 三类统一解析 + iter≥max→escalate BLOCKED（不 force-accept）+ 双重计数 | [底座总纲 §4.3](./2026-06-05-01-kdev-core底座设计总纲-v1.0.md)；**X3 反证：gate 逻辑不能埋 prompt 文本** | review/decision/acceptance 各 PASS+FAIL + 超限 escalate 单测过 |

> **R3 一次做全三类**（不止 review）——因为消费者 coding-flow 一上来就用全三类（review 代码/质量 · decision Gate-A/B · acceptance E2E）。这逼底座 R3 从第一天就完整、且被真实消费者验证。
>
> **TDD 推进**，测试设计参考 X3 的 smoke 模式（full-lifecycle + resume-after-interrupt，见 X3 `tests/test_smoke_handoff_dry_run.py`）。**Python 函数非 class**（守底座总纲 §4.1；X3 用 `StateMd` class 是反面参考）。

### 3.2 B 轨：git 托管前置（Q-009，与 A 并行）

- **实现**：kdev-memory 的 **自举 hook**（SessionStart 无 `.kdev/.git` → 从 `kdev-sync.yml` clone；有 → pull；SessionEnd/rollup → commit+push）+ `kdev-sync.yml` schema + [§9.4 托管提醒](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md)。
- **立即收益**：把**当前这套设计记忆**（`.kdev/memory/`，现 machine-local 未托管）设为 nested repo，先保命。
- **不阻塞 A**：git 托管是 kdev-memory 独立特性，与底座 TDD 互不依赖，可并行。

## 4. 阶段1 — 开发工程师 walking skeleton ★（首次真实实测）

### 4.1 refit 内容

把 [kdev-coding-flow](../../../plugins/kdev-coding-flow/) 的 13 节点 SOP 接上 kdev-core 底座：

- **R1 状态**：coding-flow 现**无**状态持久化 → 新接底座 R1，**白捡 resume**。
- **R2 节点机**：声明 coding-flow 的 13-node-table（节点 + kind + gate）。
- **R3 gate 映射**：Gate-A/B/C + E2E 映射到底座三类 —— 代码/质量评审→**review**、Gate-A/B 路线判断→**decision**、E2E/验收→**acceptance**。**精确 node/gate 映射在 writing-plans 阶段细化。**
- **Auto Mode 留 flow 侧**（守 [Q-008「执行留 flow」](../../../.kdev/memory/决策日志.md)）：底座只提供 R3 的 `awaiting_human` 停靠点，Auto Mode 的"自己往下跑"逻辑留在 coding-flow prompt。

### 4.2 边界（保持骨架薄）

- **记忆**：flat 单 scope（[记忆架构 §4](./2026-06-05-02-数字员工记忆架构-分层+scope拓扑-v0.1.md) flat 态），不上 scope/JSONL。
- **评审**：开发工程师 solo，gate 由 **AI 内联判**（R3 review-kind，decider:ai 默认，跑 UED §10 自检清单）——**不引入评审专家**（阶段3）、不引入跨员工协作。**竞赛分由外部 benchmark grader 出**，不是我们的 flow。
- **dogfood 任务**：**UED 6.0 改造 benchmark**（RuoYi-Vue3-FastAPI 视觉改造）。题目 + materials 在 `测试题目/`：[EXAM-PROMPT.md](../../../测试题目/EXAM-PROMPT.md) + `01-pic/`（3 原型）+ `02-ued6.0/`（UED 硬约束 + design-tokens + tailwind preset + css）。后端不允许改，纯前端改造。
- **两 pass**（用户定）：
  - **Pass 1 验底座**：砍成小切片（T0 全局主题 + T1 登录页，或 T0 + 用户管理标杆页），**不追分**，目的把 kdev-core R1/R2/R3 + 开发工程师端到端抠通。
  - **Pass 2 真 benchmark**：底座稳后，**整题原样不改**跑全量（T0–T4 全菜单），拿可比竞赛分（题目要求"原样发给每个参赛智能体"，砍题即不可比）。
- **TDD 适配**：视觉改造主体**非 TDD 可驱动** → R3 acceptance 走 **build 通过 + UED §10 自检 + 视觉 diff + 功能冒烟**（登录→首页→用户管理 CRUD）；coding-flow 接底座时需一条「前端/视觉」验证支路，不假设处处 TDD 红绿。

### 4.3 验收标准（tracer-bullet 级，不镀金）

**A. 底座机制类**（证明 R1/R2/R3 真在驱动，不是 prompt 演戏）
1. **R1 真驱动**：flow-state.json 全程结构化记 `current_node` + `history[GateResult]`；中途 kill → `resume` 从上次 node 正确续（非从头）。
2. **R2 真流转**：13 节点按邻接表推进；至少一次 gate 不过的**有界回流**（retry++ 记状态、可查），有上限不死循环。
3. **R3 三类各跑通一次**：review(代码/质量) · decision(Gate-A/B 选分支) · acceptance(E2E) 各至少触发一次正确流转；其中至少一次 **FAIL→修→重评**闭环；GateResult 是结构化 JSON（带 request_id/iter/verdict），下游不靠正则解析。
4. **escalate 不 force-accept**：构造一次评审超限(≥max)，验证转 BLOCKED 升人工、非自动降级接受。

**B. 端到端交付类**（证明它是能干活的员工，不只机制 demo）

5. **真任务端到端**：环境对齐→实施计划→（视觉改造主体非 TDD）→**build 通过 + 功能冒烟**（登录→首页→用户管理 CRUD）跑完，产出可运行前端 + RuoYi 既有功能不破。
6. **L2 画像成立**：员工自跑完整 flow，主控只在 gate 确认/决策，不替员工写代码。

**C. 工程质量类**

7. **底座 R1/R2/R3 单测齐全且过** + 接 coding-flow 后的集成 smoke（参考 X3 full-lifecycle+resume 模式）。

**UED 合规（已确认启用，dogfood 是前端任务）**：**UED §10 自检清单**作为一道 R3 review 闸——禁裸 hex / 8px 网格 / 按钮圆角 2px / 表头 44px / chart palette 定序 / 字体白名单 等，**客观可机检**（grep + 人工抽查）。Pass2 全量时还叠加题目自带的"遍历全菜单逐页截图"评审。

**明确非目标（防镀金）**：不要评审专家(阶段3)、scope/JSONL 记忆(阶段2)、CQO(阶段4)、多员工协作、HUD 完整体(阶段3)；dogfood 求打通不求规模。

## 6. X3 参考映射（reference-only）

X3 最大价值 = **反向验证 kdev-core 设计方向**：X3 赶工自造 runtime 踩的坑，恰好是 lean 底座要修的点（同一问题两次相遇）。

**搬（成熟设计，照搬当参考）**：12 评审节点表（R/D/T/F + 阻断/告警 + D2 并行）· 10 评审员 checklist（代码评审最细）· 冲突仲裁 3 步 SOP · COMPLETE.md 聚合 manifest · 轻组长 4 职责分离 · agent frontmatter + lint 规则 · smoke 测试模式 · kdev-start-feature 入口结构。

**别学（赶工/技术债/与 canonical 冲突）**：

| X3 做法 | 为何别学 / 底座正解 |
|---|---|
| 状态用 markdown + regex + 全量覆写 + class | R1 必须 **JSON + 原子写 + 函数非 class**（并发会撞）|
| **gate 逻辑埋 prompt 文本**，retry 计数只在 events.log 文字 | R3 必须**结构化 GateResult**，否则没法程序化查重试轮次、resume 全靠正则猜 |
| events.log 单文件 + tab 分隔 + 全量读 | 操作层必须 **per-scope JSONL**（跨 feature 无法 grep、加字段破格式）|
| `AGENT_GROUP`/`AGENT_TO_LEAD` 双份硬编码路由表 | **单一数据源**（已有漂移风险）|
| on-blocked hook 触发条件过宽 + stdout 注入 hack | hook framework **标准化 Blocked 信号** |
| handoffs 只有 COMPLETE.md 无中间 checkpoint | 中间节点失败靠文本推断 → 需结构化 checkpoint |
