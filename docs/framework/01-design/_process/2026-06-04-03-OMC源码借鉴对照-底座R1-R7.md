## 0. 总判断

OMC 是跑通的 TS 运行时，**编排状态机这块尤其成熟**。我们**借设计 pattern、不 fork 源码**；而且有几处**我们能比 OMC 做得更好**（structured verdict、escalate 不 force-accept）。

## 1. R1 状态 — 借 OMC state-manager

| OMC 现成的 | 借什么（落 Python R1） | 出处 |
|---|---|---|
| `_meta` envelope：写时注入 `written_at/mode/sessionId`，读时剥离 | flow-state 写入加 `_meta:{written_at, flow, slug, step_id}`，读时剥离，不污染业务字段 | `lib/mode-state-io.ts:217-252` |
| **目录隔离而非文件名**：`sessions/{id}/{mode}.json` | `.kdev/flows/<flow>/<slug>/flow-state.json`（我们已是这设计 ✅，得到验证）；多 worktree 天然隔离 | `state-paths.ts:31-121` |
| **temp-rename 原子写** + fsync + dir-fsync | Python `tempfile` + `os.replace`（我们已是 ✅，得到验证） | `atomic-write.ts:42-91` |
| **stale guard**：`active:bool + updated_at`，stale 时**写 active=false 而非删** | flow-state 加 `active + updated_at`；crash-resume 时按 `active && updated_at<阈值` 判恢复 | `state-manager/index.ts:584-663` |
| 💡 OMC **自己废弃了 StateManager class**，转简单函数 `writeModeState/readModeState` | **验证我们"Python 原生函数（flow_state.py）"方向对**，不必绕 class | `index.ts:742 @deprecated` |

**跳过**：读缓存（hook 进程短命无意义）、SQLite（只为外部 AI job）、O_EXCL lockfile spin-wait（`os.replace` 已原子）、双套 GC 扫描、XDG/global 双路径。

## 2. R2 编排/节点机 — 借 OMC Team Pipeline（最成熟，重点借）

OMC Team Pipeline = `plan→prd→exec→verify→fix` 邻接表状态机。

| OMC 现成的 | 借什么（落 R2） | 出处 |
|---|---|---|
| **邻接表 + 守卫分离**：`ALLOWED: Record<phase, phase[]>` 纯声明式边；守卫 `hasRequiredArtifactsForPhase()` 返回 `string\|null`（null=过，非null=拒因）| **强烈借**：node-table 每条 edge 附 `guard?(state)→str\|null`，与守卫逻辑解耦 | `team/transitions.ts:5-14, 50-94` |
| **三段式 advance**：① 邻接表合法性 → ② 守卫 → ③ 不可变变更 + 原子写 | 直接作我们 `advance(node, next, reason?)` 引擎模板 | `transitions.ts:50-94` |
| **有界回流 + 强制 failed**：`fix_loop.{attempt, max_attempts=3}`，溢出强跳 failed | R2 gate 回流照此：retry 溢出 → advance 到 terminal-fail，禁无限回流 | `state.ts:186-251` |
| **phase_history 审计链** `[{phase, entered_at, reason?}]` | node-table advance 时追加，免费调试轨迹 | `state.ts:51-72` |
| ⚠️ **幂等豁免**：fix→fix 专门打破幂等（同相位重入=retry++）| 我们 gate 回流（gate→同 action）要**显式标"回流"非"幂等重入"**，否则 retry 计数失效 | `state.ts:186-251` |
| 分层 phase：持久化 phase vs 从任务分布**推断**的 phase | node-table 静态声明 + `inferNodeKind()` 实时推断，二者独立 | `phase-controller.ts` |
| artifacts 指针入 state：`artifacts.{plan_path, prd_path, verify_report_path}` | R6 产物总线：产出物路径入 flow-state（轻量 handoff）| `state.ts:51-72` |

**跳过**：tmux/pane、provider 三路由 `stage-router`、sentinel-gate factcheck、merge-orchestrator、done.json polling（v1 已弃）。

## 5. 吸取的"好经验" — Top 8（直接进底座设计）

1. **邻接表 + 守卫分离**（R2）：声明式边 + `guard()→str|null`，advance 三段式。← 底座节点机的骨架
2. **有界回流 + 强制 failed + 幂等豁免**（R2）：回流标记 retry++，溢出转 terminal，禁无限回流。
3. **request_id 锁 approval**（R3）：防跨轮次 approval 串扰。
4. **escalate 不 force-accept**（R3，胜 OMC）：超限转人工 BLOCKED，不退化接受缺陷。
5. **structured verdict 不正则**（R3，胜 OMC）：JSON verdict（已有），别学 OMC 文本扫描。
6. **双重计数 + awaiting_human+TTL**（R3/R4）：本 gate + 本 flow 计数；停靠超时策略分 review/decision。
7. **mode = JSON 状态文件 + TTL + 互斥 + Stop circuit-breaker**（框架）：防死循环/幽灵模式。
8. **tier env override + provider inherit**（R5）：模型档可配、企业提供商兼容。

**两条"避坑"**（OMC 的教训）：① 别 force-accept 退化（改 escalate）；② 别正则扫文本判 verdict（改 structured）。
