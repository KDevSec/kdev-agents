## 0. 叙事主线（Approach A · 分层递进）

> 底座 →（转折）为什么要 SOP → harness 怎么支撑可编排 → 内置这版 SOP → 任意编排 → 自演进

因果链一句话：**有了「数字员工底座」，SOP 不写死、可编排；harness 工程是让 SOP 可编排的底座机制；终局是员工自己优化 SOP。**

三段命题（贯穿全篇）：

1. **底座** = 数字员工架构（能力 → 数字员工 → 6 人 AI 团队） —— 一支能端到端交付的 AI 团队。
2. **SOP** = 节点表 + gates 的可编排流程 —— 底座之上「跑什么、怎么把关、哪里停」。
3. **自演进** = 员工把复利飞轮从「反哺能力」延伸到「反哺 SOP 本身」 —— 从人编排 SOP 到员工自优化 SOP。

## 1. 逐页大纲（11 页）

### P1 · 封面
- 标题：**KDev 数字员工**　副标题：**从底座到自演进的 SOP 编排**
- meta：同行技术分享 · 2026-06
- 视觉：沿用 v0.1 cover（navy 上下色带 + 居中大标题）

### P2 · 核心主张 + 全篇地图
- **钩子横幅**：底座管「怎么跑」· SOP 管「跑什么」· 员工管「自己优化怎么跑」
- 三段命题（底座 / SOP / 自演进）三栏卡片
- **6 块导航**：①底座 ②为什么要 SOP ③harness 核心 ④harness 韧性 ⑤内置 SOP ⑥任意编排 + 自演进
- 讲法：先给地图，让混合受众知道全场走向；技术同行知道 harness 在第 3–4 块见真章。

### P3 · 底座①：数字员工架构（复用 v0.1 浓缩）
- 左：6 人 AI 团队组织（用户 → CEO 主控 → CQO 元监督 + 3 交付员工 + 评审专家收敛），28 agent 拆解一行
- 右：金字塔（通用 Agent → 底层技能 → 能力 → 数字员工）+ 落点 strip
- **落点**：底座给了我们一支能端到端交付的 AI 团队 —— 但「按什么流程交付」还没定。
- 讲法：这页是已对外讲过的概念模型浓缩，快速过，作为下一页转折的铺垫。

### P4 · 转折：光有员工不够，要 SOP 把活儿编排起来
- 员工会干活，但四个问题悬空：**按什么流程 / 什么顺序 / 哪里把关 / 哪里停？**
- 核心等式：**SOP = 节点表（做什么、什么顺序）+ gates（哪里把关、过不过怎么办）**
- bridge：这套「节点表 + gates」如果写死，就只能跑一种活；要让一个底座跑任意业务 → 需要 harness 工程让它**可编排**。
- 讲法：全篇的「转折页」，把听众从「员工是什么」带到「怎么编排员工干活」。

### P5 · harness 工程·核心：kdev-core lean 底座（R1/R2/R3）
- **R1 flow-state 状态机**：每个 flow 一份 `flow-state.json`，原子写（tmp + os.replace）+ history 流水 → 白捡 resume（断点续跑）
- **R2 通用 n 节点机**：节点表 flow 自己声明（不写死段数），底座提供流转引擎（顺进 / 回流）
- **R3 三类 gate 统一抽象**（核心难点 → 卖点）：
  | kind | 谁评 | verdict | 不过怎么办 |
  |---|---|---|---|
  | review 评审重试 | ai/both/human | PASS/FAIL | 改 → 重试（max N）|
  | decision 路线判断 | ai 默认/human 可接管 | 分支选择 | 选分支即流转 |
  | acceptance 验收 | test 自动跑 | PASS/FAIL | 回流 fix 节点 |
- **铁律 strip**：底座只记账、不判断本身（AI 评审 / E2E / 路线决策都在 flow 侧执行）→ 底座 lean、**语言模型无关**
- 代码片段：GateResult schema + `record_gate` 三类统一解析伪代码
- 讲法：harness「见真章」页。技术同行细看 R3 抽象（三类靠 `kind` 区分、共用 history + next_node 解析）；管理同行抓一句「底座只记账不判断」。

### P6 · harness 工程·韧性与人机协同
- 机制网格（6 块）：
  - **retry 断路**：同一 gate 连续 max_retries（默认 3）不过 → 升级人工 / CEO
  - **reviewer 锁定 + tier 升级**：重试时同一 reviewer 不漂移；复杂度涨了升模型 tier（sonnet→opus）
  - **decider 旋钮**：每类 gate「谁来判」默认 AI/自动，编排可改 human 接管岔路口（人有最终决定权、默认不挡路）
  - **Auto Mode 停靠**：自主推进时**只在 `decider: human` 的 gate 停**，其余 AI 自己过 —— 哪里停由编排决定
  - **Worker Preamble**：员工执行任务不套娃 sub-agent（防 token 爆炸）
  - **memory scope**：`shared/`（全员共享）+ `staff/<员工>/`（私有），路径前缀隔离
- 讲法：「可讲可跳」页。技术受众这页最有共鸣；时间紧可只点 decider 旋钮 + Auto Mode 停靠（最能体现人机协同）。

### P7 · 内置这版 SOP（showcase）：4 条 kdev flow
- 四卡片（kdev 开头插件内置的 SOP）：
  - **需求设计**（kdev-design-flow）：SR 需求 → AR 拆解 → 高保真原型 → 概要+详设；4 评审闸门（review，ai/both/human）
  - **编码开发**（kdev-coding-flow）：13 节点；Gate-A 关联度（decision）/ Gate-B 复杂度（decision）/ Gate-C E2E 通行证（acceptance）+ Per-Increment E2E
  - **测试设计**：用例设计 → 评审（review）
  - **测试执行**：E2E 跑测 → 验收（acceptance）
- 一句话：**这四条 flow 的所有 gate，都装进 R3 三类抽象** —— 抽象成立的证据。
- 讲法：这是「show 我们内置的这版 SOP」核心页。管理受众看到具体业务流程，技术受众看到三类 gate 各有落点。

### P8 · 一条 flow 端到端走通（coding-flow walkthrough）
- 横向流转图：action 节点产出 → gate 判 →
  - PASS → 顺进下一节点
  - decision（Gate-A 关联度高/低）→ 分支（单分支 / worktree）
  - acceptance（Gate-C E2E）FAIL → 回流 fix 节点（n7-tdd）
  - 全过 → status=completed
- **history 流水留痕** strip：`g-c-e2e iter1 FAIL → 回流 n7-tdd → iter2 PASS`（看见把关 + 留痕）
- 讲法：管理受众这页最有体感 —— 流程、把关、留痕一眼可见，对应「质量可控、过程可审计」。

### P9 · 任意 SOP 可编排：三层覆盖
- 三层表：
  | 层 | 谁编排 | 位置 | 改什么 |
  |---|---|---|---|
  | **L0 默认** | flow 作者（我们）| `node-table.yml` | 默认 SOP（节点表 + gates）|
  | **L1 用户编排** | 用户 | `orchestration.yml` | 开关 gate · decider ai↔human · retry 数 · 跳过/重排节点 |
  | **L2 自主编排**（未来）| 数字员工自己 | 同上（employee 写）| 基于经验提议 SOP 优化 |
- **有效 SOP = merge(L0 默认, L1 用户, [L2 自主])**；配套**编排指南文档**（能改什么 / 怎么改 / 关了的风险）
- 正面回应主张：**同一套底座，能跑任意业务 SOP**。
- 讲法：回扣 P4 的「可编排」承诺。L1 是当下可落地、可演示的（用户改 yml 就换 SOP）。

### P10 · 未来展望：数字员工自演进 SOP
- 复利飞轮**升级**：原飞轮（干活 → 沉淀知识库 → 反哺能力 → 越用越强）**延伸一条** → 不止反哺「能力」，也**反哺「SOP 本身」**
- 自演进路径：员工跑多了 → 发现某 gate 总是白过 / 某节点顺序更优 → **提议改 SOP**
- 守护栏：`orchestration.yml` 每项留 `source: default|user|self` 标记，**self 变更需人确认**（人有最终决定权）
- 挂自主度阶梯：L1–L2 人配（默认 + 用户编排）→ L3–L4 数字员工自主编排/优化（自演进）
- 讲法：从「人编排 SOP」到「员工自优化 SOP」，落在 L4 自治愿景但强调当前 L2 不实施、只预留挂钩。

### P11 · 收尾
- closing big：**从人编排 SOP → 员工自优化 SOP**
- 副线：底座（怎么跑）已落地 · SOP（跑什么）可编排 · 自演进（自己优化怎么跑）在路上
