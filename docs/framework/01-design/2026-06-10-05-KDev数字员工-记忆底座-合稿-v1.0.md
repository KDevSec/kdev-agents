## 0. 一句话

"记忆"是**从机器到人的三层数据**（控制态 → 事实流水 → 叙事），但**"记忆"这个概念 ≠ kdev-memory 这个 plugin**：下两层（控制态 / 事实流水）是 **kdev-core** 写的机器账，**kdev-memory 只管最上面的叙事层**——把事实流水**蒸馏 + 加料**（自评 / 用户评分 / 经验）成人能读、带质量信号的工程记忆。kdev-memory 是**全团队共享底座**（CEO 续航 + 评审经验喂飞轮 + 各员工 rollup），不是 CEO 私有。

## 1. 背景：为什么不能直接套现有 kdev-memory

- **kdev-memory 是"单流"设计**：单 `执行日志.md` + 单 `main-N` counter + 单 `当前状态.md`，为"一个用户 + 一个 Claude"线性工作设计。
- **数字员工是多 agent 并行**：需求架构师写 SR 时评审专家在 review，每个员工要记**自己的** Step 流，一条合并日志塞不下多员工并行。
- **全用 kdev-memory 重 markdown → token 爆**：每条 Step 是 4 段（执行事实 / 模型自评 / 用户评分 / 评分差异）~500-600 token，是给人读 + 蒸馏设计的；agent 知道"做了啥"不需要这些。`N 员工 × M 动作 × 600` 不可承受。

**结论**：既不收紧 kdev-memory（多用户协作仍要它），也不照搬——**扩 scope（多流）+ 分层（轻操作 / 重叙事）**。

```
kdev-team ──硬依赖──▶ kdev-memory   （拿叙事/续航底料 + 决策/踩坑共享）
```

## 3. 三层数据模型（从机器到人，跨两 plugin）

> 三层是"数据从机器到人"的主线；**下两层归 kdev-core，最上层才是 kdev-memory**。别把整条主线挂在 kdev-memory 名下。

| 层 | 形态 | 谁写 | 谁读 | 落位 |
|---|---|---|---|---|
| **控制态** | JSON | **kdev-core** | 引擎续跑 + HUD | `features/<slug>/flow-state.json`（Q-012）|
| **事实流水** | **JSONL**（events）| **kdev-core** | HUD + 记忆 rollup + CQO 审计 | `features/<slug>/events.jsonl`（行内带 `actor`，per-feature）|
| **叙事/蒸馏（= 记忆）** | **markdown**（4 段 Step / 决策 / 踩坑）| **kdev-memory** | 人续航 + 训练数据 + 飞轮 | kdev-memory（scope-aware） |

> **配置（非记忆）**：CEO 顶层约束（质量门槛 / 评审编排 / 合规基线）在 `flow-config.yml`，不进任何层，别混。

**关键（加料层原则）**：markdown Step **不每动作每员工写一次**，而是**从事实流水（events）周期性 rollup 派生 + 加料**——「执行事实」从流水蒸馏，「自评 / 用户评分 / 差异」是记忆**独有的加料**（机器流水里没有）。agent 全程只碰便宜的 events；重 markdown 一阶段聚合一次。**凡是镜像机器原始态（进度 / 原始事实）的，让位给底座 / HUD，不在记忆里造第二本账。**

> **events 按 feature 分、不按员工分（对齐 Q-012）**：物理落 `features/<slug>/events.jsonl`，每行带 `actor`。"看某员工在忙啥 / 给某员工 rollup" = 跨 features 按 `actor` 过滤；"看某功能全程" = 读该 feature 一个文件。一份流水，两种视图都靠过滤，不重复存。

## 4. scope 拓扑（统一多用户 / 多员工 / 多分支）

多用户、多员工、多分支本质都是 **"独立工作流 + 共享层"** —— 用统一的 scope 抽象：

**单装 kdev-memory（默认 1 scope，= 现状扁平态，UX 不变）**：
```
.kdev/memory/                ← scope=default（扁平）
├── 执行日志.md · 决策日志.md · 踩坑日志.md · skill-feedback.md · 当前状态.md
├── state/ · checkpoints/ · dataset/
```

**装了 kdev-team（多 scope）—— 完整目录（含 kdev-core 与 kdev-memory 两摊）**：
```
.kdev/                                   ← 独立记忆仓 nested repo（kdev-sync.yml 自举）
├── memory/                              ← kdev-memory 管（叙事/蒸馏层）
│   ├── shared/                          ← 项目级 scope（= 单装扁平内容归此）
│   │   ├── 决策日志.md · 踩坑日志.md · skill-feedback.md · 改进建议.md
│   │   ├── 当前状态.md（CEO 续航 + 判断/TODO；进度让位 HUD）
│   │   ├── 执行日志.md（项目时间线 rollup）· CQO 合规基线
│   │  【⚠️ P-C2/Q-019 修订·稀疏化】只为有「非机器记录」的 scope 而建（原"全家桶"已收窄，见 P-C2 §8.2）
│   ├── staff/reviewer/                  ← 评审经验（subject:review:<能力>, F-NNN）→ 喂飞轮
│   ├── staff/cqo/                       ← 审计报告
│   │     （✘ 无 dev/req/ceo —— dev/req 机器侧无 kdev-memory 足迹、ceo = shared；
│   │       员工"做了啥"在 events、"产出啥"在 handoffs，无 per-员工叙事 rollup）
│   └── state/ · checkpoints/ · dataset/ · config.yaml  ← 插件 plumbing，留 root（本机/不托管）
└── features/<slug>/                     ← kdev-core 写（Q-012 feature-first，一个功能一目录）
    ├── flow-state.json                  ← 台账 + active 控制态（我在哪步）
    ├── events.jsonl                     ← 事实流水（做了啥，行内带 actor）
    └── handoffs/<员工>/                 ← 产物总线（交付物，员工间接力）

（hud.html 由 kdev-hud 运行时派生 → 只读、本机不托管，不在记忆仓）
```

> **扁平 → scoped 迁移**：当前 dogfood 仓是**扁平态**（`执行日志.md` 等直接在 `.kdev/memory/` 下，= 默认/单装 scope）。装 kdev-team（scoped 化）时：markdown 内容文件归入 `memory/shared/`，再长出**稀疏** `memory/staff/{reviewer,cqo}/` 子 scope（⚠️ P-C2/Q-019：非每员工一目录），并出现 `.kdev/features/<slug>/`；`state/checkpoints/dataset/config` 仍留 root（机器本地）。**§5.3 master 表里残留的 `/staff/<员工>/` 写法见该表就地修订注。**

- **scope = 工作流身份**：协作者 / 员工 / 分支，统一处理；外加一个 `shared/` 共享层。
- **counter 泛化**：kdev-memory 已有 per-分支 counter（`Step main-N` / `Step <branch-slug>-N`，worktree 并发）——把 scope 从"分支"推广到"员工"是同台机器的自然延伸（`Step 开发工程师-N`），非另造。（⚠️ 本条已双重作废：**Q-020** 改时间戳原语 `Step <ts>-<who>`、slug/counter 退役（P-C2 §11）；**Q-019** 砍 per-员工叙事 Step（员工无 Step，不存在 `Step 开发工程师-N`，P-C2 §0.5））
- **借** CrewAI MemoryScope（路径前缀隔离 + shared 合并），落在**引擎级**（不止评审）。

## 5. 各员工记忆归属

| 员工 | 特性 | 会话/叙事 | 操作 | 其它 |
|---|---|---|---|---|
| **CEO（人机接口）** | 低频、续航 critical | **kdev-memory 叙事层 · shared scope**（续航/pending/checkpoint/brief）| 派单事件 JSONL（ceo scope）| 顶层约束 → flow-config |
| 需求架构师 / 开发 / 测试 | 高频、机器 | per-员工 rollup（按需）| **JSONL 操作 per-员工 scope** | 任务态 → flow-state |
| 评审专家 | 高频 + 经验飞轮 | 同上 | JSONL 操作 | **评审经验 = 评审专家的员工记忆**：`/staff/评审专家/` scope + subject:review:<能力> 子路由 + F-NNN 通道 |
| CQO | 监督 | 合规基线（shared）| 读 `features/*/events.jsonl`（跨 feature 审计）| — |

### 5.1 CEO 会话记忆 = kdev-memory 的老本行

CEO 是唯一跟人对话的员工（低频、续航 critical），它的"会话记忆"**不是存整段 transcript**（那是 harness/压缩管的），是 **distilled 续航**，正好由 kdev-memory 现成机制承载：

- **当前状态.md** frontmatter（`phase` / `pending_decisions` / `current_step`）= "到哪了 / 有啥待拍板"
- **决策日志 Q/G** = 对话里跟用户拍过的板
- **checkpoints/压缩前-\*.md**（PreCompact hook）= 上下文压缩前会话快照
- **SessionStart `<brief>` 注入** = CEO 醒来召回"昨天聊到哪"

CEO 代表项目跟人对接 → 它的会话续航是**项目级 / shared scope**（不是私有 CEO silo），CEO 是 shared 叙事的管家。

### 5.2 评审经验 = 评审专家的员工记忆（`staff/reviewer/`）

> ⚠️ **P-C2/Q-019 修订**：scope 收窄后 staff 目录稀疏化，但**本节的实质设计存活**——`staff/reviewer/` 正是收窄后**仅存的两个 staff scope 之一**（另一为 `staff/cqo/`），因为评审经验是机器流水**给不了的非机器记录**（飞轮校准），属"加料层"。下文 canonical id 用 **`reviewer`**（非中文 `评审专家`；中文 display 在 kdev-team staff.yml）；它**不是**「全家桶里的一员」，dev/req 等员工已无 per-员工叙事 scope。

评审经验（评得对/松/紧、评分调整建议）是**评审专家这个员工自己的记忆**——下次评审要 recall 自己的历史校准。所以它落在 **评审专家的 staff scope `staff/reviewer/`**，不是飘在扁平通道：

- **scope**：`staff/reviewer/`（评审专家私有 + recall 时自动并 `/shared` 通用基线）
- **子路由**：`subject:review:<能力>`（区分是哪个评审能力的经验）
- **通道**：F-NNN（不占 kdev-memory 的 R-NNN「改进建议」编号）
- **蒸馏**：`dataset-review-by-capability`（复用 kdev-memory 蒸馏管道）

其它员工拿的是评审**结果**（走 handoff），不是评审的校准记忆——校准记忆只在评审专家 scope 内。CQO 元抽查时 `recall(scope="staff/reviewer", subject:review:<能力>)` 读取。

> **`state/`（#14）≠ 任务态（#2）**：`state/` 是 kdev-memory hook 的内部记账（计数/去重/提醒），非语义、本机不托管；flow-state（#2）是 kdev-core 编排的语义任务态（到哪了/下一步），托管以便接力续作。两者层级与 git 待遇相反，不可合并。

### 5.4 TODO / 工作计划持久化（防压缩丢失）

主会话（CEO）的 TodoWrite 列表 = **CEO 级工作计划**（区别于 per-flow 的 flow-state），现状只活在上下文里、**压缩会丢**。

- **落点**：kdev-memory 叙事层 **shared scope**（随 CEO 会话续航，写入 `当前状态.md` 的工作计划段或专门 todos 快照）。
- **触发**：TodoWrite 变更时 / PreCompact hook → 落盘；**SessionStart 随 brief 恢复**。
- **效果**：上下文压缩、换会话、换机器都不丢"要做啥/做到哪"。

## 6. token 优化：两层表示 + 派生 rollup

> ⚠️ **P-C2/Q-019 修订**：本节「N 员工 × M 动作 × ~600 token 爆炸」作为**主驱动力已作废**——收窄模型下员工压根不写重 Step（员工 = events + handoffs），爆炸**被结构消除**而非「拆分缓解」。P-C2 价值收窄到 **recall 效率 + CEO 叙事覆盖委派工作 + 记录 ID 硬化**（见 P-C2 §0.5 / §1 / §12）。下表「优化」列的两层表示仍成立，但事实流水是**只读复用 kdev-core 既有 `features/<slug>/events.jsonl`**（按 actor 过滤出员工视图），非 kdev-memory 新写 per-scope JSONL。

| | 不优化（全 markdown）| 优化（JSONL 操作 + 派生）|
|---|---|---|
| 写 | N 员工 × M 动作 × ~600 token | events 由 kdev-core 既有流水承载（~30/行，零新写）+ 仅 CEO 几次 rollup |
| agent 召回 | 读整本 markdown | 跨 `features/*/events.jsonl` 按 actor + node 过滤捞切片 |

- **agent 召回** = `recall(scope=<actor>, node="D2")` 跨 `features/*/events.jsonl` 按 `actor` 过滤捞相关行（flat/off-flow 无 events → 退 transcript，P-C1b），不碰重 markdown。
- **markdown rollup** = 阶段/每日/按需，**仅 CEO/shared 一根**，从 events + handoffs + transcript 派生，一次性付费。

## 7. 不重复 / 归属边界（防双记浪费）

- **每事件只在 kdev-core 的 `features/<slug>/events.jsonl` 记一次**（⚠️ P-C2/Q-019：非"per-scope JSONL"；行内 `actor`、员工视图靠过滤、不另存 per-scope 物理文件，对齐 §3 与 §0.5 v0.2 收口 #3）；markdown 是派生（**仅 CEO/shared**），不双写。
- **评审反馈路由**（互斥 subject 空间）：
  - 对**评审能力**的反馈 → 评审专家员工记忆 `/staff/评审专家/` scope，`subject:review:<能力>`（F-NNN）
  - 对**外部 skill/工具**的反馈 → kdev-memory，`subject:plugin:X`
- **决策 / 踩坑**：项目级 → `shared/`；某员工特有 → 其 scope。

## 9. git 托管方案（已定 · Q-009）

记忆与**代码分支彻底解耦**，走**独立记忆仓（nested repo）+ 自举 hook**。

### 9.1 三件套

| 件 | 内容 |
|---|---|
| **代码仓**（KDevSec/kdev-agents）| `.gitignore` 保持 `/.kdev/`（彻底无视，不 track 不钉指针）+ tracked 的 `kdev-sync.yml`（只记"记忆仓地址"，随 clone 到任何机器，静态不刷历史）|
| **记忆仓**（独立 nested repo，自己 remote）| `.kdev/` 内容；per-scope 文件 + Step-ID 前缀（`<branch/scope>-N`）防并发撞 |
| **自举 hook**（kdev-memory ship）| SessionStart：无 `.kdev/.git` → 从 `kdev-sync.yml` clone；有 → `pull`。SessionEnd/rollup → commit + push |

### 9.2 为什么 nested repo（而非 submodule / track-in-main / 单独分支）

| 方案 | 否决理由 |
|---|---|
| 单独 kdev-sync 分支 | 切到 main 记忆不在场 + 切换复杂 |
| track 进 main | 记忆变代码分支内容 → 切分支/worktree 割裂、未提交挡 checkout、刷历史 |
| submodule | 父仓钉 per-commit 指针 → 切分支记忆跟着变 + 拿旧 SHA + 刷 main 历史 |
| **nested repo（取）** | 父仓彻底忽略 → 切分支/worktree 不碰记忆；自举 hook 给 submodule 的"自动带下来"便利，却**永远 pull 最新**、零指针污染 |

### 9.3 跨机 / worktree

- **新机器自举**：`git clone 代码仓` 带来 `kdev-sync.yml` → 首跑 SessionStart hook 读地址自动 clone 记忆仓 → 零手动。
- **多 worktree**：symlink 指向同一 canonical `.kdev/` → 记忆一份不割裂；Step-ID 前缀防并发编号撞。

### 9.4 🔴 托管提醒（kdev-memory 强制行为）

- **触发**：① 建立工程记忆时 ② 检测到 `.kdev/` 未托管 git（无 `.kdev/.git` 且 `kdev-sync.yml` 无远端）。
- **动作**：**中文提醒用户，强烈建议设为 nested repo 独立同步**（否则跨机 / 团队 / 换机将丢记忆）。
- **话术（中文示意）**："检测到工程记忆尚未 git 托管。强烈建议建立独立记忆仓（nested repo）并在 `kdev-sync.yml` 登记，以便跨机器 / 团队同步、换机不丢记忆。是否现在初始化？"
- **强烈建议、非强制**：用户可拒（仅本地，自负丢失风险）。

### 9.5 隐私待确认

记忆含用户 verbatim 原话 → 推到**团队共享**记忆仓 = 原话进团队仓（个人仓无碍）。用户留意；必要时团队仓做脱敏或限私有。

## 10.5 记录归因字段（phase）—— 并入元数据增强草案

> 来自痛点："拿到一批踩坑记录，不知道发生在哪个过程、用了哪些 skill。" 解法：给记录加**归因字段**，跟 `subject`（评谁）正交——`subject` 答"在评谁"，phase 答"发生在什么场景"。

| 字段 | 答什么 | 强制 | 例 |
|---|---|---|---|
| **phase**（coarse）| 工作性质（跨项目通用 10 值）| ✅ 必填 | `开发` / `测试` / `评审` |
| **phase_node**（fine）| 在 flow 的哪个节点 | optional | `n4-plan-review` / `n9a-code-review` |
| **phase_detail**（自由文本）| 自然语言兜底 | optional | `测试用例开发期间` |
| **active_skills**（list）| 当时在用的 skill / 方法论 | optional | `[TDD, playwright]` |

**phase 10 值（锁定）**：需求 / 设计 / 开发 / 测试 / 调试 / 评审 / 安全扫描 / 部署 / 文档 / 其他。

- **现状（2026-06-10 对照真实代码）**：`phase` 概念**已部分落地**（`hooks/lib/frontmatter.py` / `六类记录-schema.md` / `session-start-brief.py` 已引用，Step 切分按 phase 自然停顿）；完整的 **phase_node / phase_detail / active_skills 字段仍待实施**。
- ⚠️ **phase_node 词表必须对齐新节点模型**：并稿的旧草案用的是旧评审节点（上线就绪度 / AR / 无迭代拆分），**已过时**；实施时按 [编排底座 合稿](./2026-06-10-03-KDev数字员工-编排底座-合稿-v1.0.md) 的 node-table + [概念与能力 合稿](./2026-06-10-02-KDev数字员工-概念模型与员工能力-合稿-v1.0.md) 的新节点（迭代拆分 / 用户故事 / 部署检查）生成。
- **单角色逃生口**：phase 不该成负担——可裁词表 / 钉默认值跳过推断 / 整段关掉（与评分 opt-in 同思路，见 [评分污染治理 spec](../superpowers/specs/2026-06-09-kdev-memory会话污染治理+评分可配-design.md)）。
- 完整候选词表、决策序列见归档草案：[记录元数据增强 工作草案](./_archive/2026-05-30-01-kdev-memory-记录元数据增强-工作草案.md)。
