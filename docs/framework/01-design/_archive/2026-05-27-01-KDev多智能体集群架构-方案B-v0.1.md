## 0. 一句话目标

把现有「主控 + skill 原子能力库」**两层**结构，升级为「主智能体 + 4 个组（每组 TL + 工作 agent）」**三层**多智能体集群，用 Claude Code 原生 `Agent` 工具（无 experimental 依赖）实现，让用户**只跟主控对话**、**能查到子智能体进度**、**长任务不阻塞**、**跨 session 可续航**。

---

## 1. 架构总览

```
用户（VSCode Claude Code 扩展 / CLI 终端）
  ↕
主智能体（当前 session，跟用户对话）
  │
  │ Agent 工具派单（run_in_background: true，独立 context）
  ▼
┌─────────────┬─────────────┬─────────────┬─────────────┐
│  reqs-tl    │   dev-tl    │  test-tl    │ review-tl   │
│  需求组 TL  │   开发组 TL │ 测试组 TL   │ 审查组 TL   │
└──────┬──────┴──────┬──────┴──────┬──────┴──────┬──────┘
       │             │             │             │
       │ TL 在自己 context 里：
       │ 1. invoke 现有 kdev-* skill
       │ 2. 用 Agent 工具派工作 subagent
       │ 3. 写产物到 .kdev/handoffs/<group>/
       │ 4. 写状态到 .kdev/state.md::<group>
       │ 5. 写事件到 .kdev/events.log
       ▼
   工作 subagent（implementer / spec-reviewer / quality-reviewer / ...）

跨组通信：通过 .kdev/handoffs/ 文件总线（不依赖 mailbox）
状态主权：.kdev/state.md（主控权威）
进度可见：CLI statusLine + VSCode markdown preview + 对话流 /kdev:hud
跨 session 续航：state.md + handoffs/ 完全持久化，重启 session 主控读 state.md 就能续
```

## 2. 4 组 + 6 个现有 plugin 角色映射

| 组 | TL skill/plugin | 工作 agent / 下属 skill |
|---|---|---|

## 3. 共同基础

### 3.1 AR 编号统一规范

强制对齐 `kdev-test-points-v1` 的最严格格式：

```
AR-{DOMAIN}-{MAJOR}.{MINOR}.{PATCH}    例：AR-AUTH-01.001.001
```

- 所有 4 组、6 个 plugin、所有 handoff 必须按此格式
- reqs-TL 出口的 `ar.csv` 是格式权威定义点
- design-flow 内部如有不一致，加 adapter 转换

### 3.2 `.kdev/handoffs/` 跨组数据总线

```
.kdev/
├── state.md                          # 主控状态 + 4 组 section
├── events.log                        # 事件流（所有 TL 高频 append）
├── handoffs/
│   ├── reqs/
│   │   ├── ir.md
│   │   ├── sr.md
│   │   ├── ar.csv                    # ★ AR 编号权威
│   │   ├── prototype/
│   │   ├── design.md
│   │   └── COMPLETE.md               # 完成 signal
│   ├── dev/
│   │   ├── implementation-log.md
│   │   ├── commits.json
│   │   └── COMPLETE.md
│   ├── test/
│   │   ├── test-points.md
│   │   ├── test-cases.md
│   │   ├── ui-results.json
│   │   ├── api-results.json
│   │   └── COMPLETE.md
│   └── review/
│       ├── ceo.md / eng.md / design.md / security.md
│       └── COMPLETE.md
└── memory/                            # kdev-memory 现存（不变）
```

**规则**：plugin 内部状态（design-flow 的 `flow-state.json`、coding-flow 的 `sop-execution-log.md`）**降级为内部细节**；**完成态产物必须 copy 到 handoffs/**。

### 3.3 state.md schema

```markdown
# KDev State

feature: <slug>
feature_started_at: <ISO>
current_active_group: reqs | dev | test | review | idle

## reqs
status: complete | in_progress | blocked | pending
current_step: ir | sr | ar | prototype | plan
started_at: <ISO>
completed_at: <ISO or null>
last_progress: <一句话，≤80 字>

## dev
（同上结构）

## test
（同上结构）

## review
（同上结构）
```

### 3.4 events.log 格式

```
<ISO-ts> <group> <event-type> <msg>

例：
2026-05-27T16:34:12Z reqs-tl step_complete IR 完成，输出 .kdev/handoffs/reqs/ir.md
2026-05-27T16:38:45Z reqs-tl step_start  开始 SR 阶段（kdev-design-flow stage 1）
2026-05-27T16:55:02Z reqs-tl gate_pass   Gate 1 评审通过（评审模式 both）
2026-05-27T17:12:30Z reqs-tl complete    全部 5 step 完成，handoff/reqs/COMPLETE.md 已落
```

## 4. 4 个 TL Agent Definition 设计

### 4.1 reqs-tl

**位置**：`plugins/kdev-design-flow/agents/reqs-tl.md`
**model**：`opus`（需求理解判断重）
**核心职责**：编排 IR → SR → AR → Prototype → Plan 全流程

```markdown
---
name: reqs-tl
description: 需求组 TL。收到原始需求后编排 IR→SR→AR→Prototype→Plan 全流程，
  产出落 .kdev/handoffs/reqs/，回传 COMPLETE 信号给主控。
tools: Skill, Read, Write, Edit, Bash, Glob, Grep, Agent
model: opus
---

# 角色
KDev 需求组 TL。主控通过 Agent 工具派单给你，独立 context 跑完需求全流程。

# 工作流（5 step）
1. **init**：读 .kdev/state.md，初始化 reqs section（status=in_progress）
2. **IR**：Skill("superpowers:brainstorming") → .kdev/handoffs/reqs/ir.md
3. **设计流水线**：Skill("kdev-design-flow") with args <slug> --review=both
   → 跑完 SR / AR / Prototype / Plan + 3 Gate
4. **handoff 同步**：copy + rename 到 .kdev/handoffs/reqs/
   - 强制校验 AR 编号格式（AR-{DOMAIN}-XX.YYY.ZZZ）
5. **finalize**：写 COMPLETE.md，更新 state.md，返回 ≤200 字结构化摘要

# 进度可观测硬规
🔴 每完成一个 step 立刻 append state.md + events.log
🔴 last_progress 字段 ≤80 字
🔴 Gate 失败 3 次 → status=blocked，返回主控求决策
```

### 4.2 dev-tl

**位置**：`plugins/kdev-coding-flow/agents/dev-tl.md`
**model**：`sonnet`（执行编排为主；复杂时升 opus）
**核心职责**：基于 reqs/COMPLETE 启动 13 节点 SOP

```markdown
---
name: dev-tl
description: 开发组 TL。从 .kdev/handoffs/reqs/ 读 AR/spec/plan，
  跑 kdev-coding-flow 的 13 节点 SOP，产出落 .kdev/handoffs/dev/。
tools: Skill, Read, Write, Edit, Bash, Glob, Grep, Agent
model: sonnet
---

# 工作流
1. init state.md::dev
2. 读 .kdev/handoffs/reqs/{ar.csv, design.md, prototype/}
3. Skill("kdev-coding-flow")
   - 节点 0 背景对齐
   - 节点 1-5 规划 Gate-A/B
   - 节点 6a/6b 执行（简单直接 / 复杂派 superpowers:subagent-driven-development）
   - 节点 7-10 TDD + 验证 + Gate-C + 安全扫描
   - 节点 11-13 合并 + 部署 + 收尾
4. 同步 implementation-log.md + commits.json + COMPLETE.md
5. 更新 state.md，返回摘要

# 异常升档
- 复杂跨模块改动 / 鉴权 / DB schema → 该 implementer 升 opus
- BLOCKED 一次 → 升 opus 重派，不在 sonnet 反复重试
```

### 4.3 test-tl

**位置**：`plugins/kdev-test-flow/agents/test-tl.md`（新 plugin）
**model**：`sonnet`
**核心职责**：编排测试链路 design → render → 分流（UI / API）→ 报告

```markdown
---
name: test-tl
description: 测试组 TL。从 .kdev/handoffs/dev/ 读已编码功能，
  跑测试链路（设计 → 渲染 → UI/API 双分流 → 报告）。
tools: Skill, Read, Write, Edit, Bash, Glob, Grep, Agent
model: sonnet
---

# 工作流
1. init state.md::test
2. 读 .kdev/handoffs/dev/COMPLETE.md + reqs/ar.csv
3. Skill("kdev-test-points-v1") → 测试点 .md
4. Skill("kdev-test-cases-v1") → fielded 测试用例 .md
5. **UI 分流**（并行 A）：Skill("kdev-ui-autotest") → Playwright 脚本 + 执行
6. **API 分流**（并行 B）：
   - Skill("kdev-uicase-to-apicase") → API 用例 .md
   - ⚠️ Skill("kdev-api-test-scaffold")【缺，第一版可跳过仅跑 UI】
7. 合并 ui-results.json + api-results.json → COMPLETE.md
8. 更新 state.md，返回摘要
```

### 4.4 review-tl

**位置**：`plugins/kdev-review-flow/agents/review-tl.md`（新 plugin）
**model**：`opus`（评审决策重）
**核心职责**：编排多视角评审 + 决策聚合

```markdown
---
name: review-tl
description: 审查组 TL。从 .kdev/handoffs/test/ 读测试结果 + 所有上游产物，
  并行派 4 个评审 subagent（CEO / 架构 / 设计 / 安全），聚合决策。
tools: Skill, Read, Write, Edit, Bash, Glob, Grep, Agent
model: opus
---

# 工作流
1. init state.md::review
2. 读 .kdev/handoffs/{reqs, dev, test}/COMPLETE.md
3. **并行派 4 评审**（用 Agent 工具，run_in_background）：
   - Agent: ceo-reviewer → Skill("gstack:plan-ceo-review") → ceo.md
   - Agent: eng-reviewer → Skill("gstack:plan-eng-review") → eng.md
   - Agent: design-reviewer → Skill("gstack:plan-design-review") → design.md
   - Agent: security-reviewer → Skill("gstack-cso") + Skill("kdev-secure-coding") → security.md
4. 等 4 个回来 → 决策聚合（自有逻辑或调 final-merger subagent）
5. 写 final-decision.md + COMPLETE.md
6. 更新 state.md，返回摘要 + 关键决策
```

### 4.5 工作 subagent 清单

只有 review-tl 内部需要专用 subagent；其他组复用现有 skill。

**review 组需新写的 subagent**（plugins/kdev-review-flow/agents/）：
- `ceo-reviewer.md`
- `eng-reviewer.md`
- `design-reviewer.md`
- `security-reviewer.md`
- `final-merger.md`（决策聚合）

每个都是 5-10 行 prompt + 调对应 gstack skill 的包装层。

## 5. 主智能体（orchestrator）

新建 plugin：`plugins/kdev-orchestrator/`

### 5.1 `/kdev:start-feature <需求描述>`

主入口 skill。逻辑：

```
1. 解析需求，生成 feature_slug
2. 初始化 .kdev/state.md（4 组 section + feature 元信息）
3. Agent({
     subagent_type: "reqs-tl",
     prompt: "<需求描述>\nfeature_slug: <slug>",
     run_in_background: true
   })
4. 立即返回给用户："reqs-tl 已派出后台跑，预计 X 分钟。
   你可以问'进度'或继续聊别的。我会自动跟你报告。"
5. 监听 TL 回执（background 完成回调）
6. reqs 完成 → 询问用户是否继续派 dev-tl（或自动派）
7. 依次 dev → test → review，每个 TL 完成时主动汇报 + 渲染 HUD
```

### 5.2 `/kdev:hud`

状态查询 skill。输出 3 种模式：
- `markdown`（默认，给 VSCode 对话流）
- `box`（ASCII，给 CLI 终端）
- `watch`（CLI 全屏 1s 刷新）

数据源：`.kdev/state.md` + `.kdev/handoffs/<g>/COMPLETE.md` + `.kdev/events.log`（最近 5 条）

### 5.3 `kdev-statusline.sh`

shell 脚本，读 state.md 输出单行 HUD 摘要，给 Claude Code `statusLine` 字段调用。**仅 CLI 用户可见**（实测 VSCode 扩展不渲染）。

### 5.4 进度可见性（综合）

| 用户问的事 | 主控做什么 | UI 渲染 |
|---|---|---|
| "进度怎么样？" | 调 `/kdev:hud markdown` | VSCode 对话流渲染表格 |
| 长任务后台跑 | 不阻塞，主控继续陪聊 | run_in_background 的 subagent |
| TL 完成 | 主控收回调，主动汇报 + 渲染 HUD | 对话流自动出现新消息 |
| 用户改主意 | 主控 TaskStop 强杀 + 写新指令到 state.md | 用户继续对话 |

## 6. HUD 实时状态面板（含 2026-05-27 实测结论）

### 6.1 数据源

`.kdev/state.md` + `.kdev/handoffs/<g>/COMPLETE.md` + `.kdev/events.log`

### 6.2 CLI 用户主路径

| 优先级 | 形态 |
|---|---|
| ⭐ 主推 | Claude Code `statusLine`（输入框上方一行，**实测可用**）|
| 辅助 | `/kdev:hud --watch`（专屏 ASCII 1s 刷新）|

### 6.3 VSCode 扩展用户主路径（**降级方案**）

| 优先级 | 形态 |
|---|---|
| ⭐ 主推 A | VSCode markdown preview（`state.md` 写成 markdown 表格，VSCode 原生自动刷新）|

| 可选（W3+）| 专用 VSCode 扩展（StatusBar + TreeView + Webview，3-5d） |

### 6.4 用户群体推荐组合速查

| 用户类型 | 主推 | 辅助 |
|---|---|---|
| CLI / 终端 | statusLine | `/kdev:hud --watch` |
| **VSCode 扩展** | markdown preview + 对话流 `/kdev:hud` | TL 完成主控自动渲染 |
| 无 IDE / pipeline | `cat .kdev/state.md` | - |

### 6.5 工作量

- `/kdev:hud` skill（3 输出模式）：0.5d
- `kdev-statusline.sh`：0.25d（CLI 用）
- `events.log` 规范 + TL 高频写盘约束：0.25d
- `.claude/settings.json` 配 statusLine：0.1d
- VSCode markdown preview 友好的 state.md 排版打磨：0.1d
- **小计 ~1.2d**

## 7. 跨 Session 续航（B 天然支持）

### 7.1 续航原理

`.kdev/state.md` + `handoffs/` + `events.log` + `memory/` 全是**文件**，subagent 跑到一半 session 挂了：
- subagent 进程消失，但**文件落盘进度还在**
- 新 session 启动，主控读 state.md 就知道每组到哪个 step
- 可以从最后已知 step **重启对应 TL**（重新派 Agent）

### 7.2 续航流程

```
新 session 启动
  ↓
用户："继续上次的工作"
  ↓
主控调 kdev-memory + 读 state.md
  ↓
报告："上次到这里：
  - reqs ✅ 完成
  - dev 🟡 SOP 节点 6b（implementer 跑到一半就停了）
  要从 dev 节点 6b 续吗？"
  ↓
用户："好"
  ↓
主控 Agent({subagent_type: "dev-tl", prompt: "从 .kdev/handoffs/dev/ 续，从 SOP 节点 6b 起跑"})
```

### 7.3 续航容错

- TL 重启时必须**幂等**——已经完成的 step 不能重做（读 events.log `step_complete` 跳过）
- handoffs/ 里已有的产物**不能覆盖**——重启时检测到 `COMPLETE.md` 直接跳过该 TL
