## 0. 设计原则

| 原则 | 说明 |
|---|---|
| **Markdown 优先** | 持久化用 markdown（用户可读），结构化数据用 YAML frontmatter，事件流用 JSONL |
| **文件即真理** | 所有运行时状态都落在 `.kdev/` 文件，无内存状态需要恢复（SessionStart 直接读文件就够）|
| **接口契约最小化** | kdev-core 对外只暴露 4 类接口：`state.read/write` / `events.append/scan` / `hook.register/dispatch` / `review-memory.recall/write/distill` |
| **多 plugin 友好** | 多 plugin 同名 hook 注册按 order 字段排序顺序执行；events.log 写并发安全（append-only + 文件锁）|
| **范式借用 kdev-memory** | review-memory.md 沿用 markdown 主存 + 编号体系 + verbatim + subject 路由 |
| **不引入 kdev-memory 运行时依赖** | 所有借用都是 schema/文件格式层面，不调 kdev-memory plugin |
| **底座 = 编排引擎（[Q-007] 新增重心）** | 底座核心是 R1–R7 编排引擎（§0.1），尤其 **R2 节点状态机 + R4 自主模式**；下文 4 子系统是其管道支撑 |

### 0.1 底座职责 = R1–R7 编排引擎（[Q-007] 重心）

从现有 `design-flow`/`coding-flow` 反推，底座要统一提供：

| | 职责 | flows 现有实现 | 本文档对应 |
|---|---|---|---|
| R1 | 状态机持久化 + 断点续跑 | flow-state.json / `--resume` | §1 state.md / events.log |
| **R2** | **多节点状态机（编排引擎核心）** | 13 节点 SOP / 4 段 | ⚠️ 本 v0.1 缺，需补 |
| R3 | 评审闸门 + 重试 + 人介入档 | 3 闸门 × 3retry ×(ai/both/human) | §4（记录有、引擎缺）|
| **R4** | **自主模式（跑到 Gate 才停）** | coding-flow Auto Mode | ⚠️ 本 v0.1 缺，需补 |
| R5 | 模型分工派单 | CLAUDE.md 二档 | §6.1 subagent |
| R6 | 产物总线 + 阶段交接 | 三件套 / handoff | §4.1 handoffs |
| R7 | 调度已有能力插件 | `/skill` 调用 | §5 plugin 注册 |

> **R2/R4 是 v0.1 最该补的**；其余在现有 schema + flows 里基本有底。抽公共层时照 OMC 的 mode/Team Pipeline 范本，但**不 fork**。

## 1. 任务状态管理（`state.md` + `events.log`）

### 1.1 `.kdev/state.md` 完整 Schema

#### 1.1.1 文件结构

YAML frontmatter + markdown section：

```markdown
---
schema_version: 1
session_id: <UUID v4>
phase: reqs | dev | test | final | idle
current_node: <节点 ID，如 R5 / D2 / T1 / F2>
flow_config_path: .kdev/staff-memory/flow-config.yml
last_updated: <ISO 8601 UTC，如 2026-05-28T14:33:12Z>
created_at: <ISO 8601 UTC>
---

# CEO 权威状态

## 当前阶段
<markdown 自由文本，CEO 维护>

## 员工状态
<table，CEO 维护>

## warnings
<列表，CQO 写>
```

#### 1.1.2 frontmatter 字段定义

| 字段 | 类型 | 必填 | 说明 | 校验 |
|---|---|---|---|---|
| `schema_version` | int | ✅ | schema 版本号 | == 1 |
| `session_id` | string | ✅ | UUID v4 | 长度 36 / 含 4 个 `-` |
| `phase` | enum | ✅ | 阶段 | 5 选 1 |
| `current_node` | string | ⚠️ | 当前节点 ID | `^(R\|D\|T\|F)\d+(\.\d+)?$` 或 null |
| `flow_config_path` | string | ✅ | yml 配置路径 | 相对项目根 |
| `last_updated` | ISO 8601 | ✅ | 上次更新时间 | UTC + Z |
| `created_at` | ISO 8601 | ✅ | 创建时间 | UTC + Z |

#### 1.1.3 section 模板（CEO 维护，自由 markdown，参考样例）

```markdown
## 当前阶段
- 阶段: dev
- 当前节点: D2 (TDD 增量评审中)
- 重试次数: 1/3
- 启动: 2026-05-28T14:00:00Z

## 员工状态
| 员工 | 编排状态 | 最近能力 | 最近产出 |
|---|---|---|---|
| 需求架构师 | idle | 方案设计 (R5) | design.md ✅ |
| 开发工程师 | busy | TDD 实现 (D2) | src/+tests/ |
| 测试工程师 | idle | (待命) | - |
| 评审专家 | busy | 代码评审+质量评审 并行 (D2) | reviewing... |
| CQO | idle | (待命) | - |

## warnings（CQO 写）
- 🟡 [D2 代码评审] 1 次未通过，已重派；记录于 .kdev/staff-memory/reviews/D2-代码评审/评分-1.md
```

#### 1.1.4 读写接口

```python
# kdev_core/state.py

def read_state(state_path: Path = Path(".kdev/state.md")) -> StateDoc:
    """读 state.md，返回 frontmatter + section 字典。文件不存在时返回 None。"""

def write_state(state_path: Path, frontmatter: dict, sections: dict[str, str]) -> None:
    """覆盖写 state.md。自动更新 last_updated。
    校验 frontmatter 字段；不合法抛 SchemaError。
    并发保护：文件锁 fcntl.flock(LOCK_EX)。"""

def init_state(state_path: Path, session_id: str | None = None) -> StateDoc:
    """初始化 state.md（首次会话）。session_id 不给则生成 UUID v4。"""
```

### 1.2 `.kdev/events.log` 完整 Schema

#### 1.2.1 文件格式：JSONL（append-only）

每行一个 JSON 对象，UTF-8 编码，`\n` 分隔，不允许 `\r\n`。

#### 1.2.2 公共字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `ts` | ISO 8601 | ✅ | UTC 时间戳 |
| `actor` | string | ✅ | 触发者（如 `开发工程师/编排` / `评审专家/代码评审` / `CEO` / `CQO`）|
| `event` | enum | ✅ | 事件类型（见 §1.2.3）|
| `session_id` | string | ✅ | 与 state.md 一致 |
| 其他字段 | varies | 视 event 类型 | 见 §1.2.3 |

#### 1.2.3 事件类型枚举（v0.1，10 种）

| event | 触发者 | 关键字段 | 例子 |
|---|---|---|---|
| `capability_started` | 员工编排 | `capability` / `node` / `retry` | `{"event":"capability_started","capability":"TDD实现","node":"D2","retry":0}` |
| `capability_completed` | 员工编排 | `handoff` / `duration_seconds` | `{"event":"capability_completed","handoff":".kdev/handoffs/开发工程师/D2-iteration1/","duration_seconds":1316}` |
| `协作请求` | 员工编排 | `to` / `intent` / `subject`（list）| `{"event":"协作请求","to":"评审专家/编排","intent":"review","subject":["代码评审","质量评审"]}` |
| `review_scored` | 评审能力 | `node` / `capability` / `score`(0-100) / `blocking`(int) / `path` | `{"event":"review_scored","node":"D2","capability":"代码评审","score":84,"blocking":0,"path":".kdev/staff-memory/reviews/D2-代码评审/评分-1.md"}` |
| `review_aggregated` | 评审专家编排 | `node` / `passed` / `aggregated_score` / `to` | `{"event":"review_aggregated","node":"D2","passed":true,"aggregated_score":86,"to":"开发工程师/编排"}` |
| `blocked` | 业务能力（连 3 次失败）| `capability` / `reason` / `retry_count` | `{"event":"blocked","capability":"TDD实现","reason":"test suite hangs","retry_count":3}` |
| `review_retry_exhausted` | 评审专家编排 | `node` / `capability` / `final_score` / `remaining_problems` | `{"event":"review_retry_exhausted","node":"D2","capability":"代码评审","final_score":79,"remaining_problems":{"P0":0,"P1":2,"P2":3}}` |
| `escalated_to_ceo` | 员工编排 / 评审专家编排 | `reason` / `from_event` | `{"event":"escalated_to_ceo","reason":"review_retry_exhausted","from_event":"<ref to last event>"}` |
| `user_decision` | CEO（转用户拍板后）| `decision` (a/b/c) / `context` | `{"event":"user_decision","decision":"a","context":"R2 评审接受当前评分"}` |
| `cqo_audit` | CQO | `severity` (🔴/🟡/⚪) / `target` / `report_path` | `{"event":"cqo_audit","severity":"🟡","target":"R2 SR 需求评审","report_path":".kdev/handoffs/CQO/<报告>.md"}` |

#### 1.2.4 读写接口

```python
# kdev_core/events.py

def append_event(events_path: Path, event: dict) -> None:
    """append-only 写入。自动加 ts / session_id。
    并发保护：文件锁 fcntl.flock(LOCK_EX)。
    校验 event 字段；不合法抛 SchemaError。"""

def scan_events(
    events_path: Path,
    event_filter: list[str] | None = None,
    after_ts: str | None = None,
    actor_filter: str | None = None,
    limit: int = 1000,
) -> Iterator[dict]:
    """流式扫描 events.log，支持事件类型 / 时间 / actor 过滤。"""

def tail_events(events_path: Path, n: int = 100) -> list[dict]:
    """读最后 n 条事件（高频用于 CQO 抽查 / hook trigger）。"""
```

### 1.3 并发与一致性

| 场景 | 处理 |
|---|---|
| 多 agent 同时写 state.md | 文件锁（`fcntl.flock(LOCK_EX)`）+ 读-修改-写原子化 |
| 多 agent 同时 append events.log | append-only + 文件锁；单条 write 原子（`O_APPEND`）|
| 跨会话恢复 | SessionStart hook 读 state.md frontmatter 还原 phase/current_node；events.log 全留痕 |
| 文件损坏 | state.md schema 校验失败 → CEO 提示用户手工修复；events.log 单行损坏 → 跳过该行继续 |

## 2. HUD 服务

### 2.1 三个通道

| 通道 | 实现路径 | 触发频率 |
|---|---|---|
| **CLI statusLine** | Claude Code 原生 statusLine 接入（shell command 返回 markdown）| 每个用户提交 |
| **VSCode markdown preview** | 维护 `.kdev/hud.md` 文件（Claude Code 自动 watch）| 每个 capability_completed / cqo_audit 等关键事件 |
| **对话流** | CEO 在 chat 中直接输出 markdown 摘要 | 阶段聚合 / 评审升级 / 用户问询时 |

### 2.2 statusLine 接入

#### 2.2.1 用户的 settings.json 配置

```json
{
  "statusLine": {
    "type": "command",
    "command": "kdev-core hud statusline"
  }
}
```

#### 2.2.2 `kdev-core hud statusline` 输出格式

stdout 单行 markdown（最多 80 字符）：

```
🏢 KDev | dev (D2 ✏️) | 开发 busy / 评审 busy | warn:1
```

字段约定：
- 项目类型 icon（🏢 工具 / 🌐 web / 📱 mobile / ⚙️ api）
- 当前阶段 `(节点 icon)`：✏️ 进行中 / ✅ 完成 / ⏳ 待 / 🟡 告警 / 🔴 阻断
- 员工状态摘要（按 busy/idle 计数）
- warn / 错误条数

## 3. hook framework

### 3.1 hook 类型完整清单（v0.1 = 7 种）

| Hook 名 | 触发条件 | 调用契约 | 失败语义 |
|---|---|---|---|
| `SessionStart` | Claude Code session 启动 | 同步执行；可写 stdout 给 Claude（注入 brief / context）| 失败不阻断会话，记 stderr |
| `PreCompact` | 上下文压缩前 | 同步执行；可写 `.kdev/checkpoints/压缩前-<ts>.md` | 失败不阻断压缩 |
| `UserPromptSubmit` | 用户每次发言 | 同步执行；可写 stdout 给 Claude（注入 recall）| 失败不阻断对话 |
| `Blocked` | events.log 写入 event=`blocked` | 异步触发；handler 自由调度 agent | 失败重试 1 次 |
| `ReviewRetryExhausted` | events.log 写入 event=`review_retry_exhausted` | 异步触发；handler 应触发 CEO 跟用户对话 | 失败重试 1 次 |
| `LowScoreAlert`（可选）| events.log 写入 event=`review_scored` 且 score ∈ [threshold, low_score_threshold) | 异步触发 | 失败不阻断 |
| `StageComplete` | events.log 一阶段所有节点都 review_aggregated passed=true | 异步触发；handler 应生成阶段摘要 + 派 CQO | 失败重试 1 次 |

### 3.2 hook 注册 Schema

#### 3.2.1 plugin 提供的 `hooks.json`

放在 plugin 根目录 `plugins/<plugin-name>/hooks.json`：

```json
{
  "schema_version": 1,
  "hooks": [
    {
      "name": "SessionStart",
      "handler": "hooks/session-start.sh",
      "order": 50,
      "timeout_seconds": 5,
      "metadata": {
        "purpose": "kdev-core: 读 state.md 还原 phase + 检查 flow-config.yml"
      }
    },
    {
      "name": "Blocked",
      "handler": "hooks/on-blocked.sh",
      "trigger_pattern": {
        "events_log_event": "blocked"
      },
      "order": 50,
      "timeout_seconds": 30,
      "metadata": {
        "purpose": "kdev-core: 自动派员工编排应急"
      }
    }
  ]
}
```

#### 3.2.2 字段约定

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `name` | enum | ✅ | 7 种 hook 之一 |
| `handler` | string | ✅ | 相对 plugin 根的可执行路径 |
| `order` | int | ⚠️ | 默认 50；多 plugin 同名 hook 按升序执行（kdev-core=50 / kdev-memory=50 / 业务 plugin=70 等约定）|
| `trigger_pattern` | object | ⚠️ | 仅事件驱动 hook 必填（Blocked / ReviewRetryExhausted / LowScoreAlert / StageComplete）|
| `timeout_seconds` | int | ✅ | handler 超时时间 |
| `metadata.purpose` | string | ⚠️ | 人类可读的目的说明 |

#### 3.2.3 多 plugin 同名 hook 共存

例：`SessionStart` 同时被 kdev-memory + kdev-core 注册。

```
SessionStart 触发
  ↓ 按 order 升序
1. kdev-memory/hooks/session-start.sh (order=40, 注入 kdev-memory-brief)
2. kdev-core/hooks/session-start.sh    (order=50, 注入 kdev-core 当前阶段摘要)
  ↓
两个 handler 的 stdout 合并注入 Claude
```

**约定**：kdev-memory 范式优先（order=40），kdev-core 次之（order=50），业务 plugin（order=70）最后。

### 3.3 handler 接口规约

#### 3.3.1 输入（环境变量）

| 变量 | 说明 |
|---|---|
| `KDEV_PROJECT_ROOT` | 项目根（含 .kdev/ 的目录）|
| `KDEV_SESSION_ID` | 当前 session UUID |
| `KDEV_HOOK_NAME` | 当前 hook 名 |
| `KDEV_TRIGGER_EVENT` | 触发事件（仅事件驱动 hook，JSON 串）|

#### 3.3.2 输出（stdout / stderr / 返回码）

| 通道 | 用途 |
|---|---|
| stdout | （仅同步 hook：SessionStart / UserPromptSubmit / PreCompact）写入会注入 Claude 上下文 |
| stderr | 调试日志，写入 `.kdev/logs/hook-<name>-<ts>.log` |
| 返回码 0 | 成功 |
| 返回码 1 | 失败，按 `失败语义` 处理 |
| 返回码 2 | 跳过（不参与合并 stdout）|

#### 3.3.3 实施示例：on-blocked.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT="${KDEV_PROJECT_ROOT}"
```

| `provides.skills_dir` | string | ⚠️ | 提供的 skill 目录 |
| `provides.slash_commands` | list | ⚠️ | slash command 注册 |

### 5.3 kdev-core 启动加载流程

```
1. 扫描 plugins/<*>/plugin.json
2. 解析依赖关系（depends_on / kdev_core_required）
3. 拓扑排序（kdev-core 优先 / kdev-memory 独立 / kdev-staff 后）
4. 按序加载：
   - agents → 注册到 Claude Code subagent
   - standards → 加载为 prompt 资产（按需 prompt 拼接）
   - hooks → 合并到 hook framework registry
   - slash_commands → 注册到 Claude Code slash command
5. 启动 hook framework（监听 events.log + 触发 SessionStart 等同步 hook）
```

### 5.4 加载失败处理

- plugin.json schema 校验失败 → 跳过该 plugin，记 stderr，继续加载其他
- 依赖未满足（如 kdev-core 版本不够）→ 跳过该 plugin + 提示用户升级
- agents/standards 缺失 → 警告但允许 plugin 加载（部分功能）

## 6. 与 Claude Code 原生机制对接

### 6.1 Subagent 调度

kdev-core **不重新发明 subagent 调度**——直接用 Claude Code 原生的 `Agent({subagent_type: "..."})` 机制。

kdev-core 的角色：
- 启动时把 kdev-staff 的 39 个 agent 定义注册到 Claude Code
- 评分循环 / Blocked / StageComplete 等 hook 触发时，handler 直接调用 `Agent(...)` 派单

### 6.2 Skill / Slash Command 注册

kdev-staff 提供 slash command（`/kdev:hud` / `/kdev:status` / `/kdev:distill-review-memory` 等）通过 plugin.json 注册到 Claude Code。

kdev-core 不实现 skill 注册，仅汇总各 plugin 的 slash_commands 字段。

### 6.3 Claude Code 原生 Hook 包装

| 原生 Hook | kdev-core 包装 |
|---|---|
| Claude Code `SessionStart` | kdev-core hook framework 转发，按 order 分发到各 plugin handler |
| `PreCompact` | 同上 |
| `UserPromptSubmit` | 同上 |
| `SessionEnd` | kdev-core 用于落盘当前 state.md 快照（兜底）|

kdev-core 不发明新 Hook 类型，所有 hook 都映射到 Claude Code 原生 hook 或 events.log 事件驱动。
