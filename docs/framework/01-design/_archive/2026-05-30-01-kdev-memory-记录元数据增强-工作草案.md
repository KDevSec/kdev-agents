## 一、问题与定位

### 1.1 原始痛点

> "我拿到几个踩坑记录后，并不清楚这些踩坑是针对 skill 的，还是我在开发还是测试过程出现的；我想明确知道这些踩坑是在哪个过程中发生的，包括用了哪些 skill。"

### 1.2 关键区分：subject vs 情境元数据

讨论中识别出**两类语义不能混**：

| 字段 | 答什么问题 | 例 | 蒸馏用途 |
|---|---|---|---|
| `subject`（**已存在**，F-NNN 必填）| 这条评分/反馈**在评谁**？| `plugin:kdev-memory` / `tool:bash` | 按评价对象切训练集（dataset-skill-feedback-by-subject）|
| `phase` + `phase_node` + `phase_detail` + `active_skills`（**新增**）| 这条记录**发生在什么场景**？| `phase: 测试 / phase_node: T2 / phase_detail: 测试用例开发 / active_skills: [TDD, playwright]` | 复盘归因、按阶段聚类、跨记忆系统联合查询 |

强行用 subject 装 phase 会两边都污染（G-NNN 里的 bash 报错踩坑，subject 是 `tool:bash`？还是 `phase:开发`？）。

## 二、三字段 + active_skills 设计

### 2.1 字段总览

```yaml
## G-NNN: pytest fixture scope=module 导致状态污染
triggers: ["pytest", "fixture", "scope"]
日期：2026-05-30
phase: 测试                           # coarse, 必填（智能体推断）
phase_node: T2                        # fine, optional（项目 spec 节点）
phase_detail: 测试用例开发期间          # free-text, optional（自由文本兜底）
active_skills: [pytest, TDD]          # list, 当时在用的 skill / 方法论
status: open
```

### 2.2 三层 phase 的设计意图

| 层 | 粒度 | 词表来源 | 强制性 |
|---|---|---|---|
| `phase`（coarse）| 大颗粒、跨项目通用 | kdev-memory 内建 10 值 + 项目可裁剪 | ✅ 必填 |
| `phase_node`（fine）| 项目专属流程节点 | 项目 config.yaml 指定的 spec 文件（v1.5 §5.1）| optional |
| `phase_detail`（自由文本）| 自然语言描述 | 无词表 | optional |

**分层理由**：
- coarse 跨项目稳定（蒸馏可移植）
- fine 项目专属（与 v1.5 R/D/T/F 对齐，可由 spec 演进）
- detail 兜底（推断不出 phase_node 时仍可保留语义）

### 2.3 phase 词表（**锁定 10 值**）

```
需求 / 设计 / 开发 / 测试 / 调试 / 评审 / 安全扫描 / 部署 / 文档 / 其他
```

**已废弃**："元工作" 不单列为 phase。理由：在 kdev-agents 项目里，业务就是开发 skill / plugin / framework 本身——写 SKILL.md / 改 hook / 设计架构 doc 都属 `开发` / `设计` / `文档`，靠 `active_skills` 字段区分"在开发哪个工具"。`元工作` 仅在"项目有独立业务产品 + 元工具开发"双轨场景才有意义，本项目不属。

### 2.4 phase 内涵校准

| phase | 含义校准 |
|---|---|
| 需求 | 需求调研 / IR / SR / AR / 原型 |
| 设计 | 架构 / 详细 / 接口 / 数据模型 / UI/UX / 技术选型 / ADR |
| 开发 | 写 SKILL.md / 改 hook / 写 plugin / 写产品代码 / TDD / E2E / 联调 |
| 测试 | 测试点 / 测试用例 / 自动化 / 缺陷复现 / 覆盖度 |
| 调试 | 现象复现 / 日志 / 二分定位 / 根因 / 性能瓶颈 |
| 评审 | 代码 / 方案 / 安全 / 测试设计 / 终审等 17 类（详见 v1.5 §5.1）|
| 安全扫描 | SAST / DAST / SCA / 密钥扫描 / 渗透测试 |
| 部署 | 上线就绪度 / 灰度 / 全量 / 回滚 / 验证 |
| 文档 | 设计文档 / API 文档 / 用户文档 / README / CHANGELOG / ADR |
| 其他 | 复盘 / 沟通 / 排期 / 工具配置 / 学习调研 |

```yaml
    nodes:
      R1: 需求 / IR 完成

      R4: 需求 / 原型
      R4.1: 需求 / 可访问性评审
      R5: 需求 / 方案设计
      D1: 开发 / 实施计划
      D2: 开发 / TDD 增量
      D2.1: 开发 / 可观测性评审
      D3: 开发 / E2E 验收
      D3.1: 开发 / 性能评审
      D4: 开发 / 安全扫描
      D4.1: 开发 / 合规评审
      D5: 开发 / 部署前抽查
      D6: 开发 / 部署就绪度评审
      D6.1: 开发 / 文档评审
      T1: 测试 / 测试点设计
      T2: 测试 / 测试用例渲染
      T3: 测试 / UI 自动化
      T4: 测试 / API 自动化
      T5: 测试 / 覆盖度评审
      F1: 最终 / 全流程完成
      F2: 最终 / 终审聚合

  extras:
    # 用户/项目可继续追加
    # KDEV-D7: 开发 / kdev-memory dogfood 验证
```

### 3.2 phase ↔ phase_node 解耦

**phase 跟 phase_node 大类可以不一致**（关键设计）：

| v1.5 节点 | phase（工作性质） | phase_node（流程位置） |
|---|---|---|
| R1.5 IR 评审 | `评审` | `R1.5`（位于需求段） |
| D4 安全扫描 | `安全扫描` | `D4`（位于开发段） |
| D6.1 文档评审 | `评审` | `D6.1`（位于开发段） |
| T5 测试覆盖度评审 | `评审` | `T5`（位于测试段） |

phase = **做什么类的事**（蒸馏切片维度）
phase_node = **在 spec 流程的哪个位置**（联合查询维度）

## 五、单角色用户的逃生口（不构成干扰）

针对"只做开发 / 只做测试"的人，phase 字段不应成为负担。三档逃生口：

### 5.1 档 1：裁剪词表

```yaml
phase_vocab:
  coarse:
    - 测试
    - 调试
    - 评审
  # 其他 7 个值不出现在词表里，Claude 推断时不用
```

### 5.2 档 2：钉死默认值，跳过推断

```yaml
phase_vocab:
  default: 测试
  inference: skip   # 每条 G-NNN 直接填 测试，非测试场景用户事后手改
```

### 5.3 档 3：完全关掉

```yaml
phase_vocab:
  enabled: false   # G-NNN 不再含 phase 字段
```

### 5.4 初始化时自动选档

```
初始化问一句（Q 候选）：
"你这个项目主要覆盖哪些阶段？（多选）
 [ ] 需求 [ ] 设计 [ ] 开发 [ ] 测试 [ ] 调试 [ ] 评审
 [ ] 安全扫描 [ ] 部署 [ ] 文档 [ ] 其他
 全选 → 完整词表
 少选 → 自动裁剪 coarse 词表 + 单选锁定默认 + 不推断"
```

## 六、扩展到哪些文件

| 记录类型 | phase | active_skills | 理由 |
|---|---|---|---|
| **G-NNN（踩坑）** | ✅ 必加 | ✅ 必加 | 痛点直接来源 |
| **Q-NNN（决策）** | ✅ optional | ✅ optional | 决策跨阶段时可不填 |
| **Step（执行）** | ✅ optional | ❌ 已有"使用的 skill"字段 | 避免重复 |
| **R-NNN（改进建议）** | ❌ 不加 | ❌ 不加 | R 是项目内方法论反思，跨阶段 |
| **F-NNN（skill 反馈）** | ✅ optional | ❌ subject 已说了对谁 | phase 可补"在什么场景下吐槽" |

## 七、跨记忆系统联合查询（与 staff-memory 解耦不合并）

**phase 是 kdev-memory 与 staff-memory 的共享语义命名空间**，物理分目录不合并：

```
.kdev/
├── memory/                       ← kdev-memory plugin
│   └── 踩坑日志.md → G-007 phase=测试, phase_node=T2
└── staff-memory/                 ← kdev-core plugin
    ├── review-memory.md → R-001 节点=D2
    └── reviews/D2-代码评审/评分-1.md → phase=评审, phase_node=D2
```

联合查询能力示例：
- "D2 节点的所有信号"：扫两个目录找 `phase_node=D2`
- "测试阶段的踩坑 + 评审低分相关性"：按 `phase=测试` 联合
- 蒸馏切片 `dataset-by-phase/D2.md`：跨目录聚合

详见单独决策 Q-011（待落）。
