## 0. 一句话目标

把现有「主控 + skill 原子能力库」**两层**结构，升级为「主智能体 + 4 个组（每组 TL + 工作 agent）」**三层**多智能体集群，让用户**只跟主控对话**、**能查到子智能体进度**、**长任务不阻塞**。

---

## 1. 背景

### 1.1 用户原始需求

> 创建多智能体集群，包括主智能体、需求智能体组、开发智能体组、测试智能体组、审查智能体组，由主智能体进行各个组的任务编排和进度跟踪，人只需要和主智能体进行会话交流。
>
> 需求智能体组里面有 TL 智能体、需求智能体（IR→SR→AR 分解）、原型智能体（高保真原型）、设计智能体（设计方案）。开发、测试、审查也是类似情况。

### 1.2 KDev 现状（v3.0.3）

- 单主控 + state machine（`.kdev/state.md` + `pending_decision` gate）
- 原子能力库（IR/SR/AR/Prototype/Plan/Code/Test 各自是 skill）
- 开发组有雏形（`superpowers:subagent-driven-development` 已经在派 implementer/spec-reviewer/quality-reviewer）
- **缺：TL 层、测试组统一编排、审查组统一编排**

### 1.3 6 个现有 kdev plugin 在新架构里的位置

| Plugin | 现有定位 | 新架构角色 | 改造量 |
|---|---|---|---|
| `kdev-design-flow` | 4-stage 流水线（SR→AR→Prototype→Plan）+ 3 Gate | **reqs-TL 主体** | 🟢 加 IR 前置 + handoff 同步 |
| `kdev-coding-flow` | 13 节点 SOP，节点 6b 派 subagent | **dev-TL 主体** | 🟢 上游契约对齐 |
| `kdev-test-points-v1` | spec → 测试点 .md | test 组：**designer subagent** | 🟢 输入路径改 |
| `kdev-test-cases-v1` | 测试点 → fielded 用例 .md | test 组：**renderer subagent** | 🟢 0 |
| `kdev-ui-autotest` | Playwright 脚本生成 + 执行 | test 组：**ui-runner subagent** | 🟢 0 |
| `kdev-uicase-to-apicase` | UI 用例 → API 用例转换 | test 组：**api-translator subagent** | 🟢 0 |

待补：
- `kdev-test-flow`（test-TL 编排壳）
- `kdev-review-flow`（review-TL 编排壳）
- `kdev-api-test-scaffold`（API 测试执行，目前缺口）
- 主控 orchestrator skill

## 2. 共同基础（两方案都要做）

### 2.1 AR 编号统一规范

**强制对齐 `kdev-test-points-v1` 的最严格格式**：

```
AR-{DOMAIN}-{MAJOR}.{MINOR}.{PATCH}    例：AR-AUTH-01.001.001
```

- 4 个组、6 个 plugin、所有 handoff 必须按此格式
- reqs-TL 出口的 `ar.csv` 是格式权威定义点
- design-flow 内部如有不一致，加 adapter 转换

### 2.2 `.kdev/handoffs/` 跨组数据总线

```
.kdev/
├── state.md                          # 主控状态 + 各组 phase
├── handoffs/
│   ├── reqs/
│   │   ├── ir.md
│   │   ├── sr.md
│   │   ├── ar.csv                    # ★ AR 编号权威
│   │   ├── prototype/
│   │   ├── design.md
│   │   └── COMPLETE.md               # reqs 完成 signal
│   ├── dev/
│   │   ├── implementation-log.md
│   │   ├── commits.json
│   │   └── COMPLETE.md
│   ├── test/
│   │   ├── test-points.md
│   │   ├── test-cases.md
│   │   ├── ui-results.json
│   │   ├── api-results.json
│   │   └── COMPLETE.md
│   └── review/
│       ├── ceo.md / eng.md / design.md / security.md
│       └── COMPLETE.md
└── memory/                           # kdev-memory 现存
```

**规则**：plugin 内部状态留各自目录（`docs/design-flow/<slug>/` 等），但**完成态产物必须 copy 到 handoffs/**。

### 2.3 状态主权

`.kdev/state.md` 是**主控权威状态**，分 4 组 section：

```markdown
## reqs
status: complete | in_progress | blocked | pending
current_step: ir | sr | ar | prototype | plan
last_progress: <一句话，≤80 字>
started_at: <ISO>
completed_at: <ISO or null>

## dev
...

## test
...

## review
...
```

子 plugin 内部状态（design-flow 的 `flow-state.json`、coding-flow 的 `sop-execution-log.md`）**降级为内部细节**，每完成一个 step 必须回写 `state.md`。

### 2.4 共同缺口

- ⚠️ **`kdev-api-test-scaffold` 缺失**——uicase-to-apicase 提到下游，但 plugins/ 没有。两方案都需要这个，待拍板：(a) 现在补写；(b) 先只跑 UI 分流；(c) 复用现成框架薄包装
- ⚠️ **审查组没有 plugin**——两方案都要新建 `kdev-review-flow`

## 3. 方案 B：纯 Agent 工具方案

### 3.1 架构

```
用户
  ↕
主智能体（当前 session，跟用户对话）
  │
  │ ① Agent({subagent_type: "reqs-tl", run_in_background: true})
  ▼
┌─────────────────────────────────────────┐
│  reqs-tl  subagent（独立 context）      │
│  ─────────────────────                  │
│  Skill("kdev-design-flow") with args    │
│     ↓ 跑完 4 stage                       │
│  同步产物到 .kdev/handoffs/reqs/         │
│  写 .kdev/state.md::reqs section        │
│  返回 COMPLETE                          │
└─────────────────────────────────────────┘
  │ ② 主控收到回执
  ▼
继续 Agent 派 dev-tl / test-tl / review-tl
```

### 3.2 4 个 TL agent definition（plugin 内置）

```
plugins/
├── kdev-design-flow/
│   └── agents/reqs-tl.md          ← 新增
├── kdev-coding-flow/
│   └── agents/dev-tl.md           ← 新增
├── kdev-test-flow/                ← 新建 plugin
│   ├── skills/kdev-test-flow/SKILL.md  ← 薄编排
│   └── agents/test-tl.md
└── kdev-review-flow/              ← 新建 plugin
    ├── skills/kdev-review-flow/SKILL.md
    └── agents/review-tl.md
```

### 3.3 reqs-tl.md 示意

```markdown
---
name: reqs-tl
description: 需求组 TL。收到原始需求后编排 IR→SR→AR→Prototype→Plan，
  产出落 .kdev/handoffs/reqs/，返回 COMPLETE。
tools: Skill, Read, Write, Edit, Bash, Glob, Grep
model: opus
---

# 你的角色
你是 KDev 需求组 TL。主控通过 Agent 工具派单给你，
你独立 context 跑完需求全流程后回传。**你的工作不计入主控 context**。

# 工作流（5 步）
1. 初始化 state.md::reqs section
2. IR：Skill("superpowers:brainstorming") → .kdev/handoffs/reqs/ir.md
3. 跑主体：Skill("kdev-design-flow") with args ...
4. 产物同步到 .kdev/handoffs/reqs/（含 AR 编号格式校验）
5. 更新 state.md，返回结构化摘要（≤200 字）

# 进度可观测约束（硬规）
🔴 每完成一个 step 立刻 append .kdev/state.md::reqs
🔴 last_progress 字段 ≤80 字，描述当前在干什么
```

### 3.4 进度可见性实现

| 用户行为 | 主控响应 |
|---|---|
| 派单时 | 用 `run_in_background: true`，主控立即返回，不阻塞 |
| 问"进度？" | 主控调 `/kdev:status` skill → 读 state.md + handoffs/COMPLETE.md → 渲染表格 |
| 长时间无消息 | 主控用 TaskOutput 工具查后台 subagent 输出 |
| TL 完成 | background 完成回调通知主控 → 主控主动报 |
| 用户改主意 | TaskStop 强杀，或等返回纠偏 |

### 3.5 工作量（方案 B）

| 任务 | 工作量 |
|---|---|
| 共同基础（AR 规范、handoff 总线、state.md schema、契约文档）| 0.5d |
| 写 reqs-tl.md | 0.5d |
| 写 dev-tl.md | 0.5d |
| 写 test-tl.md + kdev-test-flow plugin | 1d |
| 写 review-tl.md + kdev-review-flow plugin | 1d |
| 主控 orchestrator skill（含 /kdev:status） | 1d |
| 跑通垂直切片 | 1d |
| 四组联调 | 1d |
| **合计** | **~6.5d** |

### 3.6 优劣

✅ **优势**：
- 100% 使用稳定 API（Agent 工具已 GA）
- 跨 session 续航强（state.md + handoffs/ 完全够用）
- 不需要 env var、不需要 VSCode 之外的终端（in-process 即可）
- 工作量已知、风险可控

❌ **劣势**：
- 用户**不能直接跟 TL 对话**（只能跟主控）
- 跨 TL 通信靠文件，比消息总线慢
- task list 需要自己维护
- 没有原生 hook 触发流转（要主控显式编排）

## 4. 方案 D：Agent Teams 底座方案

### 4.1 架构（含 nested 限制 workaround）

```
启用 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1（需 Claude Code v2.1.32+）
       │
       ▼
Team Lead = 主智能体（你跟它对话）
       │
       │ TeamCreate(members=[reqs-tl, dev-tl, test-tl, review-tl])
       ▼
   ┌─────────┬─────────┬─────────┬──────────┐
   │ reqs-tl │ dev-tl  │ test-tl │ review-tl│  ← teammate（subagent definition）
   └────┬────┴────┬────┴────┬────┴────┬─────┘
        │         │         │         │
        │ 共享 task list + mailbox 互发消息
        │         │         │         │
        │ 在自己 context 里用 Agent 工具派 subagent
        │（不能开 sub-team，nested 不允许）
        ▼         ▼         ▼         ▼
   普通 subagent（implementer / spec-reviewer / ...）
   或直接 invoke 现有 kdev-* skill
```

**核心折中**：**上层用 Agent Teams，下层用 Agent 工具**——既享受 Teams 的所有红利（共享 task list、mailbox、user 能直接跟 TL 对话），又避开「nested teams 不允许」的限制。

### 4.2 关键 Agent Teams 特性利用

| 特性 | 在方案 D 里的用法 |
|---|---|
| **TeamCreate** | 主控一次性创建 4 个 TL，不用挨个 Agent 派 |
| **共享 task list** | 4 组的所有任务进一个 list，pending/in_progress/completed 原生可见 |
| **mailbox 消息总线** | dev-tl 完成后**直接发消息**给 test-tl，不用文件中转 |
| **TaskCompleted hook** | dev 任务完成自动触发 test-tl 启动（替代主控手动派单）|
| **Shift+Down 切换 teammate** | 用户能直接跟某个 TL 对话给特殊指令 |
| **subagent definition 复用** | reqs-tl.md 既能作 teammate 类型，也能作普通 subagent 类型 |
| **TeammateIdle hook** | TL 空闲时主控收通知，可以自动派下一批任务 |

### 4.3 实现细节

```
plugins/
├── kdev-design-flow/
│   └── agents/reqs-tl.md          ← 同方案 B（subagent definition）
├── kdev-coding-flow/
│   └── agents/dev-tl.md           ← 同方案 B
├── kdev-test-flow/
│   ├── skills/.../SKILL.md
│   └── agents/test-tl.md
├── kdev-review-flow/
│   ├── skills/.../SKILL.md
│   └── agents/review-tl.md
└── kdev-orchestrator/             ← 新增（主控 skill）
    ├── skills/.../SKILL.md         ← /kdev:start-feature 入口
    └── hooks/                      ← TaskCompleted / TeammateIdle hook
        ├── on-reqs-complete.sh     ← 自动派 dev-tl
        ├── on-dev-complete.sh      ← 自动派 test-tl
        └── on-test-complete.sh     ← 自动派 review-tl
```

主控 `/kdev:start-feature <需求描述>` 入口逻辑：

```
1. 读 .kdev/state.md 初始化 4 组 section
2. TeamCreate(name="feature-<slug>", members=[
     {type: "reqs-tl", model: "opus"},
     {type: "dev-tl",  model: "sonnet"},
     {type: "test-tl", model: "sonnet"},
     {type: "review-tl", model: "opus"}
   ])
3. 派 reqs-tl 起跑：消息发给 reqs-tl："需求描述：<...>"
4. 主控继续陪用户对话
5. Hooks 自动驱动后续流转
```

### 4.4 工作量（方案 D）

| 任务 | 工作量 |
|---|---|
| 共同基础（同方案 B）| 0.5d |
| 写 4 个 subagent definition | 1d（比 B 简单：少一些手动 handoff 逻辑） |
| 写 /kdev:start-feature 主控 skill | 0.5d |
| 写 TaskCompleted hooks（3 个） | 0.5d |
| 跑通垂直切片 | 1d（含 env var 配置 + 调试 Teams 行为）|
| 四组联调 | 1d |
| **合计** | **~4.5d**（比 B 省 2d，因 task list/mailbox 不用自己写）|

### 4.5 优劣

✅ **优势**：
- **用户可直接跟任意 TL 对话**（Shift+Down 切换）
- task list / mailbox / hook 流转全部原生
- 工作量比方案 B 少 2 天
- 跟 Anthropic 官方路线一致，未来享受平台演进红利

❌ **劣势**：
- **Experimental 状态**——API 可能变（2026-02 才发，目前 v2.1.32 起支持）
- **No session resumption**——`/resume` 不能恢复 in-process teammates（**待确认严重性**）
- **No nested teams**——TL 不能开 sub-team（已用 workaround 规避）
- **一次只能一个 team**——不能并行做两个 feature 流水线
- **Lead 固定**——主控会话挂了整个 team 失效
- **Split-pane 需要 tmux/iTerm2**——VS Code 终端不支持分屏（用 in-process 即可）
- **subagent definition 的 `skills` frontmatter 不生效**——要在 spawn prompt 显式说

## 6. HUD 实时状态面板设计（用户已确认要做）

### 6.1 设计目标

借鉴 **oh-my-claudecode 的 `omc hud`** 思路，给 KDev 做一个**用户随时能看的状态面板**，覆盖：

1. 4 组当前状态（一目了然）
2. 当前 step 进度
3. 最近事件流（用户离开一段时间后回来能看到发生了什么）
4. 异常 / blocked 高亮

**核心设计原则**：**数据源 / 渲染层分离**——写一次数据，N 个 UI 并存。**必须同时覆盖 CLI 用户和 VSCode 扩展用户两种人群**。

### 6.2 数据源（共同基础，不分 CLI / VSCode）

```
.kdev/state.md                              # 主状态（markdown 友好格式）
.kdev/handoffs/<group>/COMPLETE.md           # 完成 signal
.kdev/events.log                             # 事件流：<ts> <group> <event-type> <msg>
.kdev/memory/*.md                            # 决策/踩坑（最近 N 条）
（方案 D 额外）~/.claude/tasks/{team-name}/   # task list 原生
```

**硬规**：所有 TL agent 完成 step 必须**立刻** append `state.md` + `events.log`，不能攒到结束。

### 6.3 渲染层（按用户群体并列两条主线）

#### 6.3.A CLI 用户主路径（终端 / tmux / 远程登录）

| 优先级 | 形态 | 说明 |
|---|---|---|
| ⭐ 主推 | **Claude Code `statusLine`** | settings.json 配 command → 输入框上方常驻一行 HUD 摘要 |
| 辅助 | `/kdev:hud --watch` | CLI watch 模式，1s 刷新，专用 ASCII box 渲染 |
| 兜底 | `/kdev:hud` | 用户在对话里主动打，单次渲染 |

CLI statusLine 输出（命令 `kdev-statusline.sh` 单行）：

```
KDev · feature-用户登录 · 📝✅ 💻🟡7/13 🧪⏳ 🔍⏳ · 23min
```

`/kdev:hud --watch` ASCII box 渲染（终端全屏，跟 OMC `omc hud` 同款）：

```
╔══════════════════════════════════════════════════════════════╗
║  KDev HUD · feature: 用户登录功能 · started 23 min ago         ║
╠══════════════════════════════════════════════════════════════╣
║  组        状态      当前 step           已花时间   最近进度    ║
╠══════════════════════════════════════════════════════════════╣
║  📝 reqs   ✅ 完成   -                   18 min    5 AR 产出   ║
║  💻 dev    🟡 进行中 TDD step 7/13       12 min    8/15 测试绿 ║
║  🧪 test   ⏳ 待命   -                   -         等 dev 完成 ║
║  🔍 review ⏳ 待命   -                   -         等 test 完成║

╚══════════════════════════════════════════════════════════════╝
```

#### 6.3.B VSCode Claude Code 扩展用户主路径（**已实测 2026-05-27**）

⚠️ **VSCode Claude Code 扩展不渲染 `statusLine` 字段**（见 §6.6 实测结论）——主推方案降级为「markdown preview + 对话流 HUD」双轨。

| 优先级 | 形态 | 说明 |
|---|---|---|
| ⭐ 主推 A | **VSCode markdown preview** | `state.md` 写成 markdown 表格 + emoji，右键 "Open Preview to the Side"，VSCode 原生自动刷新——**常驻 panel，眼角余光可见**（最接近 CLI statusLine 体验）|
| ⭐ 主推 B | **对话流 `/kdev:hud`** | 用户在 Claude Code 对话框里打命令 → markdown 表格原生渲染——**主动查询时用** |
| 主控触发 | **TL 节点完成时主控自动渲染 HUD** | 写进主控 system prompt 硬规：任何 TL 状态变化必须先输出 `/kdev:hud` 再继续 |
| 可选（W3+）| **专用 VSCode 扩展** | StatusBar + TreeView + Webview，3-5d 工作量。**实测已确认 statusLine 不可用后，本项重要性升级**——如果 markdown preview + 对话流体验不够好，W3+ 启动 |
| ❌ 不可用 | ~~Claude Code `statusLine`~~ | **实测 VSCode 扩展不渲染**（仅 CLI 用），见 §6.6 |

VSCode 对话流里 `/kdev:hud` 的 markdown 渲染效果：

```markdown
|---|---|---|---|---|
| 📝 reqs | ✅ 完成 | - | 18min | 5 AR 产出 |
| 💻 dev | 🟡 进行中 | TDD 7/13 | 12min | 8/15 测试绿 |
| 🧪 test | ⏳ 待命 | - | - | 等 dev |
| 🔍 review | ⏳ 待命 | - | - | 等 test |

**📋 最近事件**
- 12:46 dev-tl 🟢 单元测试 8/15 通过
- 12:41 dev-tl 🟡 节点 6b 派 implementer
```

### 6.4 用户群体推荐组合速查表（已实测 2026-05-27）

| 用户类型 | 主推 | 辅助 | 兜底 |
|---|---|---|---|
| CLI / 终端 / 远程 | ✅ `statusLine`（输入框上方一行，已实测可用）| `/kdev:hud --watch`（专屏 ASCII）| `/kdev:hud` |
| **VSCode 扩展（主流）** | ✅ `state.md` markdown preview 并排 + 对话流 `/kdev:hud` 双轨 | TL 节点完成主控自动渲染 HUD | （statusLine 不可用，降级方案见 §6.3.B）|
| 无 IDE / pipeline 场景 | `cat .kdev/state.md` | - | - |

### 6.5 实现要点

- TL agent 高频写盘约束（写进每个 TL 的 system prompt 硬规）
- `kdev-statusline.sh` shell 脚本（读 `state.md` 解析 4 组状态 → 输出单行，CLI 和 VSCode 共用）
- `/kdev:hud` skill 支持 3 种输出模式：`box`（ASCII，给终端）/ `markdown`（给对话流和 VSCode）/ `watch`（CLI 全屏 1s 刷新）
- `events.log` 由 TL agent append（统一格式）
- 项目级 `.claude/settings.json` 内置 `statusLine` 配置，团队 clone 后自动生效

### 6.6 实测结果（2026-05-27 已完成）

**hello-world 配置**：

```json
// .claude/settings.local.json
{
  "statusLine": {
    "type": "command",
    "command": "echo 'HELLO STATUSLINE TEST'"
  }
}
```

**实测方式**：保存配置 → 重启 Claude Code（CLI / VSCode 扩展分别测试）→ 观察 `HELLO STATUSLINE TEST` 渲染位置。

**实测结论**（已固化）：

| 运行环境 | 渲染位置 | 适合度 | 主推方案 |
|---|---|---|---|
| **CLI（终端 / iTerm2 / tmux）** | ✅ Claude Code 输入框**上方**一行（位置 🅰） | ⭐⭐⭐⭐⭐ 理想 | **statusLine 主推** |
| **VSCode Claude Code 扩展** | ❌ **完全不渲染**（位置 🅳） | 无法显示 | **降级**：markdown preview + 对话流 `/kdev:hud` 双轨主推 |

**根因推测**：VSCode Claude Code 扩展（截至 2026-05-27 版本）**未实现 `settings.json` 的 `statusLine` 字段**。CLI Claude Code 实现了。

**影响**：
- §6.3.A CLI 主推方案 ✅ 完全确认可用
- §6.3.B VSCode 主推方案 ⚠️ **必须降级**——statusLine 不可用，markdown preview + 对话流 HUD 升为主路径
- 长期跟踪：如果 Anthropic 后续给 VSCode 扩展加上 statusLine 支持，本节结论需重新评估

**踩坑落 G-NNN**（后续按 kdev-memory 制度补记）：
- 现象：VSCode Claude Code 扩展不渲染 `statusLine` 字段
- 影响：方案 B/D 文档 §6.3.B 主推方案降级
- 缓解：markdown preview + 对话流 `/kdev:hud` 替代

### 6.7 工作量

- HUD skill（含 3 种输出模式：box / markdown / watch）：0.5d
- `kdev-statusline.sh` shell 脚本：0.25d
- `events.log` 规范 + TL 高频写盘约束：0.25d（一并写进 TL agent prompt）
- `.claude/settings.json` 配 statusLine：0.1d
- VSCode statusLine 渲染位置实测：0.1d（10min）
- **小计**：~1.2d（两方案共用）
- **可选** VSCode 专用扩展（W3+ 实测后决定）：3-5d

## 7. 跨 Session 续航（TBD 待确认）

### 7.1 方案 B 的续航能力

✅ **天然支持**：
- state.md / handoffs/ / memory/ 全是文件
- 新 session 启动，主控读 state.md 就能续上
- subagent 跑到一半 session 挂了：subagent 进程消失，但**文件落盘的进度还在**，主控可以从最后已知 step 重启该 TL

### 7.2 方案 D 的续航缺陷（官方文档原话）

> No session resumption with in-process teammates: `/resume` and `/rewind` do not restore in-process teammates. After resuming a session, the lead may attempt to message teammates that no longer exist. If this happens, tell the lead to spawn new teammates.

**意味着**：用 `/resume` 恢复主控 session 时，4 个 TL teammate **不会自动恢复**，team 处于「lead 在但 teammate 不在」的悬空状态。

### 7.3 待确认问题清单（必须 confirm 才能定方案 D）

| # | 待确认问题 | 影响 | 验证方式 |
|---|---|---|---|
| Q1 | `/resume` 后能否手动 `TeamCreate` 重建 team + 从 state.md/handoffs hydrate？ | 决定方案 D 是否可用 | 实测一次：起 team → `/exit` → `/resume` → 重建 team 看是否能续 |
| Q2 | Team 配置 `~/.claude/teams/{name}/config.json` 在 session 重启后是否保留？ | 重建 team 的依据 | 看文件是否还在 |
| Q3 | Task list `~/.claude/tasks/{name}/` 在 session 重启后是否保留？ | task 状态续航 | 同上 |
| Q4 | Anthropic 官方有没有 roadmap 修复 resume？预计 GA 时间？ | 决定是「等等再上 D」还是「先上 B」 | 查 changelog + Discord |
| Q5 | KDev 实际场景里跨 session 续航频率多高？（一个 feature 跑多少天？）| 决定续航缺陷的实际权重 | 看 Sprint 0 数据 |

### 7.4 续航策略备选（不论 B/D 都适用）

**双轨保险**：
- 方案 D 享受 Agent Teams 红利，但**所有关键状态强制写 `.kdev/state.md` + `.kdev/handoffs/`**
- 一旦 session 挂、resume 失败、team 悬空 → 主控有能力**从 state.md 重建一切**（降级为方案 B 模式继续）

这等于「方案 D 是常态，方案 B 是降级容灾」。

适合：跨 session 续航是硬刚需，且 Anthropic 短期不修 Teams resume
