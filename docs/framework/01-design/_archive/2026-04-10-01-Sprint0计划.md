# KDev Sprint 0 计划

> 生成日期：2026-04-10
> 版本：v1.0
> 目的：用现有插件手动走完 KDev 完整流程，以 token-statistics 为实战项目，验证 v3.0.1 架构设计的可行性

---

## 一、背景与决策

### 1.1 已确认的产品决策

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 最终用户 | 先自用打磨，再开源 | 降低前期负担，用实战反馈打磨 |
| 狗粮项目 | token-statistics（0→1）→ kdevsec（1→N） | 先简单后复杂，渐进验证 |
| 推进节奏 | 并行：手动走完一个 Sprint，攒需求后集中实现 KDev | 用真实体感校验设计，避免闭门造车 |
| 技术形态 | Sprint 0 后再定（Skill 包 / Skill+CLI / 其他） | 需要实际痛点数据支撑决策 |
| 方案 | 最小记录框架 + Sprint 0 | Compound Engineering 从第一天开始积累 |

### 1.2 插件依赖

| 插件 | 状态 | Sprint 0 用途 |
|------|------|--------------|
| **BMAD** | ✅ 已安装 | PRD/架构/Epics/Sprint 核心流程 |
| **Gstack** | ✅ 已安装 | 四角色阶段评审 + /qa + /cso + /ship + /review |
| **Superpowers** (v5.0.7) | ✅ 已安装 | brainstorming, writing-plans, TDD, subagent, code-review, verification |
| **Playwright** | ✅ 已安装 | E2E 测试辅助 |
| **frontend-design** | ✅ 已安装 | UI 原型生成 |
| Compound / OMC | ❌ 不装 | 属于自动化层，Sprint 0 结束后决定是否引入 |

## 二、命令体系设计

> 对齐 standardized-dev-flow (IR→SR→AR) 流程 + BMAD 术语，共 14 个命令

### 2.1 Phase 1 规划阶段

产出迭代计划，粗粒度 AR 仅用于估算。

| 命令 | 做什么 | 交付物 | BMAD 对应 |
|------|--------|--------|-----------|
| `/kdev:ir` | 收集原始需求 | IR 文档 | brainstorming + product-brief |
| `/kdev:sr` | 梳理需求列表 + 粗粒度 AR 估算 | SR CSV + 粗粒度 AR CSV | create-prd |
| `/kdev:prototype` | 概览原型 | HTML 原型 | create-ux-design |
| `/kdev:review` | 人工评审三件套 (SR + AR + 原型) | 评审通过 | check-implementation-readiness |
| `/kdev:plan` | 输出迭代实施计划 | 迭代计划文档 | sprint-planning |

### 2.2 Phase 2 执行阶段

每迭代循环，详细 AR 在此阶段产出。

| 命令 | 做什么 | 交付物 | BMAD 对应 |
|------|--------|--------|-----------|
| `/kdev:ar` | 详细 AR 分解 + 高保真原型 + 技术设计 | 详细 AR CSV + HTML + 设计文档 | create-architecture + create-story |
| `/kdev:align` | 需求对齐 (O-S-U-R-A) + 修订 | 对齐确认单 + 修订后文档 | E2+E3 合并 |
| `/kdev:dev` | TDD 全循环 (Red→Green→Refactor) | 测试 + 实现代码 | dev-story + TDD |
| `/kdev:e2e` | E2E 系统测试 | 测试报告 | Gstack /qa |
| `/kdev:accept` | 系统验收 + 代码评审 | 验收报告 | code-review + acceptance |

### 2.3 贯穿全程

| 命令 | 做什么 |
|------|--------|
| `/kdev:start` | 入口：检查 state.md，选择阶段 |
| `/kdev:ship` | 发布上线 |
| `/kdev:recap` | 回顾总结 + 知识沉淀 |
| `/kdev:security` | 安全审计 |

### 2.4 状态机

```
Phase 1: 规划阶段
INIT → P1-IR → P1-SR → P1-PROTOTYPE → P1-REVIEW → P1-PLAN
                                           ↑
                                      [人工审批门禁]

Phase 2: 执行阶段（每迭代循环）
E1-AR → E2-ALIGN → E4-DEV → E6-E2E → E7-ACCEPT → NEXT-ITERATION
                                                        ↓
                                                   E1-AR（下一迭代）
```

## 三、前置工作：最小记录框架

> 工作量：半天。Sprint 0 开始前完成。

### 3.1 在 token-statistics 项目中创建目录结构

```
.kdev/
├── state.md              # 流程状态追踪（对齐 standardized-dev-flow）
├── questions-log.md      # 决策追溯
├── gotchas.md            # 踩坑记录
├── sprint0-journal.md    # 流程体验记录
└── daily-logs/           # 每日工作总结
```

### 3.2 在 CLAUDE.md 中写入智能体记录规则

智能体在走 KDev 流程时必须自动维护 `.kdev/` 下的记录文件：

| 触发时机 | 记录目标 | 记录者 |
|---------|---------|--------|
| 每次需要人做决策 | questions-log.md：问题、选项、用户选择、理由 | 智能体自动 |
| 每次踩坑/绕路/报错 | gotchas.md：现象、原因、解决方式 | 智能体自动 |
| 每步完成后 | sprint0-journal.md：执行过程、产出、体验评估、发现 | 智能体自动 |
| 每步结束后 | 主动向用户征询顺畅度评分（1-5）和主观感受 | 智能体提问，人回答 |
| 流程状态变更 | state.md：更新当前状态和交付物路径 | 智能体自动 |
| 会话结束前 | daily-logs/YYYY-MM-DD.md：汇总当天工作 | 智能体自动 |

### 3.3 不做的事（留给 Sprint 0 之后）

- ❌ knowledge-graph (Memory Level 1)
- ❌ learnings.jsonl (Memory Level 2)
- ❌ skills/ 技能提取 (Memory Level 4)
- ❌ session-state.json 自动化
- ❌ Hook 自动化
- ❌ 任何 KDev 命令的代码实现

## 五、需要验证的设计纰漏

> 来源：对 docs/ 下全部 26 份文档的逐行审读

| # | 纰漏 | 来源文档 | 风险 | Sprint 0 验证方式 |
|---|------|---------|------|------------------|
| 1 | **Token 预算管理缺失** | v2.0/v3.0/v3.0.1 三轮评审均指出 | 记忆注入可能撑爆上下文窗口 | 观察每步实际 token 消耗，记录到 journal |
| 2 | **框架降级策略缺失** | v2.0 评审指出，至今未补 | 插件调用失败时无 fallback | 如果某插件跑不通，记录并手动绕过 |
| 3 | **记忆系统碎片化** | 07-记忆系统对比、08-新增项目深度调研 | `.kdev/` vs `.omc/` vs `~/.gstack/` 多处写入 | 只用 `.kdev/`，观察其他系统是否冲突 |
| 4 | **BMAD Helper Pattern 未采纳** | 04-框架纯对比分析（81% token 节省） | 命令数多时 token 浪费 | 记录命令调用时的 token 效率 |
| 5 | **E2 需求对齐是否必要** | standardized-dev-flow E2 状态 | 可能过于仪式化 | 实际走一遍，评估是否值得保留 |
| 6 | **Spec-Kit Constitution 缺失** | 09-技能源码深度对比 | "项目原则约束"能力被丢弃 | 观察是否缺少此能力 |
