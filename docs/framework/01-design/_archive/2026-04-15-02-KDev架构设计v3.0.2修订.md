## 一、R-M1：外部 skill 定位翻转（整块调用 → 原子能力库）

### 1.1 问题陈述

v3.0.1 §三 将 BMAD / Gstack / Superpowers 的 skill **整块映射**为 KDev 命令（如 `/bmad-create-prd` → `/kdev:prd`）。Sprint 0 实战发现：

- **skill 自带流程污染主工作流**：gstack 的 5 项 onboarding（upgrade / telemetry / lake-intro / proactive / routing）每次调用必须 `touch marker` 跳过
- **token 效率低**：bmad-create-prd 12 步 preamble 约 50k token / bmad-create-architecture 8 步 40k token 在低复杂度项目里价值密度低
- **流程主权丢失**：整块调用时 skill 自己决定节点顺序，KDev state machine 沦为"记录执行结果"

### 1.2 修订结论

KDev state machine **主导编排**，外部 skill **只作原子能力库**。

```
旧模式（v3.0.1，废弃）：
KDev 命令 → 整块调用外部 skill → skill 带自己的流程跑 → 返回结果
                                ↑ 流程失控

新模式（v3.0.2）：
KDev state machine 主导
    ↓
调用原子能力（来自外部 skill 的核心能力，封装后）
    ↓
拼装成本地节点脚本（skill-atom）
```

### 1.3 实施原则

1. **原子能力提取标准**：每个原子能力必须能**脱离原 skill 独立使用**，接口为纯函数（输入 → 输出），不含 onboarding / telemetry / 次要节点
2. **两种集成形态**：
   - **承重墙**（高频 / 稳定 / 价值密度高）：允许"跳引导后整块调用"（例：`superpowers:writing-plans` / `superpowers:subagent-driven-development` / `gstack-plan-eng-review`）
   - **补缺**（场景化 / 低频 / 需裁剪）：强制拆为原子能力（例：`bmad-create-prd` 抽"FR/NFR 结构化模板"+"分阶段规划"两项原子能力）
3. **本地优先**：能用本地模板 + 少量 LLM 提示实现的节点，不调外部 skill（例：AR CSV 生成已有 conventions §1/§2 作为宪法，不需要 bmad skill）

### 1.4 交付物要求

Sprint 1 出口必须产出 **"skill 能力对照表 v1"**（`docs/01-design/2026-XX-XX-skill能力对照表.md`），列出：

| 维度 | 每条记录必填 |
|------|-------------|
| 来源 skill | 如 `bmad-create-prd` |
| 原子能力列表 | 如 "FR/NFR 结构化模板"、"Journey 梳理问答"、"分阶段 scoping" |
| 吸收决策 | 吸收（拆出）/ 承重墙保留 / 丢弃 |
| 改写需要 | 如"去掉 12 step preamble，保留 FR 模板" |
| KDev 节点映射 | 如"`/kdev:sr` 引用 FR 模板" |

### 1.5 原子能力接口契约

> 来源：gstack-plan-eng-review A2 发现 — Sprint 1 Day 1 必须撞到的"原子能力物理形态是什么"问题必须在设计阶段解决。

#### 1.5.1 物理形态

原子能力 = **markdown 文件 + frontmatter 契约**，存放于 `.kdev/skill-atoms/<source-skill>/<atom-name>.md`。

```markdown
---
atom_id: bmad-create-prd:fr-template
source_skill: bmad-create-prd
source_version: ">=4.0.0"        # 锁定兼容的 skill 版本范围
absorption: extracted              # extracted | wholesale | rewritten
input_schema:                      # 消费者（KDev 节点或 LLM）必须提供的字段
  project_goal: string
  constraints: string[]
  priority_hint: enum[MVP, Growth]
output_schema:                     # 原子能力产出格式
  fr_list:
    - id: FR-NN
      title: string
      description: string
      priority: enum[MVP, Growth]
  nfr_list:
    - id: NFR-NN
      category: enum[performance, security, availability, ...]
      requirement: string
usage_context:
  - /kdev:sr                       # 哪些 KDev 节点可调用
  - /kdev:ar                       # （详细 AR 补 FR 时复用）
token_budget: ~5k                  # 预估 token 消耗（用于 Sprint 2+ 预算管理）
---

# 原子能力：FR/NFR 结构化模板

## 使用说明
[prompt 片段 / 调用指引，供 LLM 或 CLI 消费]

## 输入模板
...

## 输出示例
...
```

#### 1.5.2 调用方式

| 调用方 | 方式 |
|--------|------|
| KDev CLI（节点脚本） | `kdev atom invoke <atom-id> --input <json>` → 读取 atom 文件 + 拼装 prompt + 调 LLM + 按 output_schema 校验 |
| LLM 对话 | CLI 在进入节点时主动 inject atom 文件内容到对话上下文，LLM 按 atom 的"使用说明" + 输入模板自主执行 |
| 手动调试 | `kdev atom show <atom-id>` 打印 atom 契约 + prompt 预览 |

#### 1.5.3 具体实例（Sprint 1 Day 1 起点）

**实例 A：`bmad-create-prd:fr-template`（补缺 / 拆出）**

- 输入：`{project_goal: "搭建团队 token 统计仪表盘", constraints: ["1C1G 部署", "10-20 人团队"], priority_hint: "MVP"}`
- 输出：`{fr_list: [{id: "FR-1", title: "Hook 自动上报", description: "...", priority: "MVP"}, ...], nfr_list: [...]}`
- 物理实现：抽出 bmad-create-prd 的 Step 10（Functional）+ Step 11（NFR）的 prompt 片段，丢弃 Step 0-9 的 onboarding
- **Sprint 1 Day 1 必须产出的第一个 atom 实例**（验证契约落地可行性）

**实例 B：`superpowers:writing-plans`（承重墙 / 整块）**

- 输入：`{tech_design_doc: <path>, ar_csv: <path>, review_adoptions: <path>}`
- 输出：`{implementation_plan_md: <path>}`（原 skill 产出不变）
- 物理实现：`atom.md` 只含"调用说明"，frontmatter 标 `absorption: wholesale`，body 实际转发给 `superpowers:writing-plans` 整块执行
- **承重墙 atom 与补缺 atom 共享同一套 frontmatter schema**，差别仅在 `absorption` 字段和 body 是否含 prompt 片段

#### 1.5.4 版本与失败策略

- **版本锁**：每个 atom frontmatter 含 `source_version`（semver 范围）。kdev CLI 启动时校验 skill 版本，不匹配则 warn，强制不匹配则 block
- **降级链**：atom 调用失败（skill 不在 / schema 校验 fail）→ CLI 自动降级到 `absorption: rewritten` 的 fallback atom（若存在）→ 仍失败则创建 `pending_decision` 要求用户裁决（提议"手工跑 / 跳过 / abort"三选项）
- **升级策略**：skill 新版本发布后，atom 的 `source_version` 由手动 PR bump（**不自动**，防 silent breakage）

## 二、R-M3：强制节点 gate（HARD-GATE 语义扩展）

### 2.1 v3.0.1 的 HARD-GATE 局限

v3.0.1 §二.1 定义 HARD-GATE 只在"Phase 1 规划 → Phase 2 执行"之间，确保用户批准设计后才进入代码实现。

Sprint 0 暴露：**LLM 在规划阶段内部也会自行跳步**（iter4 Step 2 brainstorm 偷跳 / G-008 跳 code quality review 均属此类）。只在 Phase 切换处设 gate 不够。

### 2.2 修订：节点级强制 gate

**所有关键节点必须产出显式决策**，才可 `next`。LLM 权限边界：

| LLM 可以 | LLM 不可以 |
|---------|-----------|
| **提议**降级路径（"跑完整 skill / 摘要替代 / 跳过"三选项） | **决定**降级 |
| 给每个选项写推荐 + 理由 | 沉默跳过节点 |
| 检查前置条件缺失，提示降级选项 | 因"输入已结构化"自行绕过节点 |

### 2.3 实现形态

`.kdev/state.md` 的 `pending_decision` 字段必须与 Sprint 0 已验证 `questions-log.md` 的 Q-NNN schema 对齐：

```yaml
pending_decision:
  node: P1-SR
  question: "bmad-create-prd 是否跑完整 12 step？"
  context: |                                   # 背景 1-2 句，必填
    当前项目是 brownfield + 低复杂度，输入已由 IR 结构化。
  options:                                      # 必须至少包含"三基本模式"（见 §2.4）
    - id: A
      label: 完整跑（~50k token）
      mode: full                                # full | degraded | skip
    - id: B
      label: 只取 FR 模板（~5k token）
      mode: degraded
    - id: C
      label: 跳过（需理由）
      mode: skip
  llm_recommendation: B
  llm_rationale: "输入已结构化 + 项目低复杂度，完整跑边际价值低"    # 必填
  user_choice: null                              # 用户未回答时为 null
  user_rationale: null                           # 用户选定后必填（特别是选 C 时）
  created_at: "2026-04-15T10:00:00+08:00"
```

**kdev CLI 行为**：

1. `pending_decision.user_choice == null` 时，`kdev next` 拒绝前进
2. 用户必须显式回答：对话里回答 / `kdev decide --answer B --rationale "..."` 命令 / 直接编辑 state.md
3. 决策写入 `questions-log.md` 追加 Q-NNN 条目后，清除 `pending_decision` 字段
4. **schema 校验**：CLI 在读取 `pending_decision` 时强制校验 options 必须覆盖 `mode: full`、`mode: degraded`、`mode: skip` 三种（见 §2.4），缺失则拒绝接受，要求 LLM 补齐

### 2.4 选项硬约束（防 LLM subtle 限制用户决定权）

> 来源：gstack-plan-eng-review A3 发现 — LLM 可以通过"只给 B/C 两选项"隐藏"跑完整"，用户决定权被 framing 层限制。

**硬规则**：每个 `audience: human` 的 gate，`options` 必须**同时**满足：

| `mode` 取值 | 含义 | 必填性 |
|------------|------|--------|
| `full` | 跑完整重量版（如完整调用外部 skill / 跑满流程 step） | ✅ 必填 ≥ 1 项 |
| `degraded` | 降级变体（摘要替代 / 原子能力调用 / 轻量版） | ⚠️ LLM 可提 0 ~ 多项 |
| `skip` | 跳过本节点（用户选择时必须填 `user_rationale`） | ✅ 必填 1 项 |

CLI 实现：
- `pending_decision` schema 校验时检查 `options[*].mode` 集合是否含 `full` 和 `skip`
- 缺失任一 → CLI 拒绝接受 pending_decision，打印 `❌ options 缺少 mode=full 或 mode=skip，请 LLM 补齐（防止 subtle 限制用户决定权）`
- LLM 必须重新生成 options

**例外**：`audience: agent` 节点的 gate（进入 / 结束）不受此约束，因为无用户介入。

### 2.5 哪些节点必须有 gate

按 §三 受众分类：

- **人受众节点**（IR / SR / AR 粗粒度 / 技术设计 / 验收报告 / 核对）：**每个都要 gate，且受 §2.4 硬约束**
- **智能体受众节点**（writing-plans / subagent-driven / TDD step）：只在"开始 / 结束"设 gate，内部不拦，不受 §2.4 约束
- **混合节点**（gstack 评审）：入口处 gate（受 §2.4 约束）+ 结果核对 gate（用文档留白 UX，见 §3.2）

## 三、R-M2：节点按受众二分

### 3.1 问题陈述

v3.0.1 所有节点一视同仁（同一套评分 / 核对 UX / 决策 gate）。Sprint 0 观察：

- **人受众节点**（Step 3 技术设计 6 决策核对，4/5）和**智能体受众节点**（Step 6 writing-plans，4.5/5）的差距不在 LLM 能力，而在"是否该让用户介入"
- 用户反馈：writing-plans"问题不大，这一步是智能体执行代码的输入，非常细致+不需人工干预"
- 在智能体受众节点强加核对反而打断；在人受众节点没设 gate 则 LLM 偷跑

### 3.2 受众分类定义

| 类别 | 定义 | 典型节点 | UX 要求 | 核对 UX 默认 |
|------|------|---------|---------|------------|
| **人受众（human）** | 用户做决策的节点 | IR / SR / AR 粗粒度 / 技术设计 / 验收报告 / 多决策核对 | 必须有决策 gate（受 §2.4 硬约束）；顺畅度评分必填 | **按问题数量自适应**：1-2 个决策 → 逐条 ping；3-5 个 → 批处理 AskUserQuestion；≥6 个 → **文档留白**（markdown 决策档，用户直接编辑）|
| **智能体受众（agent）** | subagent 执行的节点 | writing-plans 实现计划 / subagent TDD step / 测试脚本生成 | LLM 独立完成；人只看自查报告；跳过逐节点评分（R-005 已闭环）；只在"整体 meta"和"BLOCKED 干预点"采集 | N/A（无用户核对） |
| **混合（mixed）** | 既有 LLM 执行又有用户核对的节点 | gstack 评审（LLM 跑评审 + 用户裁决采纳）/ E4-DEV 整体（subagent 执行 + 用户看整体结果） | 入口 gate（受 §2.4 硬约束）+ 结果核对 gate；内部跳评分 | **默认文档留白**（评审产出 N 条发现 → 逐条附选项 → 用户编辑）；简单结果（如 `/kdev:security` 零发现）可退化为"摘要确认"单问 |

**核对 UX 三种模式细则**：

| 模式 | 适用场景 | 实现形态 | 证据 |
|------|---------|---------|------|
| **逐条 ping** | 关键决策 / 决策数 ≤ 2 | 每问单独 AskUserQuestion | iter4 Step 15 PRD 核对（3 处 GAP 逐条）|
| **批处理 AskUserQuestion** | 次要决策 / 决策数 3-5 | 单轮 AskUserQuestion 含多条 | iter4 Step 3 的反例（6 决策批处理 → 返工）|
| **文档留白** | 大批量 / 决策数 ≥ 6 / 需异步 / 决策混合重要度 | LLM 生成 `.kdev/decisions/<topic>.md`，用户编辑，CLI `kdev decide --collect` 自动誊抄进 questions-log | iter4 Step 5 Q-049（17 gstack 发现批量裁决的理想场景） |

### 3.3 受众字段进入节点元信息

KDev CLI 的节点定义必须显式标注：

```yaml
# .kdev/nodes/P1-SR.yaml（示意）
name: P1-SR
audience: human
gate_required: true
evaluation_required: true   # 顺畅度评分
skill_atoms:
  - bmad-create-prd:fr-template
  - bmad-create-prd:nfr-template
next: [P1-PROTOTYPE]
```

```yaml
# .kdev/nodes/E4-DEV-PLAN.yaml（writing-plans）
name: E4-DEV-PLAN
audience: agent
gate_required: false        # 内部不设 gate
evaluation_required: false  # 跳评分
skill_atoms:
  - superpowers:writing-plans  # 承重墙，允许整块调用
parent_node: E4-DEV
```

### 3.4 顺畅度评分口径调整

- **旧口径**（v3.0.1 默认）：每节点完成后必评分 → 拉平信号（人受众均值被智能体受众拉高）
- **新口径**（R-005 已闭环到 conventions §9.4）：
  - 人受众节点 → 必评分，即时采集
  - 智能体受众节点 → 跳过
  - 混合节点 → 只在"人参与点"和"整体 meta"评分
  - 统计时**按受众分别汇总**，不混合均值

## 四、R-CMD：14 命令受众标注 + 对齐 IR→SR→AR

### 4.1 命令体系重定义

v3.0.1 §八.1 定义了 23+ 命令（8 核心 + 12 进阶 + 9 专家），按 BMAD 四阶段组织。Sprint 0 计划（2026-04-10-01）已将其**压缩为 14 个**，对齐 `standardized-dev-flow` 的 IR→SR→AR 流程。本修订将 14 命令列表**确定为正式架构**，并补上受众标注。

### 4.2 14 命令受众表

#### Phase 1 规划阶段（6 个命令）

## 五、R-SKILL：承重墙 vs 补缺分类表

### 5.1 分类原则

来自 Sprint 0 改进建议 §建议 7。所有外部 skill 按"**承重墙**（已稳定 / 高频 / 不轻易替换）vs **补缺**（场景化 / 允许实验 / 可替换）"分类，作为集成决策的硬约束。

**铁规**：
1. **默认不做工具链替换** — 已稳定的承重墙 skill 不换，哪怕新方案看起来更"正统"（反例：BMAD 自称敏捷方法论 ≠ 在本项目适合敏捷 → 必须有实践数据）
2. **替换必须走控制实验** — 用一个独立 Feature 或小 Sprint 全走新链路，对照指标（顺畅度、token、返工率、bug 密度），不达标就回滚
3. **"官方自称"不是证据** — 只认 Sprint 0 / Sprint 1+ 实战数据

### 5.2 当前分类（Sprint 0 实战数据产出）

#### 承重墙（整块调用允许，作为 KDev 标配集成）

|-------|---------|---------|----------|
