为什么要 SOP
光有员工不够 —— 要 SOP 把活儿编排起来
数字员工会干活，但四个问题悬空 —— 这四个问题的答案，就是一条 SOP（标准作业流程）：
?
按什么流程
?
什么顺序
?
哪里把关
?
哪里停 / 交还人
SOP ＝ 节点表（做什么 · 什么顺序）　＋　gates（哪里把关 · 过不过怎么办）
如果这套「节点表 + gates」写死，一个底座就只能跑一种活。
要让同一个底座跑任意业务 → 需要 harness 工程让它可编排。↓

5
harness · 核心
让 SOP 可编排的底座机制 —— kdev-core lean（R1 / R2 / R3）
R1 · flow-state 状态机
每个 flow 一份 flow-state.json：原子写（tmp + os.replace）＋ history 流水。
→ 白捡 resume（断点续跑，coding-flow 接上就有）。
R2 · 通用 n 节点机
节点表由 flow 自己声明（不写死段数：design 4 / coding 13）。
底座只提供流转引擎：顺进（gate PASS → 下一节点）/ 回流（gate FAIL → on_fail）。
R3 · 三类 gate 统一抽象（核心难点 = 卖点）
kind谁评 → verdict不过怎么办
review 评审重试ai/both/human → PASS/FAIL改 → 重试（max N）
decision 路线判断ai 默认 / human 可接管 → 分支选分支即流转（无重试）
acceptance 验收test 自动跑 → PASS/FAIL回流 fix 节点
三类共用一套解析（record_gate）
# GateResult：flow 执行完喂给底座 { "gate":"g-c-e2e", "kind":"acceptance", "iter":2, "verdict":"PASS", "by":"test" } # 三类靠 kind 区分，共用 history + 流转 record_gate(result): state.history.append(result) match gate.kind: review | acceptance: if PASS: next = on_pass elif iter >= max_retries: ESCALATE else: next = on_fail # 回流 decision: next = branches[verdict].next # 分支 advance(resolve(next))
铁律：底座只记账、不判断本身。
AI 评审 / E2E 跑测 / 路线决策都在 flow 侧执行，结果喂回底座记录 + 解析下一节点 → 底座 lean、语言模型无关。

6
harness · 韧性
韧性与人机协同 —— 工程的「工程」质感
retry 断路
同一 gate 连续 max_retries（默认 3）不过 → 升级人工 / CEO，不无限打转。
reviewer 锁定 + tier 升级
重试时同一 reviewer 不漂移；复杂度涨了升模型 tier（sonnet→opus）。
decider 旋钮
每类 gate「谁来判」默认 AI/自动，编排可改 human 接管岔路口 —— 人有最终决定权、默认不挡路。
Auto Mode 停靠
自主推进时只在 decider: human 的 gate 停，其余 AI 自己过 —— 哪里停由编排决定。
Worker Preamble
员工执行任务不套娃 sub-agent（6 员工 × 多能力，防 token 爆炸）。
memory scope
shared/（全员共享）+ staff/<员工>/（私有），路径前缀隔离，CEO/CQO 可跨员工查。
「谁来判」是一个统一旋钮：低自主默认多挂 human、高自主（L4 自治）默认全 AI —— harness 把人机分工做成可配置参数，而非写死在流程里。

层谁编排位置改什么

L1 当下即可落地、可演示 —— 用户改一份 yml 就换一套 SOP，底座 R2/R3 直接吃合并后的配置。
