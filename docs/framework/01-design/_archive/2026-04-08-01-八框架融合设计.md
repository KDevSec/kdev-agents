## 三、各框架融合详情

### 3.1 BMAD 融合

| BMAD特性 | 融合位置 | 融合方式 |
|----------|----------|----------|
| 四阶段工作流 | Layer 1 规格引擎 | Phase 1-4 流程模型 |
| Gate Check | Layer 2 角色审查 | /solutioning-gate-check (≥90%) |
| 项目分级 | Layer 1 | Level 0-4 自动匹配流程 |
| Builder模块 | Layer 5 记忆扩展 | /create-agent, /create-workflow |
| Creative Intelligence | Layer 1 | /brainstorm, /research 命令 |
| Token优化 | 全架构 | Helper Pattern (70-85%节省) |

### 3.2 OMC 融合

| OMC特性 | 融合位置 | 融合方式 |
|---------|----------|----------|
| Team Pipeline | Layer 3 编排引擎 | plan→spec→exec→verify→fix 状态机 |
| 状态管理 | 全架构 | .kdev/state/ JSON持久化 |
| 修复循环 | Layer 3 | verify不通过→自动fix (maxAttempts) |
| HARD-GATE | Layer 1/2 | 人工门控点 |
| Handoff | Layer 5 | 跨会话状态恢复 |
| Notepad | Layer 5 | priority/working/manual记忆 |
| 模型路由 | Layer 3 | Haiku/Sonnet/Opus分层 |

**改造点：**
- ❌ 移除 tmux Workers → 改用 superpowers subagent-driven
- ❌ 移除 32 Agents → 精简为核心Agent
- ✅ 保留 Pipeline 状态机逻辑

### 3.3 GSD 融合

| GSD特性 | 融合位置 | 融合方式 |
|---------|----------|----------|
| Wave Execution | Layer 3 编排引擎 | 依赖感知并行执行 |
| Fresh Context | Layer 3 | 可选启用（复杂任务） |
| Context Engineering | Layer 3 | 50% context rule |
| Goal-Backward | Layer 1 规格引擎 | 需求分析方法论 |
| 极简理念 | 全架构 | 减少仪式活动 |
| Atomic Commits | Layer 3 | Task级别commit |

**不融合（基于用户选择）：**
- ❌ XML Prompt → 保持Markdown格式

### 3.4 ECC 融合

| ECC特性 | 融合位置 | 融合方式 |
|---------|----------|----------|
| Continuous Learning | Layer 5 记忆扩展 | 自动提取模式 |
| AgentShield | Layer 4 质量保障 | 安全扫描 + CVE检测 |
| 125 Skills | Layer 5 | 按需引入技能 |
| 12语言规范 | Layer 4 | 语言编码规范 |
| Multi-tool Support | 全架构 | 多平台兼容 |

### 3.5 speckit 融合

| speckit特性 | 融合位置 | 融合方式 |
|-------------|----------|----------|
| Constitution | Layer 1 | 项目原则文档 |
| Specify | Layer 1 | 需求规格生成 |
| Plan | Layer 1 | 宏观任务分解 |
| Tasks | Layer 1 | 微观任务详情 |
| 门控机制 | Layer 1/2 | HARD-GATE |

**流程映射：**
```
speckit Constitution → /kdev-init (创建项目原则)
speckit Specify → /kdev-spec (生成需求规格)
speckit Plan → /kdev-plan (任务分解)
speckit Tasks → TDD Step详情
```

### 3.6 openspec 融合

| openspec特性 | 融合位置 | 融合方式 |
|--------------|----------|----------|
| /opsx:propose | Layer 1 | 增量变更提案 |
| Delta Spec | Layer 1 | 变更规格文档 |
| /opsx:arch | Layer 1 | 架构设计 |

**流程映射：**
```
/kdev-change → openspec /opsx:propose
              → Delta Spec (变更规格)
              → 兼容性评审
              → 合并到主规格
```

### 3.7 superpowers 融合

| superpowers特性 | 融合位置 | 融合方式 |
|-----------------|----------|----------|
| brainstorming | Layer 1 | 需求澄清流程 |
| writing-plans | Layer 1 | TDD计划生成 |
| TDD | Layer 4 | RED-GREEN-REFACTOR循环 |
| subagent-driven | Layer 3 | Subagent并行执行 |
| code-review | Layer 4 | 代码审查流程 |
| verification | Layer 4 | 验证闭环 |
| systematic-debugging | Layer 4 | 调试流程 |

**核心TDD流程：**
```
RED: 写失败测试 → 运行确认失败
GREEN: 最小实现 → 运行确认通过
REFACTOR: 清理代码 → 保持测试通过
```

### 3.8 gstack 融合

| gstack特性 | 融合位置 | 融合方式 |
|------------|----------|----------|
| /plan-ceo-review | Layer 2 | CEO需求审查 |
| /plan-eng-review | Layer 2 | 架构审查 |
| /plan-design-review | Layer 2 | 设计审查 |
| /review | Layer 4 | 代码审查 |
| /qa | Layer 4 | QA测试 |
| /cso | Layer 4 | 安全审查 |
| /ship | Layer 4 | 发布准备 |

**角色审查流程：**
```
规格生成后 → /plan-ceo-review (商业价值)
          → /plan-eng-review (技术选型)
          → /plan-design-review (UI/UX)
实现完成后 → /review (代码质量)
          → /qa (测试覆盖)
          → /cso (安全漏洞)
```
