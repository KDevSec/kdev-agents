## 一、核心理念与设计目标

### 1.1 Compound Engineering 理念

> **核心理念**：Each unit of engineering work should make subsequent units easier—not harder.
> (每一单位工程工作应该让后续工作更容易，而不是更难)

```
传统工程 vs 复利工程
┌─────────────────────────────────────────────────────────────────────────────┐
│  传统工程: 每个功能 → 下一个更难                                             │
│  ├── 代码越多 = 边缘情况越多                                                │
│  ├── 更多依赖和耦合                                                        │
│  └── 积累技术债务                                                          │
│                                                                             │
│  复利工程: 每个功能 → 下一个更容易                                           │
│  ├── bug/测试失败/洞察 → 文档化                                            │
│  ├── 知识资产积累                                                          │
│  └── agent 可以复用先验知识                                                 │
└─────────────────────────────────────────────────────────────────────────────┘
```

**四步循环**:
```
Plan (80%) → Work (20%) → Review → Compound → Repeat
                                     ↑
                              知识沉淀 (记忆系统的核心)
```

### 1.2 设计目标

- **AI 主导执行**：从规格生成到代码实现的自动化
- **人主导规则**：关键决策点需要人工确认，确保方向正确
- **可追溯纠正**：任何阶段发现偏差都可回溯纠正
- **知识复利**：每次工作都让下次更容易

### 1.3 核心定位

```
"AI 主导执行，人来主导规则"

AI 负责:
  - 需求澄清（brainstorming 提问、方案探索）
  - 规格生成（PRD、Architecture、Story）
  - 计划分解（Task 分解、TDD 计划）
  - 代码实现（TDD 循环、RED-GREEN-REFACTOR）
  - 质量检查（代码审查、安全审计、测试）

人负责:
  - 设计审批（brainstorming 方案确认）
  - 规格验收（PRD、Architecture 确认）
  - 任务确认（Task 分解合理性）
  - 最终验收（功能、质量、安全）

关键决策点 → HARD-GATE → 人工确认
发现偏差   → questions-log → 回溯纠正
```

## 二、关键架构决策 (v3.0 更新)

### 2.1 HARD-GATE 定位

> **⚠️ 已被 [2026-04-15-02-KDev架构设计v3.0.2修订.md](./2026-04-15-02-KDev架构设计v3.0.2修订.md) §二（R-M3 强制节点 gate）替代，请以 v3.0.2 为准。** HARD-GATE 不再只位于 Phase 切换处，而是扩展为节点级强制 gate。

> **源码证据**: `superpowers-main/skills/brainstorming/SKILL.md`
> ```xml
> <HARD-GATE>
> Do NOT invoke any implementation skill, write any code, scaffold any project,
> or take any implementation action until you have presented a design and
> the user has approved it.
> </HARD-GATE>
> ```

**结论**: HARD-GATE 不是规格层，而是 **执行层入口门控**。

```
正确的 HARD-GATE 位置:
┌─────────────────────────────────────────────────────────────────────────────┐
│  Layer 2 规划层: /kdev:prd → /kdev:arch → /kdev:epics                        │
│                              ↓                                              │
│  Layer 3 方案层: /kdev:ready (就绪检查)                                       │
│                              ↓                                              │
│            ┌─────────────────────────────────────────────────────────────┐ │
│            │  HARD-GATE: 用户批准设计后方可进入实现阶段                      │ │
│            │  (不写代码、不调用实现skill、不scaffold)                        │ │
│            └─────────────────────────────────────────────────────────────┘ │
│                              ↓                                              │
│  Layer 4 实现层: /kdev:sprint → /kdev:dev → TDD循环                          │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 Gstack 多角色阶段评审

> **源码证据**: `gstack-main/AGENTS.md` + 23 个 SKILL.md 文件

**结论**: 每个开发阶段都有对应的评审角色介入。

| 阶段 | 评审 Skill | 角色定位 | 评审内容 |
|------|-----------|---------|---------|
| **Phase 2 规划** | `/plan-ceo-review` | CEO | 产品价值、10-star 产品定义 |
| | `/plan-eng-review` | Eng Manager | 架构、数据流、边缘情况、测试策略 |
| | `/plan-design-review` | Designer | 各设计维度评分 0-10 |
| | `/plan-devex-review` | DX Lead | 开发体验 |
| **Phase 4 实现** | `/review` | Code Reviewer | PR 代码审查，发现 CI 通过但生产失效的 bug |
| | `/design-review` | Designer | UI/UX 审计 + 修复循环 |
| **Phase 5 发布** | `/qa` | QA Lead | 真实浏览器测试，发现 bug，修复，验证 |
| | `/cso` | CSO | 安全审计 (OWASP Top 10) |

### 2.3 Gstack /learn + Compound Engineering 融合

> **源码证据**: `gstack-main/learn/SKILL.md`
> ```bash
> _LEARN_FILE="${GSTACK_HOME:-$HOME/.gstack}/projects/${SLUG:-unknown}/learnings.jsonl"
> ```
> Learning 类型: `pattern`, `pitfall`, `preference`, `architecture`, `tool`

**融合方案**:

```
Compound Engineering 四步循环:
Plan (80%) → Work (20%) → Review → Compound → Repeat
                              ↓         ↓
                         Gstack 评审    知识沉淀
                         /review        ↓
                         /qa         learnings.jsonl
                         /investigate   ↓
                              ↓      KDev 记忆层
                         gstack-learnings-log  questions-log.md
                         (自动记录洞察)        gotchas.md
```

**融合后的 KDev 命令映射**:

| Compound 步骤 | Gstack 对应 | KDev 记忆层 |
|--------------|------------|------------|
| **Review** | `/review`, `/qa`, `/investigate` | 运行时发现 |
| **Compound** | `gstack-learnings-log` 自动记录 | 持久化到 `.kdev/` |

#### 6.4.1 Bug Track 文档

**文件**: `.kdev/learnings/solutions/runtime-errors/jwt-expired-token.md`

```markdown
---
title: JWT Expired Token 处理
date: 2026-04-08
category: runtime-errors
module: auth
tags: [jwt, token, error-handling]
problem_type: bug
status: active
---

## Problem
JWT token 过期后，用户收到 401 错误但未自动刷新

## Symptoms
- 前端显示 "Session expired" 错误
- 用户需要重新登录
- 控制台输出 `TokenExpiredError: jwt expired`

## Solution
1. 在 axios 拦截器中检测 401 错误
2. 使用 refresh token 获取新 access token
3. 重试原请求

```typescript
// axios 拦截器
axios.interceptors.response.use(
  response => response,
  async error => {
    if (error.response?.status === 401) {
      const newToken = await refreshToken();
      error.config.headers.Authorization = `Bearer ${newToken}`;
      return axios(error.config);
    }
    return Promise.reject(error);
  }
);
```

## Prevention
- 设置 token 过期时间为 15 分钟
- refresh token 过期时间为 7 天
- 添加 token 即将过期的警告提示
```

## 十三、总结 (v3.0 更新)

### 核心价值

```
传统工程 → 技术债务积累 → 代码越多越难维护
复利工程 → 知识资产积累 → 代码越多越智能

KDev 记忆层 = 知识资产的"复利账户"
每次 Compound → 存入新知识 → 下次自动复用
```

| 功能 | 价值 | 复利效应 |
|------|------|----------|
| BMAD 四阶段 | 敏捷流程规范 | 每次开发都有章可循 |
| HARD-GATE | 执行层门控 | 防止跳过设计直接实现 |
| Gstack 阶段评审 | 每阶段专人评审 | CEO/Eng/QA/CSO 多视角把关 |
| questions-log | 决策追溯 | 避免重复讨论同一问题 |
| gotchas | 踩坑记录 | 新 agent 自动避开已知坑 |
| learnings | 模式/陷阱/偏好 | Gstack /learn 自动沉淀 |
| skills | 模式复用 | 新功能自动复用旧方案 |
| compound 命令 | 知识沉淀 | 每次工作都让下次更容易 |

### v3.0 架构决策总结

| 问题 | 结论 | 源码证据 |
|------|------|----------|
| HARD-GATE 定位 | **执行层入口门控**，非规格层 | `superpowers-main/skills/brainstorming/SKILL.md` |
| Gstack 多角色评审 | **每阶段都有对应评审** | `gstack-main/AGENTS.md` + 23 SKILL.md |
| /learn + Compound 融合 | **Review 阶段自动记录，Compound 持久化** | `gstack-main/learn/SKILL.md` |

### 融合策略总结

| 来源 | 融合策略 | 理由 |
|------|----------|------|
| **BMAD** | ✅ 核心底座 | 敏捷流程完整，门控机制成熟 |
| **Superpowers** | ✅ 执行层补充 | TDD、Subagent、HARD-GATE 更强 |
| **Gstack** | ✅ 质量保障 + 阶段评审 + /learn | 多角色评审、安全审计、学习沉淀 |
| **OMC** | ✅ 技术能力 | Hook、状态机、持久化 |
| **Speckit** | ⚠️ 可选 | BMAD 已覆盖其功能 |

### 设计哲学

- **精简** - 不存储噪音，只记录价值
- **可追溯** - 决策有据可查
- **可复用** - 模式提取为 Skill
- **Git友好** - 文本文件，版本控制
- **复利效应** - 每次工作让下次更容易 (Compound Engineering)
- **阶段把关** - 每个开发阶段都有对应评审角色 (Gstack 多角色)
