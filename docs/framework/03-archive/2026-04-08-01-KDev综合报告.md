# KDev Agent 综合研究报告

> 📅 2026-03-24
> 目标：打造一个「AI主导执行，人来主导规则」的SKILL集群或多智能体协同工作流
> 约束：整体框架简洁，开发工作流完整，不依赖 OpenClaw

---

## 一、背景与目标

### 1.1 为什么需要 AI 编码工作流？

当前 AI 编码工具（Claude Code、Codex、Cursor 等）的核心问题是：**缺乏系统化的编排和质量管理**。

- 单个 AI agent 可以写代码，但无法自我审查
- 没有规格驱动，AI 容易偏离需求
- 多 agent 并行需要编排引擎
- 跨会话记忆缺失，每次从零开始

### 1.2 核心愿景

**「AI主导执行，人来主导规则」**

- 人定规则（spec/架构/标准）
- AI 执行（编码/测试/部署）
- 质量闭环（review/安全/监控）

### 1.3 调研范围

本次调研覆盖 **11 个项目**，从 41k 到 81k star，涵盖：
- 多 agent 编排引擎
- 规格驱动开发框架
- AI 编码技能市场
- 单 agent 角色扮演系统
- 全家桶式 AI 开发平台

## 三、五大方案深度对比

### 3.1 gstack（41k ⭐）

**作者**：Garry Tan（YC CEO）
**思路**：一个 Claude Code 扮演多个角色（CEO/设计师/工程经理/QA/安全官等）

**28 个斜杠命令**：
- `/plan-ceo-review` — CEO 需求审查
- `/plan-eng-review` — 工程师架构审查
- `/review` — 代码审查
- `/qa` — 质量保证
- `/ship` — 发布准备
- `/cso` — 安全审查
- 等等...

**优势**：
- 简单易用，30 秒安装
- 一个人 = 一个团队
- 角色 prompt 设计精良

**局限**：
- 单 agent，角色间靠 prompt 切换
- 并行能力弱（串行执行）
- 没有规格驱动
- 没有记忆系统

### 3.2 oh-my-claudecode / OMC（11k ⭐）

**作者**：Yeachan-Heo
**思路**：多 agent 编排，多个 Claude Code 实例并行协作

**架构核心**：
```
Team Pipeline: plan → prd → exec → verify → fix
Worker Pool: tmux 多面板管理多 agent
Task Router: 任务分发 + 结果聚合 + 冲突解决
```

**优势**：
- 唯一有完整编排引擎的
- TypeScript 代码可改造
- 可独立运行（不绑定 Claude Code 插件）

**局限**：
- 配置复杂
- 依赖 tmux
- 没有规格驱动
- 记忆系统基础

### 3.3 GitHub spec-kit（81k ⭐）

**作者**：GitHub 官方
**思路**：Spec-Driven Development（规格驱动开发）

**流程**：
```
Constitution（项目原则）
→ Specify（需求规格）
→ Plan（技术设计）
→ Tasks（任务分解）
→ Implement（实现）
→ Review（审查）
```

**优势**：
- 最成熟的规格驱动框架
- GitHub 官方维护
- 流程模型优秀

**局限**：
- 偏流程管理，AI 执行层面较弱
- Python 实现，和 TypeScript 生态不统一
- 没有编排引擎
- 没有角色审查

### 3.4 OpenSpec（33k ⭐）

**作者**：Fission AI
**思路**：Spec-Driven Development for AI coding assistants

**核心命令**：
- `/opsx:propose` — 提出需求方案
- `/opsx:arch` — 生成架构设计

**优势**：
- 专为 AI 设计的规格框架
- 支持 Claude Code / Cursor / Codex
- TypeScript 实现

**局限**：
- 需要学习新的工作流
- 偏前端（Next.js 技术栈）
- 没有编排引擎

### 3.5 Superpowers（716 ⭐）

**作者**：obra
**思路**：Claude Code 插件市场

**关键插件**：
- **TDD** — 测试驱动开发
- **头脑风暴** — 思维发散
- **Subagent-Driven-Development** — 子 agent 驱动开发
- **研究优先** — 先研究再实现

**优势**：
- 模块化，按需安装
- TDD 流程设计精良
- 社区驱动

**局限**：
- 质量参差不齐
- 依赖 Claude Code 插件系统
- 没有编排能力

## 四、补充项目分析

### 4.1 everything-claude-code (ECC)（50k ⭐）

**规模**：125 skills / 28 agents / 12 种语言规则
**核心价值**：技能 + 规则 + 记忆 + 安全

**关键特性**：
- **Continuous Learning**：自动提取模式，跨会话持久化
- **AgentShield**：安全扫描 + CVE 检测
- **12 种语言编码规范**：TypeScript/Python/Go/Java/Rust/Swift 等
- **多工具支持**：Claude Code / Codex / Cursor / Kiro / OpenCode

**局限**：
- 没有编排引擎
- 系统庞大，改造成本高

### 4.2 wshobson/agents（32k ⭐）

**规模**：73 plugins / 180 agents / 212 skills / 92 commands
**核心价值**：粒度化插件 + 编排模式

**关键特性**：
- **Agent Teams**：文件所有权隔离的并行编排
- **Conductor**：Context→Spec→Plan→Implement 工作流
- **三层模型路由**：Haiku（简单）/ Sonnet（中等）/ Opus（复杂）

**局限**：
- 纯 Markdown，无代码
- 完全绑定 Claude Code 插件系统
- 没有记忆系统

### 4.3 OpenCode（28.5k ⭐）

**定位**：AI 终端 IDE（Fission AI）
**技术栈**：Go + React + esbuild
**特点**：独立终端应用，不依赖 Claude Code 插件

**对我们参考价值**：
- 终端 UI 设计
- 插件架构设计
- 但整体太独立，不适合做基座

### 4.4 三层模型路由（wshobson 的独特设计）

```
Haiku → 简单执行（格式化、翻译、简单问答）
Sonnet → 中等推理（日常编码、中等复杂度）
Opus → 复杂决策（架构设计、复杂推理、安全审查）
```

这个设计可以节省成本，值得融合。

## 五、基座选择：为什么是 OMC

### 5.1 三方对比评分

| 维度 | 权重 | OMC | ECC | wshobson |
|------|------|-----|-----|----------|
| 编排能力 | 25% | ⭐⭐⭐⭐⭐ | ⭐ | ⭐⭐⭐⭐ |
| 技能生态 | 20% | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 规格工作流 | 15% | ⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 记忆系统 | 10% | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ |
| 可改造性 | 15% | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| 多工具支持 | 10% | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 安全 | 5% | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **加权总分** | | **3.65** | **3.20** | **2.85** |

### 5.2 OMC 胜出的四个原因

1. **唯一有完整编排引擎的** — 编排是整个系统的核心，其他两个都没有
2. **TypeScript 代码可改造** — ECC 太大改不动，wshobson 是纯 Markdown 没代码
3. **可独立运行** — 不绑定 Claude Code 插件系统
4. **改造成本最低** — 去掉 tmux 就能用，其他模块直接复用

### 5.3 OMC 精简方案

从 **770 个文件**精简到 **~50-80 个文件**：

**保留的**：
- Pipeline 引擎（`src/team/pipeline.ts` + `dispatch-queue.ts` + `task-router.ts`）
- Worker 管理（`worker-bootstrap.ts` 改造为 child_process）
- 6 个核心 Agent 定义（architect / developer / reviewer / qa / security / doc）

**砍掉的**：
- 30+ skills → 只保留 2-3 个核心
- tmux 依赖 → child_process
- HUD 显示
- Autopilot / Ultrawork / Ralph 模式
- CCG 三模型顾问
- 自动研究模式（改造为独立的 Research Mode）

## 七、研究模式（Research Mode）

### 7.1 设计理念

给一个研究主题 → 自动搜集信息 → 深度对比分析 → 输出详细方案 → 人决策

### 7.2 工作流

```
/kdev:research "研究主题"
  → 阶段 1：信息搜集（web search + 拉源码 + 读文档）
  → 阶段 2：深度分析（对比维度、评分、关键发现）
  → 阶段 3：方案输出（多方案对比 + 推荐 + 实施计划）
  → 阶段 4：人工审阅（等用户反馈后迭代）
```

### 7.3 研究模式 vs 实现模式

| 维度 | Research（研究） | Implement（实现） |
|------|-----------------|-------------------|
| 输入 | 研究主题/问题 | 具体需求/规格 |
| 过程 | 搜集→分析→方案 | 规格→设计→编码→测试 |
| 输出 | 文档 + 推荐 | 代码 + 测试 |
| 人工角色 | 审阅+决策 | 定规格+审代码 |

### 7.4 与规格系统的集成

研究模式是规格驱动的**上游**——先研究清楚，再定规格：
```
/kdev:research → 输出推荐方案 → 变成 spec.md → /kdev:plan → /kdev:implement
```

## 九、核心理念总结

### 「AI主导执行，人来主导规则」

| 角色 | 职责 |
|------|------|
| **人** | 定规格、定架构、定标准、做决策 |
| **AI** | 写代码、跑测试、做研究、出方案 |
| **系统** | 编排并行、角色审查、质量闭环、记忆持久化 |

**三个核心工作流**：
1. **Research**（研究）→ AI 搜集分析，人审阅决策
2. **Specify**（规格）→ 人主导，AI 辅助
3. **Implement**（实现）→ AI 主导，人审代码

**不是从一个基座出发，而是 OMC 提供骨架（编排），其他项目提供器官（技能/规则/记忆/工作流）。**
