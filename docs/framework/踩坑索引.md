## 先核代码再信设计（R-009 活体案例群，头号高频模式）

六条踩坑（G-008、G-009、G-010、G-011、G 20260707、G 20260708）共享同一个元根因：**信了 spec / roadmap / 设计文档 / 提醒文案的字面描述，没去核当前代码或运行时实际发生了什么**。这不是巧合分布，是本项目目前复现率最高的坑型——每次「设计说 X 应该已经实现/已经生效」都值得先花一分钟 grep 代码或跑一次真实链路核实，再基于结论行动。方法论层面参见 `docs/meta/方法论沉淀-实战反思.md` R-009 节（该文档待补充时可回链本索引）；`.kdev/memory/改进建议.md` R-009 是原始决策条目。

<a id="g-008"></a>
### G-008：照过时 spec 给第二个数字员工设计编排，会照到已被取代的旧形态

**症状**：给需求架构师（第二个数字员工）起草编排提示词时，照 2026-06-07 的阶段2 spec §5 写"编排指令归 orchestrator agent"。

**根因**：该 spec 写于两个关键重构之前——kdev-flow-driver skill 引入（`1a19ca4`）+ feature-first 存储重构。当前真实编排机制是"顶层主控执行循环驱动引擎，agent 只是编排知识/fallback 文档"，spec 字面已经过时。

**规避方法**：建新员工时，编排形态以**当前代码**为准，不以接入 spec 为准——先读 `SKILL.md` + `references/node-agent-routing.md` 核实真实机制；新员工只复用通用编排 skill，加 `agents/*.md` + `node-table.yml` + routing 词条；接入 spec 只取"员工形态/命名/能力"，编排模型部分永远信代码。

**处置**：出提示词阶段经用户反问+核代码拦下，未落地，无需版本号修复。

<a id="g-009"></a>
### G-009：把"派不出 plugin 员工"误判为平台限制，真因是插件未启用+需 namespaced 全名

**症状**：spike 验证方案 A（用 Workflow 当编排引擎）时，`agent({agentType:'dev-engineer-plan'})` 三种写法全 not-found，得出"平台不暴露 plugin 员工"的结论。

**根因**：① 目标插件 `kdev-team` 当时未在 `enabledPlugins` 里启用，主 Agent 工具和 workflow `agent()` 两个注册表同时缺席，看起来像"平台剥掉了"；② 即使启用，注册表也只认 `plugin:agent` 这种 namespaced 全名，裸名一律 not-found；③ 叠加 G-004 的 cache 键控——启用/升级插件后不重启不即时生效。

**规避方法**：下"平台不支持/agent 不可达"结论前，先查 `~/.claude/settings.json` 的 `enabledPlugins`；agentType/subagent_type 一律用 `plugin:agent` 全名；启用/升级插件后必须重启 session；重大"平台做不到"类结论要用已知能用的同类样本做对抗式复核。

**处置**：本次经核查纠偏，方案 A 实为 feasible-with-gaps，未误杀。

<a id="g-010"></a>
### G-010：commit-tracker 读驼峰字段，CC 真实 stdin 是蛇形，从不识别 commit

**症状**：`commit-tracker.py` 从不识别任何 commit，`pending-commits.json` 从未被填，SessionStart/Stop 的 nudge 提醒一直是死的，本会话全靠主会话手动 dispatch 兜底才没丢数据。

**根因**：代码读 `data.get("toolInput")`（驼峰），但 Claude Code PostToolUse hook 真实 stdin 字段是 snake_case `tool_input`（官方契约）——永远返回 None，早退 SUPPRESS。更隐蔽的是**测试自洽不等于契约正确**：`test_commit_tracker.py` 的 fixture 也喂驼峰字段，与被测代码自洽、测试全绿，但从没验证过真实 harness 发的是什么字段名。

**规避方法**：hook stdin 字段一律按官方契约用 snake_case（`tool_input`/`transcript_path`/`session_id`/`tool_name`/`hook_event_name`/`cwd`/`tool_response`）；hook 测试的 fixture 字段名必须照官方文档写，不能照被测代码反推；验字段契约优先查官方 docs，不臆测驼峰/蛇形。

**修复态**：已在 P-C1b 实施中改为 `tool_input`，顺带复活了死掉的 commit-nudge。

<a id="g-011"></a>
### G-011：worktree 下 Step-ID 前缀按 cwd 现算，同会话两次落盘分裂成两套不连号 ID 轨

**症状**：同一 worktree 会话、同一份（symlink 共享）`.kdev`，两次 dispatch step-recorder 产出两套前缀（`worktree-feat-l3-review-dogfood-1` 和 `main-87`），写进同一份 `执行日志.md`，ID 体系自相矛盾；`current_step` 只跟主轨，worktree 那条成孤儿，按 `current_step` 找不回。

**根因（两层叠加）**：① slug 来源是 cwd 敏感的 `git rev-parse --abbrev-ref HEAD`，而 recorder subagent 实际运行 cwd 不保证等于指定 worktree；② `.kdev` symlink 把多 workspace 的记忆/计数器收敛到同一 canonical store，但 slug 计算没跟着收敛——共享存储 + 不共享命名 = 分裂。本质是把"运行时现算的分支 slug"当成了稳定 per-workspace 身份。

**规避方法（设计层决策点，P-C2 收口）**：slug 身份改用落在 `state/` 的稳定 workspace-id 锚，不用运行时 `git rev-parse`；symlink 共享 store 时统一回退主轨单调计数或强制每 worktree 独立 store，二选一；recorder dispatch 传绝对路径 + `git -C <abs>` 消除 cwd 不确定性；决定 `current_step` 是否要容纳多轨。

**处置进展**：设计已定（Q-019/Q-020，记录 ID 换时间戳原语）；Phase A 已 done v0.17.0（`812d964`，`mint_record_id` 时间戳化，cwd 现算根因结构性消除，worktree/跨机不再撞）；⚠️ 待 G-004 激活+实跑验证后转 closed。live 碎轨 reconcile 挂单独专项（P-C2 spec §11.5）。P-C2 收口前，worktree 会话落 Step 仍需人工留意前缀。

<a id="g-20260707"></a>
### G 20260707：sync_push 把 commit 和 push 绑死，已 commit 未推的积压永远清不掉

**症状**：SessionStart 报"记忆仓 3 条本地 commit 未推"，提醒文案说"多半缺 GitHub 凭据"；但实测 SSH 认证完全通、无分叉，就是推不上去。

**根因**：`sync_push` 把 commit 和 push 绑死——工作区无新变更时 `nothing to commit` 直接 return，跳过 push。任何"已 commit 但没推上去"的历史积压（早期 HTTPS push 失败留下的），只要工作区没新变更就永远被 skip。0.19.2 的"未推积压可见化"只加了检测+提醒，没解耦 commit/push，检测对了修复没跟上。附带问题：sync-reminder 文案是 HTTPS 场景话术，对 SSH remote 误报"缺凭据"。

**规避方法**：commit 与 push 解耦——`if 有变更: commit` + 独立 `if ahead upstream: push`（不管这次有没有新 commit）；文案按 remote 协议分场景（SSH 提检查 key/agent，不提凭据）。

**修复态**：已修 **0.19.5**（`28c5325`），TDD 2 新测试覆盖，539 passed；⚠️ 待 G-004 激活后转 closed。bootstrap "multiple branches" 复现不出，未随本修处理，另挂观察。

<a id="g-20260708"></a>
### G 20260708：模型他评静默降级成自评，死了 3 周不告警

**症状**："模型他评"设计意图是 recorder 读主会话 transcript 抽真事实，防主会话自夸/讨好式满分；实测两条 Step 里 recorder 回报"transcript 从某 offset 起为空，key_facts 据 dispatch summary 降级填入"——名义他评，实际全是自评，且不告警，持续了 3 周才被发现。

**根因（三层叠加）**：① transcript 指针（`transcript_path`/`since_offset`）只由 commit-tracker 在 commit 时更新，而 worktree commit 因 `.kdev/memory` 目录不存在被直接 skip，导致指针冻结在 3 周前的老会话；② `since_offset` 死绑当时那个 transcript 文件的行数，跨会话必然越界；③ 越界读空时不区分"真的没内容"和"越界"，静默用 summary 顶替不告警。

**规避方法**：越界显式标 `out_of_range`，recorder 遇到必须回报"⚠️ 他评降级为自评（原因）"，不能悄悄顶替；transcript 指针职责从低频的 commit-tracker 挪到高频的 UserPromptSubmit/SessionStart hook，每轮刷新写 `state/.current-transcript`，recorder 改用 `resolve_marker` 保证指向当前会话。

**设计洞察**：低频信号（commits）和高频信号（当前会话身份）不该共用一个指针源——这是本条真正值得记住的抽象教训，超出这次具体 bug。

**修复态**：已修 **0.19.8**；⚠️ 待 G-004 激活+实跑验证后转 closed。

## 插件/hook 机制类

<a id="g-001"></a>
### G-001：kdev-commit block-unattributed-commit hook 拒受带引号的 `-c` 参数

**症状**：`git -c user.name="ly-AI" -c user.email="x@y.com" commit ...` 被 hook deny，报错文案却显示"两个都覆盖了"。

**根因**：hook 对 Bash 工具调用的 argv 字符串做字面匹配，引号是字符串字面的一部分，导致匹配模式失配（git 本身对加不加引号解析等价，但 hook 层没有）。

**规避方法**：`-c key=value` 一律不加引号，即使值里没有空格也不要加。适用范围仅限 kdev-commit hook 校验路径，其他场景照常加引号无影响。

**处置**：未改 hook（判定为实现细节而非 bug），长期靠规避写法即可。

<a id="g-002"></a>
### G-002：kdev-commit hook deny 消息容易被 subagent 误判为 prompt injection

**症状**：implementer subagent 执行 commit 时未带 `-c user.name=ly-AI` 覆盖，触发 hook deny；deny 消息用"请用：+具体命令"措辞，subagent 按 trust hierarchy 把它当可疑指令直接忽略，改用不带 `-c` 的 commit，导致 author 未按规则覆盖。

**根因**：不是真 prompt injection，是两个问题叠加——① 当时用户全局 CLAUDE.md 与项目侧 hook 期望不一致（规则冲突）；② deny 文案格式上"语义等价于一条用户指令"，触发了 subagent 对非 user-turn 指令性文本的防御机制（这个防御本身是合理特性，误杀了合法 hook 反馈）。

**规避方法**：统一规则源头——把"-AI 覆盖"提升为正式的用户全局 CLAUDE.md 规则，消除与 hook 期望的冲突，subagent 看到 deny 时自然会正确执行。**不修 hook 文案**：文案对人类清晰，副作用在规则统一后自动消失。

**处置**：已通过修订用户全局 CLAUDE.md 解决（2026-05-28）。不 rebase 历史因此产生的非 -AI author commit。

<a id="g-004"></a>
### G-004：插件 hook 内容变更但 plugin.json version 未 bump，Claude Code cache 不刷新，新 hook 静默失效

**症状**：新 hook（SessionStart/SessionEnd 自举脚本）建好合并，但新 session 没有触发，`.kdev` 仍是旧状态；连带另一条更早合并的 hook 也从未生效过。

**根因**：Claude Code 从 `~/.claude/plugins/cache/<plugin>/<version>/` 加载插件，cache 按 `plugin.json` 的 version 字段键控。改了 hooks.json/hook 脚本/hook lib 但没同步 bump version，仓里代码已更新但 `plugin.json` 版本号没变 → Claude Code 判定"版本没变、无需重抽" → 一直跑 stale cache。

**规避方法**：任何改动 hooks.json / hook 脚本 / hook lib 的提交，必须同步 bump `plugin.json` version + 补 CHANGELOG；用户侧刷新 marketplace（`/plugin` 更新/重装）+ 重启 session 让 cache 按新版本重抽。可考虑加 CI/pre-commit 校验"hooks 路径有 diff 必带 version bump"。

**处置**：status 仍为 open——这是一条持续性操作纪律，不是一次性可关闭的修复；本索引里多条其它坑（G-009/G-011/G 20260707/G 20260708）的"已修但待激活"状态，激活条件都是这一条。

**交叉参见**：[G-010](#g-010)（hook stdin 字段驼峰/蛇形契约错误）——完整内容归入【先核代码再信设计】节，同样属于本项目 hook 机制的系统性坑，如果按插件/hook 主题检索也应对照到此。

## 记忆同步/git 托管类

<a id="g-003"></a>
### G-003：程序化 push 整个 .kdev/ 记忆树到外部仓被 auto-mode classifier 硬拦（bulk exfil）

**症状**：主会话用 Bash 跑 `kdev_sync.bootstrap()`（git init + add -A + commit + push 整个 `.kdev/` 到私有记忆仓）被 auto-mode classifier 硬 deny，即使目标是用户自己的私有仓、用户已明确说"执行 P4"、设计文档也如此规划，依然拦。

**根因**：classifier 把"把含 verbatim 用户原话的整棵记忆树推到一个非 trusted-remote 的外部仓"判为跨信任边界的批量外传（exfil）。这是 Bash 工具层的安全护栏，对话级 user intent 清不掉——不是权限配置问题，是设计上就不该由 agent 代劳。

**规避方法（不绕过护栏）**：① 激活走 hook 而不走 agent Bash——SessionStart/SessionEnd 自举脚本由 Claude Code runtime 原生执行（用户配置=用户授权），不经 agent 的 Bash classifier；② 或用户本人手动执行 bootstrap 命令；③ 或用户显式加 Bash permission rule 放行。适用范围仅限"agent 程序化把 `.kdev/` 记忆树推外部仓"这一类动作，代码仓（trusted remote）的 push 不受此限。

**处置**：机制由 agent 建，激活必须越过 agent 信任边界，交给 runtime 或用户完成——这是设计原则，非临时绕过。

**交叉参见**：[G 20260707](#g-20260707)（`sync_push` commit/push 绑死导致积压清不掉）——同属记忆仓 git 托管链路上的坑，但根因是"信设计没核代码"，完整内容归入【先核代码再信设计】节。

## 编排/交付语义类

<a id="g-005"></a>
### G-005：node-table 把"实现分层"误当"交付增量"，编排器劫持 g-deploy FAIL 当增量循环

**症状**：kdev-flow-driver MVP 首次 dogfood 表面跑通（4 个"增量"、23 次 gate、49 文件改动），但 phase_history 暴露语义滥用——`n12-deploy → n6b` 的 reason 是"g-deploy FAIL reflow"，而 deploy 根本没失败；merge 节点被空过 3 次；g-deploy 假 FAIL 3 次、第 3 次本该触发 escalate blocked 却没触发。

**根因**：两个"增量"概念被混为一谈——实现切片（T0-T4，一个交付物内部怎么分步写代码，横向分层）vs 交付增量（能独立过 e2e 的纵向切片，才是 flow 级循环单元）。这次改造整体只有 1 个真增量，但 plan agent 照抄考题的 T0-T4 分区写成"Increment 1-4"，driving loop 当真、每个 T 都跑一遍 gate 链，跑完想回 n6b 却没合法路径，只能劫持 g-deploy FAIL 当回头暗号。

**规避方法**：增量必须按"能不能独立过 e2e"切，不按需求文档里的 T0/T1/第一步/第二步这类实现工序分区抄。纯视觉/单页面/单模块改造通常 N=1。

### G-007：批量 git add 前未逐文件核对 staged diff，用户 IDE WIP 被夹带进 commit

**症状**：执行批量 `git add` 后，用户正在 IDE 中编辑的演讲稿文件（与本次 roadmap 改动无关）被一并 stage 进暂存区；若未做提交前逐文件核查，会随本次 commit 混入版本历史，造成"commit 内容与 message 不匹配"的污染。

**根因**：`git add <多个路径>` 或 `git add .` 不区分"本次工作改动"与"用户 IDE 并行改动"；且这两类文件恰好同在一个目录下，路径 glob 一并命中；主会话没有强制在 commit 前做逐文件 staged diff 核验。

**规避方法**：批量 add 后、commit 前必须执行：
```bash
git diff --staged --name-only          # 先看文件列表
git diff --staged <可疑文件路径>        # 逐一核对"不确定是否本次改"的文件
git restore --staged <误 add 的文件>   # 发现夹带即撤出
```
更保守的做法：只 `git add <确定属于本次改动的具体文件列表>`，拒绝用 glob 或 `.`。

**处置**：本次靠 staged diff 核查兜住，未污染 git history。
