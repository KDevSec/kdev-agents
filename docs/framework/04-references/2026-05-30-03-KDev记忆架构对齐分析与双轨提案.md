## 一、对齐基线：v3.0 已规划的 5 层记忆

来自 `2026-04-08-03-KDev融合架构设计.md §6.1`：

```
Memory Level 0: 运行时记忆 (借鉴 OMC notepad)
Memory Level 1: 知识图谱 (active/superseded 生命周期)
Memory Level 2: 学习沉淀 (solutions / knowledge 双轨)
Memory Level 3: 项目记忆 (questions-log / gotchas / session-state / daily-logs)
Memory Level 4: 技能库 (Claudeception 自动提取)
Memory Level 5: 全局记忆 (~/.kdev/ 跨项目)
```

---

## 二、当前 kdev-memory 实际覆盖 vs 5 层规划

| Memory Level | 规划 | 当前 kdev-memory 实际 | Gap |
|---|---|---|---|
| **0 运行时记忆** | OMC notepad 三层（priority/working/manual） | ❌ 无 | **整层缺** |
| **1 知识图谱** | `topics/<topic>/items.json` + active/superseded | ❌ 无 | **整层缺** |
| **2 学习沉淀** | `learnings/solutions/` + `learnings/knowledge/` + `learnings.jsonl` | ⚠️ R-NNN 改进建议算半个 | **大部分缺** |
| **3 项目记忆** | questions-log / gotchas / session-state / daily-logs | ✅ Q-NNN / G-NNN / 当前状态.md / 每日汇总/ | **已建** |
| **4 技能库** | SKILL.md + Claudeception 自动提取 | ⚠️ F-NNN 算反向（skill 吐槽存档） | **自动提取缺** |
| **5 全局记忆** | `~/.kdev/` 跨项目偏好 / 纠正 | ❌ 无 | **整层缺** |

**结论**：当前 kdev-memory plugin 实际只覆盖 **Memory Level 3 子集**。其他 4 层多数本来就是 staff 时代（kdev-staff 装入业务项目）要补的。

## 三、staff 架构对记忆的硬需求

来自 `2026-05-28-02-KDev-staff-整体架构-v0.1.md §1.4 + §3.1`：

| 路径 | 谁管 | 装 plugin 数 | 内容 |
|---|---|---|---|
| `.kdev/state.md` | kdev-core | core | 当前 CEO 权威状态 |
| `.kdev/events.log` | kdev-core | core | 事件流 |
| `.kdev/hud.md` | kdev-core | core | HUD 实时面板 |
| `.kdev/staff-memory/review-memory.md` | kdev-core | core | 评审 R-NNN（评审能力维度漏检）|
| `.kdev/staff-memory/reviews/<节点>-<能力>/评分-N.md` | kdev-core | core | 评审记录 |
| `.kdev/staff-memory/flow-config.yml` | kdev-core | core | 评审流程编排 |
| `.kdev/handoffs/<员工名>/` | kdev-core | core | 员工产物总线 |
| `.kdev/checkpoints/` | kdev-core | core | PreCompact 快照 |
| `.kdev/memory/*` | **kdev-memory** | **memory** | 工程过程记忆（本 plugin 范围）|

**关键约束**（doc §3.3）：

| 维度 | kdev-staff 依赖 kdev-core | kdev-staff 依赖 kdev-memory | kdev-memory 单独可用 |
|---|---|---|---|
| 必需 | ✅ 是 | ⚠️ 推荐但非必需 | ✅ 是 |
| 装机 | 必须装 | 可选 | 可独立装 |

**用户在对话中明确的方向调整**（2026-05-30）：

> "kdev-memory 肯定要和 staff 共用的，但是 memory 是可以单用的，而 staff 必须要有依赖才能使用，比如 core、memory，或者以后还延伸出其他的依赖。"

**重新表述**：

- kdev-memory：**可单装可联装**。单装提供 Memory Level 0-5 全部能力的子集；联装时 staff 也可读它的数据。
- kdev-staff：**必须依赖 kdev-core**，可选依赖 kdev-memory（如果装了就用，没装就只用 staff-memory）。

## 四、人 vs AI 双轨设计提案

### 4.1 哲学（来自 OMC + Letta）

**人 vs AI 不是物理分目录，而是写入语义 + 读取语义的分轨**：

| 维度 | 人主轨 | AI 主轨 | 双方都可 |
|---|---|---|---|
| **写入方** | 用户手改 markdown | hook 自动写 / skill 写 | notepad / state 字段（通过工具调用） |
| **读取方** | 用户开 IDE 看 | UserPromptSubmit hook 注入 / brief 召回 | 都读 |
| **格式倾向** | markdown / YAML 易读 | JSON / 强类型 schema | markdown + frontmatter |
| **生命周期** | 用户主动归档 | 自动过期 / 自动 supersede | 双方都可改 status |

### 4.2 物理结构提案

```
.kdev/memory/
│
├── notepad/                          ← Memory Level 0：运行时记忆（OMC 三层）
│   ├── priority.md                   ← 人为主写（始终加载，< 500 字符）
│   ├── working.md                    ← 人/AI 都可写（7 天自动清理）
│   └── manual.md                     ← 人为主写（永不过期）
│
├── knowledge-graph/                  ← Memory Level 1：知识图谱
│   └── topics/
│       └── <topic>/items.json        ← AI 抽取 / 人确认 + active/superseded
│   (引擎层借用 kdev-code-graph 的 UA bridge —— 见 04 文档)
│
├── learnings/                        ← Memory Level 2：学习沉淀
│   ├── solutions/                    ← bug fix 文档（AI 写，人审）
│   │   ├── build-errors/
│   │   ├── runtime-errors/
│   │   └── performance-issues/
│   └── knowledge/                    ← best practice / workflow（AI 写，人审）
│       ├── best-practices/
│       └── workflow-patterns/
│
├── 决策日志.md                        ← Memory Level 3：现有 Q-NNN（人/AI 共写）
├── 踩坑日志.md                        ← Memory Level 3：现有 G-NNN（人/AI 共写）
├── 执行日志.md                        ← Memory Level 3：现有 Step（人/AI 共写）
├── 改进建议.md                        ← Memory Level 3：现有 R-NNN（人/AI 共写）
├── skill-feedback.md                 ← Memory Level 3：现有 F-NNN（人/AI 共写）
├── 当前状态.md                        ← Memory Level 3：现有 frontmatter + body
├── 每日汇总/YYYY-MM-DD.md            ← Memory Level 3：现有
│
├── skills/                           ← Memory Level 4：自动提取的 skill
│   └── <skill-name>/SKILL.md         ← Claudeception 模式 + 触发关键词
│
├── state/                            ← AI-only 运行时状态（仿 OMC）
│   ├── recall-cache.json             ← UserPromptSubmit 召回缓存
│   ├── distill-status.json           ← 自动蒸馏状态
│   └── sessions/<sessionId>.json     ← 跨会话 handoff（PreCompact 写）
│
├── dataset/                          ← AI-only 蒸馏切片输出（现有）
│   ├── dataset-full.md
│   ├── dataset-misalignment.md
│   └── dataset-skill-feedback-by-subject/
│
└── config.yaml                       ← 用户配置（现有）

~/.kdev/                              ← Memory Level 5：全局记忆（跨项目）
├── user-preferences.md               ← 偏好（人写）
└── global-corrections.md             ← 全局纠正（人/AI 共写）
```

### 4.3 每个 Level 的人/AI 主写归属

| 文件/目录 | 人主写 | AI 主写 | 双方都写 | 备注 |
|---|---|---|---|---|
| `notepad/priority.md` | ✅ | - | - | 始终加载，人为关键 reminder |
| `notepad/working.md` | - | - | ✅ | 临时事项，AI 自动 7 天清 |
| `notepad/manual.md` | ✅ | - | - | 用户显式 reminder，永不过期 |
| `knowledge-graph/topics/*/items.json` | - | - | ✅ | AI 抽取，人审 + 改 status |
| `learnings/solutions/*.md` | - | ✅ | - | AI 写 bug fix 文档，人审 |
| `learnings/knowledge/*.md` | - | ✅ | - | AI 提炼最佳实践，人审 |
| `决策日志.md` (Q-NNN) | - | - | ✅ | 用户决策 + AI 提选项 |
| `踩坑日志.md` (G-NNN) | - | - | ✅ | AI 实时落，人事后改 status |
| `执行日志.md` (Step) | - | - | ✅ | 评分双轨：模型自评 + 用户评 |
| `改进建议.md` (R-NNN) | - | ✅ | - | AI 反思，人审 |
| `skill-feedback.md` (F-NNN) | - | ✅ | - | AI 识别 5 类语义自动起草，人确认 |
| `当前状态.md` frontmatter | - | ✅ | - | hook 解析，AI 维护 |
| `当前状态.md` body | ✅ | - | - | 人读为主，narrative |
| `每日汇总/*.md` | - | ✅ | - | AI 从其他文件聚合，人审 |
| `skills/*/SKILL.md` | - | ✅ | - | learner 自动提取，人审 |
| `state/*.json` | - | ✅ | - | hook / skill 写，人不动 |
| `dataset/` | - | ✅ | - | distill 命令写 |
| `~/.kdev/user-preferences.md` | ✅ | - | - | 跨项目偏好，人写 |
| `~/.kdev/global-corrections.md` | - | - | ✅ | 双方都写 |

### 4.4 哪些是"AI 直接写不需要确认"

继承现有 CLAUDE.md 的授权（项目根 CLAUDE.md "关键授权"段）：

> `.kdev/memory/` 下的所有文件 **不需要逐条请示**即可写入

新增授权清单：

- ✅ `notepad/working.md` —— AI 直接 append
- ✅ `learnings/{solutions,knowledge}/*.md` —— AI 写第一稿
- ✅ `state/*.json` —— AI 全程管理
- ✅ `skills/*/SKILL.md` —— learner 写
- ✅ `dataset/*` —— distill 命令写

需要用户介入的：

- ⚠️ `notepad/{priority,manual}.md` —— AI 建议，人决定（避免污染常驻 context）
- ⚠️ `knowledge-graph/topics/*/items.json` 中 `status` 字段从 active → superseded —— 至少 medium confidence 才自动改，low confidence 必须问

## 五、staff 架构兼容（与 kdev-core 共存）

### 5.1 物理隔离不合并

```
.kdev/
├── memory/           ← kdev-memory plugin（本文档 4.2 结构）
├── staff-memory/     ← kdev-core plugin（review-memory.md / reviews / flow-config.yml）
├── handoffs/         ← kdev-core
├── state.md          ← kdev-core（CEO 权威状态，与 memory/当前状态.md 不同）
├── events.log        ← kdev-core
├── hud.md            ← kdev-core
└── checkpoints/      ← kdev-core
```

### 5.2 共享语义命名空间（phase / subject）

参考 `01-design/2026-05-30-01-kdev-memory-记录元数据增强-工作草案.md` 第七节：

- `phase` / `phase_node` 字段在 kdev-memory G-NNN 和 staff-memory 评分-N.md 同名同义
- 联合查询能力：`phase=测试` 跨 `.kdev/memory/` 与 `.kdev/staff-memory/` 聚合

### 5.3 接口契约

| 装哪些 plugin | 数据流 |
|---|---|
| 只装 kdev-memory | `.kdev/memory/` 独立工作，正常的工程过程记忆 |
| 只装 kdev-core + kdev-staff | `.kdev/staff-memory/` + `.kdev/state.md` 独立工作，评审能力可跑 |
| 三者都装 | 通过 phase 共享命名空间联合查询；kdev-memory 的 G-NNN 可被评审能力读作"踩坑历史" |

## 六、跨平台扩展（v0.x → v1.x）

### 6.1 phased rollout

| 阶段 | 实现 | 跨平台能力 |
|---|---|---|
| **Phase 1**（v0.x，现状） | Claude Code plugin，本地 markdown 文件 | 只装 Claude Code |
| **Phase 2** | 加 `kdev-memory-mcp` 包装层，暴露 `recall / write / list / search / distill` 5 个 MCP tool（参考 OpenMemory） | 任何 MCP-aware 客户端用同一份 `.kdev/memory/` |
| **Phase 3** | per-client ACL（参考 OpenMemory User → App → Memory schema） | Qoder / CodeBuddy / TRAE 各装一份配置，URL path 区分 client_name |
| Phase 4 | 跨项目共享（Memory Level 5 全局记忆） | `~/.kdev/` 跨项目偏好同步 |

### 6.2 跟 OMC 的差异

OMC 也走 MCP server 路径，但**它的存储是 JSON + state files**，KDev 存储是 **markdown**。KDev 的 MCP server 只是 markdown 文件的 facade，不引入数据库依赖。

## 七、变更影响评估

| 模块 | 影响 | 工作量 |
|---|---|---|
| 现有 `.kdev/memory/` 的 5 个核心 markdown 文件 | ✅ 不动 | 0 |

| `当前状态.md` frontmatter schema | ⚠️ 现有字段保留，加 `notepad_pruned_at` 等 | 小 |
| skill SKILL.md / references/ 文档 | ⚠️ 需要描述新增 Level 0/1/2/4 的写入规则 | 中 |
| CLAUDE.md 接口契约段 | ⚠️ 加新的 hook 响应清单 | 小 |
| 蒸馏切片 | ✅ 现有不动，可增加 `dataset-by-level` 维度 | 小 |
