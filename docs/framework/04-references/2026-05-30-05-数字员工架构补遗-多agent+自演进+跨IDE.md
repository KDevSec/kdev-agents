## 二、最大突破：多 agent 共享记忆有了工程范本

第一轮调研结论：**5 仓全部没真正实现**。本轮补足：

### 2.1 CrewAI 的 Unified Memory + MemoryScope（**最优范式**）

```python
# 单一 Memory 实例，scope 路径前缀做隔离
Crew(memory=True)
  └── Agent1.memory = MemoryScope(root_path=f"/crew/{cid}/agent/{aid1}")
  └── Agent2.memory = MemoryScope(root_path=f"/crew/{cid}/agent/{aid2}")
  └── 团队共享 view = root_path="/crew/{cid}/"
```

**关键设计**：**不是物理隔离 silos，而是同一个 store 的层级路径前缀**。

引用：`_repos/crewAI/src/crewai/memory/memory_scope.py:38-76` + `_repos/crewAI/src/crewai/memory/storage/backend.py:11-180`

### 2.2 LangGraph 的 Store namespace（**工程化深度更够**）

```python
# 抽象层 - libs/checkpoint/langgraph/store/base/__init__.py:700+
class BaseStore:
    def get(namespace: tuple[str, ...], key: str) -> Item | None
    def search(namespace_prefix: tuple[str, ...], query, filter, limit) -> list[SearchItem]
    def put(namespace, key, value, index=None, ttl=None)
    def batch(ops: Iterable[Op]) -> list[Result]

# 命名空间 = tuple
# 示例：("user_123", "memories") / ("team_xyz", "shared")
```

**关键差异 vs CrewAI**：
- CrewAI 用字符串路径前缀（如 `/crew/cid/agent/aid`）
- LangGraph 用 tuple namespace（`("team_xyz", "shared")`），更结构化
- 二者本质都是"同一 store + 命名空间隔离"

### 2.3 KDev 怎么落地

`.kdev/memory/` 已经是 markdown 主存，直接套 **CrewAI 的路径前缀模式**最自然：

```
.kdev/memory/
├── shared/                          ← 全员工共享 view（root="/shared"）
│   ├── 决策日志.md
│   ├── 踩坑日志.md
│   └── ...
└── staff/                           ← per-员工 scope
    ├── ceo/                         ← scope="/staff/ceo"
    │   ├── pending-decisions.md
    │   └── stage-summary.md
    ├── 需求架构师/                  ← scope="/staff/req-architect"
    │   └── working-context.md
    ├── 开发工程师/                  ← scope="/staff/dev-engineer"
    ├── 测试工程师/                  ← scope="/staff/test-engineer"
    ├── 评审专家/                    ← scope="/staff/reviewer"
    └── cqo/                         ← scope="/staff/cqo"

# 查询 API（KDev 包成 thin layer over markdown grep）
recall(scope="/shared", query=...)              # 全员工可见
recall(scope="/staff/dev-engineer", query=...)  # 开发工程师私有 + /shared
recall(scope="/staff/*", query=...)             # CEO/CQO 跨员工查询
```

**不引入 Python LangGraph 运行时依赖**（守 KDev markdown 哲学），只借**命名空间设计**。

## 四、自演进 / 重试 / 升级：OMC 5 mode + LangGraph RetryPolicy

### 4.1 OMC 5 个核心自演进模式

| 模式 | state 文件 | 终止条件 | KDev 直接借 |
|---|---|---|---|
| **verify** | `.omc/state/sessions/{sid}/prd.json` | 所有 story `passes: true` + 评审通过 | KDev 评审完成 = 所有维度通过 + 双重通过（已规划） |
| **ralph** | 同上 + `progress.txt` | story 全过 OR 同问题 3 次 → 停止 | **评审循环的标杆实现** |
| **autopilot** | `.omc/autopilot/spec.md` + state | 5 个 QA cycle / 同错误 3 次 → 升级人类 | KDev 端到端自动化时参考 |
| **self-improve** | `.omc/self-improve/topics/{slug}/state/` | plateau OR circuit-breaker OR max_iter | **CQO 监督的算法骨架** |
| **ai-slop-cleaner** | `.omc/slop-cleanup-state.json` | 行为锁定 + 回归测试通过 | **修改前必加测试**——v1.5 没明确 |

### 4.2 5 大可复用 retry/upgrade 模式

| 模式 | 应用 | 阈值 | KDev 直接借 |
|---|---|---|---|
| **3-iteration 断路** | ralph / autopilot / ultraqa | 同一错误连续 3 次 | KDev 评审 retry max=3 后强制升级 CEO，跟 v1.5 已规划一致 |
| **Plateau + circuit breaker 双信号** | self-improve | `plateau_count >= window` OR `breaker_count >= threshold` | **CQO 监督拆两种信号**——停滞 vs 反复失败，触发不同动作 |
| **PRD 驱动完成** | ralph | `prd.json` 里每个 story `passes: true/false` | KDev 评审表四维度全过才允许出 |
| **Tournament selection** | self-improve | 多候选排名 + no-regression gate + 合并后重新跑 benchmark 确认 + regression 就 revert | **评审 retry 多个修订方案时**：不要选第一个就跑，重排序 + 复测 |
| **行为锁定先于修改** | ai-slop-cleaner | 改代码**前**必加回归测试 | v1.5 没明确——评审建议修订时强制要求"先加测试再改" |

### 4.3 reviewer 锁定 vs reviewer 升级（关键区分）

- **reviewer 锁定**（OMC stage-router 的 immutable snapshot 模式）：retry 时**同一个 reviewer agent**继续看，避免标准漂移
- **reviewer 升级**（ralph 的 STANDARD→THOROUGH 模式）：复杂度涨了升级**模型 tier**（Sonnet→Opus），但仍是同一个 reviewer 角色

→ KDev 评审循环建议**两层**：reviewer 锁定 + tier 升级。

### 4.4 LangGraph RetryPolicy（**形式化范例**）

```python
class RetryPolicy(NamedTuple):
    initial_interval: float = 0.5
    backoff_factor: float = 2.0
    max_interval: float = 128.0
    max_attempts: int = 3
    jitter: bool = True
    retry_on: type[Exception] | Sequence | Callable
```

引用：`_repos/langgraph/libs/langgraph/langgraph/types.py:416-436`

**对 KDev 启示**：评审 retry 配置不要散落 prompt，集中到 yml 编排或 config.yaml 的 `retry_policy:` 段，schema 借这个。

### 4.5 interrupt() 人类介入

```python
def interrupt(value: Any) -> Any:
    """Pause graph, surface value to client, resume with Command(resume=...)"""
```

引用：`_repos/langgraph/libs/langgraph/langgraph/types.py:811-870`

**特点**：interrupt 自动保存 checkpoint，resume 时从同一状态继续。比 OMC 的"escalate 后由用户在新会话决定"更连贯。

KDev 评审循环 3 次后升级 CEO 的实现：**interrupt + checkpoint** 模式。

## 五、Worker Preamble Protocol（**KDev 必须借**）

引用：OMC `src/agents/preamble.ts` （未读到源码，但 agents/executor.md:14 明确提到）

```
"Use the Worker Preamble Protocol (wrapWithPreamble() from src/agents/preamble.ts)
to ensure this agent executes tasks directly without spawning sub-agents."
```

**为什么 KDev 必须借**：6 员工 × 多业务能力 = 39 agent。如果不限制，开发工程师可能调评审专家，评审专家又调 CQO，套娃出去 token 爆炸。

**KDev 实现建议**（v0.x）：

```yaml
# .kdev/staff-memory/flow-config.yml
employee:
  开发工程师:
    capabilities: [实施计划, TDD, E2E验收, 安全扫描, 部署]
    delegation_policy:
      can_delegate_to: [reviewer]      # 只能调评审能力，不能调其他员工
      max_delegation_depth: 1          # 不能套娃
      preamble: |
        You are 开发工程师. Execute tasks directly.
        Do NOT spawn sub-agents.
        Read-only consultation via /shared scope.
```

## 六、跨平台 / MCP 跨 IDE

### 6.1 Continue 三层抽象（**未来 KDev v2.x+ 跨 IDE 范本**）

```
┌─────────────────────────────────────────┐
│ @continuedev/core (npm package)         │
│ - 14 方法 IDE 抽象接口                  │
│ - LLM provider / context / config / ... │
└─────────────────────────────────────────┘
              ↑       ↑       ↑
   in-proc    │   subprocess │ in-proc
              │   + JSON-RPC │
┌─────────────┐ ┌───────────┐ ┌────────┐
│ /extensions │ │/extensions│ │/exten. │
│ /vscode     │ │/intellij  │ │/cli    │
│ (TS, in-pkg)│ │(Kotlin)   │ │(TS)    │
└─────────────┘ └───────────┘ └────────┘
```

**IDE 接口 14 方法**（`_repos/continue/core/protocol/ide.ts`）：
- 文件系统：`readFile / writeFile / fileExists / getCurrentFile / getOpenFiles`
- 执行：`runCommand / subprocess / getTerminalContents`
- workspace：`getWorkspaceDirs / getDiff / getProblems`
- IDE：`getIdeInfo / getIdeSettings / showToast`
- LSP：`gotoDefinition / getReferences`

**关键 trade-off**：
- VSCode = in-process（快，能拿 terminal/problems）
- JetBrains = subprocess JSON-RPC（异步，**拿不到 terminal/problems**，所以 `loadContextProviders.ts:81-89` 显式过滤）
- CLI = 直接 instantiate Core

**对 KDev 启示**：v0.x 不做跨 IDE，但**v2.x+ 多 IDE 适配照抄三层**。需要先把 KDev 的 "IDE-specific" 操作抽出来（哪些是 Claude Code 特有？哪些是通用的？）。

### 8.3 评审循环（基于 OMC self-improve）

```yaml

  max_retries: 3                  # 3-iteration 断路
  reviewer_lock: true              # 同一 reviewer 锁定
```
