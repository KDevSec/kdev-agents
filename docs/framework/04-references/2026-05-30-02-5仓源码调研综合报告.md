## 三、记忆架构横向对比

## 四、跨平台 / MCP 模式

### 4.1 三种跨平台路径

| 路径 | 代表仓 | 实现 | 适合 KDev |
|---|---|---|---|
| **A. 单 MCP server 暴露全套** | OMC | `bridge/mcp-server.cjs` 暴露 notepad/state/memory/team/task 等所有运行时工具 | ⭐⭐⭐⭐⭐（v0.x 改造起点） |
| **B. MCP server + per-client ACL** | mem0/OpenMemory | FastAPI server + URL path scope `/{client}/sse/{user_id}` + Memory ↔ App ACL 表 | ⭐⭐⭐⭐⭐（v1.x 跨多平台目标） |
| **C. Core 库 + 多 IDE shim** | Continue | `/core` 共用 npm 包 + `/extensions/vscode` + `/extensions/intellij` + `/extensions/cli` 各自实例化 | ⭐⭐⭐⭐（如果 KDev 要做 VSCode/JetBrains 原生扩展） |

### 4.2 OpenMemory MCP 模型详细

**架构** (`_repos/mem0/openmemory/api/app/models.py:85-109`)：

```
User (user_id, 全局账号)
  └── App (owner_id, name)        # name = "Claude Code" / "Cursor" / "Qoder" / ...
       └── Memory (user_id, app_id)
           └── AccessControl (memory_id, app_id, allow/deny)
```

**安装** (`_repos/mem0/openmemory/README.md:135-141`)：

```bash
docker compose up        # 启 FastAPI + Qdrant + Next.js
npx @openmemory/install local http://localhost:8765/mcp/<client-name>/sse/<user-id>
```

**KDev 套用方向**（详见 03 提案文档）：

- 后端换成 markdown 文件（保留 KDev 的可审计性）
- 同样的 User → App ACL 表，让 Claude Code / Qoder / CodeBuddy / TRAE 共享 `.kdev/memory/` 但各管各的访问权

### 4.3 Continue 跨 IDE 模型详细

**三层分离** (`_repos/continue/`)：

```
/core               ← npm pkg @continuedev/core
  ├── config/      ← config 加载（YAML / JSON）
  ├── context/     ← 34 个 context provider + MCP 集成
  ├── llm/         ← 30 个 LLM provider class
  └── util/        ← 通用

/extensions/vscode      ← VSCode 扩展（用 core）
/extensions/intellij    ← JetBrains 插件（用 core）
/extensions/cli         ← CLI `cn`（用 core）
```

**Context provider per-IDE 过滤** (`_repos/continue/core/context/loadContextProviders.ts:81`)：terminal / problems provider 仅 VSCode 提供，JetBrains 加载时过滤。

## 六、多 Agent 编排 / 共享记忆

这是 **5 仓最弱的维度**——也是 KDev 数字员工架构最需要的能力。

| 仓 | 多 agent 能力 | 共享记忆 |
|---|---|---|
| OMC | 28+ agents + team-pipeline (`plan`→`prd`→`exec`→`verify`→`fix`) | ❌ 各 agent 独立，靠 prompt 协作，不共享 memory |
| Letta | `multi_agent_group` 字段 | ❌ 显式注释"only pulling latest block data if shared memory is being used" —— 未实现 |
| mem0 | 无编排 | - |
| Aider | 单 coder | - |
| Continue | 无多 agent | - |

**结论**：5 仓**没有任何一个**真正实现了"多 agent 共享 long-term memory"的范式。这是 LangGraph / CrewAI 调研的核心驱动力（详见第八节"待调研"）。
