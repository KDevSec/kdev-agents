# 原型评审标准（cap: prototype）

> **硬规（启动即执行）**：评审 agent 启动须先 `recall(scope=/staff/reviewer, subject:review:prototype)` 召回原型评审历史校准，再据本标准打分（记忆底座 v1.0 §5.2）。
> **共用骨架**：百分制 schema / 双重通过条件 / 🔴🟡⚪ 分级 / 评分表输出格式 / 评审循环，一律遵循 [通用评分模板.md](通用评分模板.md)。

---

## 评审对象

- **被评产物**：`prototype/`（req-architect 节点4 原型设计产出，高保真 HTML 原型目录）。
- **caller gate**：`req-architect:g-ar-proto-review`（原型部分）—— 此 gate 一闸两能力，story + prototype 并行评。
- **上游锚点**：用户故事列表 + 项目宪法 UI 约束（token/8px/字体白名单等）。

---

## 评分维度（4 维 × 25 = 100）

| 维度 | 满分 | checklist |
|---|---|---|
| **可用性** | 25 | □ 关键用户流程可走通 □ 信息架构清晰、找得到 □ 操作路径短、符合直觉 □ 空/错/加载等状态有设计 |
| **一致性** | 25 | □ 同类组件视觉/行为一致 □ 复用项目宪法设计 token（颜色/间距/字体）□ 8px 栅格对齐 □ 字体在白名单内 |
| **UED 合规** | 25 | □ 符合项目宪法 UI 约束（防发散）□ 无擅自引入新设计语言 □ 可访问性基本满足（对比度/焦点）□ 响应式断点合理 |
| **交互流畅** | 25 | □ 交互反馈及时（hover/active/disabled）□ 跳转/弹层逻辑清楚 □ 表单校验时机合理 □ 无明显交互死路 |

---

## 阈值

- **通过阈值 = 75**（创意类，略低于需求/代码类——给设计探索留弹性）。
- 双重通过条件：`通过 = 总分 ≥ 75 AND 🔴阻断 = 0`。

---

## 问题分级（prototype 专属判定示例，标准见通用模板 §3）

- 🔴 **阻断**：关键流程走不通 / 违背项目宪法硬 UI 约束（越白名单字体、破 token 体系）/ 严重可访问性缺陷。
- 🟡 **重要**：组件不一致、个别状态缺设计、交互反馈缺失。
- ⚪ **轻微**：像素级对齐、文案、配色微调。

---
