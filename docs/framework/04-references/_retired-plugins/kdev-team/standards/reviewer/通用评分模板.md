# 通用评分模板（评审专家所有能力共用）

> **承** [spec 2026-06-12 v0.2 §3.4 / §4.2](../../../../docs/superpowers/specs/2026-06-12-评审专家reviewer接入设计-design.md) · [概念模型与员工能力合稿 v1.0 §12.4/§12.6](../../../../docs/framework/01-design/2026-06-10-02-KDev数字员工-概念模型与员工能力-合稿-v1.0.md)
>
> 本文件定义**所有 6 个评审能力共用的评分骨架**：百分制 schema、双重通过条件、问题分级标准、评分表输出 schema、评审循环规则。各 cap standards 只声明自己的「评审对象 / 评分维度 / 阈值」，分级与输出格式一律遵循本模板。

---

## 1. 百分制评分 schema（100 分制）

每次评审给被评产物打一个 **0–100 的总分**，由 **3–5 个评分维度**累加而成。每个维度满分 **20–25 分**，各维度分值之和恰为 **100**：

| 维度数 | 每维满分 | 合计 |
|---|---|---|
| 4 维 | 25 分 | 100 |
| 5 维 | 20 分 | 100 |
| 3 维 | 33/33/34 分（取整凑 100）| 100 |

- 优先 **4 维 × 25** 或 **5 维 × 20**（整除、好打分）。
- 每个维度 standards 里都附 checklist（逐条核），维度得分 = 该维 checklist 满足度按比例折算到满分。
- **总分 = Σ 各维度得分**。维度名称、权重见各 cap standards「评分维度」段，本模板不锁死维度内容。

---

## 2. 双重通过条件（⭐ 核心硬规）

一次能力评审是否「通过」由**两个条件同时满足**决定，缺一不可：

> **双重通过条件：通过 = 总分 ≥ 阈值 AND 🔴阻断 = 0**

- **条件一（量）：总分 ≥ 阈值**。阈值默认值见各 cap standards（sr/story=80、prototype=75、design/code/security=85）；standards 可覆盖默认、request 可临时传更高阈值，但**不得低于** standards 声明值。
- **条件二（质）：🔴阻断问题数 = 0**。哪怕总分 95，只要存在任何一个 🔴阻断项 → **该能力 verdict = FAIL**（总分再高也不放行，有牙齿）。

**gate 级聚合**：一个 caller gate 可能触发多个评审能力（如 `g-ar-proto-review` 同触发 story + prototype）。**gate PASS = 所有 in-scope 能力都 PASS**；任一能力 FAIL → gate verdict = FAIL。

---

## 3. 问题分级标准（🔴 / 🟡 / ⚪）

评审发现的每个问题必须标一个级别，分级决定双重通过条件与 caller 的处置义务：

| 级别 | 名称 | 判定标准 | 对通过条件的影响 | caller 处置义务 |
|---|---|---|---|---|
| 🔴 | **阻断** | 触底线：正确性错误 / spec 违背 / 安全漏洞 / 数据丢失风险 / 验收不可达 / 架构红线 | 任一存在即该能力 **强制 FAIL**（双重条件之二）| **必须处置**——修复或经 escalate 用户拍板，不可自行 tech-debt 化 |
| 🟡 | **重要** | 明显缺陷但不触底线：可维护性差 / 边界遗漏（非数据风险）/ 设计不优 / 重复 | 计入扣分，**不单独触发 FAIL** | 建议修；caller 编排**自主判断**修 or 记 tech-debt |
| ⚪ | **轻微** | 风格 / 命名 / 文案 / 锦上添花建议 | 小幅扣分 | 可选；多数记 tech-debt 或忽略 |

> **建议为主、拦截经 gate**：评审专家只产出分级建议，从不直接命令 caller 改。🟡/⚪ 由 caller 编排自主取舍；🔴 经双重通过条件强制 FAIL，但兑现路径是 caller flow 的有界回流 + escalate（kdev-core 执行），评审专家不直接 halt/command。

---

## 4. 评分表输出 schema（每能力一份）

每个 `reviewer-<cap>` 评完写 `handoffs/reviewer/<gate>.<cap>.score.md`，正文是如下 YAML 块（照 spec §4.2，字段不可省）：

```yaml
cap: code                          # 评审能力 id（sr/story/prototype/design/code/security）
target: <diff_range 或产物路径>     # 被评对象（具体文件 / 目录 / diff 范围）
total: 82/100                      # 总分（Σ 维度得分）/ 100
dimensions:                        # 3-5 维，每维 name + score(得分/满分) + notes/issues
  - {name: spec一致性,   score: 22/25, notes: ...}
  - {name: 正确性+边界, score: 18/25, issues: [🔴 ...]}
  - {name: 风格抽象命名, score: 21/25}
  - {name: 架构一致性,   score: 21/25}
issues:                            # 全部问题清单，每条带 level + 证据锚点 + 建议
  - {level: 🔴阻断, file: src/x.py, line: 42, desc: ..., suggest: ...}
  - {level: 🟡重要, ...}
  - {level: ⚪轻微, ...}
verdict: FAIL                      # 通过 = total≥阈值 AND 🔴=0；此处 🔴≠0 → FAIL
```

- `issues` 每条**必须引证据**（`file`+`line` 或 transcript 段引用），不得空泛断言。
- `verdict` 由双重通过条件机械推出，不是评审员主观拍——`total≥阈值 且 🔴=0` 才能写 PASS。

---

## 5. 评审循环规则（有界回流 + escalate）

评审 FAIL 后不是死循环，走 kdev-core 原生有界回流（spec §6.2）：

1. **FAIL 且 `gate_iters < 3`** → `on_reflow`：reviewer-orchestrator 回函 verdict=FAIL + 分级建议 → **caller 编排自主判断**修哪些（🔴 必须处置、🟡/⚪ 可 tech-debt 化）→ 重做对应 action 节点 → **增量评**（只评修订部分，不重评全量）→ `gate_iters++`。
2. **`gate_iters ≥ 3`（3 次仍不过）** → `status = blocked` → **escalate CEO → 用户拍板**三选一：(a) 接受并记 tech-debt / (b) 续修第 4 次 / (c) 推翻重做。
3. **escalate 不 force-accept**——质量底线由 gate `status=blocked` 守，绝不自动降级放行（编排底座 R3：宁停不带病放行）。

> **冲突仲裁**（多能力对同产物相反结论）由 `reviewer-orchestrator` inline 处理（spec §6.1），不另设审查组长；编排也犹豫 → 升 CEO + 标元评审异常给 CQO。

---

## 6. 历史校准（recall）

每个评审能力 agent 启动时**必须** `recall(scope=/staff/reviewer, subject:review:<cap>)`，召回该能力的历史评审经验（评得对/松/紧、过往 🔴 模式），据此校准本次打分尺度，避免漂移（记忆底座 v1.0 §5.2）。评审经验回写 `subject:review:<cap>` 的 F-NNN，蒸馏到 `dataset-review-by-capability`。
