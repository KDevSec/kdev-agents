## 三、流程/阶段设计

### 3.1 BMAD - 四阶段工作流

```
Phase 1: Analysis (Product Discovery)
├── /workflow-init - 初始化
├── /workflow-status - 状态查看
└── /product-brief - 产品简报

Phase 2: Planning (Requirements)
├── /prd - 产品需求文档（复杂项目）
└── /tech-spec - 技术规格（简单项目）

Phase 3: Solutioning (Architecture)
├── /architecture - 系统架构设计
└── /solutioning-gate-check - 架构质量门禁（≥90%覆盖率）

Phase 4: Implementation (Development)
├── /sprint-planning - 冲刺计划
├── /create-story - 创建用户故事
└── /dev-story - 实现故事

Phase 6: Builder Module (Extensibility)
├── /create-agent - 创建自定义Agent
└── /create-workflow - 创建自定义工作流

Phase 7: Creative Intelligence (Innovation)
├── /brainstorm - 结构化头脑风暴（8种技术）
└── /research - 市场/竞争/技术研究

Phase 8: UX/Advanced
└── /create-ux-design - UX设计
```

**项目分级系统：**

| Level | Stories | 推荐流程 |
|-------|---------|----------|
| 0 | 1 story | /product-brief → /tech-spec → /dev-story |
| 1 | 1-10 | /product-brief → /tech-spec → /create-story |
| 2 | 5-15 | /product-brief → /prd → /architecture |
| 3 | 12-40 | Full BMAD (all phases) |
| 4 | 40+ | Full BMAD + sprint planning |

### 3.2 OMC - Team Pipeline状态机

```
Team Pipeline: plan → spec → exec → verify → fix (loop)

Workflow模式：
├── autopilot - 全自主执行
├── ralph - 自循环直到完成
├── ultrawork - 并行高吞吐
├── team - N协调Agent共享任务
└── ccg - Claude-Codex-Gemini三模型
```

### 3.3 GSD - 六步主流程

```
/gsd:new-project → discuss → plan → execute → verify → ship

1. Initialize Project - 初始化项目
2. Discuss Phase - 澄清需求偏好
3. Plan Phase - 研究+计划+验证
4. Execute Phase - Wave并行执行
5. Verify Work - 用户验收测试
6. Complete Milestone - 完成里程碑
```

### 3.4 ECC - 模块化工具集

```
125 Skills / 28 Agents / 12 Language Rules

核心模块：
├── Continuous Learning - 自动提取模式
├── AgentShield - 安全扫描
├── Language Rules - 12种语言规范
└── Multi-tool Support - 多工具支持
```

## 六、执行机制

### 6.1 BMAD

- **Sprint-based迭代** - 冲刺计划 → 用户故事 → 实现
- **Gate Check** - 架构质量门禁（≥90%覆盖率）
- **Story Points** - 故事点估算
- **MoSCoW优先级** - Must/Should/Could/Won't

### 6.2 OMC

- **Team Pipeline** - plan → spec → exec → verify → fix
- **Worker Pool** - tmux多面板并行
- **修复循环** - bounded by maxAttempts
- **人工门控** - HARD-GATE机制

### 6.3 GSD

- **Wave Execution** - 依赖感知并行
  ```
  Wave 1: Plan 01, Plan 02 (独立)
  Wave 2: Plan 03, Plan 04 (依赖Wave 1)
  Wave 3: Plan 05 (依赖Wave 2)
  ```
- **Fresh Context Per Plan** - 200k tokens纯净上下文
- **Atomic Git Commits** - Task级别commit
- **Goal-Backward Methodology** - 目标倒推法

### 6.4 ECC

- **粒度化插件** - 73 plugins按需组合
- **Agent Teams** - 文件所有权隔离并行
- **三层模型路由** - Haiku/Sonnet/Opus分层

## 七、Token优化策略

### 7.1 BMAD - Helper Pattern (70-85%节省)

```markdown
# 传统方式（重复200+次）:
"Load config from ~/.claude/config/bmad/config.yaml
Parse YAML to extract user_name, language..."

# BMAD方式（引用一次）:
"Per helpers.md#Load-Global-Config"
```

- **15个命令引用1个helpers.md**
- **81% token节省**
- **单一真实来源**

### 7.2 OMC

- 无特殊优化
- 依赖上下文管理

### 7.3 GSD - Context Engineering

| Context Usage | Quality | Claude's State |
|---------------|---------|----------------|
| 0-30% | PEAK | 全面、彻底 |
| 30-50% | GOOD | 自信、扎实 |
| 50-70% | DEGRADING | 效率模式开始 |
| 70%+ | POOR | 匆忙、最小化 |

**策略**: 每个Plan控制在~50%上下文内完成

### 7.4 ECC

- 语言规则按需加载
- 技能模块化
