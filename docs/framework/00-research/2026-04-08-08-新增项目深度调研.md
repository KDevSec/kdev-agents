## 二、核心项目深度分析

### 2.1 Claudeception - 持续学习系统

**源码证据** (来自 `Claudeception-main/SKILL.md`):

```yaml
name: claudeception
description: |
  Claudeception is a continuous learning system that extracts reusable knowledge from work sessions.
```

**核心能力**:
- **自动提取技能**：从工作会话中自动发现可复用知识
- **质量门槛**：只有经过验证、非显而易见的解决方案才会被提取
- **语义匹配**：技能描述优化用于未来检索

**学术研究支持**:
- **Voyager** (Wang et al., 2023) — 游戏Agent技能库
- **CASCADE** (2024) — 元技能（获取技能的技能）
- **SEAgent** (2025) — 通过试错学习新环境
- **Reflexion** (Shinn et al., 2023) — 自我反思

**与KDev的关系**:
```
Claudeception 核心流程:
1. 完成任务 → 2. 评估是否有可提取知识 → 3. 检查现有技能 → 4. 创建/更新技能

KDev Compound Engineering:
Plan → Work → Review → Compound → Repeat
                           ↑
                     Claudeception 在此步骤触发
```

**融合建议**：✅ **强烈推荐融合**
- Claudeception 的"技能提取"是 KDev Compound 步骤的具体实现
- 学术研究提供了理论支撑

### 2.2 claude-memory-3layer - 三层记忆系统

**源码证据** (来自 `claude-memory-3layer-main/SKILL.md`):

```yaml
name: claude-memory-3layer
description: Three-layer memory system — knowledge graph + daily notes + tacit knowledge
```

**三层架构**:

| Layer | 格式 | 目的 | 自动加载 |
|-------|------|------|----------|
| Layer 1 | JSON | 知识图谱（有生命周期管理） | ✅ 最近10条活跃 |
| Layer 2 | Markdown | 日记（今天发生了什么） | ✅ 最近3天 |
| Layer 3 | Markdown | 隐性知识（经验教训） | ✅ 完整文件 |

**与官方auto-memory的关系**:

| 维度 | 官方 v2.1.59 | Three-Layer Memory |
|------|-------------|-------------------|
| 知识发现 | AI语义理解 | 关键词匹配 |
| Git追踪 | ❌ `~/.claude/projects/` | ✅ `.claude/memory/` |
| 生命周期 | ❌ 只增不减 | ✅ `status` 字段 (active/superseded) |
| Token控制 | ❌ 截断200行 | ✅ ~1500 tokens/topic |
| 团队共享 | ❌ 仅本地 | ✅ 通过git |

**与KDev的关系**:
```
KDev 记忆层 vs claude-memory-3layer:

KDev Layer 2 (.kdev/):
├── questions-log.md    ← 类似 Layer 3 (隐性知识)
├── gotchas.md          ← 类似 Layer 3 (隐性知识)
└── daily-logs/         ← 类似 Layer 2 (日记)

缺失:
├── 知识图谱 (Layer 1 JSON) ← claude-memory-3layer 独有
└── 生命周期管理 (status字段) ← claude-memory-3layer 独有
```

**融合建议**：✅ **推荐融合**
- 借鉴 Layer 1 JSON 结构化知识图谱
- 借鉴 `status` 字段生命周期管理
- 与 Gstack /learn 的 learnings.jsonl 整合

**源码证据** (来自 `compound-engineering-plugin-main/AGENTS.md`):

```
This repository primarily houses the `compound-engineering` coding-agent plugin
and the Claude Code marketplace/catalog metadata used to distribute it.

It also contains:
- the Bun/TypeScript CLI that converts Claude Code plugins into other agent platform formats
- additional plugins under `plugins/`, such as `coding-tutor`
```

**核心能力**:

```
⚠️ 命名冲突！

  Plan → Work → Review → Compound → Repeat
  (知识复利理念)

compound-engineering-plugin:
  跨平台插件转换器
  (技术工具)

两者是完全不同的东西！
```

**融合建议**：🟡 **参考其插件结构和转换器**
- 可借鉴其跨平台转换思路

## 三、关键问题发现

### 问题1：记忆系统碎片化

**现状**：多个项目都有记忆系统实现

| 项目 | 记忆机制 | 存储位置 |
|------|----------|----------|
| KDev | questions-log, gotchas, daily-logs | `.kdev/` |
| claude-memory-3layer | 三层JSON+MD | `.claude/memory/` |
| Gstack | learnings.jsonl | `~/.gstack/projects/` |
| OMC | notepad, project-memory | `.omc/` |
| Claudeception | 自动提取的skills | `~/.claude/skills/` |

**问题**：功能重叠，用户需学习多套系统

### 问题2：Compound Engineering 概念冲突

**现状**：
- KDev 的 Compound Engineering 是"Plan → Work → Review → Compound → Repeat"
- compound-engineering-plugin-main 也是同名概念
- 需要确认是否同一来源、具体实现差异

### 问题3：技能提取机制重复

**现状**：
- Claudeception：自动提取技能
- Gstack /learn：learnings.jsonl
- KDev /kdev:compound：知识沉淀

三者的触发时机、存储格式、检索机制都不相同

## 四、头脑风暴：融合策略

### 策略A：统一记忆层

**方案**：将所有记忆机制统一到 KDev 记忆层

```
统一后的 KDev 记忆层:

.kdev/
├── knowledge-graph/           ← 借鉴 claude-memory-3layer Layer 1
│   └── topics/
│       └── <topic>/items.json  ← 有 status 字段
├── learnings.jsonl            ← 借鉴 Gstack /learn
├── questions-log.md           ← KDev 原有
├── gotchas.md                 ← KDev 原有
├── daily-logs/                ← 类似 claude-memory-3layer Layer 2
│   └── YYYY-MM-DD.md
└── skills/                    ← 借鉴 Claudeception
    └── <skill-name>/SKILL.md
```

**优点**：一套系统，学习成本低
**缺点**：需要迁移现有实现

### 策略B：分层整合

**方案**：按职责分工整合

| 层级 | 职责 | 使用项目 |
|------|------|----------|
| **运行时记忆** | 会话状态、临时上下文 | OMC notepad |
| **项目记忆** | 架构决策、约定 | KDev questions-log |
| **知识沉淀** | 学习模式、陷阱 | Gstack learnings + Claudeception |
| **技能库** | 可复用工作流 | Claudeception skills |

**优点**：保持各项目优势
**缺点**：需要定义清晰的边界

### 策略C：核心+插件

**方案**：KDev 作为核心，其他作为可选插件

```
KDev Core:
├── /kdev:prd, /kdev:arch, /kdev:epics (BMAD 核心)
├── /kdev:compound (知识沉淀)
└── /kdev:team (OMC Team Pipeline)

Optional Plugins:
├── kdev-memory-3layer (三层记忆)
├── kdev-auto-learn (Claudeception 风格自动学习)
└── kdev-gstack-qa (Gstack 多角色评审)
```

**优点**：灵活可扩展
**缺点**：插件组合可能导致冲突
