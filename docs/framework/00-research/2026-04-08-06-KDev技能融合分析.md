## 二、BMAD vs OMC 技术对比

### 2.1 核心定位差异

| 维度 | BMAD | OMC |
|------|------|-----|
| **定位** | 方法论框架 + Skill 集合 | 多智能体编排层 |
| **核心价值** | "先规划后执行"的流程规范 | Agent 协作 + 状态管理 + 自动化 |
| **适用场景** | 敏捷开发、产品迭代 | 复杂任务编排、团队协作 |

### 2.2 技术能力矩阵

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     技术能力矩阵                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  能力                    BMAD          OMC           KDev 需求              │
│  ─────────────────────────────────────────────────────────────────────────  │
│  Hook 自动化             ❌            ✅             ✅ 必需                │
│  状态机管理              ❌            ✅             ✅ 必需                │
│  并行 Agent 执行         ❌            ✅             ⚠️ 可选               │
│  跨会话持久化            ❌            ✅             ✅ 必需                │
│  记忆系统                ❌            ⚠️ (notepad)   ✅ 必需               │
│  门控机制                ✅ (★)        ❌             ✅ 必需                │
│  产出物驱动              ✅            ❌             ✅ 必需                │
│  敏捷流程                ✅            ❌             ✅ 必需                │
│  纠偏导航                ✅            ❌             ✅ 必需                │
│  Skill 体系              ✅ (50+)      ✅ (18+)       ✅ 必需               │
│  Agent 体系              ✅ (50+)      ✅ (32)        ⚠️ 精简               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 BMAD 缺失的关键技术能力

#### 1. Hook 自动化 (最大差距)

**OMC 有**:
```javascript
// SessionStart Hook: 自动注入上下文
hooks.SessionStart → 加载 state/, notepad.md, project-memory.json

// SessionEnd Hook: 自动保存状态
hooks.SessionEnd → 保存 session 状态, 触发通知

// PreCompact Hook: 压缩前保存
hooks.PreCompact → checkpoint 防止丢失
```

**BMAD 无**: 
- 每次都要手动调用 `/bmad-help` 查看状态
- 不会自动恢复上次进度
- 没有会话状态持久化

#### 2. 状态机管理

**OMC 有**:
```yaml
# Team Pipeline 状态机
states: [plan, prd, exec, verify, fix]
transitions:
  plan → prd (规划完成)
  prd → exec (PRD 确认)
  exec → verify (执行完成)
  verify → fix (验收不通过)
  fix → exec (修复后重试)
```

**BMAD 有**: 
- 概念上的四阶段流程
- 但**无代码级别的状态机**
- 依赖人工记忆当前在哪个阶段

#### 3. 跨会话持久化

**OMC 有**:
```
.omc/
├── state/
│   ├── sessions/{sessionId}/
│   │   ├── ultrawork-state.json
│   │   └── ralph-state.json
│   └── mission-state.json
├── notepad.md              # 工作记忆
└── project-memory.json     # 项目记忆
```

**BMAD 有**:
```
_bmad-output/
├── planning-artifacts/     # 静态文档
└── implementation-artifacts/
```
- 只有**静态产出物**
- 没有**运行时状态**
- 新会话无法自动恢复进度

### 2.4 结论

**BMAD 完全可以作为底座！** 关键是：

1. **保留 BMAD 的敏捷流程优势**
2. **借鉴 OMC 的技术实现**：
   - Hook 自动化
   - 状态机管理
   - 跨会话持久化
3. **新增 KDev 记忆系统**：
   - Compound Engineering
   - questions-log + gotchas + skills

## 七、融合总结

| 来源 | 融合策略 | 理由 |
|------|----------|------|
| **BMAD** | ✅ 核心底座 | 敏捷流程完整，门控机制成熟 |
| **Superpowers** | ✅ 执行层补充 | TDD、Subagent 更强 |
| **Speckit** | ⚠️ 可选 | BMAD 已覆盖其功能 |
| **OpenSpec** | ⚠️ 可选 | BMAD 有变更管理 |
| **Gstack** | ✅ 质量保障 | 多角色评审、安全审计 |
| **OMC** | ✅ 技术能力 | Hook、状态机、持久化 |

### 核心结论

1. **BMAD 可作为方法论底座**，其四阶段工作流和强制门控机制非常成熟
2. **BMAD 无记忆系统**，需要 KDev 新增
3. **技术实现借鉴 OMC**，状态机、并行执行、Hook 机制需要 OMC 提供
4. **融合策略**: BMAD 方法论 + OMC 技术底座 + KDev 记忆系统
