### 3.3 核心创新点

#### 3.3.1 Context Engineering

解决"context rot"问题 —— 上下文窗口质量退化：

| Context Usage | Quality | Claude's State |
|---------------|---------|----------------|
| 0-30% | PEAK | 全面、彻底 |
| 30-50% | GOOD | 自信、扎实 |
| 50-70% | DEGRADING | 效率模式开始 |
| 70%+ | POOR | 匆忙、最小化 |

**策略**: 每个Plan控制在~50%上下文内完成

#### 3.3.2 Wave Execution

```
Wave 1 (并行): Plan 01, Plan 02  ← 独立任务
     ↓
Wave 2 (并行): Plan 03, Plan 04  ← 依赖Wave 1
     ↓
Wave 3: Plan 05                  ← 依赖Wave 2
```

**原则**:
- 独立Plan → 同Wave并行
- 依赖Plan → 后续Wave等待
- 文件冲突 → 顺序执行或同一Plan

#### 3.3.3 XML Prompt Formatting

```xml
<task type="auto">
  <name>Create login endpoint</name>
  <files>src/app/api/auth/login/route.ts</files>
  <action>Use jose for JWT...</action>
  <verify>curl -X POST localhost:3000/api/auth/login returns 200</verify>
  <done>Valid credentials return cookie</done>
</task>
```

**优势**: 精确指令、无猜测、验证内置

#### 3.3.4 Goal-Backward Methodology

```
Step 1: 状态目标 → "工作聊天界面"
Step 2: 推导可观察真相 → 用户能看到消息、能发送消息...
Step 3: 推导必需产物 → Chat.tsx, /api/chat...
Step 4: 推导必需连接 → fetch in useEffect...
Step 5: 识别关键链接 → 可能出错的关键点
```

## 四、BMAD 分析

### 4.1 信息来源

BMAD仓库不可直接访问，以下信息来自GSD README的对比描述：

> "Other spec-driven development tools exist; BMAD, Speckit... But they all seem to make things way more complicated than they need to be (sprint ceremonies, story points, stakeholder syncs, retrospectives, Jira workflows) or lack real big picture understanding of what you're building."

### 4.2 特点推断

| 特点 | 说明 |
|------|------|
| Sprint ceremonies | 冲刺仪式、迭代会议 |
| Story points | 故事点估算 |
| Stakeholder syncs | 利益相关者同步 |
| Retrospectives | 回顾会议 |
| Jira workflows | Jira工作流集成 |

### 4.3 定位判断

BMAD定位为**企业级规范化**方案，适合：
- 大型团队协作
- 正式项目管理
- 需要合规审计的场景

### 4.4 对KDev的参考价值

| 可借鉴 | 不适合 |
|--------|--------|
| 完整流程覆盖 | Sprint ceremonies |
| 大picture理解 | Story points |
| 正式验收机制 | Jira集成 |
| 团队协作支持 | 企业仪式 |

## 七、底座选择建议

### 7.1 推荐方案

**首选: OMC (oh-my-claudecode) + GSD理念融合**

### 7.2 推荐理由

1. **OMC提供最强编排能力**
   - 唯一有完整Team Pipeline状态机
   - 状态管理成熟（持久化、恢复、Handoff）
   - 修复循环已实现

2. **GSD提供Context Engineering创新**
   - 解决context rot问题
   - Goal-Backward方法论
   - 极简流程理念

3. **改造成本最低**
   - OMC TypeScript代码可直接改造
   - 移除tmux即可使用
   - 其他模块直接复用

4. **融合可行性高**
   - OMC状态管理 + GSD Context Engineering
   - OMC编排 + GSD极简流程
   - OMC Agent + GSD Planner/Executor设计模式

### 7.3 融合架构

## 八、关键决策点

### 8.1 是否保留GSD的XML Prompt？

**建议**: 不保留（基于用户选择C）

- 保持Markdown格式
- 降低学习成本
- 保持与现有生态兼容

### 8.2 是否引入Wave Execution？

**建议**: 部分引入

- 当前Subagent并行已足够
- 可借鉴依赖感知概念
- 不强制按Wave执行

### 8.3 Fresh Context策略？

**建议**: 可选启用（基于用户选择B）

- 简单任务: 共享上下文
- 复杂任务: 启用Fresh Context
- 用户可配置
