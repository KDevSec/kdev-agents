## 二、仍然存在的问题

### 1. OMC 编排引擎的集成方式未定义

**位置**：§二 ~ §五

新版架构完全用"subagent-driven-development"描述 Layer 3，但之前版本明确提到"OMC 改造"。现在的文档没有说明：
- KDev 是 fork OMC 代码改造，还是从零实现？
- OMC 的 Team Pipeline 状态机是否还被使用？
- OMC 的 Worker Pool / tmux CLI Workers 如何处理？

附录A仍然列出 OMC 为参考项目，但架构中没有体现其具体集成方式。

**建议方向**：明确 Layer 3 的实现策略——是复用 OMC 核心模块，还是纯用 superpowers 替代。

---

### 2. IT 测试工具链未明确

**位置**：§六 6.2

| 层级 | 工具 |
|------|------|
| UT | superpowers:TDD |
| IT | 真实API调用（未指定工具） |
| E2E | gstack /qa + /browse |

IT 层只写了"真实API调用"，没有指定具体工具。主技术栈是 TypeScript + Node.js，需要明确 IT 层用什么：
- `supertest` + `vitest`？
- `axios` + 手动断言？
- 还是其他方案？

**建议方向**：在 IT 层补充具体工具选型。

---

### 3. `/kdev-change` 命令缺少详细设计

**位置**：§八 + §三

`/kdev-change` 在斜杠命令表中列出，但：
- §三只描述了"从1到N"的规格生成流程（Stage 2 增量迭代部分），没有独立的详细设计
- 没有说明 Delta Spec 与主 spec 的合并策略
- 没有说明增量变更的门禁配置是否与主流程一致

**建议方向**：补充 `/kdev-change` 的完整流程设计，或说明其与 `/kdev-spec` 的复用关系。

---

### 4. `correctionLoop` 触发条件的实现逻辑缺失

**位置**：§十一

```json
"correctionLoop": {
  "trigger": ["reviewReject", "execConflict", "specMismatch"],
  "action": "backToBrainstorming",
  "reference": "questions-log.md"
}
```

三个触发条件没有定义判断标准：
- `reviewReject`：什么程度的拒绝触发纠正？minor issues 还是 only critical？
- `execConflict`：acceptance.md 与代码的冲突如何检测？
- `specMismatch`：由谁判断 mismatch？reviewer Agent 还是自动规则？
- `backToBrainstorming`：是回到 Stage 1 重新全流程，还是局部纠正？

**建议方向**：为每个触发条件定义明确的判断标准和回退粒度。

---

### 5. `UnifiedSpec` 数据模型不完整

**位置**：§十二 12.3

```typescript
interface UnifiedSpec {
  features: FeatureSpec[];
  // ...
}
```

`FeatureSpec` 接口没有定义。此外缺少：
- `acceptance.md` 的数据结构映射
- `questions-log.md` 的数据结构映射
- Delta Spec 的结构定义（从1到N场景）

**建议方向**：补充 `FeatureSpec`、`DeltaSpec`、`QuestionsLog` 等核心数据结构定义。

---

### 6. 三层模型路由的实现机制缺失

**位置**：§十 10.2 + 附录E

Agent × Layer 映射矩阵定义了每个 Agent 的默认模型和升级/降级场景，但没有说明：
- 模型切换由谁触发？Agent 自身判断还是编排引擎分配？
- 切换依据是什么？文件数量？复杂度评分？用户指令？
- 切换时上下文如何传递？

**建议方向**：补充模型路由的判断规则和实现机制。

---

### 7. 试点计划阶段编号与架构 Layer 编号混淆

**位置**：§一 1.1 表格 + §六 6.2 表格

试点计划用"阶段1-5"，§六测试层用"E4/E5/E6/E7"——这些编号体系与 Layer 1-5 容易混淆。例如：
- "E4/E5 内部" 指的是什么？
- "E5 完成后" 是哪个阶段？

**建议方向**：统一编号体系，或在使用非 Layer 编号时加前缀说明。

## 五、总体评价

v2.6 相比 v2.1 是一次**架构层面的重构**，核心改进：

1. **五层职责清晰分离**——规格/计划/执行/质量/记忆各司其职
2. **Agent × Layer 映射矩阵**——解决了"谁做什么"的问题
3. **questions-log 纠正闭环**——任何阶段发现偏差都可回溯
4. **试点计划落地**——有了具体的时间线和验证目标

**剩余两个关键缺口**：
1. OMC 集成方式（fork vs 从零）——影响 Layer 3 实现路径
2. UnifiedSpec 数据模型完整性——影响规格引擎的数据层

**文档质量**：结构清晰、内容丰富，核心架构问题已基本解决。建议在解决上述三个高优先问题后，可以进入实施计划的细化阶段。
