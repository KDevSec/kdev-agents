## 一、仍然存在的问题

### 1. 架构层职责边界重叠

**位置**：§十五 13.1 五层架构 v2.1

Layer 2（角色审查）和 Layer 4（质量保障）存在职责交叉：

| 能力 | Layer 2 描述 | Layer 4 描述 |
|------|-------------|-------------|
| 代码审查 | "Security 安全审查" | "代码审查：两阶段评审" |
| 安全审计 | "Security 安全审查" | （未提及 code-audit） |

**问题**：Security 角色在 Layer 2 做安全审查，Layer 4 又做代码质量评审——谁先谁后？是串联还是并联？reviewer Agent（Layer 2）和 superpowers code-reviewer（Layer 4）是什么关系？

**建议方向**：明确 Layer 2 是"角色视角审查"（偏业务/架构），Layer 4 是"技术质量审查"（偏代码/测试），两者串联执行，Layer 2 先于 Layer 4。

### 2. 从 0 到 1 / 从 1 到 N 的切换机制未定义

**位置**：§三 3.1 Layer 1

两种流程并列展示，但没有说明：
- `/kdev-spec` 如何自动判断走哪条路径？
- 是用户参数指定（`/kdev-spec --new` vs `/kdev-spec --incremental`）？
- 还是系统根据项目状态自动推断（`.kdev/` 是否已存在 constitution）？
- 两种流程的输出数据格式是否统一？

**建议方向**：补充切换判断逻辑和 `UnifiedSpec` 的具体数据结构定义。

---

### 3. Subagent-Driven-Development 与 Team Pipeline 的关系不清

**位置**：§十二 10.1 + §十三 11.2

OMC Team Pipeline 有自己的 `team-exec` 阶段，superpowers 的 subagent-driven-development 也有独立的执行流程。两者在 `/kdev-exec` 阶段如何协作？

| 方案 | 说明 |
|------|------|
| A. Team Pipeline 包裹 subagent | `team-exec` 阶段内部调用 subagent-driven-development |
| B. 两者独立并行 | Team Pipeline 管宏观编排，subagent 管单任务执行 |
| C. 替换关系 | 用 subagent-driven-development 替代 `team-exec` |

文档没有明确选择哪个方案。

---

### 4. IT 测试工具链与技术栈不匹配

**位置**：§六 6.2

| 项目 | 值 |
|------|-----|
| 主技术栈 | TypeScript + Node.js |
| IT 测试工具 | `pytest + requests` |

用 Python 的 pytest 做 TypeScript 项目的集成测试，会引入额外的 Python 运行时依赖。如果主项目是 Node.js 生态，IT 层应该用 `supertest` + `vitest` 或 `jest`。

**建议方向**：确认 IT 层是否真的需要 pytest，还是应该统一到 Node.js 生态。

---

### 5. 门禁配置缺少错误恢复和超时

**位置**：§七

对比设计决策文档（`design-decisions.md`）的门禁配置，本文档缺少：

| 缺失项 | 设计决策文档有 | 本文档 |
|--------|---------------|--------|
| `onFailure` 策略 | ✅ return_to_xxx + maxRetries | ❌ |
| `timeout` 配置 | ✅ 各阶段超时阈值 | ❌ |
| `concurrency` 控制 | ✅ file-lock | ❌ |
| `retry` 退避策略 | ✅ exponential backoff | ❌ |

门禁配置应该与设计决策文档对齐，否则实现时会产生歧义。

### 6. 记忆系统的触发时机和数据流未定义

**位置**：§十五 13.1 Layer 5

新增了"技能注入: triggers 匹配 → 自动加载"，但没有说明：
- **触发时机**：每个阶段结束时提取？还是整个流程完成后？还是实时监听？
- **提取内容**：提取什么类型的 knowledge？代码模式？错误修复？架构决策？
- **注入机制**：triggers 匹配的规则是什么？正则？语义相似度？
- **数据流**：`.kdev/memory/` 的文件结构是什么？单文件还是按类别分目录？

**建议方向**：补充记忆系统的数据流图和存储格式定义。

### 7. 缺少降级策略的实现细节

### 9. 缺少端到端 Happy Path 示例

文档有很多分层设计，但缺少一个完整的端到端示例，展示：

```
用户输入: /kdev-flow "实现用户登录功能"
  → /kdev-spec: 输出 IR/SR/AR
  → /kdev-plan: 输出任务列表
  → /kdev-exec: TDD 实现
  → /kdev-review: 两阶段评审
```

每个阶段的输入/输出示例可以帮助验证架构的可行性。

## 四、总体评价

文档从初版扩展了 1 倍以上，**核心改进是补充了 OMC 和 superpowers 的深度分析**，这是正确的方向。新增的 v2.1 架构明确了"统一数据模型"和"两阶段评审"，比 v2.0 更具可操作性。

**三个关键缺口需要优先解决**：
1. Layer 2 和 Layer 4 的职责边界——架构层面的并发/串联关系
2. 从 0 到 1 / 从 1 到 N 的切换机制——规格引擎的核心入口
3. 门禁配置与 design-decisions.md 对齐——避免实现歧义

**文档质量**：内容丰富但结构需要整理（编号混乱），建议在解决上述问题后做一次整体重构。
